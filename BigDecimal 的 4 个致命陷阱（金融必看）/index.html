    BigDecimal 的 4 个致命陷阱（金融必看）¶

    BigDecimal 的 4 个致命陷阱（金融必看）

    讨论视角（这期在解决什么）¶

    • 从“金融/计费系统数字计算容错率为 0”切入，强调 double 的二进制表示会导致十进制小数精度丢失，直接造成账目对不上
    • 给出 BigDecimal 作为标准解法，但强调它不是开箱即用，存在一套高频“踩雷点”
    • 用 4 个致命陷阱串成“防御矩阵”：构造、比较、除法、不可变性

    核心论点与分论点¶

    核心论点¶

    BigDecimal 能解决 double 的精度问题，但正确使用需要遵守四条硬规则：避免用 double 作为构造入参；数值比较不要用 equals；除法必须指定保留位数与舍入模式；所有运算返回新对象必须接住返回值。否则仍会出现精度污染、比较误判、运行时异常或计算无效等线上事故。

    分论点 1：为什么 double 会“看似正确，实际出错”¶

    • 例子：0.1 + 0.2 直觉应为 0.3，但计算机可能得到 0.30000000000000004
    • 原因：底层用二进制科学计数法表示，无法精确表示 0.1 这类十进制小数
    • 影响：金融/计费场景会出现累计误差，导致账不平

    分论点 2：陷阱一——用 double 构造 BigDecimal：初始值已经不精确¶

    • 错误写法本质：new BigDecimal(0.1)
    • 问题根源：double 的 0.1 在进入构造前就已经是近似值，BigDecimal 只是“忠实记录”这个不精确的近似值
    • 正确方向：
    • 用字符串构造：new BigDecimal("0.1")
    • 或用 BigDecimal.valueOf(0.1)（更常用的防踩坑写法）

    分论点 3：陷阱二——equals 会比较 scale，导致“1.0 != 1.00”¶

    • 现象：数学上相等，但 equals 返回 false
    • 原因：equals 不仅比数值，还比标度（scale）；1.0 的 scale=1，1.00 的 scale=2
    • 正确方向：业务数值比较用 compareTo
    • compareTo 只关注数值大小，会忽略 scale 差异

    分论点 4：陷阱三——除法可能产生无限循环小数：不指定 scale/rounding 会异常或资源耗尽¶

    • 典型案例：1 / 3 是无限循环小数
    • 风险：若直接除法运算不限制精度，BigDecimal 会尝试用有限资源表达无限小数，最终抛异常或导致程序崩溃
    • 正确方向：除法必须显式指定保留位数与舍入模式（示例：保留 2 位并四舍五入得到 0.33）

    分论点 5：陷阱四——BigDecimal 不可变：运算返回新对象，不接返回值等于没算¶

    • BigDecimal 的加减乘除不会修改原对象，而是返回新的 BigDecimal
    • 风险：如果不接住返回值，业务计算结果不会生效
    • 正确方向：把每一步运算的返回值赋回变量或传递下去

    关键数据、案例与结论（逐条可复述）¶

    • double 精度案例：0.1 + 0.2 可能得到 0.30000000000000004
    • 构造结论：严禁 new BigDecimal(double)，优先 new BigDecimal(String) 或 valueOf
    • 比较结论：业务比较不要用 equals，用 compareTo
    • 除法结论：必须指定 scale 与 roundingMode，否则 1/3 这类会抛异常
    • 不可变结论：所有运算要接返回值，否则计算无效
