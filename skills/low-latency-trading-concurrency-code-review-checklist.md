# 低延迟交易系统：并发代码评审 Checklist（Atomic / SOP）

目标：面向低延迟交易系统的并发层面代码评审标准操作程序（SOP），强调可预测延迟（tail latency）、正确性与资源泄露防护。每一条均为可执行的检查动作（能落到代码位置并给出“是/否”结论）。

---

## 1) 并发模型与线程边界

- [ ] 检查每个关键链路（下单、撮合、风控、行情、出入金、落库/落盘）的“线程拥有者”是否明确：该链路的状态更新是否被限定在单线程/串行执行器/Actor/事件循环中（避免无意共享可变状态）。
- [ ] 检查所有跨线程共享的可变状态是否都有唯一的同步策略：要么线程封闭（不共享）、要么不可变（final + 只读）、要么受同一把锁保护、要么使用原子类/并发容器；禁止“读不加锁、写加锁”等不对称做法。
- [ ] 检查对外暴露的服务对象（例如单例 Service、Controller、Handler、Filter、回调对象）是否假定会被并发调用；如会并发调用，检查其所有字段是否满足线程安全策略（不可变/同步/线程封闭）。
- [ ] 检查业务关键状态是否存在“多处写入点”：若同一字段/对象可能被多个线程写入，检查是否已通过架构约束（单写多读/分区/消息队列/分片）消除并发写入。
- [ ] 检查是否在关键链路引入不受控的并发（例如并行 stream、隐式异步回调、框架默认异步执行器）；如存在，检查是否明确执行器、背压与容量上限。

---

## 2) 锁与临界区（正确性 + 尾延迟）

- [ ] 检查每个 `synchronized`/`ReentrantLock`/读写锁的保护对象是否明确（锁的“保护域”可在代码/注解中定位），并验证所有访问点都使用同一把锁。
- [ ] 检查每个加锁点是否使用 `try/finally` 释放锁；检查 `lock()` 与进入 `try` 之间是否存在可能抛异常的语句（避免漏解锁）。
- [ ] 检查临界区内是否调用外部/可覆写/不受控代码（回调、接口实现、SPI、listener）；如存在，检查是否能移出锁外，或用快照复制隔离。
- [ ] 检查临界区内是否包含 I/O（网络/磁盘/日志/指标上报/数据库）、阻塞等待（`Future.get()`/`join()`/`await()`）、或可能触发重试的远程调用；如存在，判定为高风险并要求移出锁外或改为异步/批处理。
- [ ] 检查是否存在嵌套锁（一个锁内再获取另一个锁）或嵌套临界区；如存在，检查是否有固定的锁顺序并在所有路径一致遵守（可定位到顺序定义与所有调用点）。
- [ ] 检查是否存在“锁升级/降级”或锁重入依赖；如存在，检查是否会与第三方库/框架锁交叉导致潜在死锁。
- [ ] 检查是否在高频路径使用公平锁（`new ReentrantLock(true)` / 公平 `Semaphore`）；如存在，检查是否有明确的低延迟理由（公平锁通常增大上下文切换并恶化尾延迟）。
- [ ] 检查锁粒度是否过大：同一把锁保护多个无关资源时，检查是否可拆分为分段锁/按 key 锁（striped/per-key locking）或按分区线程封闭。

---

## 3) 条件等待与协作（禁止“睡眠轮询”）

- [ ] 检查是否使用 `Thread.sleep()` 进行等待/节流/协作；如存在，要求改为显式同步原语（`Condition.await`、`CountDownLatch`、`Semaphore`、`BlockingQueue`、`LockSupport.parkNanos` 等）并可配置超时。
- [ ] 检查是否使用 `wait/notify` 实现条件队列；如存在，检查是否可替换为并发工具类（避免丢通知/虚假唤醒处理错误/难以维护）。
- [ ] 检查任何等待操作是否包含超时与失败路径（例如 `await(timeout)`、`Future.get(timeout)`）；检查超时后的补偿动作是否明确（取消/降级/回滚/熔断）。
- [ ] 检查是否存在“轮询等待共享字段变化”的循环；如存在，检查该字段是否为 `volatile` 或在锁内读取，并优先替换为事件驱动/条件变量。

---

## 4) 内存可见性与原子性（volatile / 原子类 / 竞态）

- [ ] 检查每一个 `volatile` 字段的用途是否可被证明：仅用于可见性/发布（publication）与有序性，不用于复合原子性；如出现 `volatile` 的 `x++`/check-then-act，判定为错误并要求加锁或原子更新。
- [ ] 检查所有“发布对象”的方式是否安全：构造完成后再发布；禁止在构造器中将 `this` 逸出到其他线程可见处（注册回调、放入共享容器、启动线程等）。
- [ ] 检查双重检查锁（DCL）是否满足前提：实例字段为 `volatile` 且构造过程无外部可见副作用；否则要求改为类初始化/静态 holder/显式锁。
- [ ] 检查使用 `Atomic*` 的位置是否真的需要 CAS 语义；若只有 `get/set`，检查是否可以用 `volatile` 或线程封闭降低开销与复杂度。
- [ ] 检查热点计数/累加是否使用 `LongAdder`（或等价分段累加）替代 `AtomicLong`；如仍使用 `AtomicLong`，检查是否因语义要求必须线性化读写。
- [ ] 检查是否存在“分别读取多个原子字段并组合出状态”的独立 getter；如业务需要一致快照，检查是否通过锁/不可变快照/版本号机制实现一致性。

---

## 5) 并发容器与复合操作（避免 check-then-act）

- [ ] 检查对 `ConcurrentHashMap`/`ConcurrentMap` 是否存在 `get/containsKey` 后再 `put/remove` 的复合逻辑；如存在，要求改为 `compute/computeIfAbsent/merge/putIfAbsent/replace` 等原子方法。
- [ ] 检查是否在 `ConcurrentHashMap.compute*` 的 lambda 中再次访问同一个 map（包含 `get/put`）；如存在，要求重构以避免潜在死锁/递归更新与性能问题。
- [ ] 检查并发容器是否以过度抽象的类型保存而导致 API 受限或误用：若依赖 `ConcurrentHashMap` 特定语义，变量是否声明为 `ConcurrentHashMap`（而非 `Map`/`ConcurrentMap`）。
- [ ] 检查 `Collections.synchronized*` 容器的迭代/stream/`toArray`/批量操作是否在其 monitor 锁上包裹整个复合操作（而非只同步单次调用）。
- [ ] 检查线程安全类中是否把非线程安全集合包装成 `Collections.unmodifiable*()` 返回；如存在，判定为错误（不可修改不等于线程安全），要求返回快照副本或在锁内复制。

---

## 6) 线程池、队列与背压（容量上限 + 可预期延迟）

- [ ] 检查是否使用无界线程池/无界队列（例如 `newCachedThreadPool()`、无界 `LinkedBlockingQueue`）；如存在，判定为高风险（OOM/尾延迟不可控）并要求设置容量与拒绝策略。
- [ ] 检查每个线程池的核心参数是否显式：core/max、队列类型与容量、拒绝策略、线程命名（便于排障与压测定位）。
- [ ] 检查拒绝策略是否与交易语义一致：被拒绝时是丢弃/失败返回/降级/转同步执行/入隔离队列；禁止“默认吞掉”或静默丢单。
- [ ] 检查是否存在在业务线程中同步等待线程池结果（`Future.get()`）而造成“线程池反压传播为阻塞”；如存在，要求改为异步回调/批处理/拆分执行器。
- [ ] 检查线程池是否在应用关闭/组件卸载时调用 `shutdown()`/`shutdownNow()` 并等待终止；检查等待超时后的处理策略是否明确（避免资源泄露/进程无法退出）。
- [ ] 检查是否在关键路径使用 `ForkJoinPool.commonPool()`（并行 stream、CompletableFuture 默认执行器）；如存在，要求显式指定专用执行器并隔离与限流。

---

## 7) ThreadLocal 与上下文污染（线程池场景必查）

- [ ] 检查所有 `ThreadLocal` 是否为 `static final` 且有清理策略；如在线程池线程中写入，检查任务结束是否 `remove()`（防止脏数据与内存泄露）。
- [ ] 检查是否使用可继承线程本地变量（`InheritableThreadLocal`）在异步/线程池场景传播上下文；如存在，检查是否会导致跨请求/跨订单污染。
- [ ] 检查是否存在以 `ThreadLocal` 暗含并发模型（把共享状态塞 ThreadLocal）但又在异步链路跨线程访问；如存在，要求显式传参或使用上下文载体并明确生命周期。

---

## 8) 线程安全的时间与随机（热点与正确性）

- [ ] 检查是否在并发场景共享 `DateFormat`（`SimpleDateFormat` 等）实例；如存在，判定为线程不安全并要求改为线程安全替代或加锁（并评估锁对尾延迟影响）。
- [ ] 检查是否在并发压测/并行任务中共享 `java.util.Random` 实例；如存在，要求改为 `ThreadLocalRandom` 或每线程独立实例以避免竞争。

---

## 9) 可伸缩性与热点识别（避免“热字段/热锁”）

- [ ] 检查是否存在全局单点锁/全局队列/全局缓存导致的串行瓶颈；如存在，要求按 key 分区、分段锁或拆分执行器。
- [ ] 检查是否存在“热字段”频繁写入（计数、状态位、时间戳）导致 cache line 争用；如存在，要求评估分片/批量更新/减少写频率（尤其在高频路径）。
- [ ] 检查是否在高频路径进行跨线程同步的日志/指标上报；如存在，要求异步聚合或本地缓冲后批量上报（避免锁竞争与 I/O 抖动）。

---

## 10) 并发测试与可复现性（交易系统必备）

- [ ] 检查关键并发组件是否有多线程单测或压测用例覆盖：包含并发启动协调（例如 `CountDownLatch`）、断言收集与超时防挂死。
- [ ] 检查并发测试是否覆盖最坏后果场景：重复撮合/错单、资金不一致、状态回退、幂等失效、重复扣减等，并能稳定复现。
- [ ] 检查 `await()`/`join()`/`Future.get()` 的返回值与中断处理是否被正确检查（例如 `await` 返回 false 的路径必须处理）。
