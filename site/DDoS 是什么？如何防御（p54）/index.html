    DDoS 是什么？如何防御（p54）¶

    DDoS 是什么？如何防御（p54）

    讨论视角（这期在解决什么）¶

    • 从后端线上故障视角切入：前端看似正常，但监控全红、CPU 飙升、请求超时，典型 DDoS 现象
    • 重点不在“重启/扩容”，而在理解攻击原理与资源瓶颈（网络连接资源、CPU、线程池、数据库连接）
    • 以“漏斗式分层防御”给出工程落地：基础设施清洗流量 + 边界识别拦截 + 应用韧性兜底

    核心论点与分论点¶

    核心论点¶

    DDoS（分布式拒绝服务）的核心杀伤力在于：攻击者通过控制大量“肉鸡/僵尸网络”制造压制性流量洪峰，迅速耗尽服务端连接资源与计算资源，导致合法用户无法获得服务。防御必须是分层体系：在入口侧分散/清洗大流量，在边界侧识别与限流恶意请求，在应用侧通过熔断与削峰填谷保护主链路与核心数据层；本质是成本博弈，靠架构设计提高攻击者成本。

    分论点 1：DDoS 的组织方式：控制端 + 僵尸网络，攻击者不亲自下场¶

    • 攻击者隐藏在暗处，通过控制端下发指令
    • 执行者是数量巨大的僵尸网络节点（被黑的服务器、路由器、摄像头等）
    • 结果是同时对目标发起高频请求，形成压制性“流量洪峰”

    分论点 2：网络层典型手法：TCP SYN Flood 用半连接拖死内核资源¶

    • TCP 正常建连需要三次握手
    • 攻击模式：海量来源只发第一步 SYN 后“失联”，不完成握手
    • 服务端为了维护半开连接需要占用队列/内核资源并重试
    • 效果：像“总机被占线”，新的合法连接无法接入

    分论点 3：应用层攻击：伪装成正常 HTTP，把最耗资源的业务点打穿¶

    • 特征：请求形态看起来“合法”
    • 常见目标：复杂查询（如全表扫描）、大文件导出、慢连接长时间占用线程
    • 穿透后果：线程池被占满、数据库连接被拖垮、CPU 被跑满，最终全站超时

    分论点 4：防御策略：漏斗式三重过滤（基础设施 → 边界 → 应用韧性）¶

    • 外层（基础设施）：CDN/边缘节点分散与清洗大流量，避免流量直达源站
    • 中间层（WAF/网关）：识别恶意特征，封禁异常 IP，做高频限流，把攻击挡在业务逻辑之外
    • 内层（应用自身韧性）：
    • 熔断/降级保主链路
    • 消息队列等手段做削峰填谷
    • 保护核心数据层，避免被击穿

    关键数据、案例与结论（逐条可复述）¶

    • 关键机制点：TCP 三次握手与半开连接（SYN Flood 通过不完成握手消耗连接资源）
    • 典型症状：CPU 飙升、请求全超时、服务不可用（合法流量无法被处理）
    • 结论：DDoS 防御不是“单点手段”，而是分层漏斗 + 应用韧性，核心目标是提高攻击成本与保护关键链路
