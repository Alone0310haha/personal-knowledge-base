    视频文案综合分析报告（信息提炼）¶

    使用redis出现缓存击穿雪崩穿透应该怎么解决

    讨论视角（这期在解决什么）¶

    视频从“高并发系统里 Redis 作为数据库挡箭牌”的工程视角出发，讲清三类典型缓存故障：缓存击穿、缓存雪崩、缓存穿透。重点不是概念背诵，而是面试/实战中如何用“读缓存 → 读库 → 回填缓存（带过期）”这一基本流程，补齐并发与异常场景下的防护策略，确保数据库不被打穿、系统不雪崩。

    核心论点与分论点¶

    核心论点¶

    三类问题的本质都是“缓存未命中导致请求回源数据库”，差别在于未命中的范围与原因不同；解决思路也围绕三条主线展开：减少关键数据过期、用互斥/排队削峰、用随机 TTL/高可用架构避免集中失效，并对非法/不存在数据做拦截或短路。

    分论点 1：缓存击穿（单个热点 key 失效）要用“热点永不过期 + 互斥回源”¶

    • 现象：某个原本冷门 key 突然变热点，恰好过期时大量并发请求同时 miss → 直接打到数据库
    • 影响：数据库压力飙升；并发线程重复回填缓存造成额外浪费
    • 方案 A：热点数据永不过期
    • 可预期热点：直接设置永不过期
    • 不可预期热点：监控访问量突增，达到阈值后把 TTL 调整为永不过期/更长
    • 方案 B：加锁排队（互斥回源）
    • 关键做法：双重检查锁（拿锁后再查一次缓存）
    • 解释：绝大多数请求仍走“直接命中缓存”路径；只有在 key 过期瞬间才短暂串行
    • 锁选型：单机/少量实例可先用进程内同步锁；更严谨可用分布式锁

    分论点 2：缓存雪崩（大量 key 同时失效 / 缓存整体不可用）要用“打散过期 + 高可用”¶

    • 现象：大量 key 集中过期或缓存服务器宕机，导致大规模回源数据库
    • 影响：数据库扛不住 → 应用整体不可用，形成“雪崩”
    • 方案 A（针对集中失效）：随机过期时间（在基础 TTL 上加随机抖动）
    • 目标：不让大量 key 在同一时间点过期，削弱同一时刻的回源洪峰
    • 方案 B（针对宕机/断电）：提升缓存系统可用性
    • Redis 集群/哨兵：单点挂掉可自动切换
    • 容灾：多机房/灾备，机房级故障可切换

    分论点 3：缓存穿透（缓存与数据库都没有）要用“拦截非法 + 缓存空值/布隆过滤”¶

    • 现象：请求的数据 Redis 没有，数据库也没有；大量出现时常见于恶意请求（如负数 ID、超大 ID、随机 ID）
    • 影响：每次都回源数据库，持续消耗资源，可能打挂系统
    • 方案 A：参数校验
    • 拦截明显非法参数（负数、格式不合法等）
    • 局限：无法覆盖“合法但不存在”的请求（例如某个 ID 恰好库里没有）
    • 方案 B：缓存空对象（空值也缓存，但要设置 TTL）
    • 做法：库里查不到也写缓存（空值/占位符），并设置过期时间（可叠加随机 TTL）
    • 目的：把“反复 miss 回源”变成“快速命中空值”
    • 方案 C：布隆过滤器
    • 作为前置判断结构：请求先过布隆过滤器
    • 两种思路：
      • 黑名单：把“不存在”的 ID 放入布隆过滤器，命中直接返回空
      • 白名单：把“存在”的 ID 放入布隆过滤器，不存在则直接拒绝后续查询

    关键数据、案例与结论（逐条可复述）¶

    • Redis 主要作用：给数据库做挡箭牌，减轻数据库压力
    • 击穿定义：单个热点 key 失效导致大量并发回源数据库
    • 雪崩定义：大量 key 集中过期或缓存宕机导致大规模回源，系统可能整体不可用
    • 穿透定义：请求数据缓存与数据库都不存在，常见于恶意攻击流量
    • 双重检查锁：拿锁后再查一次缓存，避免重复回源与重复回填
    • 经验性判断：大多数请求命中缓存；只有 key 过期瞬间才需要互斥回源
    • TTL 策略：基础过期时间 + 随机抖动可显著缓解“集中失效”
    • 空值缓存：必须设置过期时间，避免后续数据补齐后长期读到空
