    第 22 课：自定义业务指标¶

    学习时长：2-3 小时
    难度等级：⭐⭐⭐ 进阶
    先修要求：完成第 21 课 - Java 监控指标基础

    学习目标¶

    完成本课程后，你将能够：

    • ✅ 知道哪些业务指标值得做（可观测性优先级）
    • ✅ 能为一个业务场景设计指标：指标名、类型、标签、单位
    • ✅ 能避免常见坑：高基数标签、错误的指标类型、重复统计
    • ✅ 能把“指标 → 查询 → 告警/面板”串成闭环
    • ✅ 理解业务指标和系统指标（JVM、HTTP、DB）的分工边界

    22.1 为什么要做业务指标¶

    系统指标（CPU、内存、GC、线程）能告诉你“系统快不快、稳不稳”，但很难直接回答：

    • 用户是否在失败？
    • 哪个业务链路在慢？
    • 哪个接口在抖？
    • 订单/支付/下单是否异常？

    业务指标的目标就是把“业务状态”量化成可查询的时间序列。

    22.2 业务指标的优先级：从 RED / USE / 四个黄金信号入手¶

    22.2.1 Web 服务（推荐 RED）¶

    • R（Rate）：请求量/吞吐
    • E（Errors）：错误率
    • D（Duration）：耗时分布（P95/P99）

    22.2.2 资源型组件（推荐 USE）¶

    • U（Utilization）：利用率
    • S（Saturation）：饱和度（排队/拥塞）
    • E（Errors）：错误

    22.2.3 先把“黄金信号”补齐¶

    • 延迟（Latency）
    • 流量（Traffic）
    • 错误（Errors）
    • 饱和（Saturation）

    业务指标优先从这四个做起，最容易产生价值。

    22.3 指标设计四件套：Name / Type / Labels / Unit¶

    22.3.1 命名（Name）¶

    建议：

    • 小写 + 下划线
    • 带单位后缀：_seconds、_bytes、_total
    • 避免多义：一个指标只表达一个含义

    例子：

    • order_create_total
    • order_create_failures_total
    • order_create_duration_seconds（Histogram）

    22.3.2 类型（Type）¶

    常用选择：

    • Counter：累计事件（下单次数、支付失败次数）
    • Gauge：瞬时值（队列长度、在线用户数）
    • Histogram：分布（耗时、大小）——最常用

    经验建议：

    • 延迟优先用 Histogram（能做 P95/P99，能做 SLO）
    • 不要用 Gauge 做“累计量”

    22.3.3 标签（Labels）¶

    标签用于切维度，但最容易踩坑。

    推荐标签（低风险）：

    • service / job
    • endpoint（注意控制数量）
    • method、status
    • result（success/fail）

    避免标签（高风险，高基数）：

    • user_id、order_id、trace_id、request_id
    • error_message（错误文本会爆炸）

    22.3.4 单位（Unit）¶

    必须明确单位，避免“数值看不懂”：

    • 时间：seconds
    • 大小：bytes
    • 个数：total（计数）

    22.4 具体场景示例：下单接口怎么做指标¶

    假设你有接口：POST /api/orders

    22.4.1 要回答的问题¶

    • QPS 是否异常？
    • 错误率是否升高？（按错误类型/状态码）
    • P95/P99 是否变慢？
    • 是否集中在某个渠道（channel）或某个商品类型（sku_type）？

    22.4.2 指标设计¶

    1) 请求总量（Counter）

    • order_create_requests_total{endpoint="/api/orders",result="success|fail",channel="app|h5"}

    2) 耗时（Histogram）

    • order_create_duration_seconds_bucket{endpoint="/api/orders",channel="app|h5",le="..."}

    3) 业务失败类型（Counter）

    • order_create_failures_total{reason="insufficient_stock|invalid_coupon|..."}

    注意：reason 只能是“有限集合”，绝不能把错误消息直接塞进去。

    22.5 从指标到查询：把闭环跑通¶

    22.5.1 QPS¶

    sum(rate(order_create_requests_total[5m])) by (channel)

    22.5.2 错误率¶

    sum(rate(order_create_requests_total{result="fail"}[5m]))
/
sum(rate(order_create_requests_total[5m]))

    22.5.3 P95 / P99（Histogram）¶

    histogram_quantile(
  0.95,
  sum(rate(order_create_duration_seconds_bucket[5m])) by (le, channel)
)

    22.6 常见坑：90% 的问题都在这里¶

    • 把高基数信息做成 label（user_id / order_id / trace_id）
    • 指标名没有单位，或者单位混用（ms 和 s 混）
    • 计数类用 Gauge（重启后归零造成误解）
    • Histogram buckets 设计不合理（过少/过多）
    • 同一事件被多次统计（中间件 + 业务各算一遍）
    • 只做指标不做查询/告警（没有闭环就没有价值）

    22.7 实践建议：如何小步落地¶

    1) 先给每个关键接口补齐 RED（QPS/错误/耗时） 2) 先把标签控制在 3-5 个以内，并且每个标签值集合可控 3) 先做面板，再做告警（用面板验证指标是否可靠） 4) 对指标规模进行预算（抓取量、series 数、写入量）

    课后小结¶

    • 业务指标要回答业务问题：先选问题，再选指标
    • Name/Type/Labels/Unit 是设计核心，其中 Labels 最容易出事故
    • 先把 RED/黄金信号做对，再做更细的业务拆分
    • 最终一定要形成闭环：指标 → 查询 → 面板/告警 → 迭代
