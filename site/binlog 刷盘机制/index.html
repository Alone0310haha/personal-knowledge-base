    binlog 刷盘机制¶

    binlog 刷盘机制

    讨论视角（这期在解决什么）¶

    • 从 MySQL 服务层视角解释 binlog 的刷盘机制：binlog 属于 MySQL Server 层日志，面向所有存储引擎（并非 InnoDB 专属）
    • 核心关注：事务提交后 binlog 先写到哪里、什么时候 fsync 持久化到磁盘，以及一致性与性能的取舍

    核心论点与分论点¶

    核心论点¶

    binlog 的“写入/刷盘”由参数控制，常见有三种模式：由 OS 决定刷盘（风险更大但性能高）、每次提交都刷盘（最安全但更慢）、每 N 次提交刷一次盘（折中方案）。

    分论点 1：binlog 的层级与定位¶

    • binlog 由 MySQL Server 提供，属于服务层日志
    • 因此它对所有引擎通用（用于复制、恢复等场景），而不是只属于 InnoDB

    分论点 2：三种刷盘策略（视频口径）¶

    • 参数=0：
    • 事务提交后写入 OS 的 page cache
    • 何时持久化到磁盘由操作系统调度决定（通常系统更“空闲”时刷）
    • 性能最好，但宕机/掉电时更可能丢失最近提交的 binlog
    • 参数=1：
    • 实时写 + 实时刷：事务提交后写入 binlog 并 fsync 落盘
    • 安全性最高（代价是更高的刷盘开销）
    • 参数=N（例如 5）：
    • 先写入 page cache
    • 每累计 N 次提交，再刷到 binlog（fsync）
    • 性能与数据安全折中：可能丢失最近未刷盘的一段提交

    分论点 3：默认值的版本变化（视频口径）¶

    • MySQL 5.7.7 以前：默认值为 0
    • MySQL 5.7.7 以后：默认值为 1

    关键数据、案例与结论（逐条可复述）¶

    • binlog 归属：MySQL Server 层（引擎通用）
    • 刷盘策略：
    • 0：OS 决定刷盘（风险高、性能高）
    • 1：每次提交刷盘（最安全）
    • N：每 N 次提交刷盘（折中）
    • 默认值（视频口径）：5.7.7 前默认 0，5.7.7 后默认 1
