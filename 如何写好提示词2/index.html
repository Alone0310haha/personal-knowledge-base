    如何写好提示词2

    如何写好提示词2

    你要解决的核心问题¶

    提示词技巧太多记不住并不可怕。真正高效的做法是：先理解大模型的本质、优缺点与失误模式，再推导出“写提示词的策略”，最后按策略自然长出 one-shot / few-shot / 思维链 / 伪代码 / 角色 等技巧，而不是死记硬背。

    讨论视角（作者从什么角度出发进行讨论）¶

    作者从“提示词技巧太多记不住”的学习痛点切入，主张不要背技巧清单，而要回到大模型的本质（概率预测生成）与典型优缺点（语言强、事实/推理/记忆/上下文受限），用“策略→技巧”的推演方式来掌握提示词：先理解模型会在哪些环节失败，再用策略提高命中概率与稳定性，最后自然长出 one-shot/few-shot、思维链、伪代码、角色等具体技巧。

    核心论点与分论点¶

    核心论点¶

    提示词写作的关键不是掌握更多“招式”，而是用策略系统性地提升模型生成期望结果的概率：让模型准确理解你的意图、让概率收敛到更少的可行路径，并用外部工具/分解流程弥补模型先天短板（幻觉、推理有限、上下文窗口有限、无记忆）。

    分论点 1：先认清大模型本质与失误模式，才能推导出有效策略¶

    • 本质：根据输入预测下一个 token 的概率生成（类比“锄禾日当午”的接龙）。
    • 强项：语言理解与生成强、知识覆盖广（但为压缩后的知识）。
    • 短板：事实可能不准（幻觉/编造）、推理尤其是复杂数学不稳、知识更新滞后、上下文窗口有限、严格意义上无记忆（每次请求都要把历史再发一遍/由产品做摘要滑窗）。

    分论点 2：策略 1——写清晰指令，让模型先“知道你要什么”¶

    作者强调：如果模型没理解指令，后面再多技巧也很难补救。清晰指令的核心是减少歧义、设边界，避免模型“脑补”。

    代表性技巧（由策略自然衍生）：

    • 把目标说成可执行任务（你要它做什么、做到什么程度）。
    • 明确约束与边界（哪些不做、哪些必须做）。
    • 在提示中写出你已知的关键路径/步骤（为后面的“概率收敛”做铺垫）。

    分论点 3：策略 2——给模型“思考时间”，让推理可见以提高正确率¶

    • 作者引用“思维链”观点：要求模型把思考过程输出出来，能显著提升推理结果与输出质量（尤其在数学/多步推理场景）。
    • 实操表达：在提示里明确“先写步骤/计划，再给结论”，而不是一句话就要它一步到位。

    分论点 4：策略 3——把复杂任务拆成简单任务，让概率收敛、让上下文变短¶

    作者把“概率收敛”类比为登山找路：排除掉错误路径，留下少数可行路径，找到正确结果的概率就更高。

    分解带来的收益：

    • 每个子任务更简单，失败概率更低。
    • 任务拆小后提示更短，缓解上下文窗口限制，也提升质量。
    • 可从最简单版本开始迭代：先做出可用结果，再逐步加约束、加质量要求。

    分论点 5：策略 4——借助外部工具/外部资料，绕过模型的“事实/计算/更新”短板¶

    作者给出的典型做法：

    • 降幻觉：接入搜索/引用外部资料，把关键事实作为上下文提供（尤其是公司内部资料、最新事件等模型本身拿不到的）。
    • 提升计算正确率：遇到复杂数学，不强行让模型心算；让它调用计算器、或生成代码执行计算，再把结果回填让模型负责解释与总结。
    • 选对模型：不同任务适合不同模型（写作/翻译/推理/数学），不要迷信“最新一定最好”。

    分论点 6：为什么叫 Prompt Engineering，而不是 Prompt Tricks¶

    作者认为之所以常说“提示工程”，是因为完成任务往往需要流程设计与组合拳：分解任务、分阶段提示、引入工具与验证，而不是单条提示词解决所有问题；工程强调“过程”和“系统性扬长补短”。

    一套可复用的“提示词策略推演流程”¶

    第 1 步：先判定任务类型（决定后续策略）¶

    • 生成类：写文章、写方案、写文案、写代码
    • 变换类：翻译、改写、润色、结构化整理
    • 分析类：抽取要点、对比优劣、风险评估、找因果
    • 决策类：方案选择、排序、权衡取舍

    输出要求写在第一句：产物是什么、用于什么场景、最终交付形态（报告/表格/JSON/代码）。

    第 2 步：识别模型在这个任务里的“天然缺点”（选策略的起点）¶

    把“会出错的地方”提前列出来：

    • 需求不清会脑补：指令模糊时会自动补全你没说的假设
    • 上下文缺失会瞎猜：缺业务数据/偏好/约束时容易偏航
    • 细节一致性差：长文本/长项目容易前后矛盾
    • 复杂推理不稳：一步到位容易跳步或漏条件
    • 格式不稳定：没约束时表格/JSON/接口形态容易飘

    第 3 步：把缺点映射为策略（从策略自然导出技巧）¶

    • 为了防脑补：把指令写具体 + 明确边界 + 写验收标准
    • 技巧落地：明确受众/风格/结构；列出“必须包含/禁止包含”
    • 为了补上下文：提供必要材料 + 允许 AI 反问补信息
    • 技巧落地：给参考样例；让 AI 先问 3–5 个问题再开始
    • 为了稳推理：拆步骤 + 让过程可见 + 中途自检
    • 技巧落地：思维链（让它输出步骤）；计划→执行→检查
    • 为了稳格式：先规定结构 + 给样例 + 给 schema
    • 技巧落地：one-shot / few-shot；伪代码/类型定义/JSON schema
    • 为了稳风格：指定角色与写作视角
    • 技巧落地：设定“资深XX工程师/苏格拉底导师/审稿人”等

    第 4 步：把策略落到一个固定模板（让你每次都能复用）¶

    【任务】
请你……（一句话目标）

【输入/上下文】
1) 背景：……
2) 数据/素材：……
3) 约束：……

【输出要求】
格式：……（Markdown/表格/JSON）
必须包含：……
禁止包含：……

【过程要求】
先列计划，再执行；每一步输出证据/理由；最后自检是否满足“必须包含/禁止包含”。

【信息不足时】
如果信息不足，请先问我 3-5 个问题。

    你可以直接照做的练习（从“记技巧”转为“推策略”）¶

    1. 选一个你常做的任务（写周报/写需求/写代码修 bug/翻译）
    2. 写下这任务中模型最常犯的 3 个错（脑补/漏条件/格式飘等）
    3. 为每个错误写一个对应策略（补边界/补上下文/加自检/给样例）
    4. 把策略嵌到模板里，保存成你的“默认提示词骨架”
    5. 连续用 3 次：每次只改上下文与验收标准，不改骨架

    常见错误与解决方案（快速排障）¶

    • 结果“看起来对但不是你想要的”
    • 先检查：指令是否具体、边界是否写清、验收标准是否存在
    • 结果前后矛盾/细节漂移
    • 先做：要求先给计划；输出中途自检；最后按清单复核
    • 格式不稳定（表格/JSON/字段缺失）
    • 先做：给一个样例；再给 schema/type；必要时要求严格按 schema 输出

    关键数据、案例与结论¶

    • 类比案例：用醋除水垢 → 背后原理是酸碱中和；理解原理后，你能衍生出更多等价技巧（苹果皮/柠檬水等）而不是背配方。作者用此类比“策略→技巧”。
    • 模型选择案例（作者口径）：有些任务用更强模型先“实现”更重要，之后再用成本更低的模型替换优化；作者举例把翻译等任务从更贵的模型替换为成本约为原先十分之一的选择（口述示例）。
    • 结论落点：
    • 提示词学习的最短路径：掌握模型短板 → 推导策略 → 用策略选择技巧。
    • 提示词的核心目标：提高期望结果的生成概率，并降低不确定性与返工。
