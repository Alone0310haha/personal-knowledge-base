    AI编程 Spec开发不是Coding的终点——使用spec的正确姿势（视频总结）¶

    AI编程 Spec开发不是Coding的终点——使用spec的正确姿势

    1. 听众画像¶

    • 想用 AI 提升研发效率，但对“Spec 能否替代 Coding”存在误解的开发者（尤其是做业务交付的工程师）。
    • 在团队里推动文档先行 / 规范化协作，希望降低 AI“自由发挥”带来的不确定性的人。
    • 关注 SDD（Spec Driven Development）/TDD（Test Driven Development）等工程方法，想把它们放到 AI 协作流程里的人。

    2. 讨论角度与要解决的问题¶

    视频从“工程实践与项目交付”的角度讨论 Spec 与 Code 的关系：Spec 不是用来替代开发，而是用来降低不确定性、保障架构与关键设计不跑偏，让 AI 协作变得更可控。

    它要解决的核心问题是：

    • 为什么“写一份足够详细的 Spec 让 AI 直接生成项目”在本质上会逼近“写代码本身”。
    • 在真实商业项目中，Spec 应该写到什么粒度，写多写少各有什么风险。
    • 如何把 Spec、Coding、SDD、TDD 放在同一条交付链路里协同工作，而不是互相替代。

    3. 核心论点与分论点（含关键概念解释）¶

    核心论点¶

    Spec 和 Code 不是割裂关系；如果把 Spec 写到足以自动生成完整项目，它本质上就会变成另一种形态的 Code。正确做法是让 Spec 聚焦架构与关键约束，用它来限制 AI 的不确定性，再由 Code 完成具体实现。

    分论点 A：Spec 并不一定比写 Code 更简单（与项目复杂度强相关）¶

    视频用“项目复杂度”拆解了一个常见误解：认为“写 Spec 比写代码更舒服/更简单”。

    • 场景 A（简单 demo / 玩具项目）
    • 少量描述即可生成可运行的雏形，确实会让人感觉“Spec 更轻松”。
    • 场景 B（常规商业项目）
    • 需求、业务规则、边界条件更复杂，很难用一两句话描述清楚，Spec 也会变得复杂。
    • 描述代码逻辑的最佳形式往往仍然是 Code 本身（行业多年沉淀出的可执行表达）。

    分论点 B：为了可控，Spec 必须“冻结”关键内容；但 Spec 也不应无限膨胀¶

    视频给出的关键工程判断是：LLM 的输出不可 100% 复现，因此需要在 Spec 里固化关键点，减少 AI 的自由发挥。

    • 概念：LLM 输出不可复现
    • 同一提示词在不同时间/不同上下文可能生成不同实现细节，导致结果不可预测。
    • 概念：冻结（固化）关键内容
    • 指在 Spec 中明确写死不允许 AI 改动或自行脑补的部分，例如：关键数据结构、核心业务规则、关键函数/接口签名、关键调用链路（谁调用谁、先后顺序、输入输出）。
    • 目标是把“可变空间”压缩到安全范围内，让实现更稳定、可验收。
    • 但 Spec 天然应该更少：不是越多越好
    • 如果把所有细节都塞进 Spec，本质上等价于“换一种语言写代码”，不合理。
    • Spec 更应关注架构与重点实践：抽象方向、关键约束、关键流程，然后由 Code 落到实现细节。

    分论点 C：Spec 并不天然比写 Code 更“深思熟虑”¶

    视频把这个误解拆成“客观”和“主观”两条线：

    • 客观上：文档先行有利于方案讨论
    • Spec 适合讨论架构、数据流、请求流、目标逻辑，避免具体代码实现干扰“核心方案讨论”。
    • 主观上：是否深思熟虑取决于人
    • 优秀工程师写 Spec 或写 Code 都会专注和严谨；反之亦然，因此不能得出“Spec 天然更深思熟虑”的结论。

    分论点 D：Spec 不是银弹；它在交付链路中只覆盖部分问题¶

    视频强调 Coding 只是项目管理的一部分，交付还包括 Debug、Test、Optimization 等。

    • 结论 1：Spec 不是魔法、也不是银弹
    • 需求不清晰时，不管写 Spec 还是写 Code，产出都不会好；AI 也无法“读心”补全需求。
    • 结论 2：Spec 不一定缩短交付时间
    • 因为多了一道写 Spec 的成本，整体时间未必减少；但在 AI 协作时代，Spec 仍可能成为“必要的补全能力”。

    4. 实践操作：视频隐含的“正确使用 Spec”做法（按场景拆解）¶

    场景 A：简单 demo / 试验性功能¶

    • 目标：快速产出可运行雏形。
    • 操作要点：
    • 用简短 Spec 明确范围与验收标准（做什么、不做什么）。
    • 允许 AI 在实现细节上自由发挥，但保留快速回滚与重写的心理预期。

    场景 B：商业项目 / 需要稳定交付的功能¶

    • 目标：降低 AI 不确定性，确保架构与关键设计不偏离。
    • 操作要点（先冻结关键，再让实现展开）：
    • 冻结关键数据结构：核心实体/字段/约束（例如必填、唯一性、状态枚举）。
    • 冻结关键流程：主流程、异常分支、边界条件（请求流/数据流）。
    • 冻结关键接口与调用关系：模块边界、函数/接口签名、调用顺序与返回约定。
    • 保持 Spec 克制：只写“必须一致”的部分，把可替换的实现细节留给 Code。

    场景 C：人机协作的工程化流程（SDD + TDD + Coding）¶

    • 概念：SDD（Spec Driven Development）
    • 用 Spec 先把架构与重点设计定住，确保实现不出现大偏差，尤其适合约束 AI 的“自我发挥”。
    • 概念：TDD（Test Driven Development）
    • 用测试先把重点功能的正确性定住，减少做完才发现不符合预期的返工。
    • 组合关系（视频给出的顺序关系）：
    • SDD 定“方向与边界” → TDD 定“正确性” → Coding 定“实现细节”。

    5. 现实启示：把“Spec 不是终点”落到团队协作里¶

    • 把 Spec 当成“协作接口”而不是“替代开发的魔法咒语”：用来对齐架构、边界、关键约束与验收标准。
    • 对 AI 友好但不纵容：通过冻结关键点缩小自由度，把不确定性从实现阶段前移并显式化。
    • 不追求“文档越全越好”：Spec 的价值在于减少分歧与跑偏，而不是覆盖一切细节。
    • 当需求说不清时，先解决“需求表达”，否则 Spec/Code 都会失败。

    6. 可直接复用的 SOP：AI 协作下的 Spec 使用流程（两条流程不混用）¶

    SOP-A：快速原型（适用于 demo / 试验）¶

    1. 写 5 行以内范围说明：目标、非目标、可接受的粗糙点。
    2. 写 3 条验收标准：看到什么算成功。
    3. 让 AI 生成实现并快速跑通。
    4. 结果可用则补充最小化的“关键冻结点”（例如接口名、数据字段），结果不可用就直接重写而不是修补到复杂。

    SOP-B：稳定交付（适用于商业功能）¶

    1. 需求澄清：把需求写成可验收的输出（输入、输出、约束、失败场景）。
    2. 写 Spec（只写必须一致的部分）：
    3. 冻结数据结构（实体/字段/约束/状态）
    4. 冻结流程（请求流/数据流/异常分支）
    5. 冻结接口与调用关系（模块边界、签名、调用顺序、返回约定）
    6. 写测试（TDD 思路）：优先覆盖最关键的正确性与边界条件。
    7. Coding 实现：允许 AI 辅助生成，但以 Spec + 测试作为“护栏”。
    8. 交付校验：
    9. 测试全部通过
    10. 关键流程与冻结点不偏离 Spec
    11. 迭代维护：如果实现被迫偏离 Spec，先更新 Spec 的冻结点，再继续扩展实现，避免“文档与现实分叉”。
