    通用 Java 代码评审 Checklist¶

    目标：将 Java 代码评审标准化为可执行的检查动作，侧重语法正确性、资源泄露、代码可读性、可维护性与工程风险。

    使用规则¶

    1. 按分类逐条执行检查动作；每条必须能在代码中定位到“是/否”结论。
    2. 发现问题时，记录影响面、触发条件和最小修复方向。
    3. 对资源泄露、并发与安全问题，优先给出可复现路径或触发条件。
    4. 风格类条目默认作为非阻塞建议；只有会导致正确性、资源、安全或维护风险时，才升级为 blocking issue。

    命名规约¶

    • 检查包名是否全小写，并且点分隔的每一段是单个有语义的英文单词。
    • 检查类、接口、枚举、注解名是否为 UpperCamelCase，缩写按“词”处理。
    • 检查方法、参数、成员变量、局部变量是否为 lowerCamelCase，且名称表达业务语义而非类型或实现细节。
    • 检查常量（static final）是否为 UPPER_SNAKE_CASE，且命名能表达完整约束语义。
    • 检查 boolean 属性名是否避免以 is 作为字段前缀，并核对 getter 命名与字段语义一致。
    • 检查数组声明是否使用 Type[] name 形式，而不是 Type name[]。
    • 检查子类字段名是否与父类字段重名；检查同一方法不同代码块中的局部变量是否重名。

    代码格式¶

    • 检查每个 .java 文件是否仅包含一个顶级类，并且文件名与顶级类名一致。
    • 检查源文件结构是否按顺序组织：license（如有） -> package -> imports -> class；各 section 之间是否恰好一个空行。
    • 检查缩进是否统一，且不存在 Tab 缩进字符。
    • 检查 import 是否无通配符（包括 static wildcard），并按规范分组排序。
    • 检查大括号是否统一，且控制语句即使单行也使用大括号。
    • 检查每条语句是否独占一行，避免同一行串联多条语句。
    • 检查 switch 是否显式处理 default；如存在 fall-through，是否有清晰标识并确保不会误触发。
    • 检查是否有行尾注释；注释应独立成行并与代码内容表述对齐。

    OOP 规约¶

    • 检查类、方法、字段可见性是否最小化。
    • 检查覆盖父类或接口方法时是否使用 @Override。
    • 检查 equals / hashCode 是否同时实现且语义一致；用于集合键时，其参与字段是否不可变或生命周期内不变。
    • 检查可变内部状态（数组、集合、Date 等）是否被直接暴露；构造入参和 getter 是否需要防御性拷贝或不可变视图。
    • 检查是否在构造器或工厂方法中完成参数校验，避免对象处于部分初始化但可用的状态。
    • 检查资源型对象（连接、流、线程池、锁等）是否有明确生命周期，且关闭动作在异常路径也可达。
    • 检查是否在 equals / hashCode / toString 中执行可能阻塞或有副作用的操作。
    • 检查抽象层次是否合理，是否违背单一职责、开闭原则或在尚未稳定的变化点上过早抽象。

    集合处理¶

    • 检查遍历 List / Set / Map 时是否在循环体内直接 remove / add 修改同一集合；如需要删除元素，是否使用 Iterator.remove() 或先收集后批量变更。
    • 检查是否在 Map.get() / containsKey() 之后再 put / remove 形成 check-then-act 逻辑；如需要原子更新，是否使用 compute / merge / putIfAbsent / replace 等方法。
    • 检查对可能为 null 的集合返回值是否统一为 Collections.emptyList() / emptySet() / emptyMap()。
    • 检查是否把内部可变集合直接作为返回值或 public 字段暴露；如必须暴露，是否提供不可变视图或快照副本。
    • 检查对 subList() 的使用是否明确其视图语义，必要时是否复制为新集合。
    • 检查集合容量是否在热点路径做了合理预估，以减少扩容与 rehash 抖动。

    并发编程¶

    • 检查有并发访问可能的类（如 Controller、Servlet、Filter、单例服务）是否明确线程安全属性或共享状态保护策略。
    • 检查是否使用 synchronized + wait/notify 做条件等待；如存在，是否可替换为 java.util.concurrent 工具。
    • 检查临界区是否包含外部、可重入、可覆写调用；如包含，是否能移到锁外。
    • 检查是否存在嵌套锁或嵌套临界区；如存在，是否定义并遵循固定加锁顺序。
    • 检查锁获取与释放是否使用 try/finally 保证释放。
    • 检查是否在锁内执行大块计算或长循环；如存在，是否能缩小临界区。
    • 检查每一个 volatile 字段是否只承担可见性与有序性职责，而不误用作复合原子操作。
    • 检查共享可变字段是否有明确保护规则：要么为 final，要么为 volatile，要么被同一把锁保护。
    • 检查被锁保护的字段或集合是否清晰标注保护关系。
    • 检查 ThreadLocal 生命周期是否可控；若配合线程池使用，任务结束时是否 remove()。
    • 检查是否使用 Thread.sleep() 作为线程协作或等待手段；如存在，是否应改为显式同步原语。
    • 检查线程池是否避免无界队列或无界线程创建，并显式配置拒绝策略、队列容量和线程命名。
    • 检查线程池或调度器是否在生命周期结束时显式关闭。
    • 检查日期格式化或解析是否使用线程安全方式。

    异常与日志¶

    • 检查是否使用 try-with-resources 关闭 InputStream、OutputStream、Reader、Writer、Socket、Channel 等可关闭资源。
    • 检查 finally 中是否可能抛出异常覆盖原始异常；若处理关闭异常，是否保留原始异常为主因。
    • 检查是否捕获过宽异常（Exception / Throwable）并吞掉或仅打印信息；如捕获，是否记录上下文、异常栈并做出明确恢复决策。
    • 检查异常转换或包装是否保留 cause。
    • 检查日志是否使用参数化写法而不是字符串拼接。
    • 检查错误日志是否包含足够定位上下文，并避免记录密码、token、密钥、证件号、卡号等敏感信息。
    • 检查日志级别是否匹配影响程度，避免把可预期业务分支打成 error。
    • 检查循环或高频路径中的日志是否受限，避免 I/O 放大与性能退化。

    其他检查¶

    • 检查是否出现循环依赖或职责反转不清。
    • 检查是否引入仅为本地调试服务、但会污染构建或运行环境的临时配置变更。
