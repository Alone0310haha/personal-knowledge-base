    JPA vs MyBatis（架构视角）¶

    JPA vs MyBatis（架构视角）

    讨论视角（这期在解决什么）¶

    • 以“为什么很多团队用 JPA 一段时间后又哭着改回 MyBatis”为问题切入，从架构师视角拆两者底层设计取舍
    • 用“对象优先 vs SQL 优先”作为主对比轴：封装程度、可控性与性能风险
    • 结论导向：没有万能框架，核心是匹配业务特征，并给出可落地的混用方案（读写分离）

    核心论点与分论点¶

    核心论点¶

    JPA（以 Hibernate 为代表）的设计哲学是“对象优先、隐藏数据库”，能显著提升复杂业务建模与开发效率，但会引入执行过程不透明与性能陷阱；MyBatis 的设计哲学是“SQL 优先、控制执行”，更适合高并发与强性能诉求场景。架构上最优解常常不是二选一，而是按职责混合使用：写入与复杂事务用 JPA，查询与性能敏感读用 MyBatis。

    分论点 1：JPA 的核心价值：对象优先 + 自动映射，适合复杂业务建模¶

    • 你主要操作 Java 对象，框架负责映射与生成 SQL
    • 优势：在复杂业务逻辑、复杂对象关系中更省心，领域模型约束更强，代码更易维护
    • 代价：SQL 生成与执行细节被“封装起来”，需要理解其机制才能避免踩坑

    分论点 2：MyBatis 的核心价值：SQL 优先 + 过程可控，适合性能敏感场景¶

    • 把方法当作 SQL 的入口，SQL 怎么写、怎么调优由你决定
    • 优势：对执行过程“绝对控制”，更容易做针对性性能优化
    • 代价：业务建模与对象关系表达更依赖人为约束与工程规范

    分论点 3：JPA 典型性能陷阱：N+1 查询（自动关联导致隐式爆炸）¶

    • 现象：代码看似一次查询，但因对象关联/懒加载等配置，主查询会在后台裂变出大量子查询
    • 示例：查 100 个订单，可能额外触发 100 次 SQL 去取关联用户数据，导致数据库被打爆、延迟飙升
    • 含义：JPA 并非不能优化，但优化成本高，需要持续“盯住框架行为”，防止隐式副作用

    分论点 4：选型建议：按业务形态做取舍，再用读写分离做融合¶

    • 互联网电商/金融等：用户量与并发高，甚至需要为性能调整范式与表结构
    • 更应优先 MyBatis（极致性能，减少额外封装与不确定性）
    • 企业内部系统（OA/ERP/CRM 等）：难点是业务逻辑复杂、对象关系繁多
    • 更适合 JPA（领域建模强约束，保证复杂逻辑正确性与可维护性）
    • 同时“逻辑复杂 + 查询海量”：
    • 建议混合使用并做读写分离：写用 JPA，读用 MyBatis

    关键数据、案例与结论（逐条可复述）¶

    • 核心对比轴：对象优先（JPA） vs SQL 优先（MyBatis）
    • 典型坑：N+1 查询
    • 案例：查 100 个订单 -> 可能触发 100 次额外 SQL 拉关联用户数据
    • 架构结论：工具不是非黑即白
    • 输入/写入：用 JPA 守住事务与模型一致性
    • 输出/读取：用 MyBatis 突破读性能上限
