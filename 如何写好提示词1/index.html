    如何写好提示词1

    如何写好提示词？(上）

    核心结论¶

    写好提示词的底层方法是把“你脑中的任务描述”拆成 4 个可控变量：指令、上下文、输出格式、角色。指令决定你要它做什么；上下文决定它凭什么做对；输出格式决定产物可不可以直接拿来用；角色决定它用什么行为模式与专业默认值来完成任务。

    四大要素速览（定义 + 价值）¶

    • 指令（Instruction）：你让 AI 遵循的命令，用来完成任务/回答问题；越清晰、越具体、越有边界，越不需要 AI 猜。
    • 上下文（Context / Context window）：完成任务所需的背景信息与输入材料；你输入的和 AI 输出的都算上下文，上下文不足时它只能猜。
    • 输出格式（Output format）：让结果“可读/可复制/可解析”，减少你二次整理成本；需要结构化就给样例或 schema/type。
    • 角色（Role）：让 AI 采用特定身份/行为模式；通常不如前三个关键，但在“行为模式很重要”或“专业默认值能省约束”时很值。

    你照着写就能变稳的 Prompt 流程（四要素）¶

    提示词最核心的四个要素：

    • 指令：让 AI 做什么
    • 上下文：完成任务需要的背景信息
    • 输出格式：你希望它怎么组织结果
    • 角色：让它以什么专业身份/行为模式来做

    把它们落成一个可复用模板：

    【指令】
请你……（任务目标、范围、边界）

【上下文】
背景：……
素材/数据：……
约束：……

【输出格式】
请按……格式输出（Markdown/表格/JSON 等），包含……

【角色】
你是……（可选）

【补问机制】
如果信息不足，请先问我 3-5 个问题再开始。

    1) 指令：先把“模糊愿望”变成“可执行任务”¶

    指令（Instruction）的目标是让 AI 像接到一份清晰的任务单：做什么、给谁、做到什么程度、别做什么、怎么验收。

    写法清单（直接照抄改字）¶

    • 主题：写/做什么
    • 受众：给谁看
    • 风格：语气、长度、结构偏好
    • 边界：不要什么、不要包含什么
    • 评价标准：什么算好

    例子：把“写一篇科技文章”写具体¶

    请写一篇科技科普文章，主题是“注意力机制在大模型中的作用”。
受众：高中生，有基础但不懂深度学习。
风格：通俗、类比多、避免术语堆叠。
结构：引言（50字）→核心概念→两个例子→总结（3条要点）。
边界：不要引用论文，不要写公式。

    2) 上下文：补齐 AI 不知道但你知道的“关键背景”¶

    上下文（Context）的作用是把“你任务相关的专属信息”喂给 AI：项目/公司/偏好/材料/已有实现/约束。不提供它就只能猜。

    判定问题（每次写 prompt 前自检）¶

    • AI 知道我知道的这些信息吗？
    • 哪些信息我不提供，它就只能猜？
    • 我再多给一个“参考样例/规范”，结果会不会更贴合？
    • 这次任务需要的“项目偏好/验收口径”，是否已经写进上下文？

    常用补上下文手段（视频给出的做法）¶

    • 直接给素材：财报、文章原文、你写过的代码片段、范文
    • 让 AI 先解释再做：先解释概念/难点，再翻译/再总结（解释本身会进入上下文）
    • 让 AI 主动提问补信息：在 prompt 末尾加一句“如果还需要信息，请先提问”
    • 控制上下文窗口：对话太长时，让 AI/你自己先摘要关键点，再开新对话继续（输入越长，质量通常越下降）
    • 内容过长时换“更大的上下文窗口”：同样是摘要，也可以让更大窗口的模型先做信息压缩再执行

    关键提醒（很容易踩坑）¶

    • “请一步一步思考”只有在它把步骤输出出来时才真正形成上下文；只让它“心里想”没有可验证价值。

    3) 输出格式：让结果“可读 / 可复制 / 可解析”¶

    输出格式（Output format）决定了产物能不能直接拿去用：读起来舒服、复制粘贴就能落地、或能被程序稳定解析。

    首选格式建议¶

    • Markdown：报告/总结/教程最通用（标题、列表、加粗、表格、链接、图片、公式都好用）
    • CSV：要导入 Excel、二维表最方便（纯文本）
    • JSON / YAML / XML：要被程序解析、需要结构化字段时用（JSON 最常见）
    • Mermaid（流程图/时序图等）：需要“图”的时候用纯文本图

    强制格式的 3 种方式（从弱到强）¶

    1) 文字描述输出结构 2) 给一个“你期望的样例” 3) 给 schema/类型（工程场景） - 例如给一段 TypeScript type 或 JSON Schema - 如果你用的 API 支持结构化输出/JSON mode，把 schema 传进去，能极大提升格式稳定性

    4) 角色：可选，但在两类场景很值¶

    角色（Role）是给 AI 一个“默认行为模式 + 专业默认值”。通常不如“指令/上下文/输出格式”关键，但加了经常能减少你补约束的成本。

    适合用角色的场景¶

    1) 行为模式很重要：例如“苏格拉底式导师”，用提问引导你思考，而不是直接给答案 2) 专业背景能省约束：例如“你是资深前端工程师”，它会默认更倾向 TypeScript/React 等常见栈，减少你重复交代

    角色模板¶

    你是【角色】，你的目标是【目标】。
你必须遵守：【约束1/2/3】。
你输出前必须自检：【验收清单】。

    常见错误与解决方案（对照修正）¶

    • 错误：指令太泛（“写个科技文章/取个名字”）
    • 解决：补齐主题/受众/风格/边界/结构/评价标准
    • 错误：缺上下文（AI 只能猜你的公司/项目/偏好）
    • 解决：给素材/给参考样例/让 AI 先提问补信息
    • 错误：上下文越堆越多，质量反而掉
    • 解决：对话过长时先摘要，再开新对话；把“关键点摘要”作为新上下文
    • 错误：输出格式不稳定（表格/JSON 经常乱）
    • 解决：给样例；工程场景给 schema 或开启结构化输出模式
    • 错误：希望它“逐步思考”但看不到过程
    • 解决：要求它把步骤显式输出，作为可见上下文供下一步使用

    速用小抄（最短可用版）¶

    请你完成任务：【一句话目标】。
背景/素材：【贴在这里】。
约束：【不要做什么/必须遵守什么】。
输出格式：请用 Markdown，包含：引言、要点列表、结论。
如果信息不足，请先问我 3 个问题再开始。
