# 强一致还是保生存？BASE 理论（p64）

[强一致还是保生存？BASE 理论（p64）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=64)

## 讨论视角（这期在解决什么）

- 以“强一致还是保生存”为核心矛盾，解释分布式系统在流量洪峰与网络分区下必然要做取舍
- 先用 CAP 把“取舍的物理定律”讲清，再用 BASE 给出互联网系统可落地的工程解法
- 强调 BASE 不是“摆烂妥协”，而是用可控边界换吞吐与可用性，并通过机制把风险收口到可治理范围

## 核心论点与分论点

### 核心论点

在分布式系统中网络分区（P）不可避免，因此分区容错是前提；当分区发生时，如果执着于强一致（C），就只能拒绝服务以避免错乱，系统可用性（A）会崩。电商等业务往往必须优先保可用（A），因此只能在分区时放弃强一致，转向“最终一致”。BASE 理论就是对这种 AP 取向的工程化延伸：核心链路强一致，非核心链路异步化与最终一致，并通过幂等、重试/补偿、死信等兜底机制把一致性风险控制住。

### 分论点 1：CAP 的底层约束：P 必然存在，问题是 C 与 A 怎么选

- 分布式网络中节点通信中断是客观必然，P（分区容错）必须接受
- 一旦发生分区：
  - 选择 C：要求各节点数据实时同步一致 -> 只能拒绝服务，避免错乱
  - 选择 A：保证系统能继续对外提供服务 -> 必然牺牲强一致
- 互联网典型场景（如电商下单）中，“不可用”往往比“短暂不一致”更致命

### 分论点 2：ACID vs BASE：刚性事务与柔性事务的架构思想碰撞

- 传统数据库偏 ACID：
  - 追求强一致与刚性事务
  - 风格更“悲观”：为了不出错，愿意牺牲吞吐
  - 银行核心交易等强一致场景必需
- 互联网架构偏 BASE：
  - 风格更“乐观”：允许数据存在延迟与中间态
  - 以牺牲强一致换取高吞吐与高可用
  - 前提是“最终能收敛到一致”，且窗口可控

### 分论点 3：BASE 三要素：基本可用、软状态、最终一致

- 基本可用（Basically Available）：
  - 流量洪峰时允许降级/限流/屏蔽非核心功能
  - 目标是保住核心链路稳定（可用性优先）
- 软状态（Soft State）：
  - 接受数据存在中间状态与短暂不一致（例如支付成功后积分未到账）
- 最终一致（Eventual Consistency）：
  - 系统必须保证在一个时间窗口后，各副本最终对齐
  - 关键在于界定与治理这个“可容忍窗口”（多大延迟算可接受）

### 分论点 4：典型落地：核心链路 ACID + 非核心链路 MQ 异步解耦（刚柔并济）

- 核心环节（如下单/扣库存/资金）：
  - 仍采用数据库事务保证主流程强一致
- 主流程结束后（如积分/通知等非核心业务）：
  - 不做同步调用，改为写入消息队列
  - 积分服务异步消费消息完成更新
- MQ 作为“软状态载体”，让系统在可用性与一致性之间达到工程平衡

### 分论点 5：稳定运行的两道兜底：幂等 + 重试/补偿（含死信与人工对账）

- 幂等：
  - 网络抖动/重试会导致消息重复投递或重复消费
  - 业务层必须做去重校验，确保同一消息多次处理结果只生效一次
- 重试与补偿：
  - 消费失败不能丢弃，需要重试机制（分级重试）
  - 引入死信队列承接长期失败消息，并配合人工对账作为最终防线

## 关键数据、案例与结论（逐条可复述）

- 案例：分区时若坚持强一致 -> 系统拒绝服务；电商类场景不可用更致命 -> 必须保可用
- 案例：支付成功后积分未到账属于软状态与最终一致窗口内的合法现象
- 结论：
  - CAP 是“硬约束”，BASE 是“工程解法”
  - 主链路强一致，支链路最终一致
  - 一致性风险必须通过幂等、重试/补偿、死信队列与对账闭环治理
