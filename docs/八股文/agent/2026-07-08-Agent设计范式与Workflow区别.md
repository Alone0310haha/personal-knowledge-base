# AI Agent 工程架构

## 了解哪些其他的 Agent 设计范式？Agent 和 Workflow的区别是什么？

**15秒简答**

Agent 和 Workflow 的核心区别是“下一步由谁决定”：Workflow 是开发者提前写死的确定性流程，LLM 只作为某个节点执行任务；Agent 是把决策权交给 LLM，让它运行时动态选择工具、步骤和结束时机。常见 Agent 设计范式包括 ReAct、Plan-and-Execute 和 Reflection。工程上最常用的不是纯 Agent，而是 Agentic Workflow：用 Workflow 管住主流程，在需要灵活判断的节点嵌入 Agent，特点是可控性优先、局部灵活。

**3分钟详答**

这道题本质考的是 Agent 工程里的架构取舍：什么场景该用确定流程，什么场景该把决策权交给模型。Workflow 是确定性的流程图，开发者提前定义好每一步做什么、失败走哪个分支、何时结束，LLM 可以参与检索、重排、生成等节点，但不负责决定流程走向。它的优点是行为可预测、容易测试、出问题好定位；缺点是灵活性低，遇到没覆盖的情况容易卡住。

Agent 则相反，它把“下一步做什么”的决策权交给 LLM。开发者给目标、工具和约束，模型根据当前状态判断是搜索、计算、调用工具，还是直接输出最终答案。它能处理预料之外的复杂情况，但同样的输入可能走出不同路径，线上复现、测试覆盖、成本和延迟控制都更难。

在设计范式上，ReAct 是把 Reasoning 和 Acting 交替进行，每轮包含 Thought、Action、Observation，让模型先显式分析，再调用工具，再读取反馈进入下一轮；它简单直观、调试友好，但容易“走一步看一步”，复杂任务中可能缺少全局规划。Plan-and-Execute 则先规划完整步骤，再逐步执行，并在成熟实现里支持动态重规划：每执行一步都根据结果判断后续计划是否需要调整；它适合有依赖关系的复杂任务，但规划和重规划会增加 LLM 调用成本。Reflection 是在执行后增加质量评估环节，失败就重试或换策略；Reflexion 还会把失败原因和改进建议写成“反思总结”放入下一次尝试的上下文，适合代码生成、文案写作等高质量输出场景，但代价是 token 消耗和延迟增加。

原文的代码对比强调：Workflow 风格中，向量检索、rerank、LLM 生成都是固定步骤；Agent 风格中，循环里的 `llm.decide()` 决定下一步走搜索、计算还是最终回答。文章给出的工程结论是：生产环境里优先从 Workflow 开始，能用 Workflow 解决就不要上 Agent；只有某个节点确实需要灵活决策、写死逻辑覆盖不了时，再升级为 Agent。更主流的落地方式是 Agentic Workflow，用固定主链路保证可控，在局部节点引入 Agent 能力。

**追问：Workflow 和 Agent 最核心的区别是什么？**

**15秒简答：** Workflow 是开发者硬编码控制流，确定性高、好测试、好排查；Agent 是 LLM 运行时动态决策，灵活性高但路径不确定、调试和成本控制更难。

**追问：ReAct 范式是什么，优缺点是什么？**

**15秒简答：** ReAct 是 Thought -> Action -> Observation 的循环，让模型先推理、再行动、再读取反馈；优点是决策更稳、过程可调试，缺点是走一步看一步，复杂长任务容易迷失全局目标。

**追问：Plan-and-Execute 解决了什么问题？**

**15秒简答：** Plan-and-Execute 把规划和执行解耦，先产出全局步骤再逐步执行，并可根据执行结果动态重规划；适合复杂依赖任务，但会增加规划、重规划带来的延迟和成本。

**追问：Reflection 和 Reflexion 有什么价值和代价？**

**15秒简答：** Reflection 在执行后加入质量评估，不通过就重试或换策略；Reflexion 进一步记录失败原因和改进建议作为下次上下文，能提升高质量任务表现，但会增加 LLM 调用、token 消耗和延迟。

**追问：为什么生产环境更推荐 Agentic Workflow？**

**15秒简答：** 纯 Agent 行为不确定、难复现、成本易失控；Agentic Workflow 用固定流程兜住主链路，只在需要灵活判断的节点嵌入 Agent，兼顾生产可控性和局部智能。
