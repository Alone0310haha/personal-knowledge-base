## Workflow 常见编排模式有哪些？Agentic Workflow 为什么是生产主流？

**15秒简答**

Workflow 的编排模式本质是在代码里提前定义 LLM、Tools、Agent 的执行顺序和分支逻辑。常见模式包括：
- Prompt Chaining：把大任务拆成多个串联步骤；
- Routing：先分类，再把请求路由到不同处理分支；
- Parallelization：多个子任务并行执行，最后汇总；
- Orchestrator-Workers：由编排者拆分任务，再分配给多个 Worker；
- Evaluator-Optimizer：先生成，再评估，不通过就反馈优化。

这些模式不是互斥关系，真实项目里通常混合使用。生产环境更常见的是 Agentic Workflow：用 Workflow 固定主流程骨架，在少数需要灵活判断的节点嵌入 Agent，从而兼顾可控性、可调试性和局部智能。

**3分钟详答**

Workflow 不是“多个 Agent 串起来”，而是一种控制流编排方式。它的核心特点是：下一步去哪、失败后怎么处理、什么时候结束，主要由开发者在代码里提前定义；LLM、Tool、Agent 都只是流程里的节点。

纯 Agent 的优点是灵活，适合路径未知、需要动态决策的复杂任务；但缺点也明显：执行路径不稳定、线上复现困难、LLM 调用轮次不可控，成本和延迟都容易膨胀。纯 Workflow 正好相反，路径清晰、可测试、可追踪，但面对预料之外的输入时容易变脆。因此工程上通常采用 Agentic Workflow：主链路用确定流程兜住，只在局部节点让 Agent 自主判断。

常见 Workflow 编排模式可以从“控制流形态”来理解：

| 模式 | 核心思路 | 适合场景 | 注意点 |
| --- | --- | --- | --- |
| Prompt Chaining | 多个 LLM 调用串联，前一步输出作为后一步输入 | 文档生成、信息抽取、分阶段推理 | 每一步最好有明确输入输出和校验标准 |
| Routing | 先识别类型或意图，再进入不同分支 | 客服、工单分流、查询改写、多模型选择 | 分类错误会导致整条链路走错 |
| Parallelization | 可独立执行的子任务并行跑，最后合并结果 | 多数据源检索、多维度评审、批量分析 | 需要定义结果聚合和冲突处理方式 |
| Orchestrator-Workers | 中央编排者拆解任务，多个 Worker 分别执行 | 复杂任务拆分、代码修改、多模块分析 | 编排者要能拆得清楚，Worker 输出要可合并 |
| Evaluator-Optimizer | 生成者产出结果，评估者检查质量，不通过则反馈重做 | 代码生成、法律文本、营销文案、高质量回答 | 评估标准必须清楚，避免无限循环 |

Prompt Chaining 最像流水线。比如生成一篇技术文章，可以先让 LLM 提纲规划，再逐段扩写，再做事实检查，最后统一润色。它的优点是每一步职责清晰，比一次性让模型完成所有事情更稳定；缺点是链路变长后，前面步骤的错误会向后传播，所以关键节点要加校验。

Routing 适合“不同输入要走不同处理逻辑”的场景。比如客服系统先判断用户是咨询商品、申请退款还是投诉，再分别进入商品查询、订单退款、人工作业流程。这里 LLM 可以参与意图分类，但真正的分支跳转仍然由代码决定，所以它是 Workflow，不是 Agent。

Parallelization 适合子任务之间相互独立的场景。比如针对一个技术方案，同时让多个节点从性能、安全、成本、可维护性几个角度评审，最后汇总成一个综合结论。它的价值是降低整体延迟，并减少单个模型视角过窄的问题；但要提前设计好结果合并规则。

Orchestrator-Workers 更适合任务规模大、拆分方式不固定的场景。Orchestrator 负责理解任务、拆分子任务、分配给不同 Worker，Worker 只负责完成局部工作。它比普通并行更灵活，因为子任务数量和内容可以动态产生；但也更依赖编排者的任务拆解质量。

Evaluator-Optimizer 是质量闭环模式。一个 LLM 负责生成结果，另一个 LLM 或同一模型的评估角色负责检查是否满足标准；如果不满足，就把反馈交回生成者继续优化，直到通过或达到最大重试次数。这个模式适合对输出质量要求高的任务，本质上是把“写作-审稿-修改”的过程自动化。

工程选型时可以抓住一句话：流程稳定、边界清晰的部分用 Workflow；局部需要动态判断、写死规则覆盖不了的部分再升级为 Agent。这样既不会让系统完全失控，也不会把所有异常情况都硬编码到 if/else 里。

**追问：为什么生产环境不推荐一上来就做纯 Agent？**

**15秒简答：** 纯 Agent 的执行路径由 LLM 动态决定，灵活但不可控；线上问题难复现、难测试，调用轮次也容易带来成本和延迟失控。生产环境通常先用 Workflow 固定主链路，只在必要节点嵌入 Agent。

**追问：Prompt Chaining 和普通多轮对话有什么区别？**

**15秒简答：** Prompt Chaining 是开发者预先设计好的多步骤流水线，每一步有明确输入输出；普通多轮对话更多依赖模型上下文自然延续，控制流不一定被代码显式管理。

**追问：Routing 模式的关键风险是什么？**

**15秒简答：** 关键风险是路由分类错误。一旦第一步意图识别错了，请求就会进入错误分支，所以通常要设计置信度阈值、兜底分支和人工升级机制。

**追问：Evaluator-Optimizer 和 Reflection 有什么关系？**

**15秒简答：** 两者都利用“评估-改进”闭环。Evaluator-Optimizer 更偏 Workflow 编排模式，强调生成者和评估者的固定流程；Reflection 更偏 Agent 设计范式，强调模型对执行结果进行反思并调整后续行为。
