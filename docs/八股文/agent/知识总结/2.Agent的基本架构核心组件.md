## Agent 的基本架构由哪些核心组件构成？

### 15秒简答

Agent 的基本架构由四大组件构成：
- LLM（大脑，负责理解与决策）
- 工具系统（外部交互执行入口）
- 记忆系统（分短期/长期保持任务状态）
- 规划模块（把复杂目标拆解为可执行步骤）。- 

四者协同构成"规划→决策→执行→记忆→再决策"的循环闭环，常见误区是只提到 LLM 和工具，漏掉记忆和规划。

### 3分钟详答

Agent 架构可以拆成四个协作模块：
- LLM 是统一指挥决策的大脑，靠 System Prompt 定义角色与行为边界；
- 工具系统是对外交互的手脚，负责真正执行具体操作；
- 记忆系统负责保持任务状态，分短期和长期两层；
- 规划模块负责把复杂目标拆解成一步步可执行的序列。

整体运行是一个循环：规划模块先把目标拆成步骤，短期记忆初始化后，LLM 结合短期记忆和长期检索做决策，判断是调用工具还是任务已完成，工具执行结果再写回记忆，驱动下一轮决策，直到任务结束。

这套设计是为了补齐单一大模型的短板：模型本身没有持续状态，上下文窗口又有限，复杂任务跑十几步就容易把 context 塞爆，所以需要短期记忆（context window，任务结束即清空）和长期记忆（向量数据库，跨任务保留）分层存储，短期靠摘要压缩或滑动窗口应对爆满（但都有信息丢失风险），长期靠"存入前做重要性评估"来控制噪音，避免什么都往里存。规划模块存在的原因是复杂目标不能指望模型一步想清楚，需要 CoT（把中间思考过程显式写出来，扩展思考空间）或更贵但更适合复杂决策的 ToT（多分支展开、评估后选最优路径，类似走到岔路口先看几条路再选）这类技术辅助推理；执行上 Plan-and-Execute（先定计划、结构清晰可审核，但中途偏离时调整成本高）和 ReAct（边执行边规划、灵活但容易局部最优走偏）各有取舍，工程上常见做法是先粗略规划方向、执行中再动态微调。

文中给出的伪代码把整个循环概括为：规划拆解目标 → 初始化短期记忆 → 循环（LLM 融合短期+长期记忆做决策 → 判断动作类型 → 工具调用则执行并存入记忆，任务完成则返回答案），主流框架如 LangChain、LlamaIndex、AutoGen 也都是围绕这四个组件做封装，只是实现细节不同。


### 追问：工具系统是怎么定义和运作的，为什么工具描述很重要？

**15秒简答：** 工具用名字、描述、参数说明三要素定义；模型只负责基于描述判断调用时机并输出 JSON 格式的调用决策，真正执行和反馈结果是由程序代码完成的（模型是大脑不是手脚）；描述含糊会导致误触发，精确描述能显著提升准确率，其重要性不亚于 prompt 工程。生态上 Anthropic 提出的 MCP 协议把 Tools/Resources/Prompts 三类能力标准化，已捐给 Linux 基金会。

### 追问：记忆系统如何解决上下文爆满和"该存什么"的问题？

**15秒简答：** 短期记忆存在 context window 里，容量有限，复杂任务跑十几步就容易爆满，靠摘要压缩或滑动窗口缓解（都有信息丢失风险）；长期记忆存向量数据库，靠"存入前做重要性评估"控制噪音，并借助时间衰减因子（相关性分数 = 语义相似度 × 时间衰减）让记忆的权重随场景需要（如客服快衰减、法律合规慢衰减）动态调整。

### 追问：CoT 和 ToT 有什么区别？

**15秒简答：** CoT（思维链）是线性的，让模型把中间思考过程显式写出来，作为下一步输入以扩展思考空间；ToT（思维树）是非线性的多分支展开，每个节点评估多个可能分支再选最优路径，更适合创造性思考或复杂决策，但计算开销更高。

### 比喻式记忆

LLM：老板，负责拍板，决策
工具：外包，真正干活的人，比如：去搜某个网页，去调用某个脚本
记忆：公司档案室，数据库。存储历史说过的内容，可以跨任务保存。
规划：项目经理，将一个需求拆解为可执行的任务。
