
## Workflow，Agent，Tools 这三个的概念和区别介绍一下？

**15秒简答**

Workflow、Agent、Tools 不是三选一关系，而是粒度不同、可以相互嵌套的三层结构。
- Tools 是暴露给 LLM 调用的能力函数，只执行不决策；
- Agent 是以 LLM 为大脑的自主决策系统，会动态判断调哪个工具、是否继续、何时结束；-
- Workflow 是开发者预先写好的流程编排，把 LLM、Tools 或 Agent 组织成确定路径。
 
核心区别看“下一步去哪”由谁决定：Tools 不决定，Agent 由 LLM 决定，Workflow 由开发者在代码里提前决定。整体特性是 Tools 简单确定、Agent 灵活但难控、Workflow 可控可调试但灵活性低。

**3分钟详答**

这三个概念的本质区别，不在于谁更高级，而在于控制流由谁掌握。Tools 是最小能力单元，本质是带 schema 的函数，schema 里说明工具名、用途和参数，让 LLM 知道它能调用什么；但工具本身不知道什么时候该被调用，只负责在被调用时完成具体动作。

Agent 是更完整的决策系统，LLM 负责理解任务、规划下一步、选择工具、读取工具返回结果，并决定是否继续循环。它适合路径不固定、需要临场判断的复杂任务，但因为每一步都可能由模型动态决定，所以执行路径不稳定，调试和成本控制也更难。

Workflow 是更上层的编排方式，节点可以是普通 LLM 调用、Tool 或 Agent，关键是流程顺序和分支由开发者提前写在代码里。比如客服场景中，可以先让 LLM 做意图分类，再由 if/elif 把产品问题、退款问题、其他问题分发到不同分支；这里 LLM 只是流程节点，“下一步去哪”仍由代码控制，所以它是 Workflow，不是 Agent。

这种设计存在的原因是工程取舍：纯 Agent 灵活，但线上行为难控、排查困难、LLM 多轮调用容易带来成本和延迟失控；纯 Workflow 可预测、好调试，但无法穷举所有异常输入，遇到未预料情况会变脆。因此生产环境里更常见的是 Agentic Workflow：用 Workflow 固定主流程骨架，在需要灵活判断的关键节点嵌入 Agent，其余固定节点直接用 LLM 或 Tools，从而兼顾可控性和灵活性。

原文中的代码示例主要说明两点：Tool 示例强调工具只有描述和执行逻辑，没有决策逻辑；客服 Workflow 示例强调 LLM 可以参与意图分类和回答生成，但分支流转由普通代码决定。

**追问：Workflow 是不是多个 Agent 串联？**

**15秒简答：** 不是。Workflow 的节点可以是 LLM、Tools 或 Agent，关键不在节点类型，而在控制流是否由开发者提前写死；多个 Agent 串联只是 Workflow 的一种可能形态。

**追问：Tools 应该怎么设计，LLM 才更容易用好？**

**15秒简答：** 好的 Tool 要职责单一、描述精确、参数 schema 清楚、错误信息可理解；因为 LLM 主要靠工具描述判断是否调用，描述含糊会导致误调用或不会用。

**追问：Agent 和 Workflow 最核心的差异是什么？**

**15秒简答：** 核心差异是“下一步去哪”由谁决定：Agent 由 LLM 在运行时动态决策，同输入可能走不同路径；Workflow 由开发者在代码里预先定义，路径清晰、行为可预测。

**追问：为什么生产环境更常用 Agentic Workflow？**

**15秒简答：** 因为纯 Agent 难控、难调试、成本可能失控，纯 Workflow 又太死板；Agentic Workflow 用确定流程管住主干，在关键节点嵌入 Agent，兼顾可控性、灵活性、成本和调试效率。

**追问：常见 Workflow 编排模式有哪些？**

**15秒简答：** 常见模式包括 Prompt Chaining 串联步骤、Routing 按分类分流、Parallelization 并行执行子任务、Orchestrator-Workers 由编排者分配任务，以及 Evaluator-Optimizer 生成后评估再优化；这些模式可混合使用。
