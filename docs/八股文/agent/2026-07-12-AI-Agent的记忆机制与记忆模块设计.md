# AI Agent 记忆机制

## 请你介绍一下 AI Agent 的记忆机制，并说明在实际开发中应该如何设计记忆模块？

**15秒简答**

Agent 记忆机制的本质是把当前输入、任务上下文、长期知识和结构化事实分层管理，让 Agent 从单次问答变成能保持状态、复用经验的助手。核心机制是四层记忆：感知记忆接收当次输入，短期记忆维护 context window，长期记忆通过外部存储和语义检索跨任务复用，实体记忆沉淀结构化事实。它的特性是连贯、可个性化、可沉淀，但要控制存储噪音、检索时机和上下文成本。

**3分钟详答**

AI Agent 的记忆机制，核心是把不同生命周期、不同访问方式的信息分层管理。没有记忆的 Agent 只能做一次性问答：今天用户说过的代码风格、任务背景、历史决策，明天重新打开对话就全部消失；多步任务中如果没有短期状态，它也不知道上一步查到了什么、当前执行到哪里。

原文把 Agent 记忆分成四层。第一层是感知记忆，也就是当次调用的原始输入，比如用户消息、截图、上传文档，生命周期只有一次调用。第二层是短期记忆，也就是 context window 里的 messages，保存当前任务中的用户输入、模型输出和工具返回结果，帮助 Agent 维持多步任务状态。第三层是长期记忆，存在外部数据库里，通常用向量数据库、关系数据库或 Key-Value 存储，任务结束后仍然保留，下次通过语义检索或精确查询取回。第四层是实体记忆，它把对话中的关键事实提炼成结构化字段，比如用户偏好、项目配置、客户预算、截止日期，信息密度高、查询快。

实际设计记忆模块时，关键不是“都存到向量库”，而是回答三个工程问题：存什么、怎么存、什么时候取。存什么要看这条信息下次任务是否仍然有价值，通常包括用户偏好、任务中的关键结论与决策、外部知识或历史案例；中间推理过程、工具原始日志、闲聊内容通常不值得长期保存，因为会降低检索信噪比。怎么存要按信息类型选择介质：非结构化文本和知识文档适合向量数据库做语义检索，结构化偏好和状态字段适合关系数据库或 Key-Value 做精确查询，主流实践是混合存储。

什么时候取也有两种策略。主动检索是在任务开始前，根据当前任务描述检索相关长期记忆，并把用户偏好和背景注入 system prompt；按需检索是在任务执行过程中，把“查记忆”封装成工具，让 Agent 在需要历史知识、专业文档或过往决策时临时检索。两者结合效果最好：开始时加载用户画像和背景，执行中遇到具体知识缺口再查。

短期记忆还要解决 context window 不够大的问题。常见做法有三种：滑动窗口只保留最近 N 轮，简单但可能丢掉早期关键信息；摘要压缩用 LLM 把早期历史压成任务摘要，节省 token 但可能损失细节；记忆卸载则把暂时不用但后续可能有用的信息写入长期记忆，需要时再检索回来。

完整链路可以概括成“读 -> 用 -> 写”：任务开始前先从实体记忆和长期记忆读出相关背景；执行中用短期记忆保持连贯，并按需检索长期知识；任务结束后，把新偏好、新结论、新经验写回持久化存储。这样 Agent 才会越用越了解用户，而不是每次都从零开始。

原文提到的图表主要是在说明四层记忆的载体、生命周期和访问方式差异，以及完整记忆模块的读写闭环。框架如 Mem0、Letta、Zep 属于工程拓展：它们分别强调记忆服务层、类操作系统的分层记忆管理、时间感知与图谱化记忆，了解即可，不必把它们当成主问题展开。

**追问：四种记忆类型分别是什么？**

**15秒简答：** 感知记忆是当次原始输入，短期记忆是 context window 里的当前任务历史，长期记忆是外部数据库里的持久知识，实体记忆是从对话中提取出的结构化事实。

**追问：设计记忆模块时最核心的三个问题是什么？**

**15秒简答：** 核心是存什么、怎么存、什么时候取：只存下次任务有价值的信息，按结构化或非结构化选择存储介质，再用任务开始主动检索和执行中按需检索结合使用。

**追问：长期记忆为什么不能全部塞进向量数据库？**

**15秒简答：** 因为不同信息适合不同存储方式：文档、摘要、历史经验适合向量检索，用户偏好、项目状态、配置字段更适合关系数据库或 Key-Value 精确查询，混合存储更稳。

**追问：context window 不够大时怎么管理短期记忆？**

**15秒简答：** 常见方案是滑动窗口、摘要压缩和记忆卸载：滑动窗口简单但会丢早期信息，摘要压缩省 token 但可能损失细节，卸载到长期记忆适合保留暂时不用但后续可能用到的信息。

**追问：为什么要做记忆整合？**

**15秒简答：** 长期记忆会积累重复、过期和冲突信息，必须定期去重、冲突消解和抽象提炼，把零散情节记忆沉淀成更高密度的语义记忆，避免越存越乱。
