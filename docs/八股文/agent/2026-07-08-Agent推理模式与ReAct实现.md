# AI Agent 工程架构

## Agent 推理模式有哪些？ReAct 是啥？具体是怎么实现的？

**15秒简答**

Agent 常见推理模式包括直接回答、CoT、ReAct 和 Plan-and-Execute。CoT 的本质是把隐式推理显式写出来，降低多步推理误差；ReAct 在 CoT 上加入工具行动，让 LLM 按 Thought -> Action -> Observation 循环推进任务。ReAct 的关键实现不是模型自己循环，而是模型每次输出下一步思考和动作，代码负责解析动作、执行工具、回填观察结果并再次调用模型。它的特点是推理可见、能接外部工具，但长链路容易漂移、成本随轮次增长。

**3分钟详答**

推理模式要解决的是 LLM 多步任务里的累积误差问题。LLM 本质上是按上下文逐 token 预测，如果让它直接“一口气”给复杂问题答案，中间推导都藏在隐式状态里，某一步出错很难被纠正。CoT 通过让模型先写推理步骤再给答案，把中间结论放进上下文，后续生成就能基于已写出的步骤继续推断，类似把心算改成笔算。

CoT 的局限是只能做纯文字推理，不能自然地和外部世界交互。它拿不到实时数据，不能执行计算，也不能访问数据库；即使 prompt 里让模型“建议调用 API”，工具执行和结果回填仍要靠额外胶水。ReAct 就是在这个基础上，把 Reasoning 和 Acting 结合起来：每一轮先 Thought 分析当前情况，再 Action 选择工具和参数，工具执行后系统把 Observation 写回历史，模型基于新观察进入下一轮，直到输出 Final Answer。

ReAct 和普通 LLM 调用最大的差别在实现分工。模型不会自己“转循环”，它每次只根据当前历史输出一段结构化文本，通常包含 Thought 和 Action；外层代码负责检查是否出现 Final Answer，如果没有，就解析 Action 和 Action Input，调用对应工具，把工具返回值作为 Observation 追加到历史里，再把完整历史传回模型。也就是说，模型负责推理和下一步决策，代码框架负责驱动循环和执行真实动作。

原文用“查询 2024 年苹果和谷歌市值谁更高”的例子说明：ReAct 会先判断需要实时数据，搜索苹果市值，拿到 Observation 后再搜索谷歌市值，最后基于两个结果计算差距并输出答案。代码示例的意图则是说明 ReAct prompt 只是约束输出格式，真正的运行机制是外层循环不断执行“调用 LLM -> 解析输出 -> 执行工具 -> 回填 Observation”。

ReAct 的优点是推理过程可见，工具调用可动态调整，适合开放问答、信息搜索、任务边界不清的探索场景；缺点是每一步都要带历史调用 LLM，步骤越多 token 和延迟越高，而且走一步看一步容易漂移或把前面错误继续传播。更复杂、目标明确的任务可以用 Plan-and-Execute：先规划完整步骤，再逐步执行，并在每步后根据结果动态重规划。工程上常见组合是用强模型做规划，用较便宜的小模型或 ReAct executor 执行具体步骤，在质量和成本之间折中。

**追问：为什么需要 CoT 这种推理模式？**

**15秒简答：** 因为直接回答会把中间推理藏在隐式状态里，容易累积误差；CoT 把推理步骤显式写进上下文，让后续生成基于清楚的中间结论继续推进。

**追问：Zero-shot CoT 和 Few-shot CoT 有什么区别？**

**15秒简答：** Zero-shot CoT 只在 prompt 里加“让我们一步步思考”，成本低但格式和深度不稳定；Few-shot CoT 提供带完整推理的示例让模型模仿，输出更稳定但需要准备高质量样例并占用 token。

**追问：ReAct 为什么比纯 CoT 或 Act-only 更适合 Agent？**

**15秒简答：** 纯 CoT 不能自然获取外部事实，容易受过时知识和幻觉影响；Act-only 只行动不显式推理，动作序列脆弱；ReAct 用 Thought 指方向、Action 做操作、Observation 校准事实，形成闭环。

**追问：ReAct 的循环到底是谁驱动的？**

**15秒简答：** 不是模型自己循环，而是代码框架驱动：模型输出 Thought 和 Action，代码解析并执行工具，把结果作为 Observation 加回历史，再次调用模型，直到检测到 Final Answer 或达到步数上限。

**追问：ReAct 和 Plan-and-Execute 分别适合什么场景？**

**15秒简答：** ReAct 灵活，适合开放式问答、搜索和探索型任务，但容易漂移、成本随轮次增长；Plan-and-Execute 有全局规划，适合目标明确的复杂多步任务，但简单任务上规划本身可能是额外开销。
