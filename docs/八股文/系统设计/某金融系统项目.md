
一、资金系统核心业务逻辑与流程
针对新罗投资集团资金归集系统的业务需求，双方从以下 3 个维度进行了对齐：
1. 系统定位与用户角色
核心功能定位：系统主要用于满足国资委要求的资金归集管理，实现集团对下属子公司资金的统一调度与监管，支持上划下拨、结算及融资管理。
用户群体划分：主要使用方为集团及子公司的财务部门，具体涉及财务总监、会计及出纳。其中，出纳负责实际支付操作，会计负责业务单据与流水的关联。
系统架构模式：采用一套系统多组织架构，集团与子公司共用同一套系统，但通过组织权限进行隔离。
2. 资金计划与归集机制
计划上报周期：子公司需按月上报资金计划，最细颗粒度为半个月一次，但实际操作中通常以月为单位进行上报。
资金下拨逻辑：集团根据子公司上报的月度支出计划全额下拨资金，不扣除收入部分，主要关注支出需求。
资金归集规则：月末进行资金归集，通常采用“留底金额”模式，即保留子公司账户最低限额，超出部分全部归集至集团账户。
3. 业务源头与数据流转
业务源头识别：讨论了融资租赁等业务的源头逻辑，明确了业务部门（如融资部）生成合同及还款计划后，需生成收付款单传递给财务部门执行。
数据来源确认：计划数据来源于业务系统（如融资还款计划、水务收费系统），但本次调研未深入挖掘具体的数据源头细节。
二、系统技术架构与数据规模
4. 上下游系统对接
下游系统明确：系统下游对接金蝶、用友等记账系统，用于生成会计分录和凭证。
上游系统模糊：上游系统尚不明确，推测可能涉及业务系统（如ERP、SRM）或融资合同系统，需进一步确认。
5. 系统性能与规模
数据量级评估：整体数据量较小，每月收付款单约 2000-3000 笔，涉及十几家子公司。
并发量预估：并发用户数较低，预计不超过 50 人，主要集中在财务部门。
6. 信创环境挑战
数据库选型：客户环境采用信创数据库（如达梦），存在迁移适配问题。
迁移风险暴露：分享了涛哥在迁移过程中遇到的科学计数法显示异常等问题，指出信创数据库工具链不完善，缺乏踩坑经验。
三、AI 应用现状与工具选型
7. 宝龙 AI 组项目局限性
功能限制严重：通威项目的 AI 功能被限制在特定工作流内（如仅能变更贷款合同到期日），缺乏灵活的数据查询能力。
架构复杂低效：AI 组采用多层 BSP 架构进行接口对接，导致联调流程繁琐，且前端界面由人工开发，缺乏 AI 原生体验。
产品定位偏差：AI 组过于担心幻觉问题，导致产品功能受限，未能实现真正的 Agentic 能力，被评价为“交差式”产品。
8. 开发工具生态变化
Claude Code 风险：讨论了 Claude Code 因数据安全及合规问题被工信部下架的风险，建议转向 Codex 或其他国内模型。
