# 常见的哈希碰撞解决方案有哪些？

## 15 秒简答（贴标签）

- 定结构（它是什么）：哈希碰撞解决方案就是“多个 key 落同一桶时，如何在同一位置继续存放并可被准确查找”的策略集合。
- 讲机制（它怎么动）：核心思路是两条路：要么把冲突元素“挂在同一桶里”（拉链），要么“换个位置放”（开放寻址/探测），要么“换个 hash 再算一次”（再哈希）。
- 说特性（它怎么样）：工程上常见、各有取舍；拉链易扩展，开放寻址更省指针但对装载因子敏感。

## 3 分钟简答

开场白（润色版）：哈希碰撞是不可避免的，真正考点是：当不同 key 落到同一个桶位时，散列表怎么继续存、怎么继续找。常见方案主要有三类：拉链法、开放寻址法、再哈希法（双重哈希），不同方案在性能、内存和实现复杂度上各有权衡。

带着场景讲核心流程（以“往 HashMap 里连续 put”举例）：

- 假设有两个不同 key 经过 `hash(key) % size`（或位运算取模）后得到同一个 index。
- 方案一：拉链法（链地址法）
  - 这个 index 对应一个 bucket，bucket 里不是只放一个元素，而是放一个链表/树结构的头结点。
  - put 时把新节点追加到桶内结构里；get 时先定位桶，再在桶内用 `equals` 精确比对找到目标。
  - Java 的 HashMap（JDK 8）就是这条路：默认链表，冲突严重时会树化为红黑树降低最坏复杂度。
- 方案二：开放寻址法（如线性探测、二次探测）
  - 冲突了就不在当前桶“挂起来”，而是按探测序列继续找下一个空位置（例如 index+1、index+步长等）。
  - put 时沿着探测序列找空位；get 时同样沿着探测序列找，直到命中或遇到真正的空位为止。
  - 这类方案对装载因子非常敏感：表越满，探测越长，性能下降越明显；删除也更复杂（常需要墓碑标记）。
- 方案三：再哈希法（双重哈希）
  - 第一次 hash 冲突后，用第二个 hash 函数再次计算新的步长/位置，再进行探测或重定位。
  - 目标是让探测序列分布更分散，减少聚簇问题。

为什么是这样设计的（核心取舍）：

- 拉链法：实现直观、扩容/删除相对简单；代价是每个节点有指针开销，且冲突集中会变长（需要树化/扩容兜底）。
- 开放寻址：结构紧凑、缓存局部性好；代价是表一旦接近满载性能会急剧下降，删除复杂，扩容搬迁成本显著。
- 再哈希：用“更随机的探测”缓解聚簇；代价是需要更多 hash 计算、实现复杂度更高。

实战中我用过的地方：

- 业务开发里最常遇到的是 HashMap（拉链法思路），我会关注 key 的 `hashCode/equals` 是否合理，避免大量 key 落在少数桶导致链表变长。
- 在高并发/外部输入场景会额外关注“哈希洪水攻击”风险：依赖 JDK 8 树化兜底，同时在接口层做参数规模限制/限流等防护。

## 比喻理解（帮助记忆）

- 把桶（bucket）当成“停车位编号”：
  - 拉链法：同一个车位旁边拉一条“临停队列”，车都排在这条队列里（链表/树）。
  - 开放寻址：车位被占了就按规则去找下一个空车位（探测）。
  - 再哈希：找下一个车位时不走固定步子，而是用第二套规则算步长，让大家分散开，避免都挤在同一片区域。
