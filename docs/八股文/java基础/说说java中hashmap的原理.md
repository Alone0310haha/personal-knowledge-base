# 说说 Java 中 HashMap 的原理

## 15 秒简答（贴标签）

- 定结构（它是什么）：HashMap 本质是“数组（table）+ 链表/红黑树”的散列表，用 key 的 hash 把数据分到不同桶里存。
- 讲机制（它怎么动）：put/get 都是先算 hash 和桶下标 `(n - 1) & hash`，桶里再用 `equals` 精确定位；发生冲突就链式挂载，链太长会树化，元素太多会扩容 rehash。
- 说特性（它怎么样）：平均 O(1)、冲突可控；线程不安全、扩容有成本。

## 3 分钟简答

开场白（润色版）：HashMap 是 Java 最常用的 Map 实现，本质是散列表：底层用 table 数组做桶，用 key 的 hash 决定落在哪个桶；桶内用链表或红黑树保存冲突元素，从而在大多数场景下实现接近 O(1) 的读写。

结合场景讲核心流程（以 JDK 8 为例）：

- put(key, value)
  - 先对 key 的 `hashCode()` 做一次扰动（高 16 位参与运算），得到更均匀的 hash，减少冲突。
  - 根据 `(n - 1) & hash` 算出桶下标 index（n 是 table 长度，通常为 2 的幂）。
  - 如果该桶为空，直接放入。
  - 如果桶不为空，说明发生 hash 冲突：比较 key 是否相同（先比 hash，再比 `==` 或 `equals`）。
    - 命中同一个 key：覆盖旧 value（返回 oldValue）。
    - 没命中：以链表方式追加在桶后（或插入到合适位置）。
  - 追加后如果链表长度达到阈值（典型是 8），会尝试树化为红黑树；但如果 table 还太小（典型 < 64），通常优先选择扩容而不是树化。
  - 每次插入后都会检查是否超过阈值 `threshold = capacity * loadFactor`（默认 loadFactor=0.75），超过就 resize：容量翻倍并把元素重新分配到新桶。

- get(key)
  - 同样先算 hash 和 index。
  - 先看桶的第一个节点是否命中；不命中则在桶内继续找：
    - 桶是链表：顺着 next 逐个 `equals`；
    - 桶是红黑树：按树结构查找，复杂度约 O(log n)。

为什么这么设计：

- 用数组定位桶 + 在桶内小范围查找，换取平均 O(1) 的性能。
- table 长度设计成 2 的幂，可以用位运算 `(n - 1) & hash` 快速取模。
- 扰动函数让 hash 更均匀，降低热点桶概率。
- 链表处理冲突结构简单、空间开销小；当冲突严重时（链过长）树化，避免最坏 O(n) 退化到 O(log n)。
- loadFactor=0.75 是时间/空间的折中：桶更稀疏则更快但更耗内存，太满则冲突多性能掉得快；扩容保证整体冲突率可控。

实战中我用过的地方：

- 单线程数据处理里做“去重/索引/聚合统计”很常见，比如把用户列表按 `userId -> UserDTO` 建索引、把订单按 `orderId` 去重、把埋点按维度聚合计数。
- 需要强调的是：HashMap 线程不安全；在并发写场景我会改用 ConcurrentHashMap，或者用外部同步（如 synchronized/Lock）保证安全。

## 比喻理解（帮助记忆）

- 把 HashMap 想成“酒店房间表”：table 数组是房间走廊，index 就是房号。
- hash 计算像“导航定位房号”：把 key 先算出一个定位码，再用 `(n - 1) & hash` 找到具体房间。
- 冲突像“同一间房住了多个人”：房间里用链表把人按顺序排队挂起来；如果队伍太长，就把房间改造成“带目录的档案室”（红黑树），查找更快。
- 扩容像“酒店扩建并重新分房”：房间变多后，大家要按新房号重新入住（rehash），所以扩容是有成本的。
