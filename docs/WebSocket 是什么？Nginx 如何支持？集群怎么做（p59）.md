# WebSocket 是什么？Nginx 如何支持？集群怎么做（p59）

[WebSocket 是什么？Nginx 如何支持？集群怎么做（p59）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=59)

## 讨论视角（这期在解决什么）

- 以“实时通信为什么别再用 HTTP 轮询”切入，从性能（CPU/带宽/延迟）解释 WebSocket 的价值
- 从协议层拆到工程落地：WebSocket 如何借助 HTTP 完成握手/升级，再进入二进制长连接通信
- 延展到生产部署与集群：Nginx 如何打通 Upgrade 隧道；多节点 WebSocket 如何解决“连接孤岛”

## 核心论点与分论点

### 核心论点

HTTP 轮询（尤其短轮询）是一种“伪实时”方案：为了不漏消息必须高频请求，但大多数请求为空，浪费带宽且反复握手/解析 header 消耗 CPU，延迟还停留在秒级。WebSocket 通过建立全双工 TCP 长连接，让服务端可以主动推送，把重复的 HTTP 头部开销降到最低，并将通信延迟压到毫秒级。要在生产环境用好 WebSocket，需要正确配置 Nginx 支持协议升级与长连接超时，并在集群中引入消息总线（如 Redis Pub/Sub）解决跨节点推送。

### 分论点 1：轮询为什么低效：空请求占比高 + 握手/头部解析成本高

- 短轮询为了“实时”只能高频请求
- 绝大多数请求拿不到新数据（空响应），但仍消耗带宽与连接资源
- 每次请求的握手与 header 解析对服务器 CPU 是持续的无效消耗

### 分论点 2：WebSocket 的本质：全双工长连接，服务端拥有主动推送能力

- 建立的是 TCP 长连接，链路打通后可双向持续通信
- 服务端不再被动响应请求，有数据即可主动推送
- 避免重复 HTTP 头部开销，延迟可从秒级下降到毫秒级

### 分论点 3：协议升级流程：先用 HTTP 握手，再切换到 WebSocket（101 是分水岭）

- WebSocket 握手阶段依赖 HTTP
- 客户端在 header 中携带：
  - `Upgrade: websocket`
  - `Connection: Upgrade`
- 服务端支持则返回 `101 Switching Protocols`，表示协议升级成功
- 之后连接保持打开，但传输内容不再是 HTTP 文本，而是更高效的二进制帧（“借船出海”：用 HTTP 建连，脱离 HTTP 限制做持久化通信）

### 分论点 4：生产部署关键点：Nginx 默认不识别 Upgrade，需要三类配置打通

- 关键问题：Nginx 作为反向代理时，若不支持/不转发升级信息，会导致握手被拦截或降级，连接失败
- 需要做的三件事：
  - 强制使用 HTTP/1.1（默认 HTTP/1.0 不支持 keep-alive 与协议升级）
  - 显式转发 `Upgrade` 与 `Connection` 相关 header，让后端收到升级请求
  - 调大读取超时（默认 60s 对长连接过短），生产常设置到 3600s+，并配合应用层心跳避免空闲被切断

### 分论点 5：集群化挑战：连接是有状态的，需用消息总线解决跨节点推送

- WebSocket 连接状态保存在节点内存里，不能像 HTTP 一样随意路由
- “连接孤岛”示例：
  - 用户 A 连接在节点 1，用户 B 连接在节点 2
  - A 给 B 发消息，请求达到节点 1，但节点 1 内存里没有 B 的连接信息
- 标准解法：引入 Redis 发布订阅作为消息总线
  - 节点 1 发现目标不在本地 -> 发布到全局频道
  - 节点 2 订阅该频道收到消息 -> 在本地内存中找到用户 B 完成推送

## 关键数据、案例与结论（逐条可复述）

- 关键状态码：`101 Switching Protocols`（协议升级成功的分水岭）
- 关键 Nginx 配置点：HTTP/1.1 + Upgrade/Connection 头转发 + 长连接超时（常见 3600s+）+ 心跳
- 架构结论：
  - 单机：WebSocket 长连接 + 服务端推送解决实时性与性能
  - 多机：Redis Pub/Sub（或同类消息总线）解决跨节点推送
