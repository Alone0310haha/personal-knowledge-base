# 分布式锁：Redis 还是 Zookeeper（p66）

[分布式锁：Redis 还是 Zookeeper（p66）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=66)

## 讨论视角（这期在解决什么）

- 以“分布式锁选 Redis 还是 Zookeeper”切入，强调不是组件选型题，而是 CAP / CP vs AP 的一致性与可用性权衡题
- 从根因解释：单机时代的本地锁只能保证单进程内线程安全，横向扩展到多节点后无法跨进程边界控制并发
- 输出工程落地结论：大多数高并发互联网场景用 Redis（高吞吐），强一致/资金级场景用 Zookeeper（更稳健）

## 核心论点与分论点

### 核心论点

分布式锁的核心价值是把跨进程并发请求“强制串行化”，避免多个实例同时读写共享资源导致竞态与脏数据。Redis 方案更偏 AP：内存读写性能极强，适合大多数高并发业务，但需要额外机制保证锁的活性与正确释放；Zookeeper 方案更偏 CP：基于临时顺序节点与监听机制实现公平排队与自动释放，天然避免死锁并降低惊群，但牺牲部分性能换取一致性与安全性。最终选型取决于业务对一致性风险的容忍度与吞吐诉求。

### 分论点 1：为什么必须要分布式锁：本地锁无法跨进程边界

- 单体应用中，本地锁能控制线程安全
- 微服务多实例后：
  - 每个实例的本地锁只对本进程有效
  - 多个实例可能同时操作同一份共享资源（如库存）
  - 必然出现竞态条件与脏数据
- 因此需要“独立于应用之外”的全局互斥机制，把并行请求转为串行执行序列

### 分论点 2：Redis 分布式锁的两大难点：原子性与活性（避免死锁/误删）

- 活性（避免死锁）：
  - 常见手段：设置 TTL（过期时间）
  - 新问题：业务执行超过 TTL，锁被自动释放
    - 后续请求可能并发进入导致冲突
    - 原持锁线程再解锁可能误删“后来者”的锁（误伤）
- 原子性（避免多步操作被打断）：
  - 用 Lua 脚本把“校验锁归属 + 删除/续期”等动作封装成原子指令
  - 抵御网络抖动导致的指令中断与竞态

### 分论点 3：Redis Watch Dog：把锁生命周期与业务执行时长解耦

- 痛点：TTL 固定，业务耗时不确定
- 解决：引入“看门狗”守护线程
  - 周期性检测业务线程仍存活/仍持有锁
  - 自动续期延长锁有效期
- 效果：锁不会因为业务跑得久而过期，同时避免人为把 TTL 设置得过大导致资源长期占用

### 分论点 4：Zookeeper 分布式锁：临时顺序节点 + 公平排队 + 断连自动释放

- 机制：
  - 通过创建临时顺序节点实现“排队”
  - 最小序号节点获得锁（公平锁）
  - 客户端断开连接，临时节点自动删除，从根源避免死锁

### 分论点 5：Zookeeper 的监听优化：只监听前一个节点，避免惊群效应

- 如果所有等待者都监听同一个锁节点，释放时会广播唤醒造成惊群
- Zookeeper 的关键优化：
  - 每个节点只监听它前一个节点
  - 当前一个节点释放，仅唤醒下一个等待者
- 价值：高并发下显著降低集群网络负载与无效唤醒

### 分论点 6：落地选型：按业务容忍度决定“高吞吐”还是“强一致”

- 数据库悲观锁：
  - 受磁盘 IO 限制，更适合低并发遗留系统
  - 在高并发架构里应视为“保底方案”
- Redis（偏 AP，高吞吐）：
  - 适用：秒杀、社交互动等绝大多数互联网业务
  - 风险：在极端故障/主从切换瞬间存在极低概率丢锁/不一致，需要业务容忍
- Zookeeper（偏 CP，更稳健）：
  - 适用：资金流转、分布式事务协调等强一致场景
  - 代价：性能与运维复杂度相对更高，但换来数据安全

## 关键数据、案例与结论（逐条可复述）

- 案例：库存扣减逻辑从单体迁移到多实例后出现并发错乱，本地锁无法控制跨进程并发 -> 必须分布式锁
- Redis 关键机制：TTL + Lua 原子校验/删除 + Watch Dog 自动续期
- Zookeeper 关键机制：临时顺序节点（断连自动释放）+ 只监听前序节点（避免惊群）
- 结论：90% 高并发业务优先 Redis；强一致、资金级与协调类场景优先 Zookeeper
