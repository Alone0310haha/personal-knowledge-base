# 分库分表 ID 冲突解决方案

[分库分表 ID 冲突解决方案](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=31)

## 讨论视角（这期在解决什么）

- 以“分库分表后主键 ID 冲突怎么解决”的面试题视角，系统梳理分布式 ID 生成方案
- 重点落在：能否保证全局唯一、是否递增有序、对索引/写入性能影响、扩容与单点风险

## 核心论点与分论点

### 核心论点

分库分表后，各分片若继续用自增主键会从 1 开始导致冲突；解决思路要么在 DB 层“规划自增号段/步长”，要么在应用侧生成全局唯一 ID（UUID/Redis 自增/Snowflake），再结合可用性与性能做取舍。

### 分论点 1：自增主键设置“步长 + 起始值”

- 思路：不同分片使用不同起始值，并把 `auto_increment_increment` 设置为分片数（步长）
  - 2 张表：A 起始 1 步长 2 => 1,3,5…；B 起始 2 步长 2 => 2,4,6…
  - N 张表：步长设为 N，每张表起始值设为 1..N
- 致命缺点：不易扩容
  - 例如从 2 分片扩到 3 分片，历史数据的序列规律已形成，基本无法平滑变更

### 分论点 2：数据库“号段模式”（Segment/Hi-Lo）

- 做法：在数据库额外维护一张号段表（公共表），存储：
  - 当前号段开始值、号段长度
  - 用于并发安全的版本号（乐观锁）
  - 业务类型（不同业务用不同段）
- 运行方式：
  - 应用启动/首次获取时，把号段加载到应用缓存里
  - 号段用完后，再去 DB 申请下一段
- 问题：存在单点风险（号段表不可用会影响发号）
  - 可用 DB 集群等方式缓解，但系统复杂度上升

### 分论点 3：UUID

- 优点：应用侧生成，使用极其方便
- 代价：
  - 主键字段需改为字符串类型，查询/存储/索引开销更大
  - UUID 非递增，会导致 B+Tree 插入频繁在中间位置插入，引发页分裂/重平衡，写入性能下降
- 改造建议：
  - 保留递增的数值型主键 ID（只做内部存储顺序，不做业务含义）
  - UUID 作为唯一索引对外使用
  - 若需要跨表合并数据，内部自增 ID 仍需重新生成

### 分论点 4：Redis 原子自增（INCR）

- 优点：实现简单，性能好，天然并发安全（原子自增）
- 风险：Redis 故障可能导致已分配序列丢失
- 持久化讨论：
  - RDB（快照）：两次快照间的增量可能丢失，不适合做发号的唯一保障
  - AOF：记录每次命令，恢复慢但更可靠
  - 另一种兜底：每次发号先检查 key 是否存在，不存在则从 DB 查询最大 ID 回填到 Redis（需要额外保证并发安全，可能影响吞吐）

### 分论点 5：雪花算法（Snowflake）

- 特点：64-bit 数值型且整体有序，兼顾查询与写入性能
- 组成（视频口径）：
  - 41 bit 时间戳（毫秒级，能覆盖约 69 年）
  - 10 bit 机器/节点标识（最多 1024 节点）
  - 12 bit 序列号（同一毫秒最多 4096 个）
- 主要缺点：时间回拨（机器时间倒退）可能导致 ID 冲突

### 分论点 6：现成轮子（开源方案）

- 百度 UIDGenerator：基于雪花算法变体（缩短时间戳，给节点/序列更多 bit），较久未维护
- 美团 Leaf：支持号段模式 + 雪花模式，并处理了雪花的时间回拨问题（需要结合 ZooKeeper）
- 滴滴 TinyID：基于数据库号段模式，减少自维护号段的复杂度

## 关键数据、案例与结论（逐条可复述）

- 步长方案：简单，但扩容困难
- 号段模式：高性能、可控，但要处理公共号段表的可用性
- UUID：最方便但对索引写入不友好；推荐“自增主键 + UUID 唯一索引”的组合
- Redis INCR：简单高性能，但必须认真设计持久化/故障恢复
- Snowflake：
  - 41bit 时间戳（毫秒级，约 69 年）
  - 10bit 节点（1024）
  - 12bit 序列（4096 / ms）
  - 注意时间回拨风险
