其实对于编程这件事来说，我理解是多个需求的组合。既然是多个需求的组合，那么把每个你不会的需求拆分出来，每个需求让ai去生成。ai生成之后，你自己去解构一下ai的写法（注重逻辑性）并整理成文档。等下一次遇到类似的需求时，根据文档自己去写。这样大概就是完成了解构-建构-实践的流程。坚持一年左右就差不多了。

我也天天被这个坑。我的做法是不在 diff 里硬看——让 agent 每次改完先产出三样东西再给我看:① 一句话讲它改了啥、动机是什么 ② 一张「改之前 vs 改之后」的行为对比(截图/录屏/测试输出都行)③ 它自己没把握的地方列出来。这样 review 从"逐行读几百行"变成"看证据做判断",快很多也不会越来越不熟。最近在专门折腾这块,你要是也想聊可以扣我~
