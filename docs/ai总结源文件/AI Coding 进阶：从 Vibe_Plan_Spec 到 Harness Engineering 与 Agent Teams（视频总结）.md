# AI Coding 进阶：从 Vibe/Plan/Spec 到 Harness Engineering 与 Agent Teams（视频总结）

## 1. 听众画像：这期内容写给谁

- 主要面向已经在用 AI Coding 的研发人员（偏中高级）：用过 Coder / Claude Code / Codex / Cursor 之类工具，开始追求“稳定交付”而非“能跑就行”。
- 也面向团队/技术负责人：关心如何把 AI 从个人效率工具，升级为可复制、可治理、可验收的工程体系（含成本与流程约束）。

## 2. 讨论角度：它在解决什么问题

视频从“工作流进化 + 工程化治理”的视角讨论 AI Coding：

- 先澄清 Vibe/Web、Plan、Spec 三种模式在需求澄清、技术方案、任务执行三个阶段的边界与适用场景，解决“概念混用导致工作流失效”的问题。
- 再引入 Harness Engineering，强调把 AI 产出变成“可验证、可回路、可恢复”的工程系统，解决“长任务跑偏、质量不可控、上线不可持续”的问题。
- 最后讨论 Agent Teams（多智能体），从并行与架构角度解决“单 Agent 上下文与吞吐瓶颈”的问题，并补充成本权衡（credits 消耗）。

## 3. 核心论点与分论点

### 核心论点

AI Coding 的“进阶”不是换更炫的工具，而是把研发从“一把梭”升级为“可对齐（Plan/Spec）、可验证（Harness）、可并行（Agent Teams）”的系统化工作流；不同场景选择不同模式，并叠加治理能力，才能稳定提效。

### 分论点 A：Web / Plan / Spec 的差别是“人类介入点”与“工件粒度”

- Web（或 Vibe/Web coding）：人类只介入一次，一句话同时覆盖需求澄清 + 技术方案 + 执行；剩下靠 Agent 自行理解与推进，适合低风险、低复杂度任务。
- Plan：在“技术方案”阶段加一道刹车，先产出 plan 文档供人校准，再执行；适合日常开发中等复杂度需求。
- Spec：在每个阶段产出并校准工件（requirements.md / design.md / task list），流程更重但更可控；适合系统级复杂任务与长程异步任务。

### 分论点 B：Spec 是否要提交到 Git，本质是“定位与维护成本”的权衡

- 反方：Spec 主要用来校准人类意图、以及在长任务上下文压缩后重新注入作为强提醒，因此不一定需要落库。
- 正方：Spec 代表设计与决策历史，未来更值得 review 的可能是 spec（spec review）而非逐行 code review；有团队（视频提到 AWS 内部倾向）会推 spec 入库。
- 代价：一旦 spec 进入仓库并被 Agent 作为上下文来源，就需要持续维护与同步，否则会带来更高治理成本；在 token/credits 成本敏感、工具不统一的现实下，强制推行难度更高。

### 分论点 C：Harness Engineering 的重点是“把反馈回路做成系统”，而不是“让模型更努力”

视频将 Harness Engineering 概括为连接、编排与反馈回路：给 AI 工程套围栏，让系统能持续做对事。并给出四类实践提炼：

1. 地图是导航：入口文档（如 AGENTS.md）应短而可导航，通过目录引用按需下钻，避免把所有约束塞成“厚手册”。
2. 嵌入而非外挂：知识应尽量沉淀在仓库内，而不是散落在 Notion/钉钉/飞书等外部系统；外挂即使能用 MCP 读到，也通常不如“仓库内可检索、可版本化”的效果稳定。
3. 机械式验证而非人工检查：让 AI 在写完代码后直接跑项目与验证脚本（build/test/curl/数据库校验），而不是止步于 lint/build 通过后再由人手动启动验证。
4. 迭代自愈：验证失败就让 AI 自动修复并重试，尝试多次仍失败时才升级到人类介入；人类更多是兜底与方向扣。

### 分论点 D：Agent Teams 的价值是突破单 Agent 瓶颈，但“架构与成本”决定收益

- 最朴素的多智能体：多开会话窗口/多个 worktree 并行推进任务，本质已经是在做 Agent Teams。
- 进一步的显性化架构：以子 Agent（sub agent）承接探索/计划/编码等子任务，主 Agent 汇总结果以节省主上下文，并支持异步执行。
- 通信结构要与任务类型匹配：
  - 严谨垂直、需要强约束与中心化决策的任务，中心化（如 leader + mailbox）更稳。
  - 探索性、没有唯一正确答案、需要平等 peer 共同发散的任务，去中心化网状通信更合适，但成本更高、也更容易跑偏。

## 4. 关键数据、案例与结论（视频原话口径整理）

- 模式成本与效果：
  - Spec 在每轮会做上下文收集与对齐，整体更慢；但适合长程任务“白天/睡前设计，夜里运行，第二天验收”。
  - 示例：睡前花约 30–40 分钟列清需求后启动，让系统跑约 1–3 小时，第二天起床拿到较完整的梳理与产出。
- 长任务上下文问题：
  - 示例口径：上下文达到约 200K 后需要压缩；压缩后可将 spec 文档重新注入上下文作为强提醒，以校准方向。
- 入口文档长度建议：
  - AGENTS.md 作为地图导航建议不超过约 300 行；如果觉得太短，也尽量不要超过 500 行。
- 成本与并行的现实约束（credits）：
  - 团队版示例口径：2000 credits 不够用时，很难强制团队统一工具与流程。
  - 实测口径：某编码需求用 experts 约 400+ credits 跑完；对比另一类去中心化多智能体尝试，出现更快跑偏的情况且更烧 credits。
- 人类并行上限：
  - 演讲者自测：一下午最多提交约 17 个 PR；日常能稳定管理的并行会话大概 3–6 个，再多上下文切换成本显著上升。

## 5. 实践型内容：具体怎么操作（可直接照做的流程）

### 5.1 先选对模式（第一步就决定成败）

1. 判断任务规模与风险：bugfix/小改动 → Web 或 Plan；中等需求 → Plan；系统级/长任务/高不确定 → Spec。
2. 明确人类介入点：你愿意在哪一步做校准与验收，就选哪种模式把“刹车”放进去。

### 5.2 Spec 的“睡前设计，睡醒验收”工作流

1. 梳理需求（requirements）：把所有需求、边界、成功标准一次性罗列出来。
2. 需求对齐：让 Agent 提问补齐缺口，你回答后再迭代 requirements。
3. 产出设计（design）：基于 requirements + codebase 生成 design 文档并校准。
4. 产出任务列表（task list）：拆分执行步骤与依赖关系并确认。
5. 启动长程执行：让系统在较长时间内自动推进实现与验证。
6. 次日验收：对照 requirements 与验证结果，做修正或二次迭代。

### 5.3 Harness Engineering 的落地顺序（从最小可行开始）

1. 做“地图型入口”：写一个短 AGENTS.md（或同类入口）只放导航与关键原则，用目录引用把细节分拆到文档里。
2. 把关键知识嵌入仓库：启动命令、构建命令、认证方式、数据库连接方式、架构规约、常见故障定位路径。
3. 把验收机械化：把 build/test/lint/接口回归（curl）/数据校验写成可执行脚本，让 Agent 能直接运行并读懂结果。
4. 建闭环：执行 → 失败 → 自动修复 → 重试（多次）→ 仍失败再让人介入。
5. 提升可观测：让 Agent 能查询日志/指标，必要时接浏览器自动化做 UI 层验证。

### 5.4 Agent Teams 的两步用法（先朴素，再结构化）

1. 朴素并行：按主题拆任务，多开会话/多 worktree 并行跑，自己只做边界控制与结果合并。
2. 结构化并行：引入子 Agent 分工（探索/计划/实现/验证），主 Agent 负责协调与汇总，减少主上下文占用并提升吞吐。

## 6. 现实应用启示（把它落到日常工作里）

- 先把“对齐与验收”做成默认动作：Plan/Spec 的价值在于把关键决策显式化，减少返工与跑偏。
- 把“验证权”交给系统：让 AI 自己跑项目与回归，而不是人肉点点点；人类把精力放在目标、边界与验收标准。
- 文档不是越多越好：入口要短、可导航；细节要可索引、可分层、可按需加载，否则约束会失效。
- 并行不是免费午餐：多 Agent 提升吞吐，但会放大成本与跑偏风险；先用最朴素的多会话验证收益，再逐步上更重的协作架构。
- 护城河更像“品味 + 迭代速度”：持续试用新工具、理解底层逻辑、形成适配场景的分寸感，比死磕单一范式更重要。
