# 别再误解：Spring 的单例是容器视角，不是 JVM 视角

[别再误解：Spring 的单例是容器视角，不是 JVM 视角](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=46)

## 讨论视角（这期在解决什么）

- 针对面试高频题“Spring 的单例是不是单例模式”，用线上并发/缓存翻车案例说明：把 `@Component/@Service` 当作“JVM 里永远只有一个对象”会导致隐蔽 bug
- 把“单例设计模式（代码层强制唯一）”与“Spring 单例作用域（容器缓存视角唯一）”拆开解释，强调两者约束边界完全不同
- 最终给出实战选择建议：绝大多数业务用 Spring 单例即可；只有少数底层资源/跨容器全局场景才需要回归原生单例模式

## 核心论点与分论点

### 核心论点

Spring 的 singleton 不是“JVM 进程级物理单例”，而是“同一个 IOC 容器内部的逻辑单例”：在一个容器的单例缓存里同名 Bean 只有一份，但同一 JVM 进程可以存在多个容器（父子容器等）从而出现多个实例；甚至在同一容器里也可以通过不同 BeanName/ID 指向同一个 class 来创建多个实例。因此，Spring 的单例是容器视角，而单例设计模式是 ClassLoader/JVM 视角的强制唯一，两者不能等同。

### 分论点 1：单例设计模式的约束边界：同一 ClassLoader 下的物理唯一

- 单例模式在代码层面通过私有构造器 + 静态方法控制实例创建
- 目标：在同一个 ClassLoader / JVM 进程内强制保证只有一个实例（“物理唯一”）

### 分论点 2：Spring singleton 的含义：单 IOC 容器的缓存唯一（托管态）

- Spring 的 singleton 表示：在同一个 IOC 容器的单例缓存中该 Bean 唯一
- 这是容器托管的生命周期管理策略，不是 JVM 级别的强制规则

### 分论点 3：高频翻车场景 A：同 JVM 启多个容器（父子容器）会产生多个实例

- 同一个 JVM 进程中可以启动多个 ApplicationContext
- 容器 A 初始化会创建一个实例
- 容器 B 即使读取相同配置，也会再创建一个实例
- 对 Spring 来说两套容器各自自洽；但对开发者来说内存里确实存在两个对象
- 如果类里有“全局计数器/缓存状态”等，会导致并发或缓存 bug 难以排查

### 分论点 4：高频翻车场景 B：同一容器内也能通过“两个 ID 指向同一 class”创建两个实例

- Spring 的 singleton 更接近“按 Bean 定义唯一”，不是“按 class 唯一”
- 只要配置两个不同的 BeanName/ID 指向同一个 class，Spring 也会创建两份实例

### 分论点 5：实战选择：99% 业务用 Spring 单例，少数底层资源回归单例模式

- 业务开发主要诉求是依赖注入、生命周期管理与开发效率，Spring singleton 足够
- 只有极少数“底层资源”场景需要“跨容器/跨上下文”的强制唯一，例如：
  - 硬件驱动、连接句柄/全局资源管理
  - 跨容器共享的全局配置/全局状态（视频口径）
- 这些场景更适合用原生单例设计模式或其他全局资源管理手段

## 关键数据、案例与结论（逐条可复述）

- 结论句：Spring 的单例是容器视角的逻辑单例，不是 JVM 视角的物理单例
- 两类“打破你以为的单例”的方式：
  - 同 JVM 多容器（父子容器）→ 多实例
  - 同容器不同 BeanName/ID 指向同 class → 多实例
- 业务建议：默认用 Spring singleton；只有底层资源/跨容器强唯一诉求才考虑单例模式
