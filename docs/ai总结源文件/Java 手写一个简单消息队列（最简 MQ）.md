# Java 手写一个简单消息队列（最简 MQ）

[Java 手写一个简单消息队列（最简 MQ）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=35)

## 讨论视角（这期在解决什么）

- 把“用 Java 实现一个简单消息队列”的面试题，归约为可落地的生产者-消费者模型
- 重点不在“中间件能力”，而在并发正确性：有界队列防 OOM、同步保证原子性、wait/notify 控制生产/消费节奏
- 用一段手写实现 + 运行过程推演，解释线程竞争与唤醒后不一定立刻执行的现象

## 核心论点与分论点

### 核心论点

一个“最简消息队列”可以用生产者-消费者模型实现：用一个有界队列承载消息，生产者在队列未满时入队、满时等待；消费者在队列非空时出队、空时等待；两者在同一把锁保护下通过 wait/notify 协调，从而在多线程并发下保证不超容量、不空消费、且不发生线程不安全问题。

### 分论点 1：为什么必须是“有界队列”

- 队列不能无限塞消息，否则会导致 OOM
- 因此生产前要判断：队列当前长度 < 最大长度（临界值），否则生产者进入等待

### 分论点 2：生产者/消费者的关键是“同一把锁下做判断 + 操作”

- 生产者在“判断是否已满、入队、唤醒消费者”这一整段需要持有锁，保证原子性
- 消费者在“判断是否为空、出队、唤醒生产者”这一整段需要持有锁，保证原子性
- 否则会因为并发交错导致判断失真，引发线程不安全

### 分论点 3：生产者逻辑（视频手写口径）

- 若队列长度已到上限：wait 等待，直到消费者消费后唤醒
- 若未到上限：
  - 往队列 add 一条消息（视频用 int 模拟消息，并用长度+1 生成 1、2、3…）
  - 输出“生产者生产一条消息 + 当前队列长度”
  - notify 唤醒消费者
  - sleep 1 秒模拟业务处理，避免过程过快不便观察

### 分论点 4：消费者逻辑（视频手写口径）

- 因为消费者需要持续工作：用死循环持续消费
- 若队列为空：
  - wait（视频称“睡眠”）避免死循环空转浪费 CPU
  - wait 建议 try/catch：遇到 interrupt 等异常时要处理，避免线程“永远睡下去”
- 若队列非空：
  - 消费一条消息并输出“消费者消费一条消息 + 当前队列长度”
  - notify 唤醒生产者继续生产
  - sleep 1 秒模拟业务处理

### 分论点 5：运行推演：notify 后不一定立刻消费，取决于锁竞争

- 测试方式（视频口径）：
  - 先 start 消费者线程
  - 循环 10 次创建 10 个生产者线程去生产 10 条消息
- 现象解释：
  - 生产者/消费者都要竞争同一把 synchronized 锁
  - notify 只是唤醒等待线程，是否立刻执行取决于后续竞争锁的结果
  - 因此执行时序可能交错，也可能出现生产者连续生产多条后消费者才拿到锁开始消费

## 关键数据、案例与结论（逐条可复述）

- 队列最大长度：10（视频示例）
- 测试并发：1 个消费者线程 + 循环创建 10 个生产者线程（视频示例）
- 节奏控制：生产/消费后各 sleep 1 秒模拟业务处理
- 关键结论：简化版 MQ 的重点是“正确的并发协作”，核心机制是 synchronized + wait/notify + 有界队列
