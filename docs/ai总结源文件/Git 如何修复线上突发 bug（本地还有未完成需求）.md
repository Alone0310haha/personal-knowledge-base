# Git 如何修复线上突发 bug（本地还有未完成需求）

[Git 如何修复线上突发 bug（本地还有未完成需求）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=36)

## 讨论视角（这期在解决什么）

- 从“线上突发 bug 必须立刻修，但本地新需求没写完也没测试”这一常见冲突切入，强调不能把两类代码混在同一开发分支里上线
- 用 Git 多分支模型解释“线上代码分支”和“本地开发分支”应当隔离，通过临时切分支完成紧急修复
- 给出团队常见的分流策略：按故障严重程度决定回滚、直接合并上线分支、走发布分支、或并入开发分支随版本上线

## 核心论点与分论点

### 核心论点

线上突发 bug 的正确修复方式是：先把本地未完成的新需求代码暂存，基于线上分支切出独立的 hotfix 分支完成修复并提交，再根据紧急程度选择合并目标分支（线上/发布/开发）与测试上线策略；这样可以避免未完成需求代码混入紧急修复，保证可控上线。

### 分论点 1：为什么不能直接在本地开发分支修线上 bug

- 本地新需求未完成、未测试，代码与线上修复混在一起会导致上线不可控
- 线上修复需要最小变更、可快速验证与回滚

### 分论点 2：Git 多分支能力是解决方案的前提

- 线上代码通常有单独的分支管理（视频示例：线上分支）
- 本地开发也有单独的开发分支
- 发生线上问题时，应该从“线上分支”派生出专门的修复分支，而不是直接在开发分支修

### 分论点 3：标准处理流程（视频口径）

- 第一步：暂存正在开发的代码（先把工作区的改动收起来，避免切分支影响当前开发）
- 第二步：判断故障严重程度
  - 特别严重：系统不可用/核心业务受影响
    - 优先回滚到上一个版本，先恢复线上可用性
  - 非特别严重：走“单独修复分支”修复线上紧急 bug
- 第三步：从线上分支拷贝/派生一个 hotfix 分支
  - 在 hotfix 分支上修复代码并提交
  - 视频强调：通常不会直接在“线上分支”上改

### 分论点 4：修复完成后的合并策略（按紧急程度分流）

- 特别紧急：可能直接合并到线上分支，优先恢复使用（极少发生）
- 一般紧急：合并到 release 发布分支，走测试与上线流程
- 非紧急：合并回开发分支，随当前版本一起发布

### 分论点 5：合并冲突的处理要点

- 若 hotfix 分支与开发分支修改了同一个文件，会产生冲突
- 解决方式是对比两边修改内容，决定保留顺序与最终代码形态（视频示例：开发分支代码在前，修复分支代码在后）

## 关键数据、案例与结论（逐条可复述）

- 面试结论（经验判断）：该问题能难倒大量“自称有经验但缺少真实团队协作经历”的候选人
- 严重程度示例：
  - 特别严重：系统不可用、影响核心业务 → 回滚优先
  - 其他：切 hotfix 分支修复并按流程合并
- 合并目标分支三选一：
  - 线上分支（极少、极紧急）
  - release 发布分支（一般紧急）
  - develop 开发分支（非紧急）
