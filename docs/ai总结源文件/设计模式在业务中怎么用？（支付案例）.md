# 设计模式在业务中怎么用？（支付案例）

[设计模式在业务中怎么用？（支付案例）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=39)

## 讨论视角（这期在解决什么）

- 用“支付策略选择”这个最贴近业务的案例，把设计模式从概念拉回到“如何解决 if-else/switch 分支膨胀与扩展困难”
- 以“逐步重构”的方式串联三种常用模式：策略模式（去分支）、简单工厂（统一创建）、模板方法（抽公共流程）
- 进一步落到 Spring 工程实践：为什么工厂里 `new` 出来的对象无法自动装配，以及如何通过 Spring 容器拿到真正的 Bean

## 核心论点与分论点

### 核心论点

设计模式不是为了“多写类”，而是为了把变化点隔离出来：用策略模式把业务分支转为可替换的策略实现；用简单工厂集中管理策略实例的创建；当多个策略存在通用流程（如参数校验、日志、通用前置/后置步骤）时，用模板方法把“固定流程 + 可扩展步骤”固化下来，从而在业务持续迭代时保持代码可读、可扩展、易维护。

### 分论点 1：问题根源是分支膨胀导致可读性与可维护性下降

- 初版伪代码：声明支付接口，根据 `payType` 做分支判断（if-else/switch）创建不同支付对象，再调用支付方法
- 逻辑正确，但随着支付类型增加：
  - 分支越来越多，阅读性变差
  - 修改风险变高（改动集中在同一个分支块）

### 分论点 2：策略模式——把“分支选择”变成“策略替换”

策略模式三类角色（视频口径）：

- 策略接口：定义统一的支付策略接口与 `pay()` 方法
- 策略实现：例如支付宝/银联/微信等具体实现
- 策略上下文（环境）：根据 `payType` 产出对应策略对象，并对外提供统一的支付入口

收益：

- 上层调用从“多分支 + 多行”变成“少量固定调用”（视频示例：上层只剩两行）
- 新增一种支付类型：增加一个策略实现 + 增加映射/分支即可，上层不变

### 分论点 3：简单工厂——把“上下文里的冗余调用”收敛为统一的创建入口

- 观察：策略上下文里“根据 payType 取策略对象 + 再调用 pay”存在冗余
- 改造：引入简单工厂
  - 工厂方法返回策略接口类型
  - 根据 `payType` 创建并返回具体策略实现
- 调用侧：直接通过工厂拿到策略对象，再调用支付方法，结构更直观（可一眼看到有哪些策略）

### 分论点 4：Spring 工程落地——工厂里直接 `new` 的对象不是 Bean，无法自动装配

- 问题：简单工厂里 `new` 出来的支付对象，不在 Spring 容器中
- 结果：支付对象内部如果需要注入依赖（例如 mapper 做数据库操作）无法自动装配
- 解决思路（视频口径）：
  - 通过 Spring 提供的 `ApplicationContextAware` 拿到 `ApplicationContext`（Spring 上下文/容器）
  - 工厂不直接 `new`，而是通过 `applicationContext.getBean(...)` 获取对应类型的 Bean
  - 这样策略实现类就可以正常使用自动装配

### 分论点 5：模板方法——抽取通用流程，避免每个策略里重复写“公共逻辑”

场景：不同支付类型有共同流程，例如：

- 支付前参数合法性校验（如金额必须 > 0）
- 支付完成后的日志记录等通用后置动作

如果在每个 `pay()` 里重复写，会造成重复代码与一致性风险。

模板方法改造要点（视频口径）：

- 定义抽象类（模板类），让它实现统一的策略接口
- 在抽象类中定义模板方法（固定流程），例如：
  - `check(amount)`（公共校验）
  - 调用子类实现的抽象步骤（完成各自支付实现）
  - 可继续追加后置步骤（如日志）
- 将校验等关键步骤方法声明为 `private`：
  - 目的：防止子类绕过/覆盖关键校验步骤，保证流程契约与安全性
- 具体策略实现类继承该抽象模板类，只实现差异化步骤

### 分论点 6：最终形态的扩展方式（视频总结口径）

- 新增支付类型：
  - 增加一个新的支付策略实现（继承模板类/实现接口）
  - 在工厂中增加一个分支/映射即可
  - 上层应用无需改动
- 新增通用流程步骤：
  - 只需在模板方法中追加步骤（例如支付后统一写日志）
  - 不需要修改每个策略实现

## 关键数据、案例与结论（逐条可复述）

- 案例：按 `payType` 选择支付方式（支付宝/银联/微信）
- 结论 1：策略模式把分支膨胀隔离到策略体系中，上层调用更简洁
- 结论 2：简单工厂统一策略创建，让调用侧更直观，便于集中管理策略列表
- 结论 3：工厂直接 `new` 会导致对象不是 Spring Bean，无法自动装配；应从 Spring 容器获取 Bean
- 结论 4：模板方法把“校验/日志”等公共流程抽到抽象父类，提升扩展性与一致性；关键步骤用 `private` 固化契约
