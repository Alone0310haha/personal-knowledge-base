# 不再写DDD之后，我好像会用DDD了（视频总结）

[不再写DDD之后，我好像会用DDD了](https://www.bilibili.com/video/BV12pddBgEUx/)

## 1. 视频的听众针对的是哪一类人或人群

- 读了很多 DDD 概念/术语（聚合根、值对象、限界上下文等）但一写代码还是懵的人。
- 在项目里“对号入座式”套 DDD 分层/分包，结果代码更复杂、业务逻辑更难找的人。
- 想把 DDD 真正用在日常开发，而不是完成一套“仪式”的后端/架构关注者。

## 2. 视频是从什么角度出发进行的讨论，它要解决什么样的问题？

视频从“DDD 不是规范，而是一种思考习惯”的角度出发，重点讨论“怎么用 DDD 的思维写出更贴近业务意图、边界更清晰、复杂度更可控的代码”。

它要解决的问题是：

- 为什么懂术语却不会落地，甚至越做越复杂。
- 怎么判断边界、聚合、分层该不该引入、何时引入。
- 如何把产品流程图翻译成业务语言，从而自然浮现实体、动作、事件与边界。

## 3. 视频的核心论点和分论点

### 核心论点

DDD 的核心不是“按规范写出 DDD 形状的代码”，而是用一套思维习惯去表达业务意图、发现业务边界，并跟着业务复杂度逐步引入结构，避免过度设计。

### 分论点

- 业务语言优先：代码应该说业务的语言，而不是让业务去理解技术语言；类名/方法名应该能让人一眼看出业务场景。
- 边界是被发现的：边界不是设计阶段一次画准的，而是业务长期运转中逐步暴露；设计时先抓“谁对什么负责、谁有权利改什么”。
- 跟着复杂度走：如无必要勿增实体；分层/模式是解决复杂度的工具，不是必须遵守的仪式。业务简单时，允许更“朴素”的写法。

## 4. 列出视频中提到的关键数据，案例和结论

### 关键数据/结构化结论

- DDD 落地的底色：3 个要点（业务语言优先、边界被发现、跟着复杂度走）。
- 复杂度分 3 个阶段逐步引入结构：
  - 阶段 1：业务简单，服务里完成校验 + IO 即可，不必引入实体行为/领域事件。
  - 阶段 2：单实体状态流转变多，服务里出现大量状态判断，把状态机与业务规则收进实体，拆分 IO 层与业务层。
  - 阶段 3：多聚合复杂协作、跨聚合副作用增多，引入领域事件与更严格的应用层编排。

### 案例

- 商品的价格 vs 库存：
  - 数据上强相关（都属于商品，放一张表很自然），但业务上修改方、频率、一致性要求不同。
  - 若强绑在一起，高并发扣库存可能锁住整个商品记录，连价格字段也被影响 → 典型“数据相关但业务应分开”。
- 订单收货地址 vs 用户收货地址：
  - 数据长得像，但语义不同：用户改地址不应该改变历史订单地址。
  - 订单地址是下单时的快照，不是用户地址的引用 → 看似一个概念，其实是独立概念。
- 审核系统“提交申请”的分层误解：
  - 领域层的“提交”只做状态流转（草稿→已提交）与业务校验，不知道数据库存在。
  - 写库是应用层的 IO：先加载对象→调用领域提交→再持久化。
  - 分离后领域层不依赖框架与基础设施，单元测试可直接 new 对象调用方法完成。

### 关键结论

- 边界判断的三个实用问法：
  - 谁有权利改它？如果不同流程/不同角色以不同理由改同一概念，边界可能画大了。
  - 改一个是否必须连带改另一个？如果不需要，耦合可能是假的，可拆。
  - 变化频率是否一致？变化节奏不一致（天天改 vs 半年改一次）是强烈拆分信号。
- DDD 容易走偏：把它当规范会让你“对着概念找代码”，把它当工具会让你“对着问题找答案”。

## 5. 如果是带有实践性质的内容，请你阐述他具体是怎么操作的，第一步是什么，第二步是什么？

视频给出的落地路径，更像一套“建模与演进”操作方法：

1. 先用业务语言审视代码：打开现有代码，检查类名/方法名是技术语言还是业务语言；优先把关键流程的命名改到“业务一眼懂”。
2. 用问题发现边界：围绕“谁能改、是否连带改、变化频率”去做初步边界划分，并接受边界会在运行中被不断校准。
3. 避免为分层而分：业务简单时不强行引入聚合根/领域事件/应用层编排；复杂度到了再加结构。
4. 按复杂度三阶段演进结构：
   - 简单期：服务直写校验与持久化。
   - 状态流转多：把状态机/规则收进实体，拆 IO 与业务。
   - 跨聚合副作用多：引入领域事件与应用层编排协调。
5. 把流程图翻译成业务语言（建模前置步骤）：
   - 确定主体（实体）：让业务决策当主语，而不是“系统/流程”当主语。
   - 确定动作（业务动词）：把技术动词改成业务动词。
   - 确定结果（事件表达）：用“发生了什么”的事件表达结果，而不是只描述“做了什么”。
   - 验证：把翻译结果念给产品/业务听，确认理解一致。

## 6. 在实际生活中应用的时候的启示

- 不要用“DDD 形状”判断好坏，用“业务意图是否清晰、边界是否可演进、复杂度是否可控”判断。
- 过度设计的代价很高：为术语对号入座会让逻辑藏起来，难读难改，反而背离 DDD 初衷。
- 边界不可一次性设计完美：关键是建立一套持续发现/校准边界的问法与机制。
- 分层的收益来自可测试与可维护：领域层不依赖基础设施，能让单元测试更轻、更可靠，但前提是复杂度值得。

## 7. 可直接落地的 SOP（把 DDD 变成日常习惯）

### SOP-1：命名与语言（每次需求开始前）

1. 写下本需求的核心业务名词（实体/概念）与业务动词（行为）。
2. 把关键接口/类/方法命名为业务语言，避免 convert、handler、manager 等“看不出业务意图”的泛化名。

### SOP-2：边界初划（建模/拆分时）

1. 谁能改它：不同角色/流程是否会改同一概念？如果会，先怀疑边界是否过大。
2. 是否必须一起改：改 A 是否必须连带改 B？不必须就先拆。
3. 变化频率：A 与 B 的变化节奏是否一致？不一致就先拆。

### SOP-3：按复杂度引入结构（实现时）

1. 阶段 1（简单）：服务里完成校验 + 持久化，不强行 DDD。
2. 阶段 2（状态多）：把状态机与规则收进实体，领域层不做 IO，应用层做编排与持久化。
3. 阶段 3（跨聚合）：引入领域事件/更严格应用层编排，处理副作用与一致性边界。

### SOP-4：流程图翻译（拿到产品流程图时）

1. 主体：把“系统做了什么”改写成“谁（业务角色）做了什么决策”。
2. 动词：把“更新/调用/发送”改成业务动词（支付完成、库存被预扣、用户收到通知等）。
3. 事件：把结果写成“发生了什么”，并用它反推完成标准（例如“用户收到通知”而非“发送通知”）。
4. 验证：把翻译后的句子读给产品/业务，确认语义一致后再开始建模。
