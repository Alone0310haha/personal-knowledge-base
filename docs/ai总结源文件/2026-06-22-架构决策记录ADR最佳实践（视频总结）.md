# 架构决策记录 ADR 最佳实践（视频总结）

[【IT老齐899.59】架构决策记录ADR最佳实践](https://www.bilibili.com/video/BV1ymjN6YE4Z/)

## 1. 目标听众

本视频面向有一定工作经验的**软件工程师和架构师**，尤其是：

- 负责或参与过技术选型、架构设计的开发者
- 在团队中承担架构治理/文档规范工作的技术负责人
- 遭遇过"为什么当初选这个框架？当初的架构师都离职了，没人说得清"此类困境的团队成员

## 2. 问题背景与讨论角度

视频从**架构知识的流失**这一切实痛点出发，提出一个核心问题：

> 当一个架构决策被做出之后，决定它的人和理由会消失，但这个决策的影响还会长期存在——团队该怎么管理这种"知识债务"？

作者将此归纳为两个不同步的生命周期：

- **技术生命周期**：这套方案/组件能用多久，取决于团队规模、业务规模、应用能力等客观条件
- **知识生命周期**：当时做出这个决定的理由能存活多久，极易随人员流动消失

两者不同步，是架构管理中最常被忽视的问题。ADR（Architecture Decision Record，架构决策记录）正是为弥合这一裂缝而生。

## 3. 核心论点与分论点

### 核心论点

ADR 是架构设计与管理中不可缺少的关键工具。它以文档化的方式，记录架构决策的**完整来龙去脉**，而不仅仅是"选了什么"，更是"为什么选、当时考量了什么、什么时候应该放弃"。

### 分论点一：不记录 ADR 的四种代价

不做架构决策跟踪，团队会在以下场景中反复付出代价：

| 问题 | 表现 |
|---|---|
| **重复的争论** | 同一个技术选型问题被反复讨论，因为当初的决策依据已无从查起 |
| **新人盲区** | 新人看到某个组件/设计，完全不知道为何如此，只能猜测或乱改 |
| **迁移瘫痪** | 想换方案却不知当初约束是什么，不敢动也不知道怎么动 |
| **架构决策偏差** | 在对旧决策一知半解的情况下做新决策，方向越走越偏 |

### 分论点二：ADR 的五大核心要素

一份 ADR 文档并不复杂，核心包含以下五要素：

1. **背景（Context）**  
   当时是什么业务或技术背景促成了这次决策？有哪些约束条件？例如：系统每日审批量大、规则变更频繁、平均交付周期过长等。

2. **决策（Decision）**  
   具体选择了什么方案？选择的依据是什么？不做什么（范围界定）？

3. **后果（Consequences）**  
   - 正面影响：e.g. 修改规则的时间从 3~5 天缩短到 2 小时内；规则条目增加但维护成本未显著增长
   - 负面影响：e.g. 性能下降；回滚机制不完整；简单场景下不如 if/else 直接

4. **替代方案（Alternatives）**  
   当时还考虑过哪些其他方案？为什么最终没选它们？  
   *这部分尤为关键：记录"为什么不选"，能防止未来有人重复踩同一个坑、或在不了解背景的情况下重新提出已被否定的方案。*

5. **撤销条件（Revocation Conditions）**  
   什么情况下当前决策应该被重新评估？例如：
   - 风控规则大幅简化，不再需要规则引擎
   - 大模型取代了规则引擎的职责
   - 当前组件的依赖停止维护
   - 审批逻辑移交第三方系统
   - 性能下降已超过可接受阈值

### 分论点三：ADR 的四个生命周期状态

一份 ADR 在其生命周期中会经历以下状态流转：

```
提议（Proposed）→ 接受（Accepted）→ 弃用（Deprecated）→ 被新 ADR 取代（Superseded）
```

- **提议**：有人提出这个方向，ADR 草稿完成
- **接受**：团队决策通过，文档合并入仓库
- **弃用**：决策不再适用，但没有直接替代
- **被取代**：有新的 ADR 覆盖了本 ADR 的范围

### 分论点四：ADR 的不可变原则

> **一旦 ADR 被接受并合并入仓库，就不应该再被修改**（低级拼写错误除外）。

ADR 是当时决策的**历史快照**，代表特定时间点的真实状态。如果产生了新的变化，应该**创建一份新的 ADR** 来取代旧的，而非在原有文档上修改。旧 ADR 文末需注明"已被 ADR-XXX 取代"，新 ADR 同样需要注明"取代了 ADR-XXX"，形成**双向链条**。

### 分论点五：ADR 状态矩阵

在 ADR 目录下创建一个 `README.md`，作为**ADR 状态看板**，以表格的形式汇总所有 ADR 的当前状态（已接受/弃用/被取代等）。无需手动维护，交给 AI 在每次 ADR 状态变更时自动更新即可。

## 4. 具体操作步骤

### Step 1：在项目中建立 ADR 目录

```
项目根目录/
└── docs/
    └── ADR/
        ├── README.md       ← ADR 状态矩阵（看板）
        ├── ADR-001-xxx.md
        ├── ADR-002-xxx.md
        └── ...
```

纳入 Git 统一托管，与代码同版本管理。

### Step 2：为每次架构决策撰写 ADR

使用以下模板（可结合 AI 辅助生成初稿，再人工补充）：

```markdown
# ADR-XXX：[决策标题]

## 基本信息
- 状态：提议 / 接受 / 弃用 / 被取代
- 日期：YYYY-MM-DD
- 决策人：张三、李四

## 上下文
[描述当时的业务背景、技术现状和约束条件]

## 决策
[说明选择了什么方案，采用了哪些组件/技术，具体的方式是什么，以及明确不在范围内的部分]

## 后果
### 正面影响
- ...

### 负面影响
- ...

## 替代方案
| 方案 | 描述 | 未选原因 |
|------|------|--------|
| 方案 A | ... | ... |
| 方案 B | ... | ... |

## 撤销条件
当以下任一条件出现时，应重新评估本决策：
- ...

## 变更历史
| 日期 | 人员 | 变更内容 |
|------|------|---------|
| YYYY-MM-DD | 张三 | 初始创建 |
```

### Step 3：管理 ADR 的状态流转

- 有新想法 → 创建 ADR，状态设为"提议"
- 团队确认 → 合并到仓库，状态改为"接受"
- 决策过时但无直接替代 → 状态改为"弃用"
- 有新 ADR 覆盖 → 旧 ADR 状态改为"被取代"，末尾注明新 ADR 编号；新 ADR 头部注明取代了哪份旧 ADR

### Step 4：维护 ADR 状态矩阵

每次 ADR 状态变更时，同步更新 `docs/ADR/README.md` 中的状态表格。可用 AI Prompt 完成：

> "扫描 ADR 目录下所有文件，汇总每份 ADR 的编号、标题、状态、日期，生成/更新 README.md 中的状态矩阵表格"

### Step 5：定期回顾

每季度或半年做一次 ADR 全面回顾：
- 检查现有"接受"状态的 ADR 是否达到了撤销条件
- 检查各 ADR 所描述的实际落地情况是否仍与文档一致
- 对过时的 ADR 启动更新流程

## 5. 实际应用启示与 SOP

### 什么粒度的决策应该写 ADR？

**核心判断原则**：如果这个决策的"为什么"六个月后会让人产生疑问，就应该写 ADR。

**应该写 ADR 的决策**

- **技术选型**：选 Redis 而不是 Memcached、引入规则引擎而不是硬编码 if/else
- **架构模式**：微服务还是单体、事件驱动还是同步调用
- **跨系统约束**：统一的认证方案、日志格式、API 版本策略
- **有真实替代方案被否决的决策**：讨论过 A/B/C 三种方案，最终选了 A，B 和 C 为什么没选要留档
- **难以逆转的决策**：数据库分库分表、消息队列引入、服务拆分
- **受非显性约束驱动的决策**：因合规要求、性能瓶颈、第三方限制而做的选择——这类"为什么"最容易随人员流动消失

**不需要写 ADR 的决策**

- 命名规范、代码风格（交给 lint 工具）
- 具体功能的实现细节（注释或 PR 描述即可）
- 没有真实权衡的显而易见的选择

**实操判断问题**：如果一个新人来问"为什么这里用 X 而不是 Y"，这个答案是靠回忆还是靠文档能得到？靠回忆 → 写 ADR；靠已有文档 → 不需要另起 ADR。

### 核心启示

- **ADR 的核心价值不是"记录选了什么"，而是"记录为什么选、什么情况下要换"**，这是知识得以延续的关键。
- 写 ADR 的负担不需要很高：AI 可以基于现有工程代码/会议记录生成初稿，人工只需确认和补充。
- 不可变原则 + 链条机制，使 ADR 成为可信赖的历史档案，而非随意被改写的说明文档。

### 落地 SOP

```
1. 发生架构决策时
   └─ 立刻或决策结束后 24 小时内创建 ADR 草稿（状态：提议）

2. 团队评审通过后
   └─ 提 PR，合并入仓库（状态：接受）
   └─ AI 自动更新 README.md 状态矩阵

3. 引入新同学/新团队成员时
   └─ 指引其先阅读 ADR 目录，了解各架构决策的来龙去脉

4. 技术架构评审/迁移讨论时
   └─ 首先查阅相关 ADR，避免重复争论，明确现有约束

5. 架构发生变化时
   └─ 创建新 ADR（状态：提议）→ 评审 → 接受
   └─ 旧 ADR 末尾标注"被 ADR-XXX 取代"
   └─ 新 ADR 头部标注"取代 ADR-YYY"

6. 每季度/半年
   └─ 执行 ADR 全量回顾
   └─ 检查撤销条件是否触发
   └─ 对不再准确的 ADR 启动更新流程
```

### 与 CI/CD 集成（进阶）

现代化流水线可集成 ADR 检查：
- 检测 ADR 状态是否与实际代码/配置一致
- 当检测到撤销条件已满足时，自动标记对应 ADR 并通知相关人员重新评估
- 定期触发 ADR 评审提醒（如每季度）
