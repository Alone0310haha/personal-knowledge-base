# B 树 vs B+ 树 & InnoDB 为什么选 B+ 树

[B 树 vs B+ 树 & InnoDB 为什么选 B+ 树](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=23)

## 讨论视角（这期在解决什么）

- 以“面试高频题”的角度解释：B 树 vs B+ 树的差异，以及 InnoDB 为什么默认使用 B+ 树做索引结构
- 关键评判标准围绕：磁盘 IO 次数（树高/树宽）、大数据量下的检索效率、范围查询效率

## 核心论点与分论点

### 核心论点

InnoDB 选择 B+ 树而不是 B 树，核心是为了在“大数据量 + 磁盘 IO”场景下：让树更“宽”、层级更少，从而减少 IO 次数，并且天然更适合范围查询。

### 分论点 1：共同点（检索的基本依据）

- 都满足：小 key 在左，大 key 在右
- 查询会从根节点逐层向下定位（每下探一层基本就是一次磁盘 IO）
- IO 次数越少，检索越快

### 分论点 2：结构差异（数据放在哪里）

- B 树：每个节点同时存“索引 + 数据”
- B+ 树：
  - 非叶子节点只存索引
  - 所有数据都放在叶子节点
- 推导结果：
  - B+ 树非叶子节点更“轻”，同一页/节点能放更多索引
  - 树会更“宽”，在同等数据量下高度更低 → 下探层数更少 → IO 更少

### 分论点 3：为什么不是“B 树命中根节点更快”就选 B 树

- 反例（视频提到的直觉）：如果要查的 key 正好在根节点，B 树可能 1 次 IO 就命中
- 但数据库关注的是“大量数据”下的整体检索效率，而不是少量数据的个别命中场景
- 数据量少时，任何结构（甚至链表）也能很快；数据量大时，减少层高才是关键

### 分论点 4：范围查询能力（B+ 树明显更强）

- B+ 树叶子节点通过双向链表相连
- 范围查询（如 3~5）：
  - 先定位到起点 3
  - 再沿链表顺序扫到 5
- B 树做范围查询的代价更高（需要反复从根节点出发定位多个 key），整体效率差

## 关键数据、案例与结论（逐条可复述）

- 案例：按 id 建索引后，查询从根节点逐层下探；每次下探对应一次磁盘 IO
- 结论 1：B+ 树非叶子节点不放数据 → 树更宽更矮 → IO 次数更少 → 点查效率更稳
- 结论 2：B+ 树叶子节点链表 → 范围查询效率高，是数据库核心诉求之一
