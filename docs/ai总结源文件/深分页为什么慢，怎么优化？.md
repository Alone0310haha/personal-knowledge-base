# 深分页为什么慢，怎么优化？

[深分页为什么慢，怎么优化？](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=32)

## 讨论视角（这期在解决什么）

- 从 MySQL 执行流程与 B+Tree 索引访问路径出发，解释“深分页（Limit Offset）为什么慢”，以及在二级索引回表场景下为何会雪上加霜
- 给出从低成本到高成本的优化路径：索引覆盖/子查询延迟关联、基于主键的 Keyset 分页、游标分页、分表分区、ES，以及业务侧限制深翻页

## 核心论点与分论点

### 核心论点

`LIMIT offset, size` 的深分页慢在“并不会直接跳到 offset 位置”，而是需要从第 0 条读取到 `offset+size` 再在 Server 层丢弃前 offset 条；当走二级索引且需要回表时，会把 `offset+size` 这一大批主键逐条回表，导致额外的索引树访问与 IO，性能进一步恶化。

### 分论点 1：深分页在引擎层的真实读取方式

- 执行流程：SQL 进入 Server 层解析/预处理/优化后进入引擎层取数据
- 关键点：引擎层不是“定位到第 50000 条再取 100 条”，而是从 0 取到 `50000+100`（示例口径）
- Server 层再根据 `LIMIT`、条件、分组等做过滤，最终返回 size 条

### 分论点 2：二级索引 + 非覆盖查询会更慢（回表放大）

- 二级索引 B+Tree 叶子节点通常存“索引列 + 主键”
- 若查询列不被二级索引覆盖（例如还要查 name），需要回表：
  - 先从二级索引拿到 `offset+size` 批数据（索引列 + 主键）
  - 再用这些主键到主键索引（聚簇索引）逐条取整行
  - 回表完成后再在 Server 层丢弃 offset，保留 size
- 结论：回表发生在大批量数据上，深分页时成本被成倍放大

### 分论点 3：优化方向一——索引覆盖 / 延迟关联（子查询）

- 优先让查询变成“索引覆盖”，减少回表
- 若必须查二级索引以外的字段：
  - 先用子查询只查索引覆盖字段（拿到 size 条主键）
  - 再用这批主键去主键索引查询所需字段
- 效果：把回表规模从 `offset+size` 降为 `size`

### 分论点 4：优化方向二——Keyset 分页（用主键当页码）

- 用主键（ID）定位，不用 offset：
  - 通过 B+Tree 直接定位到某个 ID 附近，再取后续 size 条
- 前提：主键有序（自增等）
- 代价/限制：
  - 中间删除会导致“某页不足 size 条”（例如只返回 98 条）
  - UUID 等无序主键不适用

### 分论点 5：优化方向三——游标分页（上一页最大 ID）

- 取下一页时携带上一页最大 ID：
  - `WHERE id > last_id ORDER BY id LIMIT size`
- 优点：稳定避免 offset 扫描
- 代价/限制：更适合顺序翻页，不适合任意跳页；同样要求 ID 递增有序

### 分论点 6：更高成本的方案与业务侧优化

- 分库分表/按范围分表：缩小单表范围，让“深翻页”落到更小的数据集
- 引入 ES 等检索系统：数据量大且检索复杂时的常见方案（成本更高）
- 业务限制深翻页：例如电商/内容平台限制翻到极深页码，降低技术成本

## 关键数据、案例与结论（逐条可复述）

- 示例：`LIMIT 50000, 100` 会取到 `50100` 再丢弃前 `50000`
- 回表放大：若走二级索引且非覆盖查询，可能对 `50100` 条都进行回表，再筛出 100 条
- 结论：深分页优化的核心是“减少被扫描与回表的数据量”，优先从 SQL 与索引设计入手，再考虑架构与业务策略
