# 泛滥的单元测试，如何规避“失焦”问题（测试方法论）

## 1. 核心痛点与认知纠偏

### 1.1 行业通病：覆盖率崇拜正在污染测试目标

视频指出的核心问题很尖锐：很多团队把“测试覆盖率”当成质量本身，甚至要求 98% 这种近乎仪式化的覆盖率指标。结果不是质量变高，而是测试开始失焦。

典型后果：

- 为了覆盖率去测 getter、setter、equals、hashCode 这类低价值代码。
- 为了让覆盖率数字好看，写大量没有业务断言的空转测试。
- 测试用例数量暴涨，但真正关键的业务分支、异常路径、边界场景反而覆盖不足。
- 每次重构都要同步改一堆测试，测试代码变成开发负担，而不是质量护栏。
- 团队开始把“测试没报错”误认为“业务正确”。

这是一种很隐蔽的工程内耗：看起来流程很规范，实际上是在用大量测试代码制造虚假的安全感。

### 1.2 正确认知：测试不是为了覆盖代码，而是为了验证业务行为

作者给出的破局思路很明确：单元测试的主线不是“执行过哪些代码”，而是“在稳定输入下，业务输出是否符合预期”。

比如转账场景，正确的测试应该验证：

- A 账户是否减少 300 元。
- B 账户是否增加 300 元。
- 余额不足、账户冻结、金额非法等业务分支是否按规则处理。

错误的测试却只验证：

- `update()` 方法是否被调用了一次。
- 某个 DAO 是否执行了某个过程。
- 某个内部方法是否被调用三次。

这类测试盯的是实现过程，不是业务结果。一旦代码重构，业务没变，测试却炸了；这说明测试绑定错了对象。

### 1.3 破局原则：用风险和价值决定测试投入

测试不是越多越好，测试是成本。真正专业的测试观不是“所有代码都测”，而是：

> 把测试火力集中到高风险、高复杂度、高业务价值、频繁变化的地方。

覆盖率可以作为体检指标，但不能作为质量目标。质量目标应该是：关键业务行为被验证，关键边界被锁死，关键回归能快速暴露。

## 2. 判断边界与决策模型（核心）

### 2.1 必须写单元测试的场景

以下场景必须写，而且要认真写：

| 场景 | 判断标准 | 测试重点 |
|---|---|---|
| 核心业务规则 | 钱、账、库存、权限、计费、利率、风控、订单状态流转 | 正常路径、异常路径、边界值、状态变化 |
| 分支复杂 | if/else 多、规则组合多、策略分派多 | 每个业务分支的结果是否正确 |
| 边界敏感 | 日期、金额、精度、区间、临界阈值 | 临界点前后是否符合业务规则 |
| 高频变更 | 经常被改、经常被需求影响 | 防止重构和需求迭代引入回归 |
| 缺陷高发 | 历史上出过事故或线上 bug | 把 bug 固化成回归测试 |
| 纯业务算法 | 不依赖数据库、不依赖外部服务，可独立计算 | 输入输出是否稳定正确 |

典型例子：

- 定期存款提前支取利息计算。
- 转账余额变化。
- 优惠券叠加规则。
- 订单状态机。
- 权限判定。
- 金额、税费、汇率、手续费计算。

这类代码一旦错，后果不是“页面不好看”，而是直接损失钱、数据、信誉或合规安全。

### 2.2 少写或不写单元测试的场景

以下场景不应该成为单元测试主战场：

| 场景 | 推荐策略 | 原因 |
|---|---|---|
| getter/setter、纯 DTO、简单 Entity | 不写或极少写 | 无业务判断，测试价值低 |
| equals/hashCode 等框架生成代码 | 通常不写 | 除非有手写复杂逻辑 |
| 日志输出 | 不写或少写 | 日志不是核心业务行为 |
| 页面样式渲染 | 不放在单元测试里 | 应交给 UI/E2E 或视觉检查 |
| 简单参数透传 | 少写 | 测试成本大于收益 |
| 序列化/反序列化辅助代码 | 只测关键协议字段 | 不要为了覆盖率全铺 |
| 调用次数验证 | 谨慎使用 | 容易把测试绑死在实现细节上 |

注意：不是说这些永远不能测，而是它们不应该挤占核心业务测试的预算。

### 2.3 测试投入决策矩阵

可以用四个维度判断要不要写、写多少、写哪一层：

| 维度 | 高分表现 | 测试策略 |
|---|---|---|
| 业务重要性 | 金额、账户、库存、权限、计费、合规 | 必须写，优先单元测试 + 集成测试 |
| 逻辑复杂度 | 分支多、组合多、算法多、状态多 | 必须写，覆盖主路径和关键边界 |
| 变更频率 | 经常被需求修改、经常重构 | 必须写，作为回归保护 |
| 外部依赖程度 | DB、MQ、HTTP、缓存、第三方服务多 | 单元测试 mock 依赖，集成测试验证协作 |

决策口径：

- 高业务价值 + 高复杂度：重点单元测试，边界和异常要打满。
- 高业务价值 + 多外部依赖：单元测试验证规则，集成测试验证协作。
- 低业务价值 + 低复杂度：不追覆盖率，别浪费时间。
- 遗留系统 + 高耦合：不要硬拆大量代码追单测，优先关键模块单测 + 集成测试 + 端到端测试。

### 2.4 测试分层边界

作者强调：不同测试解决不同问题，不能把所有东西都塞进单元测试。

| 测试类型 | 解决什么问题 | 边界 |
|---|---|---|
| 单元测试 Unit Test | 单个类、单个方法、单个业务路径是否正确 | 外部依赖用 mock，目标是快、稳、隔离 |
| 集成测试 Integration Test | Controller、Service、DAO、数据库、组件之间协作是否正确 | 可以启动容器、数据库、真实组件 |
| 端到端测试 E2E | 从用户视角验证完整业务链路是否跑通 | 模拟 UI 或接口全链路，不追求覆盖所有分支 |
| UAT/验收测试 | 用户是否接受业务结果 | 贴近真实业务流程和验收标准 |

核心边界：

- 单元测试不要干集成测试的活。
- 集成测试不要伪装成单元测试。
- 端到端测试不要追求单元级分支覆盖。

## 3. 实操策略与落地指南

### 3.1 单元测试采用 “Arrange-Act-Assert”

每个测试都要清楚分三段：

```java
@Test
void transfer_shouldMoveMoney_whenBalanceIsEnough() {
    // Arrange: 准备数据
    Account from = new Account("A", new BigDecimal("1000"));
    Account to = new Account("B", new BigDecimal("500"));

    // Act: 执行业务动作
    transferService.transfer(from, to, new BigDecimal("300"));

    // Assert: 验证业务结果
    assertEquals(new BigDecimal("700"), from.getBalance());
    assertEquals(new BigDecimal("800"), to.getBalance());
}
```

重点在最后一步：必须有断言。没有断言的测试，只是把代码跑了一遍，不是测试。

### 3.2 命名围绕业务场景，而不是内部实现

推荐命名表达三件事：被测行为、业务场景、预期结果。

```java
transfer_shouldDeductFromSourceAndAddToTarget_whenBalanceIsEnough()
calculateInterest_shouldUseLowerRate_whenWithdrawBeforeThirtyDays()
createOrder_shouldReject_whenInventoryNotEnough()
```

不要写成：

```java
testUpdateCalledOnce()
testCalculateMethodCalledThreeTimes()
testSetter()
```

前者描述业务行为，后者描述实现细节。实现细节会变，业务契约才是测试应该守住的东西。

### 3.3 单元测试遵循 FIRST 原则

| 原则 | 含义 | 落地要求 |
|---|---|---|
| Fast | 快速 | 单个单元测试尽量控制在毫秒级，视频中建议约 10ms 以内 |
| Independent | 独立 | 测试之间不能互相依赖，不能要求先跑 B 再跑 A |
| Repeatable | 可重复 | 不依赖时间、网络、数据库脏数据、执行顺序 |
| Self-validating | 自验证 | 必须用断言表达成功/失败，不靠肉眼看日志 |
| Timely | 及时 | 代码开发或重构时同步补测试，而不是上线前补覆盖率 |

### 3.4 外部依赖用 mock，业务协作用集成测试

单元测试中，数据库、HTTP、MQ、缓存、第三方服务都应该 mock 掉。单元测试只验证本类业务逻辑在稳定输入下的输出。

但 mock 不能替代真实协作。真实 SQL、事务、序列化、容器启动、配置装配，应放到集成测试里验证。

### 3.5 70% 以上测试火力聚焦主线业务

视频里给了一个非常实用的比例感：**70% 以上的单元测试应该围绕主线业务逻辑、业务分支和业务异常来做。**

剩下的测试资源，再根据情况覆盖参数校验、边界值、辅助模块。

这句话的价值在于，它把“测试投入”从覆盖率指标拉回了业务风险指标。

### 3.6 遗留系统不要硬追纯单测

遗留系统通常有高耦合、逻辑散落、框架老旧、没人完全掌握的问题。如果为了写单元测试，强行重构几十万行代码，风险可能比收益更大。

更务实的策略：

- 关键业务模块补少量单元测试，例如计息、计费、权限。
- 中等规模集成测试覆盖主要协作链路。
- 适当端到端测试覆盖关键用户流程。
- 对遗留代码先加保护网，再小步重构。

这是风险管理，不是技术洁癖。

## 4. 反模式（Anti-Patterns）警示

### 4.1 覆盖率 KPI 化

把覆盖率当硬 KPI，很容易诱导团队写垃圾测试。覆盖率数字越极端，测试越可能变形。

危险信号：

- 大量测试只为“跑过代码”。
- 覆盖率很高，但线上核心业务仍频繁出 bug。
- 测试报告很好看，开发没人愿意维护测试。

### 4.2 测试 getter/setter 来刷覆盖率

这是典型的为了测试而测试。它占用时间、污染测试集、制造维护成本，却几乎不提供业务安全感。

### 4.3 验证调用过程，不验证业务结果

例如转账后只验证 `accountRepository.update()` 被调用了一次，却不验证两个账户余额是否正确。

这类测试最大的问题是：实现换了就失败，业务错了却可能通过。

### 4.4 没有断言的测试

只要没抛异常就算通过，这是伪测试。很多业务场景本来就应该抛异常，没有断言就无法表达预期。

### 4.5 单元测试塞进数据库、容器和真实外部服务

这会让单元测试变慢、变脆、变得不可重复。需要真实组件协作时，请诚实地把它叫集成测试。

### 4.6 测试代码比业务代码还难维护

如果每次重构业务逻辑，都要成片修改测试，说明测试可能过度绑定实现细节。好的测试应该保护业务契约，而不是锁死代码结构。

### 4.7 用端到端测试替代所有测试

端到端测试很有价值，但它慢、贵、定位问题差。它适合覆盖关键流程，不适合覆盖所有业务分支。

## 5. 核心金句或思维模型

### 金句一

> 覆盖率和质量相关，但覆盖率不是质量本身。

### 金句二

> 单元测试的主线是验证业务行为，不是证明某段代码被执行过。

### 思维模型：测试焦点三问

写任何测试前，先问自己三句话：

1. 这个测试保护的是业务结果，还是实现细节？
2. 这个场景失败后，真实业务损失有多大？
3. 这个测试未来会降低重构风险，还是增加重构负担？

如果三个问题都答不上来，这个测试大概率只是覆盖率噪音。

## 6. 日常开发决策清单

- [ ] 这个测试是否验证了明确的业务结果？
- [ ] 是否有断言，而不是只跑代码？
- [ ] 是否避免测试 getter/setter、日志、简单透传这类低价值代码？
- [ ] 是否把数据库、HTTP、MQ 等外部依赖从单元测试中 mock 掉？
- [ ] 是否把真实组件协作放到集成测试，而不是伪装成单元测试？
- [ ] 是否优先覆盖核心业务主路径、异常路径、边界值？
- [ ] 是否避免验证内部方法调用次数这类实现细节？
- [ ] 单个测试是否只验证一件事？
- [ ] 测试之间是否独立、可重复、可自验证？
- [ ] 遗留系统是否采用“关键单测 + 集成测试 + E2E”的务实策略，而不是硬追覆盖率？
