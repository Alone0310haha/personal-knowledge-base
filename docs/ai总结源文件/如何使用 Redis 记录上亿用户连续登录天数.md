# 视频文案综合分析报告（信息提炼）

[如何使用 Redis 记录上亿用户连续登录天数](https://www.bilibili.com/video/BV1qb6EB2EGi?p=13)

## 讨论视角（这期在解决什么）

视频围绕“面试高频场景题：如何使用 Redis 记录上亿用户连续登录天数”展开讨论。核心挑战在于海量用户数据的存储压力（MySQL 存储成本过高）以及高效的连续登录统计需求。

## 核心论点与分论点

### 核心论点
针对“二值统计”（如登录/未登录、签到/未签到）的海量数据场景，**Redis 的 Bitmap（位图）** 是最优选方案。它利用极小的内存占用实现高性能的二进制位操作，支持多达 42 亿位的存储。

### 分论点 1：为什么不选 MySQL？
- **存储成本**：单表记录每个用户每天的登录状态会导致数据量爆炸式增长。
- **性能瓶颈**：海量数据的查询与统计效率难以满足高并发需求。

### 分论点 2：Bitmap 的核心优势
- **原理**：由二进制位（bit）组成的数组，每位仅占 0 或 1。
- **容量**：Redis 的 String 类型实现，最大 512MB，理论可存储约 42.9 亿个 bit 位。
- **适用场景**：登录、签到、点赞等海量二值统计。

### 分论点 3：两种存储维度及选型建议
- **方案 A：以日期为 Key（天维度）**
  - **实现**：每个日期一个 Bitmap，用户 ID 作为偏移量（Offset）。
  - **统计**：从当天开始向前回溯，遇到 1 计数加 1，遇到 0 停止。
  - **优点**：统计某天所有用户的登录情况效率极高。
  - **缺点**：统计单个用户长期登录需跨多个 Key 循环，略显不便。
  - **建议**：适用于统计周期在 30 天内、用户量极大的场景。设置 30 天过期时间可自动清理旧数据。
- **方案 B：以用户为 Key（用户维度）**
  - **实现**：每个用户一个 Bitmap，日期（如当月第几天）作为偏移量。
  - **优点**：方便查询单个用户的详细签到历史。
  - **缺点**：Key 的数量会随用户量增加而激增。

## 关键结论

- **存储上限**：单笔 Bitmap 最高支持 42 亿+ 数据位。
- **性能平衡**：建议统计周期控制在 **30 天** 左右，配合 Redis 过期机制优化内存。
- **选型逻辑**：
  - 看重全平台用户活跃度统计 -> **按天存储**。
  - 看重个人签到历史/连登奖励 -> **按用户存储**。
