# RAG（检索增强生成）基本原理（面试复习资料）

## 概念定位

- **概念名称与领域**：RAG（Retrieval Augmented Generation，检索增强生成），属于大模型应用工程领域
- **非技术语言**：就像考试前临时查资料——不是让 AI 变得更聪明，而是在它回答前先帮它找到相关页面，让它对着资料作答
- **技术语言（面试可直接说）**：RAG 是一种在大语言模型（LLM）生成回答之前，先通过检索系统从外部知识库中取回相关上下文片段，再将其拼入 Prompt 共同输入 LLM 的技术框架

---

## 背景与动机

### LLM 本身的三大缺陷

| 问题 | 表现 |
|------|------|
| 幻觉 | 训练数据中没有答案时，LLM 不会说"我不知道"，而是预测下一个 token，编造合理答案 |
| 知识过时 | 训练数据有截止日期，之后的事件完全不知道 |
| 无法访问私有数据 | 企业内部文档、业务数据等从未进入训练集 |

### "直接把整个知识库塞进 Prompt"为什么不行？

1. **上下文窗口限制**：内容太多时 LLM 注意力分散，精度下降
2. **计算成本高**：每次请求都处理海量文本，费用不可接受
3. **响应速度慢**：用户无法接受等待分钟级别的回答

### RAG 的针对性解法

> "开卷考试不是从第一页翻到最后一页，而是先定位到相关页，再作答。"

RAG 在 LLM 回答前插入一个「先检索相关片段，再基于片段生成」的步骤，既解决了知识缺口，又控制了输入规模。

## 核心机制拆解

RAG 分两个阶段：**数据准备** 和 **用户提问**。

### 阶段一：数据准备（离线）

#### Step 1：文档切分（Chunking）

- **做了什么**：将原始文档切成若干小块，每块尽量表达一个完整语义
- **切分方式**：按章节 / 段落 / 固定字数（如每 500 字一块）
- **为什么必须切**：整篇文档向量化后语义混杂，检索不准；切分让每块聚焦单一主题
- **如果省略**：一个超大 chunk 含有 ABCDEFG 多个主题，检索到它但实际上大多数内容都不相关

#### Step 2：向量化（Embedding）

- **做了什么**：用 Embedding 模型把每个文本块转成高维向量（实际几百到上千维）
- **为什么必须有**：计算机不理解文字，只理解数字；向量空间中语义相近的内容距离更近
- **直觉理解**：想象一个坐标轴，"张三是男的"和"张三30岁"的坐标会比"销售额1万元"更接近
- **如果省略**：无法度量"问题和片段语义有多近"，检索退化为关键词匹配

#### Step 3：存入 Vector Database

- **做了什么**：将（文本内容, 对应向量）成对存入向量数据库
- **为什么必须有**：向量数据库支持高效的相似度查询，是后续检索的基础设施
- **如果省略**：检索时没有索引，每次都要暴力全量比对，速度无法接受

---

### 阶段二：用户提问（在线）

#### Step 4：Query Rewrite（问题重写）

- **做了什么**：将用户口语化的短问题扩写/改写，使其更贴近知识库的语言风格
- **例子**：用户问"张三是谁" → 改写为"详细介绍张三的性别、年龄、家庭住址等情况"
- **为什么要做**：用户提问往往短、碎、口语化，直接拿去向量检索命中率低
- **如果省略**：召回率下降，漏掉相关片段

#### Step 5：向量检索（召回，Retrieval）

- **做了什么**：将改写后的问题也转成向量，在数据库中按相似度找出 Top-K 个最近的片段
- **相似度算法**：余弦相似度（计算向量夹角的 cos 值）或欧式距离（直线距离）
- **这一步的局限**：只是粗筛，判断"片段和问题像不像"，不能保证片段真的回答了问题
- **如果省略**：直接把全部知识库喂给 LLM，回到"不可行"的原始方案

#### Step 6：Rerank（精排/重排序）

- **做了什么**：用更强的模型把 Top-K 候选片段与问题逐一深度比对，选出最能回答问题的 Top-M 片段
- **和上一步的区别**：检索看的是"像不像"（向量距离），Rerank 看的是"有没有回答"（语义理解）
- **如果省略**：召回的片段里可能有和问题"长得像"但实际答非所问的干扰项

#### Step 7：构建 Prompt，交给 LLM 生成答案

- **做了什么**：将用户问题 + 检索到的参考片段 + 系统提示词（如"严格按资料回答，不知道就说不知道"）拼成 Prompt，送入 LLM
- **为什么要加提示词约束**：防止 LLM 忽略资料自由发挥，产生幻觉
- **用户感知**：以上所有步骤对用户完全透明，用户只看到问题和最终答案

## 与相关概念的对比

### RAG vs Fine-tuning（微调）

| 维度 | RAG | Fine-tuning |
|------|-----|------------|
| 更新知识的方式 | 更新向量数据库（实时可改） | 重新训练模型（代价高） |
| 适合场景 | 私有数据、知识频繁变更 | 让模型掌握特定风格/能力 |
| 可解释性 | 高（能看到引用片段） | 低（知识融合进权重） |
| 成本 | 较低 | 较高 |
| 核心区别 | 在推理时检索外部知识 | 在训练时把知识固化进模型 |

### RAG vs GraphRAG

| 维度 | RAG | GraphRAG |
|------|-----|----------|
| 知识组织方式 | 向量化的文本块（扁平） | 知识图谱（实体-关系-属性，结构化） |
| 擅长问题类型 | 局部精确查找 | 全局/跨片段/关系推理问题 |
| 典型问题 | "张三是谁" | "文章中所有人物的关系是什么" |
| 核心区别 | 靠向量相似度定位片段 | 靠图结构保留跨片段关系 |

### 关键词检索 vs 向量相似度检索（混合检索）

| 维度 | 关键词检索 | 向量相似度检索 |
|------|-----------|--------------|
| 比喻 | 查字典（字对字，丁是丁卯是卯） | 问朋友（懂意思但可能有出入） |
| 优势 | 精确匹配，不会"近义词误伤" | 理解语义，能处理同义词/改写 |
| 盲区 | 无法处理同义词和语义相近 | 可能被语义相似但无关的内容干扰 |
| 实践建议 | 混合使用，盲区互补 | 混合使用，盲区互补 |

## 典型应用场景

| 场景 | 为什么用 RAG |
|------|-------------|
| **企业知识库问答** | 内部文档是私有数据，LLM 未见过；文档频繁更新，不适合每次微调 |
| **客服机器人** | 需要基于最新产品手册/政策回答，不能让 LLM 自由发挥导致承诺失误 |
| **文档助手** | 用户上传合同/报告后实时问答，文档是一次性的，无法提前训练 |
| **法律/医疗查询** | 答案需要可溯源（能指向具体条文/文献），RAG 天然带引用出处 |
| **代码库问答** | 大型私有代码库，LLM 不知道内部实现；向量检索能快速定位相关文件 |

## 局限性与常见误区

### 局限性

1. **垃圾进，垃圾出**：数据清洗（PDF 解析、OCR、表格处理）是最容易被忽视的卡点，文档解析质量决定上限
2. **Chunk 粒度两难**：块太大 → 主题混杂，检索不准；块太小 → 语义断裂（"张三"和"是一个好人"被切开，模型不知道谁是好人）
3. **只能处理已有知识**：RAG 的知识边界由向量库决定，库里没有的问题依然无法回答
4. **向量检索是粗筛**：余弦相似度高≠真正能回答问题，必须配合 Rerank
5. **模型基座是底线**：前面流程再好，LLM 本身如果幻觉严重、不遵循指令，输出依然有问题

### 面试常见误区

- **误区1**："RAG 让 AI 更聪明" → 错。RAG 不改变模型本身，它只是给模型加了"查资料"的能力
- **误区2**："向量检索精度足够，不需要 Rerank" → 向量相似度只是语义接近，不等于能回答问题；Rerank 是不可缺少的精排层
- **误区3**："切块越细越好" → 太细会导致语义断裂，颗粒度要根据文档类型具体调整
- **误区4**："RAG 和微调是竞争关系" → 实践中常常结合使用：微调让模型掌握领域语言风格，RAG 提供最新/私有知识

## 面试高频问题与回答框架

**Q1：什么是 RAG？为什么需要它？**
- 考察意图：能否说清楚动机，而不只是背定义
- 回答框架：
  1. LLM 的三个固有问题（幻觉、知识过时、私有数据）
  2. 直接塞知识库不可行（上下文限制、成本、速度）
  3. RAG = 回答前先检索相关片段，再交给 LLM

---

**Q2：RAG 的核心流程是什么？**
- 考察意图：能否分阶段、有条理地描述
- 回答框架：两阶段框架
  1. 数据准备：切分（Chunking）→ 向量化（Embedding）→ 存入 Vector DB
  2. 用户提问：Query Rewrite → 向量检索（Top-K 粗筛）→ Rerank（精排）→ 构建 Prompt → LLM 生成

---

**Q3：Chunking 怎么做？切大还是切小？**
- 考察意图：是否理解粒度权衡，而非背公式
- 回答框架：
  1. 切太大：一个块多主题，检索命中了块但内容不精确
  2. 切太小：语义断裂，模型失去上下文
  3. 实践：按文档结构（章节/段落）切，辅以重叠（overlap）避免边界截断；具体策略依文档类型而定

---

**Q4：向量检索和 Rerank 有什么区别，为什么需要两步？**
- 考察意图：是否理解粗筛 vs 精排的设计逻辑
- 回答框架：
  - 向量检索（快，粗）：用余弦相似度/欧式距离，找语义接近的 Top-K，判断"像不像"
  - Rerank（慢，精）：用更强模型逐一比对，判断"有没有回答问题"
  - 为什么分两步：如果直接 Rerank 全库，计算量爆炸；先粗筛缩小候选集，再精排，效率和精度兼顾

---

**Q5：RAG 和 Fine-tuning 怎么选？**
- 考察意图：是否理解两种方案的本质差异
- 回答框架：
  - 知识频繁更新 / 私有数据 / 需要可溯源引用 → 优先 RAG
  - 需要模型掌握特定风格、专业术语、推理方式 → 考虑 Fine-tuning
  - 实践中常结合：Fine-tuning 固化风格，RAG 提供动态知识

---

**Q6：RAG 的常见优化点有哪些？**
- 考察意图：是否有实际工程经验或深度理解
- 回答框架：
  1. 数据清洗：PDF/表格/图片解析质量是上限
  2. Chunking 策略：粒度、重叠（overlap）、按结构切
  3. Query Rewrite：将口语化问题扩写，提升召回
  4. 混合检索：关键词检索 + 向量检索互补
  5. Rerank：精排提升准确率
  6. Prompt 设计：约束 LLM 严格基于资料回答，减少幻觉
  7. 模型基座选型：幻觉少、遵循指令、优先开源（可本地部署，数据安全）

---

**Q7：RAG 有什么局限性？什么情况下不适用？**
- 考察意图：能否客观评估方案边界
- 回答框架：
  - 知识库没有的问题无法回答
  - 文档质量差时，后续步骤无法弥补
  - 需要跨多个片段做复杂推理时，标准 RAG 效果下降（可考虑 GraphRAG）
  - 追求极低延迟的场景，多步流程带来额外 RT

---

**Q8：混合检索是什么？为什么比单一向量检索更好？**
- 考察意图：是否了解实际落地中的工程优化
- 回答框架：
  - 关键词检索：精确匹配，适合专有名词、型号、人名等
  - 向量检索：语义匹配，适合意思相近但用词不同的情况
  - 两者盲区互补：关键词处理不了同义词；向量可能被语义相似但无关的内容干扰
  - 混合 = 先查字典 + 再问朋友，综合判断

## 用自己的话解释（费曼测试）

> 面试中我会这样说：
>
> "RAG 要解决的问题是：大模型训练完就'冻住'了，它不知道你公司内部的文档，也不知道最新发生的事。但如果直接把整个知识库塞给它，上下文太长，又慢又贵。所以 RAG 的思路是——在回答之前先去'查资料'：把知识库切成小块存进向量数据库，用户提问时先检索最相关的几段，再把这几段塞进 Prompt 一起给模型。模型就对着这几段资料作答，就像开卷考试。整个过程对用户完全透明，用户只看到问题和答案。所以它解决的核心问题是：如何让 LLM 能够基于它原本不知道的、可能随时更新的私有知识，给出有据可查的回答。"
