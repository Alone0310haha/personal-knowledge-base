# 前模糊索引失效（LIKE '%xxx'）

[前模糊索引失效（LIKE '%xxx'）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=30)

## 讨论视角（这期在解决什么）

- 以“索引为什么失效、怎么优化”的面试题视角，解释 MySQL `LIKE '%xxx'`（前模糊/左模糊）导致索引失效的原因与常见替代方案
- 核心围绕：B+Tree 的有序性决定了“必须从前缀开始才能走索引”，否则会退化成全表扫描

## 核心论点与分论点

### 核心论点

前模糊（例如 `LIKE '%abc'`）因为首字符不确定，无法按 B+Tree 的顺序从前缀定位索引范围，通常只能全表扫描；要优化只能改变检索方式（让查询能够从前缀命中索引）或缩小扫描代价。

### 分论点 1：为什么前模糊会导致索引失效

- 索引底层是 B+Tree，叶子节点按 key 有序存储
- 如果第一个字符不确定：
  - 无法确定从哪一段范围开始查
  - 不能利用“有序范围查找”的能力
  - 结果常常退化为全表扫描，性能显著下降

### 分论点 2：解决方案 1——反向存储 + 后向模糊

- 把待检索字段额外存一份“反转字符串”
  - 例：把 `徐叔` 反转存成 `叔徐`
- 查询时把“前模糊”转换成对反转字段的“后模糊”（可走索引）

### 分论点 3：解决方案 2——限制扫描范围

- 即便不得不全表扫描，也尽量加过滤条件缩小范围
  - 例：限定“当天创建的数据 / 当月创建的数据”
- 本质：减少参与扫描的数据量来换性能

### 分论点 4：解决方案 3——索引覆盖，避免回表

- 即使发生全表扫描，也尽量让查询只返回索引列
- 通过索引覆盖减少回表成本，从而获得一定性能收益

### 分论点 5：终极方案——引入搜索引擎

- 如果上述方案都不适用，使用专门的搜索引擎/检索框架（视频举例：Elasticsearch）来支持模糊检索需求

## 关键数据、案例与结论（逐条可复述）

- 失效场景：`LIKE '%xxx'`（左模糊/前模糊）通常无法走 B+Tree 索引
- 原因：首字符不确定，无法按有序索引定位范围
- 结论：优化方向要么“把查询变成能用前缀命中的形式”，要么“降低全表扫描成本”，再不行就用专用检索系统
