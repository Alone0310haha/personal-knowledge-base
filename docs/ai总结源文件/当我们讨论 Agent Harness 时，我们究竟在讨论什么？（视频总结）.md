# 当我们讨论 Agent Harness 时，我们究竟在讨论什么？（视频总结）

## 1. 听众画像：这期内容写给谁

- AI/Agent 一线开发者与工程负责人：关心“怎么把模型变成可交付的生产力”，以及多 Agent、记忆、工具编排、验证闭环这些工程问题。
- AI 产品/创业者：关注通用/垂直 Agent 的边界、Agent Infra 的机会点，以及开源生态里“范式迁移/产品打磨/分发”如何决定胜负。
- 对 Agent 热潮好奇的泛用户：从 OpenClaw（文本里亦称 Open Crow/“龙虾”）到 Hermes/AMIES 的迁移，理解为什么“能用、能记住、能持续变好”的体验会改变人机关系与信任阈值。

## 2. 讨论角度：它在解决什么问题

视频从“模型能力已接近可用 → 工程系统要把能力释放出来”的角度，讨论 Agent Harness 与相关范式：

- 解释 Harness 的本质：不是“更聪明的模型”，而是让模型在真实世界里稳定行动的脚手架（工具编排、状态管理、错误处理、反馈回路、约束与自由的平衡）。
- 回答为什么要 Multi-Agent：一方面是信息交换效率（模型之间的高密度沟通远高于人+模型），另一方面是克服单 Agent 长上下文带来的退化与跑偏。
- 澄清 Self-Evolution/记忆/Skill 的价值：让 Agent 对“正确路径”形成可复现的技能与记忆，从而提高一致性与可迁移性（同一 harness + skills 下可更自由切换模型）。
- 扩展到产业层：通用 vs 垂直 Agent 的“最后一公里交付”、Agent Infra（身份、支付、沙箱等）与创业机会，以及开源产品如何避免代码膨胀并保持可用性。

## 3. 核心论点与分论点

### 核心论点

Agent 的竞争不只在模型，而在 Harness：把模型当“引擎/大脑”，Harness 当“机甲/手脚/缰绳”，通过工具、约束、记忆与验证闭环，把智能转化为可交付、可复制、可持续进化的能力；Multi-Agent 与 Self-Evolution 是突破人类瓶颈与长任务稳定性的关键路径。

### 分论点 A：Harness 是“约束 + 能力 + 反馈”的系统，而不是提示词

- Harness 要同时做到：给 Agent 足够的工具/环境/自由度去完成任务，并设置边界与机制让它不会轻易跑偏（类比“同事”：要约定能做什么、不能做什么，并给到电脑/邮箱/电话等工具）。
- Harness 的关键在反馈链路：让 Agent 能拿到真实世界反馈（测试、A/B、部署结果、外部系统响应）并把经验沉淀为技能/规则（技能、COI、hooks 等），从而减少“人类反复确认”的成本。

### 分论点 B：Multi-Agent 的价值在于“效率 + 稳定性”，不是角色扮演

- 效率：人给 Agent 的信息量往往很少，而模型输出很长；模型之间可以更高密度、秒回交换信息，整体推进更快。
- 稳定性：长上下文会导致退化与跑偏；视频给出的经验口径是“上下文用量超过约 50% 后智能会指数级下降”，因此用多 Agent 做 cross check、分治与重置上下文，能降低“钻牛角尖”概率。

### 分论点 C：Self-Evolution 的本质是“把正确路径压缩成可复现的技能”

- 典型痛点：同一件事第一次做对了，第二次又做错，导致无法信任 Agent。
- 解决方式：通过记忆/技能把正确 workflow 固化，一旦确认“这是正确的”，后续可稳定复现；从而增强可预测性与信任。
- 迁移收益：当你有同一个 harness + skills，即便切换不同模型，也能更接近一致输出（把“稳定性”从模型能力部分外置到系统与记忆层）。

### 分论点 D：模型与 Agent 互相促进，但通用应用会被“内化”

- 应用在真实世界 unlock 出的用法，会反哺模型训练；很多 workflow/skill 最终可能被模型内化。
- 这使得纯通用应用存在“被内化掉”的风险；但对开发者而言，最有价值的是持续触碰真实世界边界、把反馈闭环跑起来。

### 分论点 E：Agent Infra 的“底层基建”门槛高，上层工具/环境有机会

- 关键基建被认为主要是身份认证与支付；这类基础设施往往需要巨大信任与资源，不一定是创业公司最容易打穿的层。
- 更可行的机会在上层：面向 Agent 的工具与环境（例如让 Agent 以更低摩擦完成挂号、打车、缴费等的 COI/接口与流程产品），以及围绕“AI-ready 土壤”出现的新交互/新范式。

## 4. 关键数据、案例与结论（视频口径整理）

- Hermes/Nous 的增长数据（OpenRouter 口径）：从约 2B tokens/day 增长到约 20B tokens/day（约一个月内），并提到某天接近 300B tokens/day 的量级。
- 模型使用数据（口径）：MiniMax M2.7 在 OpenRouter 上达到约 25B+ tokens/day（对 Hermes 用户使用很突出）。
- 训练/研发自动化（口径）：MiniMax M2.7 的 RL pipeline 中约 70%–80% 的工作由模型/Agent 自己完成；人类更多做方向判断与 taste。
- 长任务跑偏与纠偏：提到长任务研究结论——一旦某处偏离正确路线，后续容易“越跑越偏”；实践中用 cross check（多 Agent）与“换脑子”式重启能显著改善。
- Agent Infra 的关键要素：identity（身份认证）与支付被反复点名为最底层、最关键但最难的基础设施。
- 社会实验案例：YOYO agent 被设定目标“打败 Claude Code”，主人承诺不再维护代码也不再给 token，Agent 通过写日记/发推/开打赏自我生存（文本里提到已运行到第 43 天）。
- 身份/实名的讨论：对 Claude Code 强制实名，给出两种解释视角——一是“AI 请求归因与安全”，二是“以 safety 之名做封闭与垄断”的担忧。

## 5. 实践型内容：具体怎么操作（可直接照做的流程）

### 5.1 从 0 到 1 搭一个能跑的 Harness（第一步）

1. 定义边界：明确 Agent 能做什么、不能做什么（尤其是外部系统/权限/资金相关操作）。
2. 配齐工具：为 Agent 提供可执行的工具与环境入口（本地/云、浏览器/脚本、仓库/CI、第三方系统接口等）。
3. 设定交付标准：把“完成”的定义写清楚（验收条件、失败条件、回滚条件）。

### 5.2 把反馈回路做成系统（决定稳定性的第二步）

1. 让 Agent 获取真实反馈：跑测试、验证接口、读日志/指标、A/B 或灰度结果等。
2. 失败即纠偏：要求每轮行动后做一次复盘式再思考（文中提到的 Int Live Thinking：行动后再思考并调整路径，而非一条 plan 一走到底）。
3. 固化正确路径：把被验证为“正确”的做法沉淀为可复用资产（记忆/技能/COI/hooks），避免下次重复犯错。

### 5.3 何时引入 Multi-Agent（第三步）

1. 分治：把探索、实现、验证拆开，让不同 Agent 并行推进。
2. Cross check：关键决策/方案/结论至少两路产出交叉验证，降低单 Agent 跑偏与长上下文退化。
3. 人类聚焦 taste：把人的时间用在目标定义、方向判断与审美把关，而不是每一步“批准继续”。

## 6. 现实应用启示（把它落到日常工作里）

- 把 Harness 当作产品与工程系统来做：可用性、可部署、少折腾、避免“代码膨胀”，往往比堆功能更能带来规模化采用。
- 从“AI 适应人”转向“人重构工作方式”：当能力与信任阈值变化后，人会围绕 AI 重构流程，像电力/蒸汽机一样改变生产方式。
- 关注身份归因与权限边界：AI 能发起请求很容易，但“自证是谁/归属谁/责任怎么划分”会成为关键问题（实名、身份、授权、审计）。
- 通用 vs 垂直的关键在最后一公里交付：很多垂直机会不是模型不会，而是缺“那一步”让流程跑通；同时高风险行业（如法律）对责任与合规要求更高。
- 不要把焦虑当结论：视频用 ATM 的类比强调“提效会带来更多服务与更多岗位形态”，更现实的策略是尽早拥抱并把 AI 用起来。
