[工程技术：在智能体优先的世界中利用 Codex](https://openai.com/zh-Hans-CN/index/harness-engineering/)

## 文章讲了什么

OpenAI 团队复盘了一次“极限约束”的工程实验：在约 5 个月里交付一个真实产品（内部 beta + 外部 alpha，经历发布、部署、故障与修复），并坚持“人类不手写任何代码”，让 Codex 负责从应用逻辑、测试、CI、文档、可观测性到内部工具的全部代码产出。

在这个过程中，团队发现瓶颈并不在模型是否会写代码，而在于：工程团队是否为智能体构建了足够清晰、可导航、可强制执行、可反馈的运行环境与约束体系。作者把这种面向智能体的软件工程方法，概括为 Harness Engineering。

## 核心论点

当编码由智能体完成时，工程速度与质量的主要决定因素，会从“人写代码的效率”转移为“系统是否让智能体可靠地做对事”：

- 给智能体的是“地图”，不是“1000 页说明书”：上下文是稀缺资源，过度堆叠指令会挤占任务与代码信息，导致误优化与规则腐烂。
- 把知识、计划与决策沉淀到仓库内、版本化、可索引的工件中：对智能体来说，运行时拿不到的内容等同于不存在。
- 用机械化约束替代微观管理：通过结构化架构、依赖边界、lint/结构测试，把不变量写进系统，让智能体在边界内自由发挥。
- 把反馈回路产品化：让智能体可直接读取并操作 UI、日志、指标、追踪，能复现问题、验证修复、产出可回放证据。
- 规模化后瓶颈会变成人类 QA 与注意力：吞吐高时“等待比纠错更贵”，要降低阻塞合并门槛，并用自动化审查与持续“垃圾回收”控制漂移与技术债。

## 关键做法（文章中反复出现的模式）

### 1) 以仓库为“记录系统”（System of Record）

- AGENTS.md 从百科全书变成目录页（约 100 行），指向结构化 docs/ 知识库
- 文档被编目、交叉链接、可验证（新鲜度、覆盖率、所有权），并由 CI/linter 强制执行
- “doc-gardening” 智能体定期扫描过期文档并自动发起修复 PR
- 计划是一等工件：轻量计划用于小改动，复杂工作用执行计划，带进度与决策日志并提交到仓库

### 2) 为智能体的可读性而设计代码与抽象

- 更偏好可组合、API 稳定、行为透明的依赖与抽象
- 在部分场景中，宁可实现小而可控的内部工具（可观测性集成、100% 测试覆盖、行为可预测），也不绕过上游不透明行为

### 3) 早期就强制架构边界，避免漂移

- 每个业务域固定分层，并严格限制依赖方向与允许的边
- 通过自定义 linter + 结构测试机械强制执行
- lint 错误信息直接带修复指引，便于智能体在上下文中自我修复

### 4) 让验证变成默认路径（可观测 + 可复现 + 可回放）

- 每个任务基于 git worktree 在隔离实例中运行；任务结束后环境、日志与指标可清理
- 把 Chrome DevTools 协议接入运行时，提供 DOM 快照、截图、导航等能力，使智能体能驱动 UI 并推理行为
- 日志/指标对智能体可查询（如 LogQL/PromQL），使“性能/旅程时延”这类目标可直接变成验收条件

### 5) 用“黄金原则 + 后台清理”持续治理

- 承认智能体会复制既有模式并导致漂移
- 把主观但重要的工程品味编码成机械规则（黄金原则），并定期后台扫描偏差、发起重构 PR，像垃圾回收一样持续偿还技术债

## 如何运用到实际开发（把它落到团队流程里）

### 1) 把“智能体工作所需的知识”迁移进仓库

- 把关键讨论与决策从聊天/文档孤岛迁移为仓库内工件：ADR、架构地图、边界规则、命名约定、验收标准
- 让入口文档变成“地图 + 路标”：短、稳定、强索引，避免指令大杂烩

### 2) 把质量变成系统不变量，而不是评审风格

- 用 lint/结构测试强制关键边界：分层、依赖方向、日志结构、文件大小、命名约定、可靠性要求
- 让 lint 报错可操作：错误信息里写清楚修复路径与示例

### 3) 让验证与反馈回路成为“智能体的默认工具链”

- 为关键用户旅程提供可执行的验收脚本（E2E/集成测试/性能检查）
- 将日志、指标、追踪、UI 驱动能力开放给智能体，让它能复现-修复-验证闭环

### 4) 重新设计 PR 与合并门槛，适配高吞吐

- 缩短 PR 生命周期，降低阻塞性门禁（在保证关键不变量的前提下）
- 将更多审核工作转为“智能体对智能体”的自动审查与回归验证
- 把人类注意力投入到需求边界、验收标准与系统能力补齐，而不是逐行改代码风格

## 可执行步骤（按文章的方法，从零搭一套 Harness）

### Step 1：建立“地图式入口”

- 写一份短 AGENTS.md，明确：目标、边界、不变量、以及知识库入口
- 建 docs/ 目录作为记录系统：架构地图、领域文档、运行手册、计划与决策日志

### Step 2：把不变量写成可执行约束

- 为架构分层与依赖边界写结构测试
- 为编码规范与可靠性要求写自定义 lint，并把修复指引写进报错

### Step 3：让智能体能在真实环境里验证

- 支持按 worktree 启动隔离实例
- 接入 UI 驱动（截图/DOM/导航）与可观测性查询（日志/指标/追踪）
- 把关键旅程和性能目标转成验收条件

### Step 4：把知识与质量维护自动化

- CI 校验知识库结构、交叉链接与新鲜度
- 周期性运行“doc-gardening / repo-gardening”任务：扫描漂移、补文档、发重构 PR

### Step 5：持续迭代系统，而不是让智能体“更努力”

- 智能体卡住时，优先问“缺什么能力/工具/信息/约束”，并把答案沉淀回仓库
- 把复盘结论编码进规则、工具或文档入口，让下一次运行自动受益
