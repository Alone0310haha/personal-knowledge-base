# 为什么大厂弃用 MyBatis 二级缓存

[为什么大厂弃用 MyBatis 二级缓存](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=52)

## 讨论视角（这期在解决什么）

- 以面试问题“二级缓存性能这么好，为什么生产不启用”切入，强调这不是配置题，而是分布式一致性与架构取舍题
- 用“单体时代可用的本地缓存机制”对比“分布式集群下的数据一致性风险”，说明二级缓存的边界条件
- 给出可落地的标准回答：关闭 MyBatis 二级缓存，缓存上移到中心化 Redis 集群

## 核心论点与分论点

### 核心论点

MyBatis 二级缓存本质是“每个应用节点各自持有的本地缓存（namespace 级）”。在单体/单节点时代它能减少重复查询，但在分布式集群下会造成节点间缓存无法同步失效，从而产生脏读与数据割裂风险；生产环境更可靠的做法是关闭二级缓存，把缓存统一交给中心化的 Redis 等专业缓存系统作为全局一致的缓存层。

### 分论点 1：一级缓存的定位与局限：会话级、短命、只能挡住本次会话内的重复查询

- 一级缓存绑定在 SqlSession（会话）上
- 优点：同一会话内重复查询可直接命中，减少数据库访问
- 局限：会话提交/关闭后缓存即失效，无法跨会话、跨线程共享

### 分论点 2：二级缓存的设计初衷：打破会话壁垒，扩大到 namespace 级共享

- 二级缓存把缓存范围扩大到 mapper namespace 维度
- 能在不同会话/不同线程间共享数据（以序列化等方式暂存）
- 在单体应用时代通常是“好方案”，但它依赖的前提是“同一份缓存能覆盖所有请求”

### 分论点 3：分布式集群下的致命问题：节点本地缓存无法联动失效 -> 脏数据

- 二级缓存是本地缓存：每个节点各自维护一份
- 典型事故链路：
  - 节点 A 更新数据库并清空自己的缓存
  - 节点 B 的本地缓存不受影响，仍保留旧数据
  - 用户打到 B 节点会读到脏数据
- 在支付/库存等强一致敏感场景，这类数据割裂属于高等级线上事故风险

### 分论点 4：标准工程解法：关闭二级缓存，缓存上移到中心化 Redis

- 配置上显式关闭 MyBatis 二级缓存
- 用 Redis 集群承接缓存能力：作为全局唯一的“缓存真理源头”
- 无论哪个节点发生数据变更，都统一去 Redis 做更新/失效通知，避免各节点“互不通气”

## 关键数据、案例与结论（逐条可复述）

- 关键结论：二级缓存的问题不在“性能”，在“分布式一致性”
- 案例：双节点部署时，A 更新并清空自身缓存，但 B 仍持有旧缓存 -> 用户从 B 读到脏数据
- 落地结论：生产更偏向“中心化缓存层（Redis）+ 明确的一致性策略”，而不是依赖 ORM/框架自带的节点本地二级缓存
