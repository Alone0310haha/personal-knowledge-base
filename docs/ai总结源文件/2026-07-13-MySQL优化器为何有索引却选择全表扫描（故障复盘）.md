# MySQL 优化器为何有索引却选择全表扫描（故障复盘）

## 1. 故障现场与影响（案发现场）

本次案例来自转转团队的一次 MySQL 查询优化复盘。业务表已经提前建立了索引，其中包括：

- `source_type, update_time, source_id` 相关联合索引
- `source_type, state` 相关联合索引

从常规经验看，查询条件中包含 `source_type`，且它处在联合索引左侧，至少应该成为候选索引；如果查询中还涉及 `source_id`，也可能命中另一个索引。

但线上实际表现是：某条 SQL 执行耗时 **10 秒以上**。执行 `EXPLAIN` 后发现结果与直觉相反：

```text
possible_keys: 有相关候选索引
key:           PRIMARY
Extra:         Using where
```

也就是说，优化器确实知道存在可用索引，但最终选择了主键索引扫描，再在 Server 层或引擎返回后做条件过滤。核心矛盾是：

**明明 `possible_keys` 里有业务索引，为什么 MySQL 最终 `key` 选择了 `PRIMARY`，表现得像全表扫描？**

这类问题的危险点在于：开发者容易把“建了索引”误认为“必然使用索引”，而真实的 MySQL 优化器是基于成本估算做选择的。索引只是候选路径，不是强制执行路径。

## 2. 排查路径与关键线索（侦探过程）

### 2.1 第一步：先用普通 `EXPLAIN` 确认执行路径

排查的第一步不是猜索引是否失效，而是先看执行计划。

```sql
EXPLAIN
SELECT ...
FROM your_table
WHERE source_type = ...
  AND ...;
```

普通 `EXPLAIN` 中最关键的字段是：

```text
possible_keys  优化器认为可能可用的索引
key            优化器最终选择的索引
type           访问类型，通常从 const/ref/range/index/ALL 逐步变差
rows           预估扫描行数
filtered       条件过滤后的剩余比例
Extra          额外动作，如 Using where、Using index、Using filesort
```

本案的关键线索是：**`possible_keys` 有候选索引，但 `key` 选了 `PRIMARY`，`Extra` 出现 `Using where`。**

这推翻了第一个初步假设：不是索引不存在，也不是优化器完全没识别到索引，而是优化器识别到了索引后仍然主动放弃。

### 2.2 第二步：使用 `EXPLAIN FORMAT=JSON` 观察成本信息

普通 `EXPLAIN` 只能看到最终选择，但解释不了“为什么这么选”。因此作者进一步使用 JSON 格式执行计划：

```sql
EXPLAIN FORMAT=JSON
SELECT ...
FROM your_table
WHERE source_type = ...
  AND ...;
```

`EXPLAIN FORMAT=JSON` 比普通 `EXPLAIN` 多出一批成本估算信息，尤其是：

```json
{
  "query_block": {
    "cost_info": {
      "query_cost": "..."
    }
  }
}
```

这里的 `cost_info` 是理解优化器选择的关键。它不是实际耗时，而是优化器基于统计信息、扫描行数、CPU 成本和 IO 成本估算出来的相对成本。

### 2.3 第三步：用 `FORCE INDEX` 验证“强制走索引是否更优”

为了验证优化器是否误判，作者尝试强制使用指定索引：

```sql
EXPLAIN FORMAT=JSON
SELECT ...
FROM your_table FORCE INDEX (idx_source_id)
WHERE source_type = ...
  AND ...;
```

字幕中将该操作识别成了 `false index`，实际应理解为 MySQL 的 `FORCE INDEX`。

强制索引后的执行计划显示：走业务索引的成本并没有明显低于默认计划。视频中提到，作者对强制索引路径做了手工成本验算，最终成本约为：

```text
3865.2
```

这个值与执行计划中的成本估算基本一致。

这一步推翻了第二个初步假设：不是优化器“瞎选”，而是从它的成本模型看，走业务索引并没有明显收益。

### 2.4 第四步：结合数据分布解释扫描行数

作者继续从数据分布角度解释成本估算。视频中给出的关键数据是：

```text
总数据量：约 160 万
符合条件的数据量：3221
平均命中间隔：1600000 / 3221 ≈ 501
```

含义是：平均每扫描约 **501** 行，才能找到 1 条符合条件的记录。

这说明查询条件虽然看起来能用索引，但从优化器估算看，过滤效果、回表成本、扫描成本叠加后，并没有形成压倒性优势。最终 MySQL 选择主键扫描加 `Using where`，在它的成本模型里是合理的。

## 3. 根本原因分析（底层原理）

### 3.1 MySQL 优化器不是“有索引就用”，而是“谁成本低用谁”

MySQL 查询优化器的核心目标不是机械地匹配索引，而是在候选执行路径里选择估算成本最低的路径。成本主要由两部分组成：

```text
总成本 ≈ IO 成本 + CPU 成本
```

其中：

- **IO 成本**：读取数据页、索引页、回表读取聚簇索引页的成本。
- **CPU 成本**：比较条件、过滤记录、排序、计算表达式等成本。

在 MySQL 5.7 中，这些成本依赖一组默认成本常量和存储引擎统计信息。优化器会估算不同路径：

```text
路径 A：走业务二级索引 -> 找到主键 -> 回表 -> 条件过滤
路径 B：按主键或表扫描 -> 逐行判断 WHERE 条件
```

如果路径 A 的二级索引选择性不高，或者需要大量回表，路径 A 的成本就可能接近甚至高于路径 B。

### 3.2 为什么“走索引”反而可能不划算

很多慢 SQL 的反直觉点在于：索引不是免费通道。

走二级索引时，MySQL 通常要经历：

```text
二级索引 B+Tree 定位
-> 拿到主键 id
-> 回到聚簇索引按主键取整行
-> 应用剩余 WHERE 条件
```

如果命中数据很多，或者索引不能覆盖查询列，回表会被放大。此时顺着主键或数据页扫描，虽然听起来粗暴，但顺序读和批量过滤的成本可能被优化器估算得更低。

因此会出现：

```text
possible_keys 有索引
key 不选该索引
Extra 出现 Using where
```

这不是索引“失效”的唯一解释，而可能是优化器认为“使用索引的总账不划算”。

### 3.3 本案的本质：候选索引存在，但选择性和成本优势不足

本案的根因可以概括为：

**索引存在，但在当前数据分布、查询条件、回表成本和 MySQL 5.7 优化器成本模型下，业务索引路径的估算成本并不低于主键扫描路径，导致优化器选择了主键扫描加条件过滤。**

这里尤其要区分两个概念：

```text
possible_keys: 语法和索引结构上“可能能用”
key:           成本模型比较后“最终决定用”
```

开发中经常误判的地方是：只看到 `possible_keys` 里有索引，就认为 SQL 已经被索引优化。真正要看的，是 `key`、`rows`、`filtered`、`Extra` 以及 JSON 执行计划里的 `cost_info`。

## 4. 解决方案与验证（止血与根治）

### 4.1 紧急止血：必要时用 `FORCE INDEX` 验证，不要盲目上线

遇到这类问题，可以先用 `FORCE INDEX` 做验证：

```sql
EXPLAIN ANALYZE
SELECT ...
FROM your_table FORCE INDEX (idx_source_id)
WHERE ...;
```

或在 MySQL 5.7 中使用：

```sql
EXPLAIN FORMAT=JSON
SELECT ...
FROM your_table FORCE INDEX (idx_source_id)
WHERE ...;
```

但 `FORCE INDEX` 更适合作为诊断手段，而不是默认根治方案。因为数据分布会变化，今天强制索引更快，不代表未来仍然更快。强制索引会削弱优化器根据数据变化自适应选择路径的能力。

### 4.2 根治思路：改写 SQL，引导优化器先缩小候选集

视频中的最终优化方式是：将原 SQL 改写成子查询形式，先用更明确的条件拿到目标 `id`，再由外层通过主键等值查询取完整数据。

抽象形式如下：

```sql
SELECT *
FROM your_table
WHERE id IN (
    SELECT id
    FROM your_table
    WHERE source_type = ...
      AND state = ...
      AND ...
);
```

如果子查询可以被合适索引覆盖，那么优化效果来自两点：

1. 内层查询只处理更窄的索引字段，减少无效扫描和回表。
2. 外层通过主键 `id` 等值访问，访问路径更确定。

这类写法的本质不是“子查询天然更快”，而是通过 SQL 结构改变优化器的搜索空间，引导它先做高选择性的过滤，再回表或按主键取数据。

### 4.3 验证指标

视频中明确给出的优化前关键数据如下：

```text
优化前耗时：10 秒以上
总数据量：约 160 万
符合条件数据：3221
平均命中间隔：约 501 行命中 1 行
强制索引路径手算成本：约 3865.2
```

视频没有完整给出优化后的具体耗时、扫描行数或成本值，因此不能编造优化倍率。正确的验证方式应当是：

```sql
EXPLAIN FORMAT=JSON
SELECT ... -- 优化前 SQL

EXPLAIN FORMAT=JSON
SELECT ... -- 优化后 SQL
```

对比：

```text
query_cost     是否下降
rows_examined  预估扫描行数是否下降
key            是否命中预期索引
Extra          是否减少 Using where / Using filesort / 回表
实际耗时       多次执行取稳定值，避免单次缓存影响
```

如果 MySQL 版本支持，优先补充：

```sql
EXPLAIN ANALYZE
SELECT ...;
```

它能看到实际执行耗时和实际行数，比纯估算的 `EXPLAIN` 更接近生产真相。

## 5. 举一反三与自查清单（避坑指南）

### 5.1 慢 SQL 排查不要只看“有没有索引”，要看“最终用了什么索引”

检查顺序：

```text
possible_keys -> key -> type -> rows -> filtered -> Extra -> cost_info
```

重点判断：

- `possible_keys` 有值但 `key` 不用，说明优化器主动放弃。
- `key` 命中索引但 `rows` 仍很大，说明索引选择性可能不足。
- `Extra` 出现 `Using where` 不一定有问题，但要结合扫描行数判断过滤是否过重。

### 5.2 遇到反直觉执行计划，使用 `EXPLAIN FORMAT=JSON` 看成本

普通 `EXPLAIN` 告诉你“选了谁”，JSON 执行计划更接近告诉你“为什么选”。

常用模板：

```sql
EXPLAIN FORMAT=JSON
SELECT ...
FROM ...
WHERE ...;
```

重点看：

```text
cost_info.query_cost
rows_examined_per_scan
rows_produced_per_join
filtered
used_key_parts
attached_condition
```

### 5.3 不要把 `FORCE INDEX` 当万能药

`FORCE INDEX` 可以用于验证优化器是否误判，但上线前必须回答：

- 数据量继续增长后是否仍然更快？
- 数据分布变化后是否会变慢？
- 该索引是否覆盖查询列，还是会产生大量回表？
- 是否有更稳的 SQL 改写或索引重建方案？

除非压测和生产数据分布都证明强制索引稳定收益，否则不要轻易把它作为长期方案。

### 5.4 联合索引不仅看最左前缀，还要看选择性、回表和排序

命中最左前缀只是“有资格使用索引”，不代表“使用索引一定最优”。设计联合索引时要同时考虑：

- 高频等值过滤字段放前面。
- 高选择性字段优先参与索引。
- 查询列能否被覆盖，减少回表。
- 是否能同时服务 `WHERE`、`ORDER BY`、`LIMIT`。
- 低选择性字段单独放索引前缀，可能收益很弱。

### 5.5 统计信息和版本差异会影响优化器判断

本案发生在 MySQL 5.7 语境下。MySQL 8 的优化器、直方图、执行计划能力都有增强，同一条 SQL 在高版本中可能不需要改写。

日常自查建议：

```sql
SHOW INDEX FROM your_table;
ANALYZE TABLE your_table;
EXPLAIN FORMAT=JSON SELECT ...;
```

当数据分布发生明显变化、大批量导入删除、索引选择突然异常时，应检查统计信息是否过期，并在低峰期评估是否需要执行 `ANALYZE TABLE`。

## 6. 一句话复盘

**索引不是命令，只是候选路径；MySQL 优化器最终选择的是它估算成本最低的访问方式。排查“有索引却全表扫描”时，要从 `EXPLAIN` 进入 `EXPLAIN FORMAT=JSON`，用成本、扫描行数、数据分布和回表代价解释优化器的选择，再决定是更新统计信息、改写 SQL、调整索引，还是谨慎使用强制索引。**
