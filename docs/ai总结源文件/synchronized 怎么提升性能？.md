# synchronized 怎么提升性能？

[synchronized 怎么提升性能？](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=38)

## 讨论视角（这期在解决什么）

- 从“加锁保证线程安全”进一步推进到“怎么在保证正确性的前提下提升 synchronized 性能”，本质是减少锁冲突与锁范围
- 用一个“按学校统计交卷数量”的并发计数案例说明：粗粒度锁会把并行变串行；细粒度锁可以在不同 key 上并行
- 强调落地中的两个常见坑：锁对象不一致导致锁失效；字符串常量池导致跨功能意外互斥

## 核心论点与分论点

### 核心论点

提升 synchronized 性能的关键不是“少加锁”，而是“锁对地方”：把锁粒度从全局（整个接口/方法）缩小到业务冲突域（同一学校/同一 key），让互不冲突的请求并行执行；同时必须保证锁对象的身份一致，避免因为锁对象不同而失去互斥，以及避免使用全局常量池字符串造成跨接口的无谓阻塞。

### 分论点 1：粗粒度加锁能保证正确性，但会把并行退化为串行

- 案例：接口模拟教务系统，记录“每个学校的考试交卷数”
- 并发：创建 3 个线程（1 个北大、2 个清华）并行执行；为了模拟 IO 阻塞，线程 sleep 3 秒
- 现象：不加锁时出现线程不安全（清华应为 2 但只记录到 1）
- 修复：对关键代码加 synchronized（或分布式锁）可保证计数正确，但执行变为一个学生一个学生串行交卷，吞吐下降

### 分论点 2：按“冲突维度”拆锁：不同学校并行，同一学校串行

- 业务冲突域是“同一学校的计数更新”，而不是“整个接口”
- 优化思路：把锁资源从全局锁改为以 `school` 为维度的锁
- 效果：不同学校的学生交卷并行执行；同一学校的学生交卷互斥串行，既正确又提升整体并发

### 分论点 3：坑 1——锁对象不同，锁就形同虚设

- 在真实 HTTP 请求场景中，Spring 传入的可能是不同的 `String` 对象实例（即使内容相同）
- 若直接用参数对象作为锁（例如 `synchronized(school)` 但 school 实例不同），就无法形成互斥
- 结果：看起来“加了锁”，实际上仍然并行执行，线程不安全复现

### 分论点 4：坑 2——用 `String.intern()`/字符串常量当锁，会引入跨接口的隐性竞争

- 把锁对象统一到字符串常量（常量池）可以解决“同文不同对象”的互斥问题
- 但字符串常量池是 JVM 全局共享的
- 如果系统里还有另一个功能接口也用相同的学校字符串做锁（例如“办理入学”也锁 “清华”），就会发生跨接口阻塞：考试接口的清华学生会阻塞入学接口的清华学生，造成无谓的性能损耗

### 分论点 5：更稳妥的方案：为每个接口维护独立的“key -> lock 对象”映射

- 目标：同一接口内按 school 维度互斥；不同接口互不影响
- 做法：为该接口单独维护一个 `ConcurrentHashMap<String, Object>` 存锁对象
- 通过 `computeIfAbsent(school, k -> new Object())` 获取/创建锁对象
- 使用该对象作为 synchronized 的锁资源：同校互斥、异校并行，并避免常量池的跨功能竞争

## 关键数据、案例与结论（逐条可复述）

- 并发样例：3 个线程（北大 1、清华 2），模拟 IO sleep 3 秒
- 错误结果：不加锁时清华计数丢失（2 变 1）
- 性能结论：全局锁正确但串行；按学校拆锁可并行提升吞吐
- 工程结论：锁必须“对象身份一致”；避免使用全局字符串常量池作为跨系统共享锁；推荐使用接口内独立的 `ConcurrentHashMap` 锁表
