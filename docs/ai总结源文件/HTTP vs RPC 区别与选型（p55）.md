# HTTP vs RPC 区别与选型（p55）

[HTTP vs RPC 区别与选型（p55）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=55)

## 讨论视角（这期在解决什么）

- 以分布式系统的“服务间通信选型”切入：HTTP vs RPC 不是谁更高级，而是匹配内外部场景的取舍
- 用“兼容万物的通用标准（HTTP）”对比“追求极致效率的内部专线（RPC）”，强调两者通信思维不同
- 给出工程落地范式：外部用 HTTP，内部用 RPC，通过 API 网关做协议转换（外宽内快）

## 核心论点与分论点

### 核心论点

HTTP 的优势是自描述与强兼容（文本协议、门槛低、无需 SDK、调试友好），适合对外开放与接口频繁变化场景；RPC 的优势是契约驱动与高性能（二进制传输、强类型、编译期校验、长连接与序列化效率），适合内网微服务高并发与低延迟场景。最佳实践通常不是二选一，而是“对外 HTTP + 对内 RPC”，用网关隔离外部不确定性并在入口做协议翻译。

### 分论点 1：HTTP 的核心特征：自描述、通用、低门槛

- 请求与响应“人类可读”，浏览器/APP/curl 都能直接调
- 无需预装 SDK，只要网络能通即可调用，适配外部复杂客户环境
- 适合快速联调与开放平台场景

### 分论点 2：HTTP 的代价：文本臃肿 + 解析开销，在高并发内网会成为性能杀手

- header 等冗余信息多，传输体积大
- JSON 等大文本解析消耗 CPU，流量上来后延迟难压
- 在极高并发的内部调用链里成本被放大

### 分论点 3：RPC 的设计思路：让远程调用像本地函数，靠“契约”换效率与稳定

- 传输的是二进制流而非文本，数据体积更小
- 通过 IDL/契约预先定义数据结构：传输时不用传字段名，只传值
- 强类型约束能在编译期拦截类型错误，降低运行期低级故障

### 分论点 4：工程落地：外宽内快（公网 HTTP + 网关翻译 + 内网 RPC）

- 公网环境不可控：必须用 HTTP（无法要求浏览器/外部调用方解析二进制协议）
- API 网关承担“系统海关”职责：对外暴露 HTTP，对内翻译成 RPC，并在边界做治理与隔离
- 内网可信环境：订单/库存/支付等核心链路用 RPC 的长连接 + 二进制序列化，追求最低延迟与最高吞吐

## 关键数据、案例与结论（逐条可复述）

- 典型踩坑：
  - 内部服务全用 HTTP+JSON：流量上来后 CPU 花在解析大文本上，延迟难压
  - 对外强推 RPC：要求对接方安装依赖库/SDK，导致对接体验差、兼容性问题多
- 选型结论：
  - 开放平台/对接前端/业务验证期：优先 HTTP（开发效率、联调便利、接口变动成本低）
  - 微服务集群/高并发/强契约要求：引入 RPC（编译器约束换运行稳定与性能）

### 代价：臃肿

### 开放平台/前端对接/快速迭代：HTTP
### 微服务集群/高并发/强契约：RPC
