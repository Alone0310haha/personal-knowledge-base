# 视频文案综合分析报告（信息提炼）

[如何设计分布式日志存储架构](https://www.bilibili.com/video/BV1qb6EB2EGi?p=11)

## 讨论视角（这期在解决什么）

视频以“面试场景题：如何设计分布式日志存储架构”为切入点，核心是在系统从单体演进到分布式/微服务后，日志分散在多台机器上导致排障困难，需抽离统一日志系统，并在不影响业务的前提下做到高可用、可扩展与可观测。

## 核心论点与分论点

### 核心论点

分布式日志体系的关键不是“把日志写下来”，而是“让日志集中化、可检索、可扩展”，同时通过异步链路（如消息队列）把日志采集与存储从业务请求链路中解耦，避免影响线上业务。

### 分论点 1：单体阶段用文件日志最简单且稳定

- 单体项目：用 Logback / Log4j 等直接写本地文件
- 优点：部署简单、成本低、维护容易、性能高、稳定
- 适用：服务数量少、排障范围小、无需统一检索平台

### 分论点 2：分布式/微服务阶段需要“统一日志管理系统”

- 问题：仍用文件存储会导致异常排查必须逐个服务、逐台机器找日志
- 方向：抽离日志系统，形成统一入口与统一展示
- 关键点：用消息队列异步写入日志存储，提升吞吐并降低对业务的影响

### 分论点 3：日志系统设计要优先保证可用性与扩展性

- 可用性：日志系统不能影响业务系统正常运行（不可让日志链路拖垮主链路）
- 扩展性：不仅服务于当前系统，还要支持未来更多项目接入

### 分论点 4：三种常见落地方案与选型建议

1) MongoDB 日志存储方案
- 链路：业务产生日志 → MQ 异步写入 MongoDB → 自研界面展示
- 适配：日志结构灵活（NoSQL）、查询过滤需求较多、数据量大
- 关键能力：天然支持分片与复制，适合横向扩展

2) ELK（Elasticsearch + Logstash + Kibana）
- 分工：
  - Elasticsearch：存储与检索
  - Logstash：日志收集/过滤/转换/组合
  - Kibana：UI 展示
- 工程实践：Logstash 较“重”，常用 Filebeat 替代做轻量采集；采集后经 MQ 写入 ES，再用 Kibana 展示
- 适配：需要更强检索能力、功能更全面的日志平台（代价是中间件多、运维成本高）

3) Loki（配套 Grafana 生态）
- 组件：Promtail 采集 + Loki 存储 + Grafana 展示
- 适配：已在用 Grafana，希望更轻量的日志系统，不追求 ELK 级别的“全文检索/复杂处理”

## 关键结论（逐条可复述）

- 单体：文件日志足够好（简单、稳、便宜）
- 分布式：必须集中化，否则排障成本指数上升
- 解耦方式：日志采集/存储走异步（消息队列），避免阻塞业务请求
- 选型：
  - 结构灵活 + 大数据量 + 分片复制：MongoDB
  - 强检索 + 功能全：ELK（Filebeat 可替代 Logstash 做轻采集）
  - Grafana 生态 + 轻量：Loki + Promtail + Grafana

### Loki：存储
