# 给你一个亿 Redis Keys，如何统计双方的共同好友

[给你一个亿 Redis Keys，如何统计双方的共同好友](https://www.bilibili.com/video/BV1qb6EB2EGi?p=14)

## 讨论视角（这期在解决什么）

视频围绕“面试挖坑题：给你一个亿 Redis Keys，如何统计双方的共同好友”展开。核心矛盾在于面试官设下的陷阱——**数据量级（一亿用户）与存储成本/性能的冲突**。直接用 Redis 存储全量关系数据既昂贵又不合理，因此讨论重点在于如何设计高性价比、高性能的海量社交关系存储与计算架构。

## 核心论点与分论点

### 核心论点
**不要直接把一亿用户的全量好友关系塞进 Redis！** 虽然 Redis 的 Set 结构（`SINTERSTORE`）能轻易计算交集，但内存成本过高。正确的思路是**分层存储与计算**：全量数据存冷存储（MySQL/HBase/Neo4j），热点数据（如网红/大V）存 Redis 缓存，结合离线计算或专用图数据库解决复杂关系查询。

### 分论点 1：Redis 方案的局限性（面试陷阱）
- **理论可行**：使用 Set 存储好友列表，`SINTERSTORE` 计算交集。
- **实际不可行**：一亿用户的好友关系数据量巨大，全量存内存（Redis）成本极高，且大 Key 计算交集会阻塞单线程的 Redis。

### 分论点 2：MySQL 方案（基础兜底）
- **架构**：必须分库分表。
- **查询**：通过 `JOIN` 查交集。
- **优化**：并发量大时数据库扛不住，需配合定时任务预计算，将热点数据（如共同关注的网红）缓存至 Redis。

### 分论点 3：Neo4j 方案（专业对口）
- **定位**：图数据库，专为社交网络设计。
- **优势**：天然支持“共同好友”、“可能认识的人”等复杂关系查询；性能优于 MySQL，支持十亿级节点。
- **适用**：深度社交场景，关系链复杂。

### 分论点 4：HBase + Hadoop 方案（大数据通用）
- **架构**：HBase 存储海量关系数据，Hadoop 进行离线计算。
- **适用**：通用海量数据处理，不局限于社交场景。

## 关键结论

1. **拒绝全量 Redis**：除非公司预算无限，否则一亿用户的关系链不能全放内存。
2. **混合架构是主流**：
   - **MySQL**：存核心业务数据。
   - **Neo4j**：存社交关系数据（推荐）。
   - **HBase**：存海量历史/冷数据。
   - **Redis**：仅缓存热点数据（如大V的共同好友计算结果）。
3. **技术选型逻辑**：
   - 专用社交场景 -> **Neo4j**
   - 通用海量计算 -> **HBase + Hadoop**
   - 成本敏感/初期 -> **MySQL 分库分表 + Redis 缓存**
