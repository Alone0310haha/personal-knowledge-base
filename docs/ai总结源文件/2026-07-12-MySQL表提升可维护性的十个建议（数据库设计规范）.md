# MySQL 表提升可维护性的十个建议（数据库设计规范）

## 1. 核心设计哲学

数据库表设计的核心不是“把字段存进去”这么简单，而是要在性能、可演进性、数据一致性和团队协作之间建立长期稳定的约束。

这份规范背后的思想是：**提前把可维护性设计进表结构，用审计字段、命名规范、主键策略、分区规划和受控变更机制，降低线上 DDL、数据冗余、查询退化和团队协作失控的风险。**

## 2. 黄金法则（推荐做法）

### 2.1 【数据追踪与增量同步】业务主表必须预留审计字段

**具体规范：** 主要业务表默认增加创建时间、更新时间、删除时间、创建人、更新人等审计字段。审计字段不直接承载业务含义，但必须支撑数据同步、排障、追责和变更追踪。

```sql
CREATE TABLE order_info (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '代理主键',
    order_no VARCHAR(64) NOT NULL COMMENT '业务订单号',
    user_id BIGINT UNSIGNED NOT NULL COMMENT '用户ID',
    total_amount DECIMAL(18, 2) NOT NULL COMMENT '订单金额',
    status TINYINT NOT NULL COMMENT '订单状态',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    created_by BIGINT UNSIGNED NULL COMMENT '创建人',
    updated_by BIGINT UNSIGNED NULL COMMENT '更新人',
    deleted_at DATETIME NULL COMMENT '软删除时间',
    PRIMARY KEY pk_order_info (id),
    UNIQUE KEY uk_order_info_order_no (order_no),
    KEY idx_order_info_user_id_created_at (user_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订单信息表';
```

**标准判断：** 如果这张表后续可能参与数据同步、审计、问题定位、软删除恢复，就不要省略审计字段。

### 2.2 【数据量可预期增长】提前规划分区或分表维度

**具体规范：** 对明显具备时间序列特征的数据，优先按时间范围规划分区；对天然按用户、租户、设备查询的数据，可以考虑按 `user_id`、`tenant_id`、`device_id` 做哈希分区或分表路由。

```sql
CREATE TABLE device_event (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    device_id BIGINT UNSIGNED NOT NULL,
    event_time DATETIME NOT NULL,
    payload JSON NULL,
    PRIMARY KEY pk_device_event (id, event_time),
    KEY idx_device_event_device_time (device_id, event_time)
)
PARTITION BY RANGE COLUMNS(event_time) (
    PARTITION p202601 VALUES LESS THAN ('2026-02-01'),
    PARTITION p202602 VALUES LESS THAN ('2026-03-01'),
    PARTITION pmax VALUES LESS THAN (MAXVALUE)
);
```

**标准判断：** 分区键必须匹配高频查询条件，否则分区只会增加复杂度，无法减少扫描范围。

### 2.3 【低频扩展属性】用 JSON 承载弱业务属性，而不是频繁改表

**具体规范：** 对只用于展示、不参与筛选、不参与分组、不参与排序的弱属性，可以放入 `JSON` 扩展列，减少频繁 DDL。

```sql
ALTER TABLE product_info
ADD COLUMN ext_json JSON NULL COMMENT '扩展属性，仅用于低频展示字段';
```

适合放入 JSON 的属性：

- 营销展示字段
- 第三方返回的非核心扩展信息
- 低频透传属性
- 临时灰度字段

不适合放入 JSON 的属性：

- 高频查询条件
- 需要排序、分组、聚合的字段
- 需要强约束、强校验的核心业务字段
- 高频更新且 JSON 体较大的字段

如果 JSON 中某个属性后来成为查询条件，应映射为生成列并建立索引。

```sql
ALTER TABLE product_info
ADD COLUMN brand_code VARCHAR(64)
    GENERATED ALWAYS AS (JSON_UNQUOTE(JSON_EXTRACT(ext_json, '$.brandCode'))) STORED,
ADD KEY idx_product_info_brand_code (brand_code);
```

### 2.4 【数据一致性与查询效率】规范化为主，反规范化为辅

**具体规范：** 核心业务模型优先遵循第三范式，避免把国家名、城市名、用户等级名等可变维度字段大量冗余到主表；统计分析、报表、检索场景可以额外建设反规范化宽表。

推荐结构：

```sql
-- 规范化主数据
country(id, country_name)
city(id, country_id, city_name)
user_info(id, city_id, ...)

-- 面向分析或查询加速的反规范化宽表
user_profile_snapshot(user_id, country_name, city_name, ...)
```

**标准判断：** 交易链路优先一致性，分析链路优先查询效率；反规范化表必须有同步策略和数据校验机制。

### 2.5 【长期稳定性】主键使用无业务含义的代理主键

**具体规范：** 主键优先使用 `BIGINT UNSIGNED AUTO_INCREMENT` 或趋势递增的分布式 ID，不使用身份证号、手机号、学号、订单号等业务字段作为主键。

```sql
id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '代理主键'
```

业务唯一性通过唯一索引保证：

```sql
PRIMARY KEY pk_user_info (id),
UNIQUE KEY uk_user_info_id_card_no (id_card_no),
UNIQUE KEY uk_user_info_mobile (mobile)
```

**标准判断：** 主键一旦被外键、缓存、消息、日志、下游系统引用，就必须保持不可变；凡是未来可能变更的业务编码，都不应成为主键。

## 3. 避坑红线（禁止做法）

### 3.1 禁止在生产环境直接手工执行高风险 `ALTER TABLE`

直接在线上大表执行 DDL 是最高风险操作之一。它可能导致锁表、元数据锁等待、复制延迟、业务写入阻塞，甚至改错后难以回滚。

错误示例：

```sql
ALTER TABLE order_info ADD COLUMN risk_tag VARCHAR(32);
```

正确原则：

- 先评估表大小、QPS、DDL 算法、锁影响和回滚方案。
- 先备份，再变更。
- 优先使用版本化迁移工具或在线 DDL 工具。
- 大表变更优先考虑新表替换、影子表迁移、灰度双写、回放校验。

### 3.2 禁止把所有查询字段都建索引

索引是查询加速结构，但不是免费午餐。每多一个索引，`INSERT`、`UPDATE`、`DELETE` 都要同步维护更多 B+Tree，写入成本、磁盘占用、Buffer Pool 压力都会上升。

禁止做法：

```sql
KEY idx_order_info_status (status),
KEY idx_order_info_type (type),
KEY idx_order_info_channel (channel),
KEY idx_order_info_remark (remark)
```

判断标准：

- 高频过滤条件可以建。
- 高频 `ORDER BY` / `GROUP BY` 可以建。
- 低频查询、小表查询、低选择性字段不要盲目建。
- 联合索引必须考虑最左前缀、字段选择性和排序方向。

### 3.3 禁止使用业务含义字段做主键

身份证、手机号、学号、业务订单号都可能变更、脱敏、重发或迁移。一旦作为主键，后续修改会牵动外键、缓存 Key、消息幂等 Key、数据同步和审计链路。

禁止示例：

```sql
PRIMARY KEY (id_card_no)
```

推荐示例：

```sql
PRIMARY KEY pk_user_info (id),
UNIQUE KEY uk_user_info_id_card_no (id_card_no)
```

### 3.4 禁止滥用 UUID 作为 InnoDB 聚簇主键

无序 UUID 写入 B+Tree 时会随机插入叶子页，容易造成页分裂、页碎片和缓存命中率下降；同时 UUID 字符串存储空间远大于 `BIGINT`，二级索引也会被放大。

如果必须使用全局唯一 ID，应优先选择趋势递增 ID；如果使用 UUID，也应避免以普通字符串形式作为聚簇主键。

### 3.5 禁止命名随意、缩写泛滥

字段、索引、表名必须可读、可搜索、可协作。禁止使用只有个人知道含义的拼音首字母或模糊缩写。

禁止示例：

```sql
yxbc VARCHAR(32)
ddzt TINYINT
```

推荐示例：

```sql
marketing_batch_code VARCHAR(32)
order_status TINYINT
```

命名标准：

- 表名、字段名统一小写。
- 单词用下划线分隔。
- 尽量使用完整英文语义。
- 索引名前缀统一：`pk_`、`uk_`、`idx_`、`fk_`。
- 大系统表名增加模块前缀，如 `oms_order_info`、`ums_user_info`、`billing_invoice`。

## 4. 实战权衡（Trade-off）

### 4.1 范式化 vs 反范式化

- 范式化优势：减少冗余，降低更新异常，保证主数据一致。
- 范式化代价：查询需要多表 join，复杂报表性能较差。
- 反范式化优势：减少 join，提升查询和分析效率。
- 反范式化代价：需要同步机制，存在数据延迟和一致性校验成本。

判断标准：

- 核心交易表：优先范式化。
- 统计报表、搜索、画像、分析宽表：允许反范式化。
- 冗余字段必须回答三个问题：谁写入、谁同步、如何校验。

### 4.2 JSON 扩展列 vs 正式字段

- JSON 优势：减少低价值字段带来的频繁 DDL，适合弱属性扩展。
- JSON 代价：更新通常更重，字段约束弱，直接索引能力有限，查询表达复杂。

判断标准：

- 只展示、不筛选、不排序、不聚合：可以放 JSON。
- 参与核心查询、统计、约束、风控、对账：必须独立成列。
- JSON 字段变热后，应通过生成列或正式 DDL 治理。

### 4.3 索引数量 vs 写入性能

- 索引越多，读路径可能越快。
- 索引越多，写路径一定更重。

判断标准：

- 高频查询优先。
- 大表优先。
- 高选择性字段优先。
- 能被联合索引覆盖的，不要拆成多个孤立单列索引。
- 写多读少的表，索引必须克制。

### 4.4 分区设计 vs 运维复杂度

- 分区优势：减少扫描范围，便于冷热数据管理和历史数据归档。
- 分区代价：DDL、主键、唯一约束、SQL 写法、分区维护都会更复杂。

判断标准：

- 查询条件稳定包含分区键，才值得分区。
- 时间序列表优先按时间范围分区。
- 用户/租户维度明显时，才考虑哈希分区或分库分表。
- MySQL 分区需要提前规划分区生命周期，避免数据落入兜底分区后失控。

### 4.5 在线 DDL vs 新表迁移

- 小表、低峰期、明确支持 instant/inplace 的 DDL，可以走受控在线变更。
- 大表、高 QPS、核心交易表，应优先考虑影子表、新表迁移、双写校验或在线变更工具。

判断标准：

- 能否回滚。
- 是否锁表。
- 是否影响复制延迟。
- 是否有备份。
- 是否经过预发环境和低峰演练。

## 5. 日常开发自查清单

- [ ] 表是否有稳定、无业务含义的代理主键？
- [ ] 业务唯一字段是否使用唯一索引表达，而不是直接当主键？
- [ ] 主业务表是否包含 `created_at`、`updated_at`、`created_by`、`updated_by`、`deleted_at` 等审计字段？
- [ ] 表名、字段名、索引名是否统一小写、英文、下划线分隔？
- [ ] 索引命名是否使用 `pk_`、`uk_`、`idx_`、`fk_` 前缀？
- [ ] 金额是否使用 `DECIMAL`，而不是 `FLOAT` / `DOUBLE`？
- [ ] 布尔/状态字段是否使用合适的 `TINYINT` 或枚举编码，并有清晰注释？
- [ ] 是否避免了身份证、手机号、学号、订单号等业务字段做主键？
- [ ] 是否避免了无序 UUID 作为 InnoDB 聚簇主键？
- [ ] 每个索引是否都对应明确的高频查询、排序或唯一约束？
- [ ] 是否避免了为低频查询和低选择性字段盲目建索引？
- [ ] JSON 字段是否只用于低频展示型扩展属性？
- [ ] JSON 中的字段是否没有参与高频筛选、排序、分组和聚合？
- [ ] 大表是否提前评估分区、归档或冷热分离策略？
- [ ] 分区键是否出现在高频查询条件中？
- [ ] 反规范化字段是否有同步策略、校验机制和责任边界？
- [ ] 生产 DDL 是否有备份、评估、回滚方案和变更窗口？
- [ ] 是否避免了直接在生产环境手工执行高风险 `ALTER TABLE`？
- [ ] 大系统是否通过表名前缀或 schema 区分业务模块和权限边界？

## 6. 一句话准则

**数据库设计的红线是：核心字段要稳定、关键数据要可追踪、线上变更要可回滚、索引和冗余要有明确业务收益；凡是为了省一时开发成本而牺牲长期可维护性的设计，都会在生产环境加倍偿还。**
