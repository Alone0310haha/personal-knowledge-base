# MVCC（多版本并发控制）如何实现无锁一致性读？

[MVCC（多版本并发控制）如何实现无锁一致性读？](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=33)

## 讨论视角（这期在解决什么）

- 从“并发事务会带来哪些读写异常”切入，先用脏读/不可重复读/幻读把隔离级别的目标讲清楚
- 再从“读已提交为什么不靠读锁阻塞”切入，解释 InnoDB 用 MVCC 做一致性读来提高并发性能
- 用一个 ReadView + 版本链（undo log）示例，把“到底读到哪一个历史版本”落到可执行的判断规则上

## 核心论点与分论点

### 核心论点

MVCC（多版本并发控制）是 InnoDB 在部分事务隔离级别下实现“无锁一致性读”的底层机制：通过隐藏列记录版本信息、用 undo log 维护版本链、再用 ReadView 快照判断可见性，从而在并发更新存在时仍能读到“满足隔离级别要求的已提交版本”，避免读锁导致的阻塞和吞吐下降。

### 分论点 1：隔离级别是在解决并发读写的三类典型问题

- 脏读：读到其他事务未提交的数据（例：账户 1000 被改成 800，但更新事务回滚）
- 不可重复读：同一事务两次读同一行，结果不一致（例：第一次读 1000，别人提交更新后第二次读到 800）
- 幻读：同一事务两次做范围/汇总，结果不一致（例：汇总为 3000，别人插入 1000 后再汇总变成 4000）

对应关系（视频口径）：

- 读未提交：可能脏读
- 读已提交：解决脏读
- 可重复读：解决不可重复读
- 串行化：用更强的锁（如表锁）解决幻读，但并发性能最低

补充：视频提到 MySQL 默认隔离级别为可重复读，Oracle 默认为读已提交。

### 分论点 2：读已提交如果用“读锁阻塞”等待，会把并发退化为串行

- 如果更新事务持有锁，读事务为了“只读已提交”而等待更新事务提交/回滚，就会产生阻塞
- 阻塞越多，吞吐越低
- MVCC 的价值就在于把“读”改成读历史版本（已提交版本），尽量不与写操作互相等待

### 分论点 3：MVCC 的三个关键点：隐藏列、undo log、ReadView

- 隐藏列：一行记录除了业务列，还维护版本相关的隐藏信息（视频点名“三个隐藏列”）
  - 关键含义：记录由哪个事务 ID 修改；以及指向更老版本的存储指针（类似 roll_pointer）
  - 另一个隐藏列用于在没有主键时维护行 ID（视频称“不重要”）
- undo log：保存历史版本（用于回滚，也用于一致性读）；多次 update/delete 会形成版本链
- ReadView：一次一致性读的快照信息，用于从版本链中筛出“当前这次读能看到的版本”

### 分论点 4：ReadView 用“四个关键信息 + 四个判断”决定版本可见性

ReadView（视频口径）包含四类信息：

- 当前事务 ID（本次查询所在事务）
- 未提交事务列表（当前活跃/并发的事务 ID 集合）
- 最小事务 ID（活跃事务列表中的最小值）
- 最大事务 ID（“尚未开始”的事务 ID，可理解为创建快照时的上界）

可见性判断（视频示例逻辑）：

- 先拿版本链里“最新版本”的事务 ID 来做判断
- 判断 1：版本事务 ID = 当前事务 ID → 可读（自己改的自己可见）
- 判断 2：版本事务 ID < ReadView 最小事务 ID → 可读（在快照创建前已提交）
- 判断 3：版本事务 ID > ReadView 最大事务 ID → 不可读（快照创建后才出现的版本），沿指针找更老版本
- 判断 4：版本事务 ID 在最小与最大之间 → 看它是否属于未提交活跃事务列表
  - 在列表中：不可读（并发未提交版本），沿指针找更老版本
  - 不在列表中：可读（并发但已提交）

### 分论点 5：读已提交 vs 可重复读的差异，本质是 ReadView 创建时机不同

- 读已提交：每一次查询都会创建一个 ReadView，因此同一事务内两次查询可能看到不同的“最新已提交版本”
- 可重复读：同一事务对同一查询只创建一次 ReadView（通常在第一次一致性读时创建），后续相同查询沿用同一快照，所以结果一致
- 结论：视频强调 MVCC 主要用于读已提交与可重复读；读未提交与串行化不采用该机制（串行化偏向用强锁实现）

## 关键数据、案例与结论（逐条可复述）

- 脏读案例：事务 A 把 1000 改 800 未提交；事务 B 读到 800；事务 A 回滚 → B 读到的是脏数据
- 不可重复读案例：事务 A 先读 1000；事务 B 提交更新到 800；事务 A 再读变 800
- 幻读案例：事务 A 汇总为 3000；事务 B 插入 1000 后提交；事务 A 再汇总为 4000
- ReadView 示例：事务 10 更新，事务 20 查询；ReadView 活跃事务为 {10,20}，最小=10，最大=21；版本不可见时沿指针回到更老版本，最终读到满足“已提交”的历史版本
