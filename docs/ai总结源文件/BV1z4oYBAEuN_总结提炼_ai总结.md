# 视频AI总结（总结提炼）

视频地址：https://www.bilibili.com/video/BV1z4oYBAEuN/

## 一句话摘要
用一次“卖家活动平台（发权益 + 扣费用）”的真实业务场景串起分布式一致性/分布式事务的核心取舍：不盲目追求强一致，用“本地事务落库 + 消息异步驱动 + 幂等 + 定时兜底 + 对账”的工程组合把最终一致性做稳。

## 讨论角度
- 以面试对话的方式，从简历项目切入，把“分布式一致性”落到可讲清楚、可落地的方案设计与故障兜底上。
- 对比 2PC / TCC / SAGA / 可靠消息等常见方案，强调“业务一致性要求 + 系统侵入性 + 资源成本”的综合权衡。

## 核心论点与分论点
- 核心论点：分布式一致性不是背概念，而是围绕“失败一定会发生”做可恢复设计；当业务允许时，用最终一致性方案（消息驱动 + 幂等 + 补偿/对账）往往更工程化、更可用。
- 分论点 1：2PC 强一致的代价高（资源占用、阻塞、死锁/长事务风险），多数业务并不值得。
- 分论点 2：TCC 通过中间态降低阻塞，但对上下游侵入性强（需要 Try/Confirm/Cancel 及冻结/预留能力）。
- 分论点 3：SAGA 依赖补偿链路，适合“可补偿”的长流程，但会引入反向动作与复杂度；如果业务不希望出现“逆向回滚”，要慎用。
- 分论点 4：可靠消息/最大努力通知 + 定时扫表 + 天级对账，是把最终一致性做“可控、可监控、可追责”的关键组合拳。

## 关键案例/结论（视频里的关键“工程化答案”）
- 业务背景：卖家参加活动，需要同时发生两件事：
  - 发放权益（如广告券、补贴、佣金减免等）
  - 收取费用（报名费/按订单收佣金等）
- 方案选型：偏“可靠消息 + 最终一致性/最大努力通知”，而不是照搬教科书式 2PC/TCC/SAGA。
- 核心链路（Outbox/本地消息表思路）：
  - 在本地事务中同时落库：权益记录 + 扣费记录，先保证“我确认这笔业务发生过且可追踪”。
  - 将记录转换为消息，异步投递（权益发放走一个消费者，扣费走另一个消费者）。
  - 依赖 MQ 的重试能力处理短暂失败。
  - 额外加定时任务扫表兜底（解决：落库成功但消息没发出去/消费失败未完成等极端场景）。
  - 最后再做天级对账，确保账实能对齐。

## 常见追问与答题模板（按视频脉络整理）
- “本地事务提交后如何保证消息一定发出去？”
  - 不把“发消息”当成强同步步骤；用定时扫表（或 outbox relay）兜底，把“未投递/未完成”的记录重新投递或补偿。
- “为什么会出现重复消费？怎么处理？”
  - 重复投递是常态（确认机制/网络抖动导致发送方误判未成功而重试）。
  - 处理要点：端到端幂等。通过业务维度拼幂等键（如卖家ID/国家/活动ID/权益类型等），由实际执行方（扣费/发放侧）做幂等落库与判重。
- “幂等键并发下会不会仍然扣两次？”
  - 如果两个请求并行通过判重查询，确实可能竞争；执行侧需要用锁或唯一约束等方式确保同一幂等键只能成功一次。
  - 可用思路：锁内二次检查（double-check）、或直接锁内检查 + 执行；配合唯一索引/事务保证更稳。
- “缓存一致性 vs 事务一致性有什么区别？”
  - 事务一致性关注 ACID 语义下的原子性/一致性结果（例如转账总额守恒、账能对上）。
  - 缓存一致性关注缓存与数据库数据的同步与陈旧问题（读到旧数据/延迟不一致）。

## 并发与高 QPS 的要点（视频里提到的思路）
- 写并发竞争（多订单并发扣减同一余额/权益额度）：
  - 典型解法：数据库悲观锁/乐观锁；或用分布式锁（本质仍是悲观串行化同一资源的更新）。
  - 目标：优先保证数据安全（不扣错），性能优化其次。
- 缓存击穿：
  - 热点 key 过期或失效时，大并发回源数据库。
  - 解法：只允许一个并发回源并更新缓存，其余等待结果（常见思路：per-key lock / singleflight / 双重检查）。
- Bloom Filter：
  - 用小空间“近似表示”全集，快速判断 key 是否可能存在，常用于防缓存穿透。
  - 普通 Bloom 不支持删除；删除需求通常通过重建或使用支持删除的变体（如布谷鸟过滤器）。

## 业务背景
- 活动业务平台
- 同时：发权益 + 扣费用
## 方案选型
- 2PC：强一致但阻塞/资源占用/死锁风险
- TCC：中间态降低阻塞，但侵入性强（Try/Confirm/Cancel + 冻结能力）
- SAGA：补偿链路，反向动作复杂；不希望逆向时慎用
- 最终选择：可靠消息/最大努力通知（最终一致性）
## 核心链路（工程落地）
- 本地事务落库
  - 权益记录
  - 扣费记录
- 记录转换消息 -> 异步投递
- MQ 重试（短失败）
- 定时扫表兜底（消息丢/消费失败）
- 天级对账（账实一致）
## 可靠性关键
- 重复投递是常态
- 端到端幂等（幂等键 + 执行侧判重）
- 幂等并发竞争：锁/唯一约束/双重检查
## 相关扩展
- 缓存击穿：单飞/按 key 加锁/双重检查
- Bloom Filter：防穿透；普通 Bloom 不可删，删需重建或变体
## 系统边界
- 交易 vs 结算拆分：领域差异/组织与复杂度驱动
