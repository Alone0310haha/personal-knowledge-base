## 讨论角度
- 以“面试现场给一段秒杀代码，让候选人当场找问题并修复”为切入点，重点考察：并发安全、原子性、幂等、吞吐与锁粒度权衡，以及单机/集群边界条件的澄清能力。
- 视频核心不在“写出最强秒杀系统”，而在“拿到烂代码时，如何用可解释的方式定位问题、给出可落地的修复方案（允许使用 AI 但必须会验）”。
  
## 核心论点与分论点
### 核心论点
- 秒杀正确性=“不超卖 + 同一用户最多成功一次 + 数据一致性”；要做到这一点，必须把关键步骤做成线程安全且具备原子性的组合操作，而不是只把某个容器换成线程安全就完事。
  
### 分论点
- 先明确题目隐含约束：单机内存方案 vs 集群方案（数据库/Redis/分布式锁）；不澄清边界，容易“修对了代码、答错了题”。
- 单机内存实现常见坑：共享状态（库存、购买记录）在多线程下的读改写竞态；非线程安全集合导致并发异常；“先判断再扣减/再写记录”不是原子操作会导致超卖/重复购买。
- 锁要解决的不仅是“线程安全”，更是“原子性”：库存扣减与购买记录写入要么都成功，要么都失败。
- 锁粒度要和业务目标匹配：全局锁正确但牺牲吞吐；按用户维度加锁能让不同用户并发、相同用户串行。
- CAS/AtomicInteger 能解决库存并发扣减（避免超卖），但不能自动解决“同一用户只能成功一次”的幂等问题；仍需要围绕用户维度的并发控制或去重结构。
  
## 关键问题清单（面试官想听到的点）
- 状态存储边界：库存/购买记录如果只在内存里，集群下无法共享与持久化；正确做法通常落到 DB/Redis（视频里先指出“这里只能算单机版”）。
- 共享变量的竞态：多个线程同时读库存、判断>0、再扣减，若没有原子保障会超卖。
- 线程不安全容器：购买记录若用非线程安全 List/Map，在并发读写下会产生异常或脏数据；即便用 CopyOnWriteArrayList 也要理解其写入成本与适用场景。
- 检查-再执行（check-then-act）问题：先“是否买过”、再“扣库存”、再“写记录”，任何一步不在同一原子域都可能被并发打穿。
- 锁粒度选择：方法级别锁会把不同用户也串行化；更合理的是“同用户串行、不同用户并发”。
- AtomicInteger 的正确用法：用 CAS（compareAndSet）循环或原子自增/自减方法，避免“先 get 再 set”造成竞态；同时要考虑失败重试是否会死循环（库存归零会收敛退出）。
  
## 修复思路（单机版可落地方案）
### 目标语义
- 不超卖：库存永不被扣到 0 以下。
- 不重复：同一 userId 最多成功一次。
- 可并发：不同 userId 尽量并行执行，不要被全局锁拖死。
  
### 典型实现组合
- 库存：AtomicInteger + CAS 扣减（或等价的原子扣减策略），用于并发下避免超卖。
- 购买去重：并发安全的 Set/Map 结构（例如基于 ConcurrentHashMap 的 KeySet），用于记录“已成功用户”。
- 锁：按 userId 维度加锁（比如 ConcurrentHashMap<userId, Lock> 或条带锁/分段锁思想），保证“同用户请求串行”，从而把“判重 + 扣库存 + 写成功记录”包进同一临界区；不同用户走不同锁，吞吐更高。
  
## 关键取舍与面试表达方式
- 如果面试官强调“单机内存”，就把问题聚焦在：线程安全、原子性、锁粒度、CAS。
- 如果面试官追问“集群秒杀”，就补上：库存与订单要落到 DB/Redis、用分布式锁/原子脚本/消息队列削峰，内存 Map 仅能做本机缓存或本地互斥。
- 使用 AI 的正确姿势：让 AI 给方案，但你要能指出方案是否满足“原子性/幂等/锁粒度/性能”；视频里也在反复追问“这样能保证原子性吗”“会不会阻塞所有用户”。
  
## 关键结论
- 秒杀题最容易“修了一半”：只换线程安全集合或只用 AtomicInteger 都不够，必须把业务关键路径做成原子组合操作，并在锁粒度上兼顾吞吐。
- 正确答题路径是：先澄清单机/集群边界 → 列出并发风险点 → 给出最小可行修复（锁/CAS/并发容器）→ 讲清楚 trade-off 与可扩展方案。
