## 你能照着做的“AI Coding 流水线”（从需求到验收）

### 0. 目标与边界（先定规则）

- 目标：在“正确”的前提下，让 AI 尽可能无人监督地持续跑（中间返工自动化），人类只做最终验收与少量关键纠偏。
- 边界：UI 类任务天然难自动验收，优先把可命令行验证（build/test/lint）的部分流水线化；UI 验收先保留人工介入或补 UI 自动化测试后再放权。

### 1. 准备三件核心资产（强烈建议先做完再开干）

1) “契约文件”（AGENT/Agent.md 类）
- 写清楚：角色分工（主控只调度+验收，不直接写业务代码）、执行纪律（失败必须返工收敛）、证据产出要求（每步必须落盘可回放）。

2) “统一执行入口”（runner / python 包 / 脚本）
- 要求：所有真实执行都必须从同一个入口触发（包括拉代码、改代码、跑命令、跑测试）。
- 输出：自动生成执行审计目录（证据链），把每一轮执行的输入/输出/命令/关键产物落盘。

3) “执行审计目录”（audit）
- 结构要支持你事后回答三件事：
  - 它做了什么？（改了哪些文件、执行了哪些命令）
  - 为什么这么做？（每步依据/决策）
  - 做得对不对？（每步的验证结果与失败原因）

### 2. 把需求写成“可执行任务”（不要只写愿望）

- 写清任务范围：要改什么、不改什么（例如“UI 不变，只替换底层 VM 调用实现”）。
- 写清操作约束：例如“先 clone/复制到新目录避免污染；在新目录改；保留原项目对照”。
- 写清验收条件（必须写）：至少包含
  - build 必须通过
  - 单测/集成测试必须通过（给出具体命令）
  - 关键功能的验证步骤（手工/脚本都行）

### 3. 启动执行（让主控调度，让子执行器干活）

- 让主控按契约做事：分解任务、派发子任务、逐步验收、失败就返工，直到收敛。
- 让执行器负责所有落地动作：编辑代码、运行构建/测试、生成证据。
- 如果子执行器跑很久：不要默认当成卡死；用进程/日志检查它是否仍在运行，再决定是否中断重试。

### 4. 验收：用“两条线”确认结果

1) 结果验收（黑盒）
- 直接跑：build/test/运行验证步骤，确认行为符合预期。

2) 过程验收（白盒，证据链回放）
- 打开审计目录，从关键步骤开始抽查：
  - 是否按你要求的路径执行（例如是否真的先 clone 再改）
  - 是否对关键变更做了验证（例如是否编译/跑测试）
  - 失败后的返工是否合理，是否出现“掩盖问题”的修补

### 5. 返工与收敛（把“返工”自动化）

- 返工触发条件：任何验收失败（build 失败、测试失败、功能偏差）都必须回到执行器重试。
- 返工输入要结构化：把错误日志、失败截图/输出、验收标准一起交给它，并要求它在审计目录中记录修复证据。
- 收敛标准：直到所有验收条件满足，并且审计目录能解释“怎么跑到正确结果的”。

### 6. 归档：把这次成功变成“可复用基准”

- 把本次任务的关键规则沉淀到：
  - 契约文件（主控纪律、默认必须做的验证步骤）
  - License/经验库（可迁移的工程经验：验收清单、常见坑、验证命令）
- 目标：下次类似任务不需要你再提醒“至少要编译一下”这种常识性验收。

## 工具、材料与注意事项清单

### 工具/材料

- 代码仓库：建议用 git worktree / 多分支并行跑任务，互不干扰
- 构建与测试命令：明确写进任务与契约（build/test/lint）
- 执行入口：脚本/包（统一调度、统一落盘、统一审计）
- 审计目录：存放证据链（命令、日志、diff、产物、每轮结论）

### 注意事项

- 验收条件一定要前置写清楚，否则 AI 很容易“自认为完成”就停（典型：没要求编译就不编译）。
- UI 类任务要么补自动化 UI 测试，要么把验收保留人工；别指望纯文本对话能稳定验收 UI。
- 不要只靠提示词约束主控：提示词容易“摸鱼式钻空子”，关键纪律要工程化（默认执行编译/测试、默认生成审计）。

## 常见错误与解决方案

- 错误：没写验收条件，AI 做完就宣布完成（甚至没编译）。
  - 解决：把“必须编译 + 必须跑测试 + 必须输出证据链”写成默认契约。
- 错误：执行跑很久就当成卡死，强行中断导致反复返工。
  - 解决：先用进程/日志确认是否在跑，再决定是否中断；把“如何判断是否仍在运行”写进 SOP。
- 错误：审计目录只存“结论”，不存“证据”，无法回放。
  - 解决：强制落盘命令、关键日志、diff、每轮验收结果；保证可追溯。
- 错误：地基不稳就追求“主动机制/自动巡检”，最终制造噪音。
  - 解决：先把流水线工程化到稳定可验收，再做主动机制（定时扫描/预修复/推荐下一步）。
