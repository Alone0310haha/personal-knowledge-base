## 讨论视角

- 从一次线上压测现象出发：压测流量已到阈值却没有触发限流，反推“限流实现”存在逻辑/工程缺陷
- 以“滑动窗口 + Redis ZSET + Lua 脚本”为例，拆解限流正确性依赖的关键前提（尤其是时间来源）
- 讨论“准确性 vs 性能/复杂度”的工程权衡：集群强一致限流 vs 单机近似限流

## 核心论点与分论点

### 核心论点

- 基于滑动窗口的集群限流如果把“当前时间戳”交给客户端/业务节点传入，在多机时钟不一致的情况下会导致窗口统计失真，从而出现“该限流却没限”的结果；时间应在同一可信侧统一生成（例如在 Redis 端取时间）。

### 分论点 1：现有实现的基本逻辑（滑动窗口）

- 存储结构：用 Redis 的 ZSET（Sorted Set）按时间戳记录请求（score 为时间戳）
- 每次请求流程（Lua 脚本保证原子性）：
  - 清理窗口外的旧记录
  - 统计窗口内有效记录数量
  - 若小于阈值：把“当前时间戳”写入 ZSET 并放行
  - 若大于等于阈值：拒绝请求（限流）

### 分论点 2：致命问题在“时间戳来源”

- 当前实现把“当前时间”从客户端/应用侧传入脚本
- 集群环境下不同机器时钟存在偏差（快/慢不一致）
- 结果：同一逻辑窗口在不同机器上写入的时间戳不统一，导致统计出来的窗口内请求数不准，从而出现漏限流/误限流

### 分论点 3：修复方式：把时间统一到 Redis 侧

- 应用侧不再传当前时间
- 在 Lua 脚本里用 Redis 侧的本地时间函数获取当前时间，保证同一个限流键下的时间源一致
- 同时作者强调时间精度要足够细（文稿提到使用微秒级），否则滑动窗口会被“离散化”成非预期效果

### 分论点 4：进一步的工程权衡：每次请求都打 Redis 的成本

- 即使单次 Redis 操作很快，高并发下“每请求一次就访问 Redis”仍是显著开销与潜在瓶颈
- 当并发极高且对“集群限流精确阈值”要求没那么严时，可考虑近似方案：
  - 单机限流：每台机器本地按自己的 QPS 阈值限流
  - 定期同步：以固定周期从 Redis 拉取/下发配置（而非每请求访问 Redis）
- 代价与接受条件：
  - 可能出现“某台机器已满、另一台机器尚有余量”的不均衡现象
  - 若负载均衡足够均匀，该现象概率低，可接受为“工程 trade-off”，换取吞吐与可用性

## 关键数据 / 案例 / 结论

- 案例现象：压测请求量达到预定阈值仍未触发限流
- 关键结论：滑动窗口限流的正确性强依赖“统一时间源”；跨机传入时间会破坏统计
- Lua 脚本价值：能把“清理 + 统计 + 写入/拒绝”串成一个原子操作，避免并发竞态（但它无法解决时间源不一致）
- 工程原则：很多时候“牺牲一点精度”可能换来更好的性能或更高的基础稳定性，关键在于业务能否接受该误差
