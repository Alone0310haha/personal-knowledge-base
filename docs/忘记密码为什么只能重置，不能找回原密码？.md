# 忘记密码为什么只能重置，不能找回原密码？

[忘记密码为什么只能重置，不能找回原密码？](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=34)

## 讨论视角（这期在解决什么）

- 从“忘记密码为什么只能重置不能找回”这一面试题出发，把结论落到服务端密码存储机制（不可逆）与安全风险（明文/可逆会导致灾难性后果）
- 进一步把“不可逆”拆成工程可落地的算法选型：MD5 / SHA-256 / bcrypt 的差异、各自风险点、以及如何抗暴力破解
- 给出在 Java/Spring 体系下的落地方式：用 Spring Security 直接使用 bcrypt（视频口径）

## 核心论点与分论点

### 核心论点

系统不会直接“告诉你原密码”，因为服务端合规做法是不保存明文密码，而是保存不可逆的加密/哈希结果；既然不可逆，系统本就不知道原密码，只能通过“重置”让用户设置新密码。同时，从安全角度看，如果系统能还原原密码，意味着一旦数据库泄露或内部人员滥权，用户密码将直接暴露。

### 分论点 1：密码绝不能明文落库，否则安全风险不可接受

- 数据库被盗时，明文密码会被批量泄露
- 即使数据库没被盗，有数据库权限的人也可能恶意导出利用
- 因此正确方向是：服务端保存不可逆的加密/哈希结果，登录时做比对验证

### 分论点 2：MD5 的两个主要问题：碰撞风险 + 密文固定导致可被暴力破解

- 抗碰撞性不强（视频举例：两个仅 1 个字符不同的长字符串，最终 MD5 密文相同）
  - 视频补充：作者用随机 72 位字符串循环跑，约 4~5 小时跑出了碰撞样例
- 同一明文每次得到的密文相同
  - 攻击面：可以用“字典/暴力枚举”把常见密码的密文预先算好，撞库匹配

### 分论点 3：加盐可以提高破解成本，但“固定盐/私钥”仍可能被识别套路

- 做法：把一个只有服务端知道的“私钥/盐”拼到密码里再做哈希，比对时同样拼接再比对
- 风险（视频强调）：如果盐是固定且可被推断（例如总有相同前缀），攻击者可能识别后“去前缀”还原出更接近原文的内容
- 更好的做法：每个密码使用随机盐，让攻击者无法复用彩虹表/字典表

### 分论点 4：SHA-256 比 MD5 更强，但仍有“同文同密”的问题，需要随机盐

- SHA-256 的密文更长（视频口径：64 位字符），碰撞难度更高
- 但同样存在：同一明文每次密文一致 → 仍需要程序员自行引入随机盐来抵御暴力破解

### 分论点 5：bcrypt 是更适合存密码的方案：内置随机盐 + 可调成本 + 工程落地成熟

- bcrypt 生成固定长度密文（视频口径：60 位字符）
- 密文结构里包含：
  - 版本号（视频示例提到 2a）
  - cost（加密强度，视频示例提到 10）
  - 随机盐 + 加密结果
- 价值：
  - 随机盐使得“同文不同密”，难以用预计算表暴力撞库
  - cost 可提高计算成本，拉长爆破时间
- 工程实践（视频口径）：在 Spring 项目中引入 Spring Security，即可方便使用 bcrypt

## 关键数据、案例与结论（逐条可复述）

- 面试题关键回答：系统不知道原密码，所以只能重置，不能提供原密码
- 风险案例（原则性）：明文落库 + DB 泄露/内部滥权 → 用户密码直接暴露
- MD5 碰撞案例（视频描述）：
  - 两个字符串仅 1 个字符不同（a vs e），最终 MD5 密文相同
  - 作者用随机 72 位字符串跑了约 4~5 小时得到碰撞样例
- 暴力破解案例（视频描述）：针对常见字符组合，暴力枚举匹配 MD5 密文“非常快”
- bcrypt 密文示例解释（视频描述）：包含版本号 2a、强度 10、随机盐与密文
