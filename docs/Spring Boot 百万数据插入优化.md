# Spring Boot 百万数据插入优化

[Spring Boot 百万数据插入优化](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=18)

## 讨论视角（这期在解决什么）

- 以“Spring Boot 业务接口性能优化/批量写入”的视角：百万数据插入如何提速
- 重点不是“单条 insert 怎么快”，而是通过“分批 + 并行 + 正确的线程池参数”缩短整体耗时

## 核心论点与分论点

### 核心论点

把 100 万条数据拆成多个批次，用线程池并行执行批量保存，并用同步器等待所有异步任务完成后再统计耗时，可把“串行批量插入”的总耗时大幅压缩。

### 分论点 1：基线做法（慢）

- 一次性构造 100 万条数据放入 List
- 单线程调用批量保存（示例：`saveAll`）
- 演示耗时：约 90 多秒（接近 100 秒）

### 分论点 2：改造做法（快）

- 分割数据：每 1 万条为一批
  - 100 万条 => 100 份 List（每份 1 万条）
- 并行执行：循环提交 100 个批处理任务到线程池异步执行
- 结果等待：使用 `CountDownLatch`
  - 初始化计数为 100
  - 每个批处理完成后 `countDown`
  - 主线程 `await`，直到全部批处理完成再记录耗时
- 演示耗时：约 17 秒左右（相比基线快约 60~70 秒量级）

### 分论点 3：为什么用 Spring 的线程池 + @Async

- 通过 `@Async` 让方法异步执行，写法简单
- 默认 `@Async` 不建议直接用，但可以通过“自定义线程池并绑定给 @Async”来规避问题
- Spring 的 `ThreadPoolTaskExecutor` 相比 JUC 线程池，多了 `TaskDecorator` 等增强能力（示例提到可设置线程名前缀等细节）

### 分论点 4：线程池参数如何设置（核心考点）

- 先判断任务类型：IO 密集 vs CPU 密集
  - IO 密集：磁盘/网络 IO 会阻塞，CPU 容易空闲
  - CPU 密集：计算为主，CPU 长时间忙碌
- 数据库写入更偏 IO 密集（网络连接/IO 不稳定导致阻塞）
- 经验设置：
  - 核心线程数 ≈ CPU 核心数 × 2（可再 +1 预留，避免不可预料阻塞）
  - 最大线程数可在核心线程数基础上再放大（示例提到核心 32/33，最大 66）
- 结论：线程数不是越多越好，要结合任务特性与资源约束（CPU、连接池、上下文切换）来配

## 关键数据、案例与结论（逐条可复述）

- 规模：100 万条数据批量插入
- 分批：每批 1 万条 => 100 批
- 同步器：`CountDownLatch(100)` 等待全部批处理完成
- 效果：约 90+ 秒 => 约 17 秒
- 结论：批量写入提速的关键是“拆分并行 + 合理线程池配置”，而不是盲目加线程
