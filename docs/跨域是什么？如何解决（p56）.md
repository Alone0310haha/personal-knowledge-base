# 跨域是什么？如何解决（p56）

[跨域是什么？如何解决（p56）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=56)

## 讨论视角（这期在解决什么）

- 从前后端分离落地的真实问题切入：跨域不仅是控制台报错，更是浏览器安全模型（同源策略）在工程化中的体现
- 先讲“为什么会拦”（同源策略与 CSRF 风险），再给“怎么解”（CORS / 本地代理 / 反向代理网关）
- 以工程阶段做选型：开发阶段提效率用 proxy，生产阶段用 Nginx 统一域名与路由，公共/可控 API 用 CORS

## 核心论点与分论点

### 核心论点

跨域问题的根源是浏览器强制执行同源策略：当协议/域名/端口不一致时，即便请求已经到达服务端并返回，浏览器也会在“读取响应”阶段阻断数据访问。解决跨域不是绕过安全，而是在不同阶段用合适的方式建立“可信访问路径”：协议层授权（CORS）、开发期通过本地同源转发（proxy）、生产期通过反向代理统一域名（Nginx/网关）。

### 分论点 1：同源策略为什么存在：从源头切断恶意站点利用用户凭证发起未授权请求

- 浏览器要求协议、域名、端口完全一致才允许读取响应
- 目的：防御跨站请求伪造（CSRF）等攻击
- 典型风险：用户登录敏感系统后浏览器保留凭证，若访问恶意站点，脚本可借助凭证向敏感接口发起请求；同源策略阻断“恶意站点 ↔ 敏感接口”的交互链路

### 分论点 2：CORS（跨域资源共享）：标准解法，靠服务端在响应头授权

- 机制：浏览器在跨域请求前可能发起预检请求（preflight）
- 服务端需在响应头显式下发授权信息（如 Access-Control-Allow-Origin 等），把来源加入白名单
- 浏览器检测到授权后放行，业务数据可正常读取
- 适用：公共 API、服务端可控且愿意配合配置的场景

### 分论点 3：本地代理（proxy）：开发期提效的工程化方案，利用“同源只限制浏览器，不限制服务间通信”

- 方式：本地启动代理服务（如 Node 中间层），前端只请求本地同源地址
- 代理再把请求转发给远程 API，响应原路返回
- 浏览器视角：始终是同源通信，从而规避跨域检查
- 适用：前后端并行开发、接口频繁变动、强调联调效率的开发阶段

### 分论点 4：Nginx/反向代理网关：生产期基石，用统一域名从网络层面消除跨域

- 对外只暴露单一域名
- 路由分发：
  - 静态资源路径指向前端资源
  - API 路径反向代理到后端服务
- 效果：前端资源与后端接口在网络层面同源，跨域被网关层屏蔽，对客户端透明
- 同时提供能力：负载均衡、路由分发等

## 关键数据、案例与结论（逐条可复述）

- 关键定义：同源要求协议/域名/端口一致；跨域拦截发生在浏览器“接收/读取响应”阶段
- 三种方案的定位：
  - CORS：W3C 标准，架构简单但依赖后端配置配合
  - Proxy：前端工程化标准，解决开发阶段调试效率与并行开发
  - Nginx：生产部署基石，统一域名 + 路由/负载能力，跨域对客户端透明
- 实践结论：开发用 proxy 提效率，生产用 Nginx 保性能与稳定，必要时用 CORS 做协议级授权
