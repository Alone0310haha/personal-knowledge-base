# 分布式事务：是什么？有哪些方案？怎么选（p65）

[分布式事务：是什么？有哪些方案？怎么选（p65）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=65)

## 讨论视角（这期在解决什么）

- 从“微服务解耦带来的代价：一致性边界被打碎”出发，把分布式事务定义为跨进程/跨服务的一致性问题
- 用 CAP/网络不可靠解释“本地事务确定性为何失效”，强调跨进程后会出现 RPC 三态（成功/失败/未知）
- 以“场景与成本的博弈”为主线，对比三类主流方案：2PC、TCC、事务消息（半消息）

## 核心论点与分论点

### 核心论点

本地事务之所以“确定”，依赖数据库 WAL 等机制提供的可回滚/可恢复保障；但一旦业务跨越进程，网络分区与抖动使调用结果出现“未知态”，本地事务的原子性边界被打穿。分布式事务的主流解法本质是在一致性、可用性、吞吐之间做取舍：要强一致就引入协调与锁（2PC），要高并发就把控制上移到业务并做预留/补偿（TCC），要最终一致就用消息中间件把“业务执行与消息发送”绑定起来（事务消息）。

### 分论点 1：本地事务 vs 分布式事务：确定性来自数据库，引入网络后变成不确定

- 本地事务依赖数据库引擎的日志/回滚机制，提交/回滚的结果可被物理保障
- 跨进程后不可控点来自网络：
  - 请求可能未达
  - 请求已执行但响应丢失
  - 节点宕机/网络抖动导致调用方无法判断结果
- 这会把 RPC 结果变成三态：成功/失败/未知（最难处理的是未知）

### 分论点 2：2PC（两阶段提交）：靠协调者统一管控全局状态，换强一致但吞吐会被锁住

- 机制：引入协调者广播指令，要求参与者锁定资源并持久化日志，但先不提交
- 代价（视频强调的核心痛点）：
  - 同步阻塞：等待协调者最终指令
  - 资源独占：数据库连接/行锁被长时间持有，对其他请求不可见
  - 吞吐下降：节点越多，链路越长，资源占用越重
  - 风险：协调者单点问题会放大故障影响

### 分论点 3：TCC：把“锁与协调”从数据库层搬到业务层，用预留/确认/取消换高并发

- 核心思想：业务层做资源隔离与预留，避免数据库层长时间锁住
- 常见做法（视频举例）：
  - Try：不直接扣主库存，把数量转到“冻结字段”（资源预留）
  - Confirm：基于预留结果直接提交（完成扣减/结算）
  - Cancel：失败时反向补偿（把冻结资源回滚）
- 优点：无锁/高并发友好
- 代价：业务侵入性强，且要处理复杂异常
  - 空回滚
  - 悬挂等边界问题

### 分论点 4：事务消息（半消息）：用 MQ 的可靠投递 + 回查实现最终一致

- 适用：非实时强一致，允许一定延迟（最终一致）
- 机制链路（视频以 RocketMQ 为例）：
  - 先投递 half message（持久化但不可消费），用于探测中间件可用性
  - half message 成功后执行本地业务（如下单落库）
  - 本地事务提交后发送 commit 指令，消息变成可消费状态，下游订阅处理并可重试
  - 若 commit 指令丢失，MQ 通过“回查机制”查询上游事务状态表，自动修复 half message 状态
- 价值：解决“业务执行成功但消息没发出去/发出去但业务没成功”的一致性难题

### 分论点 5：选型原则：强一致/高并发/可解耦最终一致分别对应不同方案

- 强一致（金融账务等）：更偏 CP 取向，愿意牺牲性能换确定性（如 2PC 类）
- 高并发且 RT 敏感（秒杀/网关链路）：用 TCC 通过业务层精细控制换性能
- 可容忍延迟的跨域通知类：事务消息优先，保证可用性与解耦，并用重试/回查收敛一致

## 关键数据、案例与结论（逐条可复述）

- 关键概念：RPC 三态（成功/失败/未知），未知态是分布式事务难点根源
- 典型案例：
  - 2PC：参与节点锁资源等待最终提交 -> 吞吐显著下降
  - TCC：库存通过“冻结字段”实现预留 -> 高并发但业务复杂度上升
  - 事务消息：half message + commit + 回查 -> 保障最终一致
- 结论：不存在“万能分布式事务”，按一致性等级与业务实时性选最小成本方案
