# 单表最多数据量需要分表？

[单表最多数据量需要分表？](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=22)

## 讨论视角（这期在解决什么）

- 以“单表多大需要分表”的面试题为切入点，反对用单一“行数阈值”当标准答案
- 从存储结构出发（InnoDB B+Tree、页大小、行大小）推导单表容量与性能边界，给出更合理的工程判断口径

## 核心论点与分论点

### 核心论点

“单表多少行要分表”没有统一标准，必须结合行大小、索引结构（B+Tree 层高）、磁盘 IO 次数、机器资源与运维成本（备份恢复）综合判断；单纯抛开容量只谈行数是不成立的。

### 分论点 1：为什么“2000 万行必须分表”不可靠

- 网络常见说法：到 2000 万就需要分表
- 反例：同样 2000 万行，行宽 100B 与 1KB 的表，索引层高与 IO 成本完全不同

### 分论点 2：阿里开发规约的口径更偏工程化

- 视频提到的规约建议：单表超过 500 万行或单表容量超过 2GB，推荐分表
- 解释：行数阈值之外还引入了容量阈值，更贴近真实资源约束

### 分论点 3：用 B+Tree（三层）推导“理论可承载行数”

- 基本前提
  - B+Tree 由磁盘页组成
  - 页大小默认 16KB（可通过 SQL 查询）
  - 每页有页头/页尾等开销，视频按约 200B 估算
  - 有效载荷 ≈ 16KB - 200B = 16184B
- 叶子页能放多少行
  - 需要先估算“一行占多少字节”
  - 示例表：4 个 int（4B×4=16B）+ varchar(20)
    - 若字符集 UTF8MB4：varchar(20)≈ 4×20 + 2（额外开销）= 82B
  - 合计行长约 100B
  - 单页可存行数 ≈ 16184 / 100 ≈ 161 行
- 非叶子页能放多少索引项
  - 非叶子页存索引 key + 指针
  - 示例：int key 4B + 指针 6B = 10B/项
  - 单页可放索引项 ≈ 16184 / 10 ≈ 1618 项
- 三层树的数量级结果（视频推导口径）
  - 三层 B+Tree 在行宽约 100B 时，理论可达“4 亿+”级别行数
  - 若行宽约 1KB，则数量级降到约 4200 万左右（仍高于 2000 万）
- 结论：行宽不同，单表可承载行数差距巨大，因此“2000 万”只能算经验参考

### 分论点 4：工程建议的落点

- 三层 B+Tree 是较理想的范围：定位数据大致需要 3 次磁盘 IO
- 现实约束还包括：
  - 磁盘形态（机械盘 vs SSD）
  - 备份/恢复的时间与风险（单表过大不利于运维）
- 视频给出的经验判断：约 1000 万行左右是相对理想的工程数量级（不是硬阈值）

## 关键数据、案例与结论（逐条可复述）

- 页大小：16KB（默认）
- 页有效载荷估算：16184B（16KB-200B）
- 示例行宽估算：≈100B（4×int + varchar(20) UTF8MB4）
- 三层 B+Tree 规模（视频口径）：
  - 行宽 100B：4 亿+ 行
  - 行宽 1KB：约 4200 万行
- 阿里规约口径（视频提到）：500 万行或 2GB 推荐分表
- 最终结论：分表触发点应按“行宽/索引层高/IO/运维成本”来定，行数只能作为参考
