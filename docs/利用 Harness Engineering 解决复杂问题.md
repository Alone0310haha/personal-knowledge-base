## 讨论视角（作者从什么角度切入）

作者从“AI 编程在真实大型代码库（brownfield）里容易产生返工与 slop（低质量/不贴合的代码）”的生产痛点切入，把核心问题归结为：不是模型不够强，而是上下文与过程缺乏工程化控制。作者将解决方案放在 Harness/Context Engineering 的实践层面，强调通过有意识的上下文管理与工作流设计，让现有模型在复杂代码库里也能稳定产出、减少返工，并维持团队对系统演进的共同理解（mental alignment）。

## 核心论点与分论点

### 核心论点

要让 AI 在复杂代码库里“持续做对”，关键不是反复纠正模型，而是工程化地管理上下文与流程：频繁、有意地压缩（intentional compaction），配合“研究→计划→实现”（Research / Plan / Implement）工作流与可验证反馈，让模型始终尽量工作在“smart zone”，避免在上下文过载的“dumb zone”里越写越偏、越修越乱。

### 分论点 1：AI 编程的主要成本常来自返工，而非首轮生成

- 大量团队的真实体验是：AI 写完后需要不断“告诉它哪里错了→重导→再错→再重导”，在复杂任务/老代码库中尤其明显，最终返工吞噬收益。
- 绿色田野（greenfield）或小型项目更容易成功；十年老仓库/成熟系统更容易出问题。

### 分论点 2：上下文窗口不是越大越好，存在“dumb zone”

- 模型并非“纯函数”，输出强依赖对话历史与上下文质量。
- 上下文里最致命的是“错误信息”，其次是“缺失信息”，再次才是“噪声太多”。
- 当上下文占用过高（作者以 Claude Code 为例讨论 smart/dumb 的分界，给出约 40% 作为经验阈值），会出现明显收益递减甚至质量变差；过多 MCP/工具回传的 JSON、日志噪声会加速进入 dumb zone。

### 分论点 3：Intentional Compaction：把对话压成“可审阅的真相快照”

作者提出用“有意压缩”替代无止尽对话：

- 将当前上下文压缩为一份 Markdown（包含正在改的文件、关键行号、必要的测试/构建输出、当前结论与约束）。
- 人类先审阅/修正压缩结果，再用这份“压缩后的、可控的上下文”开启新的上下文窗口继续推进。
- 这样能避免模型在长对话里被自己和工具输出淹没，减少偏航，并提升后续每一步 tool calling 的正确率。

### 分论点 4：Subagents 的价值不是“角色扮演”，而是“控上下文”

- 子智能体/分支上下文用于“读与找”：让子智能体去大仓库里搜索、阅读、理解代码流，最后只回传高度浓缩的结果（文件路径、关键片段、结论）给主智能体。
- 主智能体保留干净上下文，直接进入实现与验证，从而节省 token、避免 dumb zone。

### 分论点 5：RPI 工作流（Research / Plan / Implement）= 频繁有意压缩的组织形态

作者强调 RPI 的重点不是缩写，而是三次“压缩与对齐”：

- Research：客观查明系统如何工作（基于代码“取证”），形成研究文档（压缩真相）。
- Plan：把意图写成可执行步骤（压缩意图），并明确每步如何验证（测试/构建/手工验收）。
- Implement：按计划逐步执行，频繁验证；偏航则回到压缩与重开上下文。

## 关键数据、案例与结论

- 研究/结论口径（视频提及）：在成熟大代码库中，AI 生成代码常带来反复修改与返工；复杂任务下 slop 会累积。
- 实战结果（作者口径）：
  - 团队在摸索出方法后，吞吐提升到约 2–3 倍，并且强调“no slop”。
  - 在一个约 30 万行的 Rust 代码库中，用 context engineering 的方法完成复杂修复并得到接受。
  - 另一次高强度场景中，单次工作时段交付约 3.5 万行代码并合入（其中包含一定比例的样板/更新类改动）。
- 关键结论：
  - 不要外包思考（don’t outsource the thinking）：AI 只能放大你已经做过的思考或没做过的思考。
  - 计划/研究的质量是杠杆点：研究里一个误解可能导致整条实现链路“全坏掉”；一条错误计划可能等价于大量错误代码。

## 思维导图（markmap）

```markmap
# Harness/Context Engineering 实战（视频提炼）
## 痛点
### brownfield 大仓库 -> slop + 返工
### “写→纠错→再写→再纠错”耗尽收益
## 关键概念
### smart zone vs dumb zone
#### 上下文越满，效果越差（经验阈值）
#### 错误信息 > 缺失信息 > 噪声
### Intentional Compaction（有意压缩）
#### 压缩成可审阅 Markdown（文件/行号/结论/约束/测试输出）
#### 复核后重开上下文继续做
### Subagents
#### 不是角色扮演
#### 用于读/找/理解 -> 回传高度浓缩结论
## 工作流：RPI
### Research（压缩真相）
#### 基于代码取证，形成研究文档
### Plan（压缩意图）
#### 明确步骤 + 每步验证方式
#### 计划越清晰，执行越可靠（存在甜蜜点）
### Implement（可靠执行）
#### 按计划逐步改 -> 频繁验证 -> 偏航就压缩/重开
## 工程落点
### Onboarding / progressive disclosure
#### 根目录上下文 + 分层上下文（按需加载）
#### 更偏好 on-demand 压缩快照，避免文档陈旧与“谎言”
### Mental alignment
#### 通过研究/计划/提示链路让团队保持一致理解
## 结论
### 现有模型也能在大仓库解决复杂问题
### 核心是上下文管理与流程治理（Harness）
### 文化与流程需要从技术领导层推动
```
