# InnoDB Buffer Pool 工作过程（更新语句视角）

[InnoDB Buffer Pool 工作过程（更新语句视角）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=24)

## 讨论视角（这期在解决什么）

- 从 InnoDB 引擎层视角解释一次“更新语句”的底层路径：Buffer Pool 如何命中/加载页、如何形成脏页、Undo Log 如何参与回滚与隔离
- 重点落在：内存与磁盘一致性、页级缓存、事务相关日志在更新过程中的角色

## 核心论点与分论点

### 核心论点

InnoDB 的更新不是直接改磁盘记录，而是优先在 Buffer Pool 的数据页上完成修改；修改前会写入 Undo Log 以支持回滚与隔离；被修改的数据页在落盘前会处于“脏页”状态，最终通过刷盘把内存与磁盘同步恢复一致。

### 分论点 1：请求从服务层进入引擎层

- 关闭自动提交（进入事务语义），执行更新语句
- 请求经由 MySQL 服务层后，由执行器调用引擎层能力（访问 Buffer Pool 等）

### 分论点 2：Buffer Pool 命中与页级加载

- 先在 Buffer Pool 中查找目标记录所在页是否已缓存（示例：where id=1）
  - 命中：直接在内存页上更新
  - 未命中：去磁盘表空间文件（ibd）读取对应数据页
- 关键点：加载/缓存的粒度是“整页”，不是单条记录
  - 目的：提高相邻数据（周边记录）的后续访问效率（局部性）

### 分论点 3：Undo Log 的作用（回滚 + 隔离）

- 更新前会把“修改前的数据”写入 Undo Log
- 用途：
  - 回滚：更新失败或显式回滚时，可用旧版本数据把 Buffer Pool 中的变更撤回
  - 隔离：为隔离级别相关能力提供基础支撑（旧版本可见性等）

### 分论点 4：脏页与最终一致

- 更新完成后，Buffer Pool 中的数据页成为“脏页”
  - 脏页含义：内存页与磁盘页内容不一致
- 需要把脏页刷回磁盘，才能恢复为“干净页”（内存与磁盘一致）

## 关键数据、案例与结论（逐条可复述）

- 缓存粒度：按“数据页”缓存与加载
- 脏页产生条件：Buffer Pool 页被修改但尚未同步到磁盘
- Undo Log 的核心价值：提供修改前镜像，用于回滚与隔离相关机制
- 结论：更新链路的关键是“页级缓存修改 + 旧值日志 + 脏页刷盘”，这也是 InnoDB 性能与事务能力的基础
