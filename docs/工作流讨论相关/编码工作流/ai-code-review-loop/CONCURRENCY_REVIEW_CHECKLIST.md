# Java 并发代码评审 Checklist（高并发 / 低延迟场景）

目标：面向高并发、低延迟或共享状态敏感场景的并发代码评审标准操作程序，强调正确性、可预测延迟、容量边界与资源泄露防护。

## 使用规则

1. 仅在变更涉及共享可变状态、锁、线程池、异步回调、缓存一致性、并发容器、ThreadLocal 或热点路径时读取。
2. 每一条都要能落到代码位置并给出“是/否”结论。
3. 发现问题时，优先说明竞态、死锁、阻塞、泄露或尾延迟放大的触发路径。
4. 若并发问题同时影响正确性与性能，优先按 blocking issue 处理。

## 1. 并发模型与线程边界

- [ ] 检查每个关键链路的线程拥有者是否明确；状态更新是否被限定在单线程、串行执行器、Actor 或事件循环中。
- [ ] 检查所有跨线程共享的可变状态是否都有唯一的同步策略：线程封闭、不可变、同一把锁保护，或使用原子类 / 并发容器。
- [ ] 检查对外暴露的服务对象、回调对象、单例 Bean 等是否假定会被并发调用；如会并发调用，字段是否满足线程安全策略。
- [ ] 检查关键状态是否存在多处写入点；若同一字段可能被多个线程写入，是否已通过架构约束消除并发写入。
- [ ] 检查是否在关键链路引入不受控并发（并行 stream、默认异步执行器、隐式回调）；如存在，是否明确执行器、背压与容量上限。

## 2. 锁与临界区

- [ ] 检查每个 `synchronized`、`ReentrantLock`、读写锁的保护对象是否明确，并验证所有访问点都使用同一把锁。
- [ ] 检查每个加锁点是否使用 `try/finally` 释放锁；`lock()` 与进入 `try` 之间是否存在可能抛异常的语句。
- [ ] 检查临界区内是否调用外部、可覆写或不受控代码；如存在，是否能移出锁外或通过快照隔离。
- [ ] 检查临界区内是否包含 I/O、阻塞等待、日志、指标上报、数据库或远程调用；如存在，是否应移出锁外或改为异步。
- [ ] 检查是否存在嵌套锁或嵌套临界区；如存在，是否有固定锁顺序并在所有路径一致遵守。
- [ ] 检查是否依赖锁升级、锁降级或锁重入与第三方库交叉使用；如存在，是否存在潜在死锁。
- [ ] 检查锁粒度是否过大；同一把锁保护多个无关资源时，是否可拆分为分段锁、按 key 锁或线程封闭。

## 3. 条件等待与协作

- [ ] 检查是否使用 `Thread.sleep()` 进行等待、节流或协作；如存在，是否改为显式同步原语并可配置超时。
- [ ] 检查是否使用 `wait/notify` 实现条件队列；如存在，是否可替换为并发工具类。
- [ ] 检查任何等待操作是否包含超时与失败路径；超时后的取消、降级、回滚或熔断动作是否明确。
- [ ] 检查是否存在轮询等待共享字段变化的循环；如存在，字段是否具备正确可见性，并优先评估事件驱动替代方案。

## 4. 内存可见性与原子性

- [ ] 检查每一个 `volatile` 字段是否只承担可见性 / 发布 / 有序性职责，而不是错误承担复合原子性。
- [ ] 检查对象发布方式是否安全；禁止在构造器中让 `this` 提前逸出到其他线程可见位置。
- [ ] 检查双重检查锁是否满足前提：实例字段为 `volatile`，且构造过程无外部可见副作用。
- [ ] 检查使用 `Atomic*` 的位置是否真的需要 CAS 语义；仅 `get/set` 时是否可以用 `volatile` 或线程封闭简化。
- [ ] 检查热点计数或累加是否应使用 `LongAdder` 等分段累加方案。
- [ ] 检查是否通过分别读取多个原子字段拼装一致状态；若业务需要一致快照，是否通过锁、不可变快照或版本号保证。

## 5. 并发容器与复合操作

- [ ] 检查对 `ConcurrentHashMap` / `ConcurrentMap` 是否存在 `get` / `containsKey` 后再 `put` / `remove` 的复合逻辑；如存在，是否改用原子方法。
- [ ] 检查是否在 `ConcurrentHashMap.compute*` 的 lambda 中再次访问同一个 map。
- [ ] 检查并发容器是否以过度抽象的类型保存而导致 API 语义被弱化或误用。
- [ ] 检查 `Collections.synchronized*` 容器的迭代、stream、`toArray`、批量操作是否在同一把锁上包裹整个复合操作。
- [ ] 检查线程安全类中是否把非线程安全集合只做不可修改包装后直接返回；如存在，是否应改为锁内复制或快照。

## 6. 线程池、队列与背压

- [ ] 检查是否使用无界线程池或无界队列；如存在，是否补齐容量上限与拒绝策略。
- [ ] 检查每个线程池的核心参数是否显式：core / max、队列类型与容量、拒绝策略、线程命名。
- [ ] 检查拒绝策略是否与业务语义一致，是否存在静默丢任务或默认吞掉异常。
- [ ] 检查是否存在业务线程同步等待线程池结果而把背压传播为阻塞。
- [ ] 检查线程池是否在应用关闭或组件卸载时调用 `shutdown()` / `shutdownNow()` 并等待终止。
- [ ] 检查是否在关键路径使用 `ForkJoinPool.commonPool()`；如存在，是否应显式指定专用执行器。

## 7. ThreadLocal 与上下文污染

- [ ] 检查所有 `ThreadLocal` 是否为 `static final` 且有清理策略；若在线程池线程中写入，任务结束时是否 `remove()`。
- [ ] 检查是否使用 `InheritableThreadLocal` 在异步或线程池场景传播上下文；如存在，是否会导致跨请求污染。
- [ ] 检查是否依赖 `ThreadLocal` 隐含并发模型，却在异步链路中跨线程访问同一上下文。

## 8. 时间、随机与共享工具对象

- [ ] 检查是否在并发场景共享 `DateFormat` 或 `SimpleDateFormat` 实例。
- [ ] 检查是否在并发压测或并行任务中共享 `java.util.Random` 实例；如存在，是否应改为 `ThreadLocalRandom` 或每线程独立实例。

## 9. 可伸缩性与热点识别

- [ ] 检查是否存在全局单点锁、全局队列、全局缓存或串行瓶颈。
- [ ] 检查是否存在频繁写入的热字段，导致 cache line 争用或高频同步。
- [ ] 检查是否在高频路径进行跨线程同步日志或指标上报；如存在，是否应改为异步聚合或批量上报。

## 10. 并发测试与可复现性

- [ ] 检查关键并发组件是否有多线程单测或压测用例覆盖，且具备并发启动协调、断言收集与超时防挂死机制。
- [ ] 检查并发测试是否覆盖最坏后果场景，如重复执行、资金或状态不一致、幂等失效、重复扣减、状态回退等。
- [ ] 检查 `await()`、`join()`、`Future.get()` 的返回值、超时和中断处理是否被正确检查。
