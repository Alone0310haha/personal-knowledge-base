[Lessons from Building Claude Code: How We Use Skills](https://x.com/trq212/status/2033949937936085378)

## 文章讲了什么

作者基于 Anthropic 在 Claude Code 内部大规模使用 Skills 的经验，回答三个问题：

- 什么样的 Skill 值得做（典型类型有哪些）
- 怎么写出“好用且可维护”的 Skill（最佳实践是什么）
- 技能如何在团队内分发、治理与迭代（如何避免技能泛滥）

文中强调：Skill 不是“几段提示词/markdown”，而是一整个可分发的能力包（文件夹），里面可以放说明文档、参考代码、脚本、模板、数据与配置，并可通过配置与 hooks 在会话期间动态改变 Agent 的行为。

## 核心论点

Skill 的关键价值不是把提示词写长，而是把组织的“知识 + 流程 + 验证 + 护栏 + 可复用代码”工程化成可组合、可复用、可持续迭代的能力模块；其中提升质量与稳定性的最重要杠杆是：

- 验证 Skill（把“正确”变成可重复、可证据化的验证流程）
- Gotchas（持续沉淀模型最常犯错的坑点，降低失败率）
- 渐进式披露（用目录与引用文件控制上下文与细节投喂）

## Skill 的高频类型（文章给出的分类）

### 1) 知识/用法类

让 Agent 正确使用某个库/CLI/SDK（尤其是内部工具），提供参考代码片段与常见坑点。

### 2) 验证/测试类

把端到端验证流程技能化（例如用 Playwright 驱动 UI、用 TTY 驱动交互式 CLI），并在每一步做状态断言或产出可回放证据。

### 3) 数据/监控类

对接数据源与监控栈，固化常用查询、仪表盘定位、指标与维度的正确 join 方式，把“找数据”变成可复用流程。

### 4) 工作流自动化类

把重复流程封装成一个命令（周报、standup、创建工单等），并通过落盘历史产物帮助一致性与增量更新。

### 5) 脚手架/模板类

生成符合组织规范的样板工程/handler/migration 等，把重复初始化工作变成稳定产出。

### 6) 质量/Review 类

统一代码风格、测试实践、对抗式 code review 等；更稳的做法是尽量用确定性脚本/工具驱动，并可在 hooks/CI 中自动跑。

### 7) 代码获取/发布/部署类

PR 看护、部署、灰度与回滚、cherry-pick 等，把发布链路标准化、可验证化。

### 8) 排障/调查类

从症状（告警、错误签名、Slack 线程）出发，多工具联动完成排查并产出结构化报告。

### 9) 运维/维护类（含破坏性操作护栏）

例行维护、清理资源、成本调查等，强调“护栏 + 确认 + 分阶段执行”，避免误操作放大事故。

## 如何把文章思想用到实际开发

### 1) 先做“验证 Skill”，把正确性前置

- 把你们的关键路径验收变成可执行脚本：注册/登录/下单/支付/发货/退款等
- 把“验收证据”变成产物：日志、断言结果、截图/录屏、关键 API 响应快照
- 让 Agent 的产出默认带验证，而不是“写完代码就结束”

### 2) 用 Gotchas 把失败模式固化为组织资产

- 每次模型犯错，记录成 Gotchas：错误表现、触发条件、正确做法、反例
- 把 Gotchas 放在 Skill 的高优先级位置，让 Agent 每次调用都先读到
- 随真实踩坑持续追加，而不是一次性写完

### 3) 用“文件夹 + 渐进式披露”管理上下文

- 把主说明写短，把细节拆到 references/（API、示例、枚举值、错误码、CLI 参数等）
- 输出需要模板的，就放 assets/ 模板，避免模型每次重造格式
- 能代码化的就放 scripts/ 或 lib/，让 Agent 只做编排与决策

### 4) 为技能提供“最小必要的配置记忆”

- 把渠道、环境、服务名、默认参数等放到 config.json
- 配置缺失时，引导 Agent 在开跑前补齐输入（必要时用多选问题减少歧义）

### 5) 用 hooks 做“按需启用”的强约束

- 对生产操作启用护栏：阻断 rm -rf、DROP TABLE、强推、危险 kubectl 操作等
- 对调试场景启用冻结：限制只能改指定目录，避免越修越乱
- 只在调用该 Skill 的会话中生效，避免日常开发被强约束打断

## 可落地步骤（建议按这个顺序做）

### Step 1：选一个明确的 Skill 类型

目标是让 Skill 只解决一类问题：先从验证/测试类或知识/用法类入手，收益最大、边界最清晰。

### Step 2：搭目录结构，先支持渐进式披露

- README.md：触发条件、输入输出、执行原则
- references/：API/命令/示例/枚举/错误码
- scripts/：验证/查询/生成/驱动器
- assets/：输出模板（报告、standup、PR 模板、复盘模板）

### Step 3：先写 Gotchas，再写正文

- 把你们最常见的 5–10 个坑点写清楚
- 每条坑点都给“不要做什么 + 正确做法 + 可验证的例子”

### Step 4：把重复动作代码化

- 把“查数据/跑验证/生成报告”的共性部分做成脚本或小库
- 让 Skill 调用脚本产出结构化结果，主提示只负责解释与组装

### Step 5：加配置与护栏（可选但建议）

- 对需要记忆的环境信息做 config
- 对高风险动作做 hooks

### Step 6：发布与迭代

- 先在团队内小范围试用，收集失败样例
- 用“失败样例 -> Gotchas/脚本/模板”的方式迭代
- 监控使用率，淘汰低价值或重复 Skill，避免技能膨胀
