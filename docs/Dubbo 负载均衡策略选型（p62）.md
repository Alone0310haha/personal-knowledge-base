# Dubbo 负载均衡策略选型（p62）

[Dubbo 负载均衡策略选型（p62）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=62)

## 讨论视角（这期在解决什么）

- 从微服务系统“最致命瓶颈不是代码，而是流量分发不均”切入，把负载均衡当作系统韧性最后一道闸门
- 以调用链路视角讲清 Dubbo 负载均衡的生效位置：服务发现拿全量节点 -> 路由过滤 -> LB 选一个最优节点
- 输出一个可落地的选型与配置原则：默认稳健、按接口精细化调优，有状态场景用一致性哈希

## 核心论点与分论点

### 核心论点

Dubbo 的负载均衡策略直接决定流量是否会在集群内“洪水式冲击某个节点”，从而引发级联超时与集群雪崩。没有特殊诉求时，加权随机（含预热）是最稳健的默认选择；若希望在小流量下也严格按比例分配，用平滑加权轮询；若要动态避开“变慢的机器”，用最小活跃数；若业务是有状态（本地缓存/会话粘性），用一致性哈希并配合虚拟节点降低节点上下线的震荡。

### 分论点 1：Dubbo 负载均衡在链路中的精确位置（先路由后负载）

- 请求发起后先经过服务目录获取所有可用节点
- 再通过路由规则剔除不符合条件的节点
- 最后在候选健康节点中，由负载均衡策略选出目标节点（最后一棒）

### 分论点 2：加权随机（默认稳健）：概率上按权重分流 + 预热保护冷启动

- 通过概率统计让流量在大样本下与权重成正比
- 内置“预热”：新上线机器权重逐步爬坡，避免缓存未热/状态未稳就被大流量打死
- 典型适用：无特殊约束、追求整体稳健的通用场景
- 局限：小流量下随机波动更明显，可能出现短时间“碰撞不均”

### 分论点 3：平滑加权轮询：在时间轴上把高权重请求打散，避免突刺

- 目标：在机器配置参差不齐、流量较小但仍希望按比例分配时，让分配更“可控”
- “平滑”特征：不会连续把大量请求打到同一台高权重机器，降低瞬时突刺
- 典型适用：需要更严格比例与更平滑负载曲线的场景

### 分论点 4：最小活跃数：感知实时繁忙度，把新请求发给“最空闲”的节点

- 共同盲区：随机/轮询都不知道节点当前是否在 GC/网络抖动/慢调用中“卡住”
- 最小活跃数通过感知各节点积压请求数（活跃数），把新请求给当前最空闲的节点
- 价值：自适应隔离变慢节点，提高整体吞吐与稳定性，减少超时扩散

### 分论点 5：一致性哈希：解决有状态服务的“粘性”与缓存命中率问题

- 无状态服务：请求发给谁都一样
- 有状态服务（依赖本地缓存/需要会话粘性）：随机分发会导致缓存命中率归零或会话不一致
- 一致性哈希保证相同参数的请求固定落到同一台机器
- 虚拟节点机制用于降低单点下线带来的流量震荡：节点下线时流量更均匀地分散到其他节点

### 分论点 6：落地配置原则（金字塔）：方法级优先于接口级，消费端优先于服务端

- 配置优先级遵循金字塔：方法级 > 接口级；消费端 > 服务端
- 建议：
  - 服务端配置通用策略作兜底
  - 消费端对核心高频接口做精细化调优
  - 内置策略不够时，可基于 Dubbo SPI 扩展自定义负载均衡逻辑

## 关键数据、案例与结论（逐条可复述）

- 案例 1：策略失误导致流量集中到某节点 -> 超时扩散 -> 集群雪崩
- 案例 2：新机器上线缓存未热 -> 若无预热保护，容易被大流量瞬间打死
- 案例 3：节点因 GC/网络抖动变慢 -> 随机/轮询仍持续压流量 -> 超时加剧；最小活跃数可自动避开
- 结论：先按“是否有状态 + 是否需要动态避慢 + 是否需要严格比例”三问选策略，再按“消费端方法级”做重点接口调优
