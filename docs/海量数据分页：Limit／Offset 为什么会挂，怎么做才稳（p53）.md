# 海量数据分页：Limit／Offset 为什么会挂，怎么做才稳（p53）

[海量数据分页：Limit／Offset 为什么会挂，怎么做才稳（p53）](https://www.bilibili.com/video/BV1qb6EB2EGi?spm_id_from=333.788.videopod.episodes&vd_source=c7bea6fb9f92988f548051c4ed7de1e7&p=53)

## 讨论视角（这期在解决什么）

- 以面试高频题“海量数据分页怎么查（第 100 万页）”切入，强调这是存储/IO/内存模型与数据库执行机制的综合题，而不是背 SQL 语法
- 对比三种方案在不同数据规模与业务形态下的边界：逻辑分页（应用层截断）→ 物理分页（Limit/Offset）→ 延迟关联（覆盖索引 + 回表）→ 游标分页（Keyset/Cursor）
- 输出一个架构选型结论：后台管理系统可接受页码跳转；C 端大流量场景应抛弃页码，改用游标驱动保持稳定

## 核心论点与分论点

### 核心论点

分页的关键不是“能不能写出 Limit/Offset”，而是“让数据库只返回你真正需要的一小段数据，并避免深分页带来的扫描与回表随机 IO”。当页码很深时，Limit/Offset 会退化为大量无效扫描与丢弃；更合理的策略是按场景选择：中等规模且必须跳页用延迟关联降低回表成本；海量数据与高并发场景改用游标分页（keyset/cursor），把复杂度从与总量相关的 O(N) 降到与返回条数相关的 O(1)。

### 分论点 1：逻辑分页（应用层截断）在海量数据下会直接把系统打崩

- 误区：以为“分页在数据库做”，实际可能发生在应用层（先全量查询再丢弃）
- 结果链路：
  - 全量查询触发全表扫描
  - 数据洪峰占满网络 IO
  - 大量对象反序列化进入堆内存，99.9% 最终会被丢弃
  - 内存被冲爆，触发 OOM

### 分论点 2：物理分页（Limit/Offset）只能解决浅分页，深分页会退化为“大量扫描 + 大量丢弃”

- 深分页的本质：即使只取 10 条，数据库也可能需要先遍历/扫描前面海量行才能定位到 offset
- 资源浪费：扫描一百万行，返回 10 行，绝大多数工作是无效的
- 若索引不覆盖：还会对大量行回表，触发海量随机 IO，导致耗时飙升（视频举例到几十秒量级）

### 分论点 3：折中优化：延迟关联（先用覆盖索引拿 ID，再回表取数据）

- 思路：先在索引上做子查询只取 ID（尽量走覆盖索引），再基于 ID 回表取目标行
- 价值：显著减少回表与随机 IO，属于“以空间换时间”的索引策略
- 适用：数据量较大且需要“跳页”的中等规模业务（例如百万级列表）

### 分论点 4：终局方案：游标分页（Keyset/Cursor）抛弃页码，做到与总量解耦

- 抛弃“第几页”这种人类概念，改用“从某个有序键之后继续取”（如 id/时间戳）
- 数据库可以利用 B+Tree 的有序性直接 seek 到位置，再顺序读取少量数据
- 复杂度与总量解耦：数据再大，单次查询仍可保持毫秒级稳定（视频给出的结论）

## 关键数据、案例与结论（逐条可复述）

- 案例 1：深分页（第 100 万页）用 Limit/Offset
  - 需要扫描/遍历前面大量行并丢弃，造成巨大 CPU/IO 浪费
  - 若回表严重，会引发海量随机 IO，查询耗时可飙升到几十秒
- 案例 2：延迟关联
  - 先用覆盖索引取 ID，再回表取数据，显著降低随机 IO
- 选型结论（按场景）
  - 后台管理系统：数据量相对可控，页码跳转可接受，Limit/Offset 开发成本最低
  - 百万级且必须跳页：延迟关联做优化收口
  - C 端无限流/高并发/亿级数据：游标分页保证稳定性
