    第 3 课：配置文件详解¶

    学习时长：3-4 小时
    难度等级：⭐⭐ 基础
    先修要求：完成第 2 课 - 安装与运行

    学习目标¶

    完成本课程后，你将能够：

    • ✅ 掌握 Prometheus 配置文件的 YAML 结构与语法
    • ✅ 理解全局配置（global）的各项参数含义
    • ✅ 配置抓取任务（scrape_configs）监控不同目标
    • ✅ 配置规则文件（rule_files）加载告警和记录规则
    • ✅ 使用配置验证和热重载功能
    • ✅ 实践配置 Prometheus 监控自身并使用 SIGHUP 热重载

    3.1 配置文件概述¶

    3.1.1 配置文件的位置与格式¶

    Prometheus 的配置文件默认名为 prometheus.yml，采用 YAML 格式。

    YAML 基础语法： 

    # 注释使用 # 开头
key: value              # 键值对
nested:
  child_key: value      # 嵌套结构
list:
  - item1               # 列表项
  - item2
  - item3

    配置文件位置： - Docker 容器内：/etc/prometheus/prometheus.yml - 二进制安装：安装目录下的 prometheus.yml - 自定义路径：通过 --config.file 参数指定

    3.1.2 配置文件的顶层结构¶

    一个完整的 Prometheus 配置文件包含以下顶层部分：

    # 1. 全局配置
global:
  scrape_interval: 15s
  evaluation_interval: 15s

# 2. 规则文件
rule_files:
  - "rules/*.yml"

# 3. 告警配置
alerting:
  alertmanagers:
    - static_configs:
        - targets:
          - alertmanager:9093

# 4. 抓取配置
scrape_configs:
  - job_name: "prometheus"
    static_configs:
      - targets: ["localhost:9090"]

    各部分的作用： - global：定义全局默认参数 - rule_files：指定告警和记录规则文件 - alerting：配置 Alertmanager 地址 - scrape_configs：定义要抓取的目标

    3.2 全局配置（global）¶

    3.2.1 核心参数详解¶

    global:
  # 抓取间隔 - 默认值，可被 scrape_configs 中的配置覆盖
  scrape_interval: 15s

  # 抓取超时时间 - 必须小于 scrape_interval
  scrape_timeout: 10s

  # 规则评估间隔
  evaluation_interval: 15s

  # 外部标签 - 添加到所有采集的指标和告警中
  external_labels:
    monitor: "prometheus-server"
    environment: "production"
    region: "cn-east-1"

    3.2.2 参数说明¶

    scrape_interval（抓取间隔）： - 含义：Prometheus 多久抓取一次目标 - 默认值：1m（1 分钟） - 推荐值：15s - 30s - 影响： - 间隔越短 → 数据越精细 → 存储压力越大 - 间隔越长 → 数据越粗糙 → 可能错过瞬时异常

    scrape_timeout（抓取超时）： - 含义：等待目标响应的最长时间 - 默认值：等于 scrape_interval - 推荐值：比 scrape_interval 少 5-10 秒 - 示例：scrape_interval: 15s → scrape_timeout: 10s

    evaluation_interval（评估间隔）： - 含义：多久评估一次告警和记录规则 - 默认值：1m - 推荐值：与 scrape_interval 保持一致 - 影响： - 评估越频繁 → 告警响应越快 → CPU 消耗越大

    通俗理解 scrape_interval vs evaluation_interval：

    想象你是一个**班级管理员**，有两个任务：

    参数 职责 比喻 频率建议
    scrape_interval 收集数据 每 15 分钟去教室收一次作业 15-30 秒
    evaluation_interval 检查规则 每 15 分钟检查是否有人不及格 与 scrape_interval 相同

    执行流程示例：

    假设你定义了告警规则：CPU 使用率 > 80% 持续 5 分钟则告警

    时间线：
00:00  → scrape: 收集到 cpu_usage = 75%
         ↓
         evaluate: 检查 75 > 80？否 → 不告警

00:15  → scrape: 收集到 cpu_usage = 82%
         ↓
         evaluate: 检查 82 > 80？是 → 开始计时（第 1 次）

00:30  → scrape: 收集到 cpu_usage = 85%
         ↓
         evaluate: 检查 85 > 80？是 → 继续计时（第 2 次）

... 持续 5 分钟后 ...

05:00  → evaluate: 已经持续 5 分钟都 > 80%
         → 🔥 触发告警！发送通知

    为什么推荐保持一致？

    • ❌ evaluation_interval 更短（浪费）：新数据还没收集，检查了也用旧数据，白忙活
    • ❌ evaluation_interval 更长（延迟）：00:15 收集到的异常，要等到下次评估才发现，告警延迟
    • ✅ 两者相等：每次收集到新数据后立即检查，既及时又不浪费

    external_labels（外部标签）： - 含义：添加到所有指标和告警的标签 - 用途： - 标识数据来源（哪个 Prometheus 实例） - 多 Prometheus 实例联邦查询时区分数据 - 远程存储时标识租户

    实战示例： 

    global:
  # 生产环境推荐配置
  scrape_interval: 15s
  scrape_timeout: 10s
  evaluation_interval: 15s

  # 标识此 Prometheus 实例
  external_labels:
    cluster: "k8s-prod-01"
    prometheus: "prometheus-server-1"
    team: "platform"

    3.2.3 时间格式¶

    Prometheus 支持多种时间格式：

    # 秒
scrape_timeout: 10s

# 分钟
scrape_interval: 2m

# 小时
retention_time: 15d

# 组合使用
evaluation_interval: 1m30s  # 1 分 30 秒

    3.3 抓取配置（scrape_configs）¶

    3.3.1 基础抓取配置¶

    监控 Prometheus 自身： 

    scrape_configs:
  - job_name: "prometheus"
    static_configs:
      - targets: ["localhost:9090"]
        labels:
          env: "production"

    配置说明： - job_name：作业名称，会作为 job 标签添加到指标中 - static_configs：静态目标配置 - targets：目标地址列表（host:port 格式） - labels：添加到该目标的额外标签

    3.3.2 多目标抓取¶

    监控多个目标： 

    scrape_configs:
  - job_name: "node-exporters"
    static_configs:
      - targets:
          - "node1:9100"
          - "node2:9100"
          - "node3:9100"
        labels:
          group: "web-servers"

      - targets:
          - "node4:9100"
          - "node5:9100"
        labels:
          group: "db-servers"

    Prometheus 会自动添加的标签： - job: "node-exporters" - instance: "node1:9100" - group: "web-servers"（自定义）

    3.3.3 覆盖全局配置¶

    为特定 Job 设置不同的抓取间隔： 

    global:
  scrape_interval: 15s

scrape_configs:
  # 使用全局配置（15s）
  - job_name: "prometheus"
    static_configs:
      - targets: ["localhost:9090"]

  # 覆盖为 30s
  - job_name: "slow-app"
    scrape_interval: 30s
    static_configs:
      - targets: ["app1:8080", "app2:8080"]

  # 覆盖为 5s（高频监控）
  - job_name: "critical-service"
    scrape_interval: 5s
    scrape_timeout: 3s
    static_configs:
      - targets: ["critical-api:8080"]

    3.3.4 指标路径与方案¶

    自定义指标端点： 

    scrape_configs:
  - job_name: "spring-boot-app"
    metrics_path: "/actuator/prometheus"  # 默认是 /metrics
    scheme: https                         # 默认是 http
    static_configs:
      - targets: ["app.example.com:8443"]

    常用指标路径： - /metrics：Prometheus 默认 - /actuator/prometheus：Spring Boot Actuator - /metrics.php：PHP 应用 - /_prometheus/metrics：自定义路径

    3.4 服务发现配置¶

    3.4.1 文件服务发现（file_sd）¶

    动态更新目标列表： 

    scrape_configs:
  - job_name: "dynamic-targets"
    file_sd_configs:
      - files:
          - "targets/*.json"
          - "targets/*.yml"
        refresh_interval: 30s  # 每 30 秒检查文件变化

    JSON 格式目标文件（targets/web-servers.json）： 

    [
  {
    "targets": ["web1:9100", "web2:9100"],
    "labels": {
      "env": "production",
      "type": "web"
    }
  },
  {
    "targets": ["db1:9100"],
    "labels": {
      "env": "production",
      "type": "database"
    }
  }
]

    YAML 格式目标文件（targets/apps.yml）： 

    - targets:
    - "app1:8080"
    - "app2:8080"
  labels:
    env: "staging"
    team: "backend"

    优势： - ✅ 无需重启 Prometheus - ✅ 可与其他自动化工具集成 - ✅ 支持多个文件

    3.4.2 Kubernetes 服务发现¶

    自动发现 Pod： 

    scrape_configs:
  - job_name: "kubernetes-pods"
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      # 只抓取带有 prometheus.io/scrape=true 注解的 Pod
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true

      # 使用注解中的端口
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_port]
        action: replace
        target_label: __meta_prometheus_pod_port
        regex: (.+)

      # 使用注解中的路径
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_path]
        action: replace
        target_label: __metrics_path__
        regex: (.+)

    支持的角色： - pod：发现 Pod - service：发现 Service - endpoints：发现 Endpoints - node：发现 Node - ingress：发现 Ingress

    3.4.3 其他服务发现¶

    Consul 服务发现： 

    scrape_configs:
  - job_name: "consul-services"
    consul_sd_configs:
      - server: "consul.example.com:8500"
        services:
          - "web-service"
          - "api-service"

    DNS 服务发现： 

    scrape_configs:
  - job_name: "dns-sd"
    dns_sd_configs:
      - names:
          - "web.service.consul"
        type: "A"
        port: 9100

    3.5 重标签配置（Relabeling）¶

    3.5.1 重标签的作用¶

    核心理解：Relabeling 是在**调用 HTTP 请求获取指标数据之前**执行的操作，用于对目标进行"预处理"。

    通俗比喻：想象你是一个**快递分拣中心**的主管：

    包裹从各地运来 → 分拣中心处理 → 送到最终目的地
     ↓              ↓                ↓
  目标服务     Relabeling      Prometheus 存储

    分拣中心的工作（Relabeling）包括： - 🏷️ 贴标签：给包裹贴上"北区"、"易碎"等标签 → 添加/修改标签 - ✅ 筛选：只接收重量>1kg 的包裹 → keep（保留） - ❌ 拒收：不接收寄往"测试小区"的包裹 → drop（丢弃） - 📦 改地址：某些小区改送到快递柜 → 修改 metrics_path

    执行时机（非常重要）：

    完整流程：
1. 服务发现          2. Relabeling      3. 发起 HTTP 请求    4. 存储指标
   ↓                   ↓                    ↓                    ↓
从配置文件/          处理目标标签         访问目标 URL          保存到 TSDB
K8s/Consul 等        (抓取前预处理)       /metrics             
获取目标列表                                                     

└──────────────┘    └──────────────┘     └──────────────┘    └──────────────┘
     ↓                    ↓                     ↓                  ↓
  发现目标            过滤/添加/           真正调用           带标签的
  (可能有            修改标签              HTTP               时间序列
  __address__                                                  
  等元数据)

⚠️ 关键点：Relabeling 执行时，还没有发起 HTTP 请求，看不到任何指标数据！

    使用场景： - 🏷️ 添加/删除标签：给目标添加团队、环境等标签 - ✅❌ 根据标签过滤目标：只监控生产环境，丢弃测试环境 - 🔧 修改抓取配置：动态修改 metrics_path、scheme、端口等 - 📊 从标签值构建新的标签：从地址中提取端口号、服务名

    3.5.2 重标签配置语法¶

    基础语法：

    relabel_configs:
  - source_labels: [<label1>, <label2>, ...]  # 从哪些标签读取值
    separator: ";"        # 多个值怎么拼接（默认 ";"）
    regex: "<regex>"      # 匹配什么模式（默认 "(.*)"）
    target_label: <label> # 结果写到哪个标签
    replacement: "$1"     # 怎么写入（默认 "$1"）
    action: replace       # 做什么操作（默认 "replace"）

    执行流程：

    1. 读取 source_labels 的值
         ↓
2. 用 separator 拼接成一个字符串
         ↓
3. 用 regex 进行匹配（可以捕获分组）
         ↓
4. 用 replacement 写入 target_label

    示例：从地址中提取端口号

    - source_labels: [__address__]
  action: replace
  target_label: port
  regex: ".*:(.*)"      # 匹配 "host:port" 中的 port 部分
  replacement: "$1"     # 使用第一个捕获组

    输入：__address__ = "app-server-01:8080"
         ↓ 应用 regex: ".*:(.*)"
         ↓ replacement: "$1"
输出：port = "8080"

    3.5.3 常用操作（action）¶

    用**快递分拣**的比喻来理解 5 种 action：

    Action 通俗理解 快递分拣比喻 典型场景
    replace 贴标签/改标签 给包裹贴上"北区"标签 添加团队、环境标签
    keep 只保留符合条件的 只接收重量>1kg 的包裹 只监控生产环境
    drop 拒收/丢弃 拒收寄往"测试小区"的包裹 丢弃测试环境
    labelmap 批量复制标签 把所有"xx 小区"标签复制为"配送区域=xx" 批量映射 Kubernetes 标签
    labeldrop 批量删除标签 撕掉所有"临时"标签 清理临时标签

    1. replace（替换/添加标签）

    通俗理解：给目标"贴标签"或"改标签"

    # 示例 1：添加固定值的标签
- action: replace
  target_label: team
  replacement: "backend-team"
# 效果：给所有目标添加标签 team="backend-team"

# 示例 2：从源标签提取值
- source_labels: [__address__]
  action: replace
  target_label: port
  regex: ".*:(.*)"
  replacement: "$1"
# 效果：从 "app:8080" 中提取 port="8080"

# 示例 3：修改抓取地址
- source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_port]
  action: replace
  target_label: __address__
  regex: (.+)
  replacement: ${1}
# 效果：使用注解中的端口作为抓取地址

    2. keep（保留）

    通俗理解：设置"准入规则"，不符合条件的目标直接丢弃，不会发起 HTTP 请求

    # 示例 1：只保留生产环境
- source_labels: [environment]
  action: keep
  regex: "production"

# 示例 2：只保留带有 scrape=true 注解的 Pod
- source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
  action: keep
  regex: true

# 示例 3：只保留特定服务
- source_labels: [__meta_kubernetes_pod_label_app]
  action: keep
  regex: "api|web|worker"

    执行效果： 

    目标列表：
  - app1.prod.com:8080 (env=production)  ✅ 保留
  - app2.prod.com:8080 (env=production)  ✅ 保留
  - test1.test.com:8080 (env=test)       ❌ 丢弃（不会抓取）
  - app3.staging.com:8080 (env=staging)  ❌ 丢弃（不会抓取）

    3. drop（丢弃）

    通俗理解：设置"黑名单"，匹配到的目标直接丢弃，不会发起 HTTP 请求

    # 示例 1：丢弃测试环境
- source_labels: [environment]
  action: drop
  regex: "test.*"

# 示例 2：丢弃命名空间
- source_labels: [__meta_kubernetes_namespace]
  action: drop
  regex: "kube-.*"  # 丢弃 Kubernetes 系统命名空间

# 示例 3：丢弃特定地址
- source_labels: [__address__]
  action: drop
  regex: "localhost:.*"

    keep vs drop 的区别： - keep：**白名单**思维（只保留匹配的） - drop：**黑名单**思维（丢弃匹配的）

    4. labelmap（标签映射）

    通俗理解：批量复制标签，通常用于 Kubernetes 场景

    # 将 Kubernetes Pod 的标签批量复制到最终指标中
- action: labelmap
  regex: __meta_kubernetes_pod_label_(.+)
  replacement: pod_label_$1

    执行效果： 

    原始标签：
  __meta_kubernetes_pod_label_app = "api"
  __meta_kubernetes_pod_label_team = "backend"
  __meta_kubernetes_pod_label_version = "v1.2"
         ↓ 应用 labelmap
结果标签：
  pod_label_app = "api"
  pod_label_team = "backend"
  pod_label_version = "v1.2"

    5. labeldrop（删除标签）

    通俗理解：批量删除不需要的标签，节省存储空间

    # 删除所有临时标签
- action: labeldrop
  regex: "temp_.*"

# 删除 Kubernetes 元数据标签（以 __ 开头）
- action: labeldrop
  regex: "^__.*"

    使用场景： - 清理临时标签 - 删除不再需要的元数据 - 减少存储开销

    3.5.4 实战示例¶

    示例 1：多环境标签映射： 

    scrape_configs:
  - job_name: "microservices"
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      # 根据命名空间添加环境标签
      - source_labels: [__meta_kubernetes_namespace]
        action: replace
        target_label: environment
        regex: "prod-(.*)"
        replacement: "production"

      - source_labels: [__meta_kubernetes_namespace]
        action: replace
        target_label: environment
        regex: "staging-(.*)"
        replacement: "staging"

      # 添加团队标签
      - source_labels: [__meta_kubernetes_pod_label_team]
        action: replace
        target_label: team

    示例 2：动态构建抓取地址： 

    scrape_configs:
  - job_name: "custom-metrics"
    kubernetes_sd_configs:
      - role: service
    relabel_configs:
      # 从注解中获取自定义端口
      - source_labels: [__meta_kubernetes_service_annotation_prometheus_io_port]
        action: replace
        target_label: __address__
        regex: (.+)
        replacement: ${1}

      # 从注解中获取自定义路径
      - source_labels: [__meta_kubernetes_service_annotation_prometheus_io_path]
        action: replace
        target_label: __metrics_path__
        regex: (.+)

    3.6 规则文件配置（rule_files）¶

    3.6.1 配置规则文件¶

    rule_files:
  - "rules/alerts.yml"
  - "rules/recording-rules.yml"

    文件路径规则： - 相对路径：相对于配置文件所在目录 - 绝对路径：以 / 开头的完整路径 - 支持通配符：*.yml、rules/*.yml

    3.6.2 告警规则 vs 记录规则¶

    一句话理解： - 告警规则：🚨 用来发现问题并通知人（像烟雾报警器） - 记录规则：📊 用来预计算复杂查询，提升性能（像提前做好的报表）

    告警规则（Alerting Rules）¶

    目的：发现问题，发送通知

    通俗比喻：告警规则就像**烟雾报警器**

    烟雾传感器持续检测 → 发现烟雾浓度超标 → 触发警报 → 通知你着火了
      ↓                    ↓                   ↓
  监控指标            评估条件            发送通知

    完整示例：

    groups:
  - name: example-alerts
    interval: 30s  # 覆盖全局 evaluation_interval（每 30 秒评估一次）
    rules:
      - alert: HighMemoryUsage
        expr: node_memory_MemUsed_bytes / node_memory_MemTotal_bytes > 0.9
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "高内存使用率"
          description: "实例 {{ $labels.instance }} 内存使用率超过 90%"

    逐字段解释：

    字段 作用 通俗理解 必填
    alert: HighMemoryUsage 告警名称 给这个警报起个名字 ✅
    expr: ... > 0.9 触发条件 什么情况下拉响警报 ✅
    for: 5m 持续时间 条件满足多久才真的报警 ❌
    labels 告警标签 用于分类、过滤、路由 ❌
    annotations 告警描述 报警时显示的详细信息 ❌

    告警状态流转：

    时间线示例（内存使用率变化）：
────────────────────────────────────────────────────────────→

00:00  内存 = 85%
       ↓
       评估：85% > 90%？❌ 否
       ↓
       状态：Inactive（正常）

00:15  内存 = 92%
       ↓
       评估：92% > 90%？✅ 是
       ↓
       状态：Pending（开始计时）

00:30  内存 = 93%
       ↓
       评估：93% > 90%？✅ 是
       ↓
       持续 1 分钟，还没到 5 分钟

... 持续 5 分钟后 ...

05:00  内存 = 91%
       ↓
       评估：91% > 90%？✅ 是
       ↓
       🔥 状态：Firing（触发告警！）
       ↓
       发送通知到 Alertmanager
       ↓
       你收到邮件/短信/钉钉消息

10:00  内存 = 70%
       ↓
       评估：70% > 90%？❌ 否
       ↓
       状态：Resolved（告警恢复）
       ↓
       发送恢复通知

    状态说明：

    ┌─────────────┐
│  Inactive   │ ← 初始状态，条件不满足
│  (正常)     │
└──────┬──────┘
       │
       │ 条件首次满足
       ↓
┌─────────────┐
│   Pending   │ ← 等待中，开始计时
│  (待处理)   │    for 计时器运行中
└──────┬──────┘
       │
       │ 持续时间达到 for 要求
       ↓
┌─────────────┐
│   Firing    │ ← 触发告警，发送通知
│  (触发中)   │
└──────┬──────┘
       │
       │ 条件不再满足
       ↓
┌─────────────┐
│  Resolved   │ ← 告警恢复，发送恢复通知
│  (已恢复)   │
└─────────────┘

    记录规则（Recording Rules）¶

    目的：预计算复杂查询，提升查询和告警性能

    通俗比喻：记录规则就像**提前做好的报表**

    原始数据：每天 10 万条销售记录
    ↓
提前计算：每小时汇总一次销售额
    ↓
保存到报表：sales_hourly_total
    ↓
查询时：直接查报表，不用重新计算

    为什么需要记录规则？

    场景：你有一个复杂的查询，每次都要计算很久：

    # 查询每个 job 的 5 分钟请求速率
sum by (job) (rate(http_requests_total[5m]))

    问题： - ❌ 每次查询都要实时计算 - ❌ 数据量大时很慢（2 秒+） - ❌ 仪表盘加载慢 - ❌ 告警评估也慢

    解决方案：用记录规则**提前计算好**

    groups:
  - name: recording-rules
    rules:
      - record: job:http_requests:rate5m
        expr: sum by (job) (rate(http_requests_total[5m]))

      - record: instance:node_cpu:ratio
        expr: avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m]))

    性能对比：

    没有记录规则时：
────────────────────────────────────────────────────────

用户查询：sum by (job) (rate(http_requests_total[5m]))
              ↓
         实时扫描所有 http_requests_total 数据
              ↓
         实时计算 rate()
              ↓
         实时计算 sum by (job)
              ↓
         返回结果（耗时 2 秒）


有记录规则后：
────────────────────────────────────────────────────────

Prometheus 每 15 秒自动计算一次：
  job:http_requests:rate5m = sum by (job) (rate(http_requests_total[5m]))
              ↓
         把结果保存为新的时间序列

用户查询：job:http_requests:rate5m
              ↓
         直接读取预计算的结果
              ↓
         返回结果（耗时 0.1 秒）⚡ 快 20 倍！

    记录规则的执行流程：

    时间线：
────────────────────────────────────────────────────────────→

每 15 秒（evaluation_interval）自动执行：

00:00  Prometheus 计算：job:http_requests:rate5m = sum by (job) (rate(http_requests_total[5m]))
       ↓
       保存结果：job:http_requests:rate5m{job="api"} = 125.5
       ↓
       保存结果：job:http_requests:rate5m{job="web"} = 342.1

00:15  Prometheus 再次计算（数据更新了）
       ↓
       覆盖旧结果：job:http_requests:rate5m{job="api"} = 130.2
       ↓
       覆盖旧结果：job:http_requests:rate5m{job="web"} = 350.8

... 持续进行 ...

用户查询时：
  直接查询 job:http_requests:rate5m
  ↓
  立即返回预计算的结果（非常快）

    告警规则 vs 记录规则 对比¶

    特性 告警规则 记录规则
    关键字 alert record
    目的 发现问题，发送通知 预计算，提升性能
    输出 告警事件（Inactive/Pending/Firing） 新的时间序列
    必需字段 alert, expr record, expr
    可选字段 for, labels, annotations 无
    命名规范 大驼峰：HighMemoryUsage 小写 + 冒号：job:metric:operation
    给人看还是给机器看 🧑 给人看的（告警通知） 🤖 给查询和告警用的

    配合使用（最佳实践）¶

    最佳实践：记录规则 + 告警规则配合使用

    # 1. 先用记录规则预计算（提升性能）
- record: service:http_errors:ratio5m
  expr: |
    sum by (service) (rate(http_requests_total{status="500"}[5m]))
    /
    sum by (service) (rate(http_requests_total[5m]))

# 2. 再用告警规则监控预计算的结果（简化告警）
- alert: HighErrorRate
  expr: service:http_errors:ratio5m > 0.05
  for: 5m
  annotations:
    summary: "服务 {{ $labels.service }} 错误率过高"

    好处： - ✅ 告警评估更快（不用实时计算复杂表达式） - ✅ 减少重复计算（多个告警可以用同一个记录规则） - ✅ 查询和告警都用同一份预计算数据，保证一致性

    3.8 配置验证与热重载¶

    3.8.1 配置验证（promtool）¶

    Prometheus 提供了 promtool 工具用于验证配置文件：

    # 验证配置文件语法
promtool check config prometheus.yml

# 验证规则文件
promtool check rules rules/alerts.yml

# 运行单元测试
promtool test rules tests.yml

    验证输出示例： 

    $ promtool check config prometheus.yml
Checking prometheus.yml
  SUCCESS: 1 rule files found
 SUCCESS: prometheus.yml is valid prometheus config file syntax

Checking rules/alerts.yml
  SUCCESS: 5 rules found

    3.8.2 热重载配置¶

    方法 1：发送 SIGHUP 信号： 

    # 1. 查找 Prometheus 进程 ID
ps aux | grep prometheus

# 2. 发送 SIGHUP 信号
kill -SIGHUP <pid>

# 或者
killall -SIGHUP prometheus

    方法 2：使用 HTTP API： 

    # 发送 POST 请求到/-/reload 端点
curl -X POST http://localhost:9090/-/reload

    注意：需要启动时添加 --web.enable-lifecycle 参数： 

    prometheus --config.file=prometheus.yml --web.enable-lifecycle

    方法 3：Docker 容器： 

    # 发送 SIGHUP 信号到容器
docker kill -s SIGHUP <container_name>

    3.8.3 配置重载的时机¶

    需要热重载的配置： - ✅ scrape_configs 变更 - ✅ rule_files 变更 - ✅ alerting 配置变更 - ✅ global.scrape_interval 变更 - ✅ global.evaluation_interval 变更

    不需要热重载（自动生效）： - ✅ file_sd_configs 目标文件变化 - ✅ Kubernetes SD 自动发现

    需要重启的配置： - ❌ global.external_labels 变更 - ❌ storage.tsdb.* 存储相关配置 - ❌ web.* Web 服务配置

    3.11 常见问题与排查¶

    3.11.1 配置验证失败¶

    问题：promtool check config 报错

    常见错误： 

    # ❌ 错误 1：缩进不正确
scrape_configs:
- job_name: "prometheus"  # 缩进错误
  static_configs:
  - targets: ["localhost:9090"]

# ✅ 正确
scrape_configs:
  - job_name: "prometheus"
    static_configs:
      - targets: ["localhost:9090"]


    # ❌ 错误 2：时间格式错误
scrape_interval: 15  # 缺少单位

# ✅ 正确
scrape_interval: 15s

    # ❌ 错误 3：正则表达式无效
relabel_configs:
  - regex: "[invalid"  # 未闭合的括号

# ✅ 正确
relabel_configs:
  - regex: "valid-regex"

    3.11.2 目标显示 DOWN¶

    排查步骤：

    步骤 1：检查网络连通性 

    # 测试目标是否可达
curl http://localhost:9090/metrics

# 检查端口是否监听
netstat -tlnp | grep 9090

    步骤 2：查看 Prometheus 日志 

    # 查看日志中的错误
tail -f prometheus.log | grep "scrape failed"

    步骤 3：检查 Targets 页面 

    访问 http://localhost:9090/targets
查看 "Last Error" 列的错误信息

    常见原因： - ❌ 目标服务未启动 - ❌ 防火墙阻止连接 - ❌ metrics_path 配置错误 - ❌ scrape_timeout 太短

    3.11.3 热重载不生效¶

    问题：发送 SIGHUP 后配置未更新

    排查步骤： 

    # 1. 确认 Prometheus 进程存在
ps aux | grep prometheus

# 2. 确认启动时启用了生命周期
ps aux | grep prometheus | grep enable-lifecycle

# 3. 检查重载日志
tail -f prometheus.log | grep "Reloading"

# 4. 手动测试 reload API
curl -X POST http://localhost:9090/-/reload

    解决方案： - 确保启动时添加了 --web.enable-lifecycle 参数 - 检查是否有权限发送信号 - 查看日志确认重载是否成功
