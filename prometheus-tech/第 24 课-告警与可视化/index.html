    第 24 课：告警与可视化¶

    学习时长：2-3 小时
    难度等级：⭐⭐⭐ 进阶
    先修要求：完成第 19-20 课（规则与通知）、第 22 课（业务指标）

    学习目标¶

    完成本课程后，你将能够：

    • ✅ 把“指标 → 面板 → 告警”串成可落地的闭环
    • ✅ 会为 Java 应用做一套最小可用的 Grafana 面板结构
    • ✅ 会写 3 类常用告警：可用性、延迟、错误率（包含 for 与 labels/annotations）
    • ✅ 理解 SLI/SLO 的基本概念，并把 SLO 变成可查询与可告警的表达式
    • ✅ 能用 Prometheus UI 与 Alertmanager 验证告警链路是否打通

    24.1 先定目标：你到底想看什么、告什么¶

    面板与告警不是“越多越好”，核心是回答这三类问题：

    1) 服务是否可用？（Availability） 2) 服务是否变慢？（Latency） 3) 用户是否在失败？（Errors）

    对 Web 服务来说，优先用 RED 指标（Rate/Errors/Duration）构建面板与告警。

    24.2 最小可用面板（Java 服务建议结构）¶

    建议一套面板至少包含 5 个区域：

    1) 总览 - up、版本、实例数、QPS、错误率、P95

    2) HTTP - 按 endpoint/method 的 QPS、错误率、P95/P99

    3) 依赖 - DB/Redis/外部接口的耗时与错误（如果有指标）

    4) JVM - GC、堆内存、线程、CPU（系统健康）

    5) 告警回放 - 最近 1h firing 的告警数量、Top 告警

    直觉：总览让你 30 秒内判断“是不是系统性故障”，分区让你定位“哪里坏了”。

    24.3 PromQL 模板：直接拿来用¶

    以下假设你有第 22 课那类业务指标：

    • order_create_requests_total{result="success|fail",channel="..."}
    • order_create_duration_seconds_bucket{...}

    24.3.1 QPS（吞吐）¶

    sum(rate(order_create_requests_total[5m]))

    24.3.2 错误率（Error Ratio）¶

    sum(rate(order_create_requests_total{result="fail"}[5m]))
/
sum(rate(order_create_requests_total[5m]))

    24.3.3 P95 延迟（Histogram）¶

    histogram_quantile(
  0.95,
  sum(rate(order_create_duration_seconds_bucket[5m])) by (le)
)

    24.4 告警规则设计：三条主线¶

    24.4.1 可用性告警（Availability）¶

    适合“服务挂了/抓不到/实例全灭”这类硬故障。

    - alert: ServiceDown
  expr: sum(up{job="spring-boot"} == 0) > 0
  for: 2m
  labels:
    severity: page
  annotations:
    summary: "服务实例不可用"
    description: "存在至少一个实例抓取失败（up=0）"

    24.4.2 错误率告警（Errors）¶

    - alert: OrderCreateHighErrorRate
  expr: |
    (
      sum(rate(order_create_requests_total{result="fail"}[5m]))
      /
      sum(rate(order_create_requests_total[5m]))
    ) > 0.02
  for: 5m
  labels:
    severity: page
  annotations:
    summary: "下单错误率过高"
    description: "过去 5m 错误率 > 2%，持续 5m"

    24.4.3 延迟告警（Latency）¶

    - alert: OrderCreateHighLatencyP95
  expr: |
    histogram_quantile(
      0.95,
      sum(rate(order_create_duration_seconds_bucket[5m])) by (le)
    ) > 0.5
  for: 10m
  labels:
    severity: page
  annotations:
    summary: "下单延迟过高（P95）"
    description: "P95 > 500ms 持续 10m"

    24.5 for 怎么选：防抖 vs 延迟¶

    for 的意义是“条件持续满足一段时间才报警”，用来抗抖动。

    经验规则：

    • up 类硬故障：for 1-2 分钟即可（避免短暂网络抖动）
    • 错误率：for 5 分钟以上更稳（避免瞬时尖峰）
    • 延迟：for 10 分钟左右更稳（延迟抖动常见）

    24.6 SLI/SLO：把目标变成可告警的数学表达式¶

    24.6.1 定义一个 SLO（示例）¶

    SLO：下单接口 99% 的请求在 500ms 以内。

    如果你用 Histogram，可以把“达标比例”算出来：

    sum(rate(order_create_duration_seconds_bucket{le="0.5"}[5m]))
/
sum(rate(order_create_duration_seconds_count[5m]))

    然后告警“达标比例过低”：

    - alert: OrderCreateSLOViolation
  expr: |
    (
      sum(rate(order_create_duration_seconds_bucket{le="0.5"}[5m]))
      /
      sum(rate(order_create_duration_seconds_count[5m]))
    ) < 0.99
  for: 15m
  labels:
    severity: page
  annotations:
    summary: "下单 SLO 违规"
    description: "500ms 内比例 < 99% 持续 15m"

    24.7 告警验证：你需要跑通的最小闭环¶

    1) Prometheus /rules：规则已加载、无报错
    2) Prometheus /alerts：告警状态是否在 pending/firing
    3) Alertmanager UI：是否收到对应的告警
    4) Grafana 面板：对应指标是否能查到（避免“告警表达式本身就没数据”）

    24.8 常见坑与修正方式¶

    • 告警太吵：提高 for、提高阈值、加入更合理的过滤维度、分级（page/ticket）
    • 告警不触发：先在 Expression Browser 单独跑 expr，确认有数据与阈值正确
    • 面板难看懂：先做总览，再做拆分；统一单位与图例
    • 指标维度爆炸：检查标签设计，必要时用 metric_relabel 删除高基数标签

    课后小结¶

    • 面板回答“发生了什么”，告警回答“需不需要立刻处理”
    • 先用 RED/黄金信号搭面板与告警，再逐步细化
    • 告警规则要带 for、带 labels/annotations，方便路由与定位
    • SLO 的关键是把目标转成“比例/分位数”的 PromQL 表达式
