    MySQL 一条 SELECT 的执行过程¶

    MySQL 一条 SELECT 的执行过程

    讨论视角（这期在解决什么）¶

    • 以“面试速答题”的角度梳理一条 SQL（SELECT）在 MySQL 中的大致执行链路
    • 核心是把请求从客户端一路走到存储引擎（InnoDB）并返回结果的关键环节讲清楚

    核心论点与分论点¶

    核心论点¶

    一条 SELECT 的执行过程可以按“连接与认证 →（可选）查询缓存 → 解析/预处理 → 优化与执行计划 → 执行器调用存储引擎 →（InnoDB）Buffer Pool/磁盘读取 → 返回结果 + 记录日志”的链路来回答。

    分论点 1：连接与认证¶

    • 客户端发起 SQL 后，先与远程数据库建立连接
    • 用户名/密码等认证通过后进入 MySQL 服务层

    分论点 2：查询缓存（历史机制）¶

    • 服务层曾有查询缓存：以 Key-Value 形式存储
    • Key：SQL 语句
    • Value：查询结果
    • 视频口径：MySQL 5.7 默认关闭；MySQL 8.0 已完全移除
    • 原因：数据库数据变更频繁导致缓存命中率低，维护成本高（更适合交给应用层缓存）

    分论点 3：解析器 → 预处理器¶

    • 解析器：解析 SQL，检查语法是否正确
    • 预处理器：校验表名/字段名等是否存在（以及做一些前置检查）

    分论点 4：优化器（生成执行计划）¶

    • 选择更优的执行路径并生成执行计划
    • 示例（视频举例口径）：联合索引需要遵循最左前缀原则，优化器会对条件/顺序做一定程度的优化以更接近可用路径

    分论点 5：执行器 → 存储引擎（InnoDB）¶

    • 执行器根据执行计划，调用存储引擎接口获取数据
    • InnoDB 读取路径：
    • 先查 Buffer Pool（内存缓冲区），命中则直接返回
    • 未命中再落到磁盘读取，读到后再放入 Buffer Pool

    分论点 6：日志与辅助记录¶

    • 在执行过程中会记录多类日志（视频提到如 binlog、redo log 等）

    关键数据、案例与结论（逐条可复述）¶

    • 查询缓存状态（视频口径）：
    • 5.7 默认关闭
    • 8.0 移除
    • Buffer Pool 的意义：用内存命中替代磁盘 IO，提高检索效率
    • 结论：面试回答抓住“服务层 解析/优化/执行”与“InnoDB Buffer Pool 命中/落盘”的两段主链路即可
