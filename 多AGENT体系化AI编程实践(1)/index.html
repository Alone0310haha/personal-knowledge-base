    多AGENT体系化AI编程实践(1)

    • AI Coding提效实践1 更好的编写提示词 多AGENT体系化AI编程实践

    一句话总结¶

    把 AI 编程当作“多 Agent 开发团队”来设计：先把角色、边界、判定标准、输出模板定义清楚，再用依赖/并行的任务拆解把结果落到文档与流程里，而不是停留在对话上下文中。

    核心观点（提炼）¶

    • 体系化 AI 编程面向真实项目（研发/测试/上线），不是“让模型写 demo”。
    • 多 Agent 的正确心智模型是“团队协作”：不同 Agent 承担不同角色与阶段职责，主 Agent 负责汇总、调度、拍板。
    • 提示词有效与否，关键在两点：潜在决策方向是否到位、正确/错误的判定标准是否明确。
    • 只给抽象建议（例如“考虑并发与依赖”）难以落地；把依赖关系与可并行任务写成明确结构，AI 才能执行、团队也能对齐预期。
    • 不要生搬硬套公开框架；应把框架当成“可借鉴的仓库”，结合自己团队的流程与习惯沉淀出自有版本。
    • 重要信息必须落盘（例如 tasks.md 的依赖与执行计划区块），否则会随着上下文被稀释或遗忘。

    方法框架：定义一个“Agent 角色卡”¶

    • 人格/身份：它是谁、在哪个阶段介入（例如“研发负责人/方案评审阶段”）。
    • 风格参考：参考某个公开人物/业内大佬/团队同事（可让模型补全隐含习惯与表达方式）。
    • 做什么 / 不做什么：明确边界（例如评审阶段不直接改代码，只做审查与建议）。
    • 对错标准：在什么条件下算对、算错（尽量可检验、可执行）。
    • 潜在决策方向：该角色倾向的拆解路径、关注点、优先级。
    • 输出模板：固定产出结构（发现点、拆解建议、依赖关系、并行项、需要用户决策/不需要用户决策等）。

    落地示例（视频中的关键做法）¶

    • 让“研发负责人”Agent 在方案评审阶段输出任务拆解：
    • 列出 task 之间的 depends-on 关系
    • 区分可并行执行的任务
    • 标注哪些需要用户/Leader 决策、哪些可直接推进
    • 主 Agent 汇总多方意见，团队内部先讨论消化，必要时再由 Leader 介入拍板。

    可复用的提示词骨架（精简版）¶

    1. 你是谁（角色/阶段/目标）
    2. 风格参考（可选）
    3. 你要做什么与不做什么
    4. 对错标准（可检验）
    5. 输出要求（固定模板：发现/建议/依赖/并行/需决策项）
