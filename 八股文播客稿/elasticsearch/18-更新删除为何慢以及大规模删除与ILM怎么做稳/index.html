    18 更新删除为何慢以及大规模删除与ILM怎么做稳

    【开场：共情与痛点】 小雅：【停顿1秒】老王，我有个特别折磨人的线上体验：ES 里删数据，删着删着磁盘不降，查询还变慢。更离谱的是，产品催我“你不是删了吗，怎么还占空间？”我当时真的想把电脑扣上。【笑】面试官听了就问：“ES 的更新/删除为什么看起来慢？大规模删除怎么做更稳？ILM 又是啥价值？”我发现我只会说“它有合并”，说不清整条链。

    老王：害，这题你得把一句话说清楚：Lucene 的段不可变[生活化类比：像你把账本一页页订成册，订好了就不撕不改]，所以更新和删除不是原地改，而是“先标记、再新写、最后再收拾”。【停顿1秒】你一旦接受这个规则，所有现象都解释得通。

    小雅：行，那你从“为什么删了磁盘不降”开始讲，我太需要这个答案了。【加重语气】

    【爆发：核心解析】 老王：删除在 ES 里通常是标记删除[生活化类比：像你在账本里把一条记录划掉并写个“作废”，但纸张还在那儿占位置]。它先让这条数据在查询结果里消失，但磁盘空间不会立刻回收。真正的回收要等 merge[生活化类比：像你月底把一堆作废记录的账本重新誊抄成干净的新账本，旧账本才能丢]，后台合并段的时候才会把那些被标记删除的内容清掉一部分。

    小雅：【停顿1秒】那更新呢？更新是不是也一样“看起来慢”？

    老王：对，更新本质更像“删旧写新”。旧文档被标记删除，新版本写进新段里。[生活化类比：像你写错了账，不是在原行涂改，而是划掉旧的，再写一条新记录]所以更新会带来写放大和更多的段合并压力。能理解这个逻辑吗？

    小雅：【恍然大悟】所以我删了一堆数据，deleted docs 变多，查询也可能变慢，是因为系统要绕开一堆“作废标记”？

    老王：对。deleted docs 多了[生活化类比：像你翻一本被划得密密麻麻的账本，眼睛要不停跳过作废行，当然更费劲]磁盘占用高、查询扫描成本也会上来，直到合并把它们清理掉，才会逐步恢复。能理解这个逻辑吗？

    小雅：【加重语气】那大规模删除怎么做更稳？我现在特别怕 delete-by-query 这种操作。

    老王：你怕得对。大规模逐条 delete 或 delete-by-query[生活化类比：像你要清空一个仓库，却选择一箱一箱拿出去丢，丢完地上还一堆空箱子和碎纸，整理还得花很久]会产生大量标记删除，合并压力巨大，甚至影响线上查询。

    小雅：那正确姿势是什么？你给我一个“能在面试里讲得像真干过”的方案。

    老王：稳的做法是把“删除”上升到索引粒度。时间序列数据，比如日志，按天或按周建索引，到期直接删整个索引[生活化类比：像你每个月一本账本，过期了整本扔掉，干净利落]，比在一本账本里划掉十万行省太多。

    小雅：【停顿1秒】那业务数据不是时间序列怎么办？比如要清理一大批历史订单里的某类记录。

    老王：这种更稳的是“重建视图”：新建一个索引，把你要保留的数据 reindex 过去，然后切别名，最后删旧索引。[生活化类比：像你把旧仓库里有用的货搬到新仓库，新仓库启用后直接把旧仓库关门拆掉]这样可控、可回滚，风险比在旧索引上狂删低很多。能理解这个逻辑吗？

    小雅：那 ILM 是在这个里面扮演啥角色？我怕我讲成“自动化删除工具”太浅。

    老王：ILM[生活化类比：像你给仓库制定一套生命周期规则：新货先放热区方便拿，过一阵移到冷区省成本，再过期就自动清仓]它的核心价值是把滚动、迁移、保留、删除这些策略自动化，减少你手工在凌晨做危险操作。你不用背阶段名，讲清楚“自动按规则把索引从热到冷再到删”就够。能理解这个逻辑吗？

    【反转：面试官在想啥】 小雅：所以面试官问更新删除慢，其实在考我什么？感觉不是要我背“段不可变”四个字。

    老王：生产里你确实不天天做大清理，但面试官问你，其实是在考察你的“写放大与数据治理意识”。你能不能知道删除为什么不立刻省空间，能不能选对清理策略，避免把集群拖死。给你一句能背的面试话术。

    小雅：来一句能背的，把标记删除、merge、索引级删除、ILM、reindex 都串起来。

    老王：你就说：“ES 基于段不可变，更新和删除不会原地修改，而是先标记删除再写入新段，空间回收依赖后台段合并，因此看起来会慢且 deleted docs 多了会影响磁盘与查询；大规模清理更稳的方式是按索引粒度删除，日志类配合按时间滚动索引并用 ILM 自动管理生命周期；业务数据的大范围清理通常用新索引 reindex 保留需要的数据、切别名后再删除旧索引，避免 delete-by-query 带来的合并压力。”【停顿1秒】这句很工程。

    【收尾：三点总结】 小雅：【停顿1秒】我总结一下，其实就是三件事：第一件事儿，更新删除慢是因为先标记再新写，磁盘回收要等合并，删了不立刻省空间很正常；第二件事儿，大规模逐条删会把 deleted docs 和合并压力拉满，查询也会受影响；第三件事儿，最稳是索引级删除和生命周期自动化，日志按天周滚动配 ILM，业务大清理就新索引重建再切别名删旧索引。【恍然大悟】这下我终于能跟产品解释“为什么删了还占空间”了。

    老王：对，你能把“现象→机制→策略”讲清楚，就不怕任何追问。线上做删除，永远记得一句话：别跟段不可变硬刚。
