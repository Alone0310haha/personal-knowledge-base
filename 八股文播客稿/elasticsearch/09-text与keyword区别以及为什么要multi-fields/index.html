    09 text与keyword区别以及为什么要multi fields

    【开场：共情与痛点】 小雅：【停顿1秒】老王，我又遇到那种“看起来很基础但特别容易说崩”的题。面试官问我：“text 和 keyword 有什么区别？为什么经常要做 multi-fields？”我当时脑子里就一句话：一个分词一个不分词。结果他说“那你线上 title 为啥要两个字段”，我就开始冒汗了。【笑】我怕我说不清楚“为什么”，只会说“大家都这么配”。

    老王：害，这题就专门抓“你是不是照抄配置”。你先别急，咱把它当成两个完全不同的使用场景：一个是“我想搜到”，一个是“我想算清楚”。【停顿1秒】你只要把这两个需求分开，答案就自己冒出来了。

    小雅：行，那你先说 text，但别一口气太多，我会插话确认的。【加重语气】

    【爆发：核心解析】 老王：text[生活化类比：像你把一段话切成很多关键词贴在便签上，方便你按关键词快速翻到相关内容]它是用来做全文检索的。你写进去的原始字符串，会被分析器处理成一串词，然后用这些词去匹配和算相关性。

    小雅：【停顿1秒】你说“分析器”和“相关性”，我先不展开，我怕跑偏。那 keyword 呢？

    老王：keyword[生活化类比：像快递单号，原样保存，拿来精确匹配、分组统计、排序最靠谱]它不分词，保存原值。你要做精确过滤、聚合统计、排序、去重，通常都用它。能理解这个逻辑吗？

    小雅：能。那为什么 text 不适合聚合？面试官就爱追这一句。

    老王：因为 text 分词后变成一堆 token[生活化类比：像你把一整句话撕成碎纸片，每片上只有一个词]，聚合要的是“稳定的原值”来分组。你拿碎纸片去做“按原句分组”，它根本没有那个原句的形态了，除非你强行把碎纸片再拼起来，成本也高还容易出事。能理解这个逻辑吗？

    小雅：【恍然大悟】所以聚合要的是“整句”，不是“碎词”。那我再问个实际的：一个字段为什么要同时是 text 和 keyword？这就是 multi-fields 对吧？

    老王：对，multi-fields[生活化类比：像同一个人同时有“名字的拼音索引卡”和“身份证号”，你查人可以按拼音模糊查，也可以按身份证精确查]就是让同一个业务字段同时具备两种能力。比如 title，你希望用户能搜“华为 手机”这种关键词匹配，这用 text；同时你又希望运营能按标题做去重、按标题排序、按标题聚合统计，这就得有 title.keyword。

    小雅：【停顿1秒】那面试官如果问“我什么时候用 term，什么时候用 match”，是不是就可以顺着这个说？

    老王：对。term 更像对 keyword/numeric/date 做精确匹配[生活化类比：像拿身份证号对人，一对一个准]；match 更像对 text 做分词后的匹配[生活化类比：像你只记得对方名字里带个“雅”，在通讯录里按关键词搜出来一堆候选]。你不用背 API 名字，你抓住“精确 vs 全文”这个分界线就稳。能理解这个逻辑吗？

    小雅：懂。那还有一个坑我想提前问，线上有人把 keyword 当成全文搜，或者把 text 拿去排序，会有什么后果？

    老王：把 keyword 当全文搜，召回会很差，用户体验像你在字典里只允许“完整匹配”，少一个字都搜不到。[生活化类比：你在便利店报“可乐”，收银员只认“可口可乐 500ml 这串完整名字”，你说短了他就装听不见]把 text 拿去排序或聚合，往往会走一条很费内存的路，最终表现是查询抖、内存高、GC 折腾。你不用展开细节，面试里点到“资源代价”就够了。能理解这个逻辑吗？

    【反转：面试官在想啥】 小雅：所以面试官问 multi-fields，其实不是想听我背一句“text 分词 keyword 不分词”？

    老王：对。生产里你确实不会天天改字段类型，但面试官问你，其实是在考察你的“查询意图意识”。也就是你能不能把需求拆成：我要召回、我要过滤、我要排序、我要聚合，然后知道每个需求该落在哪种字段上。给你一句能背的面试话术。

    小雅：来一句能背的，我要那种一说完就像做过搜索的。

    老王：你就说：“text 用于全文检索，会被分析器分词并参与相关性计算；keyword 保留原值，适合精确过滤、聚合、排序和去重；业务里同一个字段往往既要搜索又要统计，所以会用 multi-fields 同时提供 text 和 .keyword 子字段，让查询意图各走各的最优路径。”【停顿1秒】这句很稳。

    【收尾：三点总结】 小雅：【停顿1秒】我总结一下，其实就是三件事：第一件事儿，text 是为了“能搜到”，它会分词并算相关性；第二件事儿，keyword 是为了“算得准”，它拿原值做过滤、排序、聚合和去重；第三件事儿，multi-fields 就是一个字段两条腿走路，既满足搜索又满足统计，不用在需求之间二选一。【笑】这题我感觉终于能答得像在讲业务了。

    老王：对，就按“需求→字段→代价”这条线说，面试官很难再把你绕晕。
