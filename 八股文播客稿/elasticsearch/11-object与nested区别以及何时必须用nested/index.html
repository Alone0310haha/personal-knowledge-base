    11 object与nested区别以及何时必须用nested

    【开场：共情与痛点】 小雅：【停顿1秒】老王，我跟你讲，我最怕那种“你以为查对了，其实查错了”的坑。面试官问我：“object 和 nested 有什么区别？什么时候必须用 nested？”我当时还挺自信，说“nested 就是性能差点但更准确”。结果他问我：“那你能举个 object 会查出假结果的例子吗？”我当场沉默。【笑】

    老王：害，这题问得很到位，因为线上最恶心的 bug 就是“看起来结果对、其实错得很隐蔽”。【停顿1秒】你记住一句话：object 在数组场景里会扁平化，字段之间的“同一性”会丢；nested 是把每个子对象当成独立的小文档，能保证条件落在同一个子对象里。

    小雅：【加重语气】你别光给结论，给我一个我能讲出来的“假阳性”案例，不然我又要背。

    【爆发：核心解析】 老王：行。比如你有个订单文档里有个 items 数组，每个元素有 name 和 price。你想查“name 是 可乐 且 price 是 3”这俩条件必须是同一个商品项。用 object 的时候，ES 会把数组里的字段拆成多值集合[生活化类比：像你把一对一对的袜子全倒在床上，左袜一堆右袜一堆，你再想找‘同一双’就找不到了]，于是可能出现“可乐在第一个 item，3 块在第二个 item”，但查询仍然命中，这就是假阳性。能理解这个逻辑吗？

    小雅：【恍然大悟】卧槽，这个太直观了。也就是说 object 会把 items.name 变成一堆名字，把 items.price 变成一堆价格，然后条件就可能跨元素拼在一起。

    老王：对，你抓住关键了。nested[生活化类比：像每双袜子用夹子夹成一对，永远不会把左袜和别的右袜配在一起]会把数组里的每个子对象单独索引成隐藏文档，查询时你必须指定 path，让“name 和 price”在同一个子对象里同时成立。能理解这个逻辑吗？

    小雅：【停顿1秒】那什么时候“必须用 nested”？我怕我说得太绝对。

    老王：你就说一个判断标准：只要你需要在对象数组里做“同一个元素内的字段关联查询”，就必须用 nested。比如购物车、简历经历、地址列表、权限规则这种，一旦要保证字段之间不串台，就上 nested。能理解这个逻辑吗？

    小雅：那如果我只是存着展示，不做这种关联查询呢？

    老王：那就别硬上 nested。因为 nested 的代价是写入会变成更多内部文档，查询也更重。[生活化类比：像你为了不丢袜子，每双都夹起来，整理是更稳，但收纳和翻找也更费工夫]如果你不需要“同一性”，扁平字段或者 object 就够了。能理解这个逻辑吗？

    小雅：【加重语气】我插一句确认一下。【停顿1秒】nested 的本质不是“更花哨”，而是“保证数组里每个小对象的边界不被打散”，对吧？

    老王：完全正确。

    小雅：面试官有时候还会问“那 nested 查起来怎么写”，我又怕一说就变成念 DSL。

    老王：你别念语法，你用口语描述意图就行：我会用 nested query 指定 items 这条路径，让过滤条件在同一个 item 内成立。面试官要的是你知道“必须加 path 才不会串”。能理解这个逻辑吗？

    【反转：面试官在想啥】 小雅：所以这题面试官到底在考我什么？

    老王：生产里你确实不会天天改字段类型，但面试官问你，其实是在考察你的“数据正确性意识”。就是你能不能意识到：查询快不快是一回事，结果对不对是另一回事，尤其在对象数组这种结构里，错了还不容易被发现。给你一句能背的面试话术。

    小雅：来一句能背的，最好带上“什么时候选它”。

    老王：你就说：“object 在对象数组场景会被扁平化，多值字段可能跨元素组合导致假阳性；nested 会把每个子对象独立索引成隐藏文档，查询时通过指定 path 保证条件落在同一子对象内；只要业务需要对象数组的字段关联一致性，就必须用 nested，代价是写入与查询资源更高，不需要关联查询时用 object 或扁平字段更省。”【停顿1秒】这句很稳。

    【收尾：三点总结】 小雅：【停顿1秒】我总结一下，其实就是三件事：第一件事儿，object 在数组里会扁平化，字段可能跨元素拼起来，容易出现假阳性；第二件事儿，nested 把每个子对象当成独立的小文档，查询时能保证条件落在同一个子对象里；第三件事儿，只要你要“同一性”就用 nested，但要接受写入和查询成本更高，不要为了炫技乱用。【恍然大悟】这题我终于能举例了。

    老王：能举例就赢一半。面试官一听你知道“假阳性”这三个字，基本就不会觉得你只是会背概念了。
