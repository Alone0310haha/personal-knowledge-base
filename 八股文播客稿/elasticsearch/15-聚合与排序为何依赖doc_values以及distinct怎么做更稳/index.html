    15 聚合与排序为何依赖doc values以及distinct怎么做更稳

    【开场：共情与痛点】 小雅：【停顿1秒】老王，我有个特别真实的线上阴影。我们有个报表页面，突然就开始卡，CPU 飙、内存飙，最后 ES 报了一堆我看不懂的东西。面试官听完就问我：“聚合和排序为什么依赖 doc values？distinct 你怎么做更稳？”我当时只会说“用 keyword”。【笑】他就追：“那你知道为什么吗？”

    老王：害，这题是典型的“你会用”和“你知道为什么能用”之间的分水岭。先给你一句人话：聚合和排序本质是按字段做统计和取序，所以最怕每次都把整条文档搬出来翻。doc values[生活化类比：像你做财务报表，把每张发票的“金额”单独摘成一列，算总和、排序时只看这一列，不用把每张发票全文都摊开]就是为这个准备的。

    小雅：你这个比喻我能秒懂。【加重语气】那为什么不用 _source 直接搞？反正 _source 里啥都有。

    【爆发：核心解析】 老王：因为 _source 更像原始档案袋[生活化类比：像每个人的档案夹，里面材料很多，你只想按年龄排序却要把每个档案夹都打开翻一遍]。聚合和排序只需要某个字段的值，doc values 把同一个字段集中存成列式结构，读取更顺滑，尤其在需要扫描很多文档时差距非常大。能理解这个逻辑吗？

    小雅：【停顿1秒】能。那我看到过一个词叫 fielddata，它跟 doc values 是啥关系？我们那次事故好像就提到了 fielddata。

    老王：fielddata[生活化类比：像你临时把一堆档案夹里的“年龄”都抄到一张大表上放在桌子上用，确实方便，但桌子很快就被占满]通常出现在你拿 text 字段去排序/聚合时，因为 text 默认不走 doc values，它会把东西加载到堆内存里，堆一大就 GC 折腾，延迟就抖。能理解这个逻辑吗？

    小雅：【恍然大悟】所以“用 keyword”其实是在说“让它走 doc values 的路”，对吧？

    老王：对。keyword 默认更适合排序和聚合，就是因为它通常配套 doc values。你再加一句“不要对 text 做排序/聚合，应该用 .keyword 子字段”，面试官基本就点头了。

    小雅：【停顿1秒】那聚合本身有什么坑？我知道 terms 聚合很常用，但我听说桶多了会炸。

    老王：terms 聚合[生活化类比：像你统计“每个手机号出现次数”，手机号种类越多，分组桶就越多，桌上要摆的篮子就越多]面对高基数会产生很多桶，内存和计算都会爆。稳的做法是先 filter 缩小数据集[生活化类比：先把需要统计的票据挑出来再算，不要把全公司十年发票都摊开]，再控制桶数，别动不动就全量分组。

    小雅：那 distinct 呢？我们业务经常要“去重用户数”，还有“列出所有唯一值”。

    老王：distinct 分两种诉求。你要“去重计数”，可以用基数估算那套，强调它是近似的但很快。[生活化类比：像数人头你不一张张点名，而是按门票扫码估个大概，误差可控但效率高]你要“把所有唯一值都列出来并分页”，terms 就容易翻车，这时候更稳的是 composite 聚合[生活化类比：像你把全城商家按区分批抄到册子里，每次抄一页，用上一页的最后一个作为接力点继续抄]，它支持分页遍历桶。能理解这个逻辑吗？

    小雅：【加重语气】我插一句确认。【停顿1秒】terms 更像 topN，composite 更像“全量遍历 + 分页”，对吧？

    老王：对，这个区分很关键。

    小雅：那面试官如果问“为什么先 filter 再聚合”，我怎么讲得更有说服力？

    老王：你就说聚合是重活，先过滤能减少参与聚合的文档量，桶数也会更可控，整体成本下降。[生活化类比：先把不要的菜叶摘掉再下锅，油烟和时间都少很多]能理解这个逻辑吗？

    【反转：面试官在想啥】 小雅：所以这题面试官其实在考啥？感觉不像在考 API。

    老王：生产里你确实不天天研究存储结构，但面试官问你，其实是在考察你的“分析型查询成本意识”。你能不能知道排序聚合为什么吃资源，知道该选什么字段类型，知道桶数和基数会把集群打爆。给你一句能背的面试话术。

    小雅：来一句能背的，把 doc values、fielddata、terms、composite、distinct 都串起来。

    老王：你就说：“排序和聚合本质是按字段做列式扫描，依赖 doc values 提供高效的列存访问；对 text 做排序/聚合容易触发 fielddata 占用堆内存导致抖动，应该用 keyword 或 .keyword 子字段；terms 聚合适合 topN 但高基数桶多会吃内存，需先 filter 缩小数据集并控制桶数；需要全量遍历唯一值时用 composite 聚合分页更稳，去重计数根据精度需求选择近似或全量方案。”【停顿1秒】这句非常工程。

    【收尾：三点总结】 小雅：【停顿1秒】我总结一下，其实就是三件事：第一件事儿，排序和聚合是按字段“看一列”，doc values 就是那列式存储，不用每次翻整份文档；第二件事儿，别拿 text 去排序聚合，不然容易走 fielddata 把堆顶爆，正确姿势是用 keyword 或 .keyword；第三件事儿，terms 适合 topN，高基数要小心桶爆，真要全量 distinct 就用 composite 分页遍历，先 filter 再聚合更稳。【恍然大悟】这题我感觉我能从“事故”讲到“原理”了。

    老王：对，面试官最喜欢你这种能把事故讲成方法论的。你记住：分析型查询不是不能做，是要知道成本在哪里。
