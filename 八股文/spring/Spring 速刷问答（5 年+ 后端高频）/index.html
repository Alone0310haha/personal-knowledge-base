    Spring 速刷问答（5 年+ 后端高频）

    IoC 容器与 Bean 生命周期¶

    你怎么一句话讲清 IoC/DI？它解决了什么问题？¶

    • 15秒简答：IoC/DI 本质是“对象创建与依赖装配的控制权交给容器”；核心机制是容器按 BeanDefinition 创建 bean，并在合适时机做依赖注入；特性是解耦、可扩展，但启动期更复杂、错误多发生在装配阶段。
    • 3分钟详答：出现频次：★★★★★（Spring 必考开场题）。
    • 开场：我会说 IoC 是思想，DI 是实现手段：不再在业务代码里 new 依赖，而是声明依赖关系，让容器在运行时装配。
    • 场景：Service 依赖 Dao，如果手写 new，会把实现类写死且难以测试；IoC 下通过接口注入实现，替换实现只改配置/注解。
    • 核心流程（概念）：容器先读取配置（注解/配置类/XML）生成 BeanDefinition；创建 bean 时再做实例化→属性填充→初始化。
    • 为什么这样设计：把“对象生命周期管理”和“横向能力（AOP/事务/事件）”集中到框架，让业务只写领域逻辑。
    • 实战：我最直接的收益是可测试性（mock 注入）和可插拔（不同实现按 profile/条件装配）。
    • 可能追问：
    • 追问：IoC 和 DI 有区别吗？
      15秒简答：IoC 是控制反转的思想；DI 是把依赖注入进来的实现方式之一（构造器/Setter/字段注入）。
    • 追问：为什么很多团队推构造器注入？
      15秒简答：依赖显式、可保证非空、便于单测；字段注入隐藏依赖且更难做不可变设计。

    BeanFactory vs ApplicationContext 怎么选？面试官想听到哪个差异点？¶

    • 15秒简答：它们本质都是 IoC 容器；核心机制是 BeanFactory 提供最小化的 getBean 能力，ApplicationContext 在此之上增加事件、国际化、资源加载、自动装配与更多后处理；特性是后者更“全家桶”，绝大多数应用默认用 ApplicationContext。
    • 3分钟详答：出现频次：★★★★☆（常作为“你真的懂 Spring 吗”追问）。
    • 开场：BeanFactory 更像底座，ApplicationContext 是面向应用的完整容器。
    • 关键差异：
      • 功能：ApplicationContext 支持 ApplicationEvent、MessageSource、ResourceLoader 等企业特性。
      • 装配与扩展：更完整的后处理器链路与自动配置能力（特别是 Boot 场景）。
      • 启动策略：很多实现会在 refresh 阶段更积极地准备单例（与配置有关），便于尽早暴露问题。
    • 实战：业务服务几乎总是 ApplicationContext；BeanFactory 更多在框架底层、测试或极简容器中出现。
    • 可能追问：
    • 追问：什么是 refresh()？
      15秒简答：ApplicationContext 的核心启动流程：准备环境→创建 BeanFactory→注册后处理器→实例化单例→发布容器就绪事件。
    • 追问：ApplicationContext “更重”体现在哪？
      15秒简答：启动阶段会初始化更多基础设施 bean/后处理器/事件体系，能力更强但启动成本更高。

    Bean 的生命周期你如何讲成“可执行的链路”？（实例化→注入→初始化→销毁）¶

    • 15秒简答：生命周期本质是“容器可插拔的构建流水线”；核心机制是实例化（构造/工厂）→属性填充→Aware 回调→BeanPostProcessor 前后置→初始化方法→可用；特性是扩展点丰富但也容易被代理/AOP 改写成“看起来不按直觉”。
    • 3分钟详答：出现频次：★★★★★（面试高频且可深挖）。
    • 开场：我一般用“几个关键回调点”来记：实例化、填充、初始化、销毁。
    • 核心阶段（简化版）：
      • 实例化：构造器/工厂方法创建原始对象。
      • 属性填充：依赖注入发生在这里（按类型/名称解析依赖）。
      • Aware：拿到容器相关能力（BeanName、BeanFactory、ApplicationContext 等）。
      • 初始化：@PostConstruct、InitializingBean、init-method 等。
      • 后置处理：BeanPostProcessor 可以在初始化前后做增强，AOP 代理通常在这里产生，最终返回可能是代理对象而非原对象。
      • 销毁：DisposableBean、destroy-method、@PreDestroy 等（主要针对单例）。
    • 实战：排查“注入值不对/事务不生效/AOP 不生效”时，我会用生命周期视角判断：增强是否发生在正确阶段、拿到的是不是代理对象。
    • 可能追问：
    • 追问：@PostConstruct 和 init-method 谁先谁后？
      15秒简答：都在初始化阶段；通常 @PostConstruct 更偏标准注解回调，init-method/InitializingBean 属于 Spring 机制，具体顺序取决于容器实现但都发生在“可用之前”。
    • 追问：prototype 的销毁回调会自动触发吗？
      15秒简答：默认不会，prototype 由容器创建但不托管销毁；需要调用方自行清理或用自定义 scope 管理。

    Spring 的“三级缓存”解决了什么问题？循环依赖为什么只解决了一部分？¶

    • 15秒简答：三级缓存本质是“提前暴露单例引用以打破构造链”；核心机制是 singletonObjects/earlySingletonObjects/singletonFactories 分层缓存，使 A→B→A 这种 setter 注入能拿到早期引用（可能是代理）；特性是只主要解决单例+setter/字段注入的循环，构造器循环依赖仍无解。
    • 3分钟详答：出现频次：★★★★☆（5年+常问，能体现你理解到容器实现层）。
    • 开场：循环依赖的根因是“创建 A 需要 B，创建 B 又需要 A”，容器需要一个“半成品引用”让对方先继续走下去。
    • 三级缓存怎么用：
      • 完整对象进一级缓存 singletonObjects。
      • 早期对象引用进二级 earlySingletonObjects（尚未完全初始化）。
      • 三级 singletonFactories 存一个 ObjectFactory，用于在需要时创建“早期引用”（尤其为了让 AOP 代理能提前暴露）。
    • 为什么只解决一部分：
      • 构造器注入：必须在构造时拿到依赖，无法先给你一个“没构造完的 A”。
      • prototype：多实例生命周期由调用方驱动，容器难以用全局缓存打破环。
    • 实战：我更倾向避免循环依赖（抽象依赖、事件解耦、拆分职责）；如果必须，尽量用构造器以外方式并控制代理介入。
    • 可能追问：
    • 追问：为什么引入 AOP 后循环依赖更容易踩坑？
      15秒简答：因为需要提前暴露的可能是代理而不是原对象，早期引用与最终代理对象不一致会带来行为差异。
    • 追问：怎么快速定位循环依赖链？
      15秒简答：看启动报错的 bean 创建栈/依赖图（日志会打印依赖路径），按“谁依赖谁”画环并拆掉最外层的强依赖。

    BeanPostProcessor vs BeanFactoryPostProcessor（以及 RegistryPostProcessor）怎么区分？各在什么时候运行？¶

    • 15秒简答：它们本质是“容器扩展点的两层钩子”；核心机制是 BFPP 在实例化 bean 之前改 BeanDefinition/容器配置，BPP 在 bean 创建过程中改实例/返回代理；特性是 BFPP 更偏“改蓝图”，BPP 更偏“改成品”，二者组合支撑了注解解析与 AOP/事务等能力。
    • 3分钟详答：出现频次：★★★★★（5年+高频，尤其问 Boot 自动装配时一定会碰到）。
    • 开场：我用一句话区分：BeanFactoryPostProcessor 处理“定义”，BeanPostProcessor 处理“对象”。
    • BeanFactoryPostProcessor：
      • 发生在 BeanDefinition 都加载进来之后、单例实例化之前。
      • 典型：PropertySourcesPlaceholderConfigurer、以及各种修改 BeanDefinition 的扩展。
    • BeanDefinitionRegistryPostProcessor（更早）：
      • 能“注册新的 BeanDefinition”（比 BFPP 更底层），典型代表是解析 @Configuration/@ComponentScan/@Import 的处理器。
    • BeanPostProcessor：
      • 贯穿 bean 创建过程，初始化前后都能插逻辑。
      • 典型：AutowiredAnnotationBeanPostProcessor（依赖注入）、AOP 的代理创建等。
    • 实战：当我需要“在容器启动期批量注册一堆策略类/handler 映射”，会考虑 RegistryPostProcessor；需要“对某类 bean 做统一包装（代理/增强）”，用 BPP。
    • 可能追问：
    • 追问：为什么很多框架能力都靠 BPP？
      15秒简答：因为 BPP 能拦截 bean 创建并替换返回值（代理），是实现 AOP/事务/校验/注入的关键入口。
    • 追问：BFPP 能拿到 bean 实例吗？
      15秒简答：原则上不应该依赖 bean 实例（此时还未创建/不应触发创建），它面向的是 BeanDefinition 与容器配置。

    AOP 与代理机制¶

    Spring AOP 的底层是什么？JDK 动态代理和 CGLIB 怎么选？¶

    • 15秒简答：Spring AOP 本质是“基于代理的运行时方法拦截”；核心机制是把切面织入到代理的调用链里（Interceptor Chain），接口优先用 JDK 动态代理，没接口再用 CGLIB 子类代理；特性是侵入低但只能拦截方法调用点，且 final 类/方法对 CGLIB 不友好。
    • 3分钟详答：出现频次：★★★★★（AOP 必问题）。
    • 开场：Spring AOP 不是改字节码（默认），而是创建一个代理对象，把横切逻辑包在方法调用前后。
    • 两种代理：
      • JDK 动态代理：基于接口，生成实现同接口的代理类，调用走 InvocationHandler。
      • CGLIB：基于继承，生成目标类子类并覆盖方法织入逻辑；final 类/方法无法被代理。
    • 为什么要这样设计：代理模式足够通用且对用户代码侵入小；性能在现代 JVM 下通常不是瓶颈，关键是语义正确性与可观测性。
    • 实战：我会优先面向接口编程（JDK 代理更稳定），对类代理场景会注意 final、equals/hashCode、以及序列化等边界。
    • 可能追问：
    • 追问：AOP 能拦截 private 方法吗？
      15秒简答：默认不行，代理只能拦截可被代理调用的公共/受保护方法调用点；private 方法调用在类内部不会经过代理。
    • 追问：为什么同类内部方法调用（self-invocation）AOP 不生效？
      15秒简答：因为调用发生在目标对象内部，绕过了代理对象，拦截链没有机会介入。

    AOP 的 JoinPoint/Pointcut/Advice/Aspect 你怎么 30 秒讲清楚？¶

    • 15秒简答：AOP 概念本质是“在哪里、做什么、以什么方式织入”；核心机制是 Pointcut 选中一组 JoinPoint（方法调用点），Advice 定义增强逻辑，Aspect 把切点和通知打包；特性是横切关注点可复用，但切点过宽会引入隐式耦合与排障难度。
    • 3分钟详答：出现频次：★★★★☆（概念题，但答得工程化会加分）。
    • JoinPoint：可被拦截的点（Spring AOP 里主要是方法执行）。
    • Pointcut：匹配规则（哪些方法需要拦截）。
    • Advice：增强逻辑（Before/After/AfterReturning/AfterThrowing/Around）。
    • Aspect：切面，把“规则+逻辑”封装成一个模块。
    • 实战：我会把切点控制在稳定的边界层（controller/service/annotation），避免对细碎方法做全局切点导致不可控的性能与语义问题。
    • 可能追问：
    • 追问：Around 和其他通知有什么本质差别？
      15秒简答：Around 能决定是否执行目标方法、能包裹前后并捕获异常，控制力最强但也最容易写出副作用。
    • 追问：Advice 顺序怎么控制？
      15秒简答：通过 @Order 或 Ordered 接口控制切面顺序；同一切面内部的多个通知按框架规则执行。

    Spring AOP 的拦截器链是怎么跑起来的？你会怎么画调用链？¶

    • 15秒简答：拦截器链本质是“责任链 + 递归推进”；核心机制是代理方法调用进入 MethodInterceptor 列表，逐个执行，最终触达目标方法；特性是链路清晰但堆栈会变深，排查时要识别哪些拦截器在链上（事务/安全/日志等）。
    • 3分钟详答：出现频次：★★★☆☆（偏原理，但 5年+很常见）。
    • 开场：我会把它类比 Servlet Filter：一层层 before，最后调用目标，再一层层 after 返回。
    • 典型链路：代理方法 → AOP 框架构造拦截器列表 → Around/Before/事务拦截器等依次执行 → 调用目标方法 → 反向返回做 after/提交/清理。
    • 为什么这样设计：统一把“横切能力”抽象成可组合的拦截器，顺序可控，易扩展。
    • 实战：当出现“事务没生效/异常吞了/日志重复”时，我会检查链顺序与切点是否命中了预期方法。
    • 可能追问：
    • 追问：为什么有些异常不会触发事务回滚？
      15秒简答：因为事务拦截器在链上按规则决定回滚（默认只回滚 RuntimeException/Error），且异常可能被上游拦截器捕获吞掉。
    • 追问：如何确认当前对象是不是代理？
      15秒简答：可通过类名/调试观察（通常是 $$Proxy 或 CGLIB 后缀），或通过框架工具方法判断是否 AOP 代理。

    AOP 常见踩坑有哪些？（final、private、自调用、代理对象注入）¶

    • 15秒简答：踩坑本质是“你以为拦截了，其实没经过代理”；核心机制是代理的边界决定可拦截范围（public/protected、外部调用），final/私有/自调用都会绕过或无法增强；特性是问题隐蔽，往往上线后才发现事务/日志/鉴权没生效。
    • 3分钟详答：出现频次：★★★★★（面试官很爱让你举坑）。
    • 常见坑清单：
      • final 类/方法：CGLIB 无法覆盖，JDK 代理要求接口。
      • private 方法：外部无法通过代理调用，不会被拦截。
      • 同类自调用：内部 this 调用绕过代理。
      • 代理对象注入：注入的是接口类型还是实现类型，会影响拿到的是 JDK 代理还是目标对象。
      • 多切面顺序：事务在日志外还是内，会影响异常传播与记录。
    • 实战：我通常用“边界层 + 接口化 + 少量清晰切点”降低踩坑概率；对事务/安全这种关键能力，会写集成测试验证行为而不是靠自信。
    • 可能追问：
    • 追问：怎么解决自调用不生效？
      15秒简答：把被拦截方法抽到另一个 bean；或显式从容器拿代理调用；或启用 exposeProxy 但要谨慎（耦合与可读性变差）。
    • 追问：为什么 final 会影响代理？
      15秒简答：CGLIB 通过继承覆盖方法实现增强，final 禁止继承/重写；JDK 代理则要求接口。

    声明式事务（Spring Tx）¶

    声明式事务的本质是什么？@Transactional 到底是怎么生效的？¶

    • 15秒简答：声明式事务本质是“用 AOP 把事务边界织入方法调用”；核心机制是事务拦截器在方法前开启事务、方法后提交/回滚，并绑定资源到线程；特性是开发成本低但强依赖代理边界，自调用/异常处理不当会失效。
    • 3分钟详答：出现频次：★★★★★（Spring 事务必考）。
    • 开场：@Transactional 不是 magic，本质就是一个切点 + 事务拦截器。
    • 核心流程（概念）：
      • 调用进入代理 → 命中事务拦截器 → 根据传播行为获取/创建事务 → 绑定连接/Session 到线程（ThreadLocal） → 执行业务 → 按规则提交或回滚 → 清理线程绑定资源。
    • 为什么这样设计：把事务从业务代码里剥离（不用手写 begin/commit/rollback），同时让事务策略（传播、隔离、超时）可配置。
    • 实战：我会强调事务要“短小、只包数据库操作”，避免把 RPC/IO 放进事务里导致锁时间长与死锁概率上升。
    • 可能追问：
    • 追问：为什么事务通常绑定在线程上？
      15秒简答：因为一次调用链在同一线程内共享同一个连接/会话最简单，ThreadLocal 让下游 DAO 能拿到同一事务资源。
    • 追问：为什么有时看起来加了 @Transactional 但没开事务？
      15秒简答：没经过代理（自调用/非 public）、代理没创建（bean 未被容器管理）、或切点没命中（包扫描/配置问题）。

    事务传播行为怎么背不重要，怎么讲“用得出来”才重要：REQUIRED vs REQUIRES_NEW vs NESTED¶

    • 15秒简答：传播行为本质是“遇到已有事务时怎么处理”；核心机制是 REQUIRED 加入或新建，REQUIRES_NEW 挂起外层另起新事务，NESTED 在同一物理事务内用保存点实现嵌套回滚；特性是 REQUIRES_NEW 隔离最强但成本高，NESTED 更像局部回滚工具但依赖底层支持。
    • 3分钟详答：出现频次：★★★★★（面试高频深挖点）。
    • 开场：传播行为决定“一个业务流程拆成多个事务边界时，失败如何影响彼此”。
    • 三个最常用：
      • REQUIRED（默认）：大部分业务用它，保证一个调用链有一个事务边界。
      • REQUIRES_NEW：典型用于“主流程失败也要落库”的审计日志/补偿记录；外层事务被挂起，内层独立提交。
      • NESTED：同一个大事务里某个子步骤失败只回滚到保存点，不影响外层其他步骤（前提是数据源/事务管理器支持 savepoint）。
    • 为什么这样设计：不同业务对一致性与可恢复性要求不同，需要灵活控制影响范围。
    • 实战：我会谨慎使用 REQUIRES_NEW（可能增加锁与连接占用），并确保日志/补偿记录是幂等的，避免重试导致重复数据。
    • 可能追问：
    • 追问：REQUIRES_NEW 一定会开启新连接吗？
      15秒简答：通常会获取新的事务资源并挂起外层资源；具体取决于事务管理器与数据源，但语义上它是独立事务。
    • 追问：NESTED 和 REQUIRES_NEW 的本质区别？
      15秒简答：NESTED 用保存点，仍在同一个物理事务里；REQUIRES_NEW 是完全独立的事务，提交/回滚互不影响。

    @Transactional 回滚规则你怎么讲？为什么默认不回滚 checked exception？¶

    • 15秒简答：回滚规则本质是“异常到事务结果的映射策略”；核心机制是默认遇到 RuntimeException/Error 标记回滚，checked exception 默认认为是可预期业务异常不回滚（可通过 rollbackFor 定制）；特性是默认策略更偏工程折中，但业务上必须显式定义“哪些异常必须回滚”。
    • 3分钟详答：出现频次：★★★★★（面试很爱问“为什么你事务没回滚”）。
    • 开场：事务最终是“提交还是回滚”的决策，Spring 用异常类型做默认决策。
    • 默认规则：
      • RuntimeException/Error：默认回滚。
      • Checked Exception：默认提交（除非配置 rollbackFor）。
    • 为什么这样设计：checked exception 在 Java 语义上更多是“可恢复/可处理”的受检错误，Spring 默认不把它当作“必须回滚”的系统失败；但这不是业务正确性的保证。
    • 实战：我会把业务异常分两类：需要回滚（例如下游写失败、校验不通过会导致脏数据）就明确 rollbackFor；不需要回滚（纯查询/可补偿日志）就按默认或 noRollbackFor。
    • 可能追问：
    • 追问：try-catch 把异常吞掉会怎样？
      15秒简答：事务拦截器看不到异常就会提交；如果确实要吞异常但仍回滚，需要手动标记回滚或重新抛出符合规则的异常。
    • 追问：怎么让 checked exception 也回滚？
      15秒简答：在 @Transactional 上配置 rollbackFor=Exception.class 或指定具体异常类型。

    事务为什么会失效？列 6 个你踩过/见过的高频场景¶

    • 15秒简答：失效本质是“事务拦截器没机会执行或执行后语义被破坏”；核心机制是没走代理、自调用、方法不可代理、bean 不在容器、异常被吞、传播行为与调用链不匹配；特性是问题隐蔽但高发，必须用测试与代码规范兜住。
    • 3分钟详答：出现频次：★★★★★（面试官最爱考“你真的做过项目吗”）。
    • 高频场景（挑重点讲）：
      • 自调用：同类内部调用绕过代理。
      • 非 public 方法：默认代理拦截不到。
      • bean 不是容器管理：new 出来的对象没代理。
      • 异常被 catch 吞掉：拦截器看不到异常，提交了。
      • rollbackFor 配置不对：抛了 checked exception 但没配置回滚。
      • 传播行为不符合预期：比如你以为 REQUIRES_NEW，但实际调用链没经过代理或事务挂起不生效。
    • 实战：我会通过两类手段防：代码规范（public+拆 bean+构造注入）+ 集成测试（验证回滚/提交行为）。
    • 可能追问：
    • 追问：为什么非 public 不生效？
      15秒简答：代理通常只能拦截对外可调用的方法，非 public 在代理机制/调用路径上常无法被拦截（尤其是自调用时）。
    • 追问：怎么验证事务真的生效？
      15秒简答：写一个失败用例观察数据是否回滚，并在日志/调试中确认调用经过的是代理对象且命中事务拦截器。

    readOnly、timeout、isolation 这些事务属性你在生产怎么用？别只背定义¶

    • 15秒简答：事务属性本质是“给事务边界加约束与优化提示”；核心机制是 readOnly 提示底层做读优化/避免脏写，timeout 防止长事务占锁，isolation 选择并发一致性级别；特性是能提升稳定性，但隔离级别与超时会影响吞吐，需要结合数据库与场景取舍。
    • 3分钟详答：出现频次：★★★★☆（5年+工程化问题）。
    • readOnly：查询链路（纯读）我会标记 readOnly，减少误写风险；某些数据库/驱动可做优化（依赖实现）。
    • timeout：我会给关键链路设置超时，避免“锁住等死”的长事务（尤其是批处理、慢 SQL）。
    • isolation：通常跟数据库默认一致（例如 MySQL RR/RC 的差异要结合业务），除非明确需要更强/更弱隔离；更高隔离意味着更低并发。
    • 实战：我会优先做“短事务 + 快 SQL”，其次才靠 isolation 去硬顶一致性；并且把超时和重试/补偿策略配套起来。
    • 可能追问：
    • 追问：readOnly 能保证不写吗？
      15秒简答：它更多是提示/约束，是否强制取决于底层实现；要保证不写还得靠权限、代码规范与测试。
    • 追问：timeout 到了会发生什么？
      15秒简答：事务会被标记回滚并抛异常（具体异常类型与管理器实现相关），目的是尽快释放锁与资源。

    Spring Boot 与自动装配（面试高频）¶

    Spring Boot 自动装配到底做了什么？一句话把 @SpringBootApplication 讲清楚¶

    • 15秒简答：自动装配本质是“按条件批量注册配置类与 bean”；核心机制是 @SpringBootApplication = @Configuration + @ComponentScan + @EnableAutoConfiguration，其中自动配置通过 ImportSelector 挑选满足条件的 AutoConfiguration 并注册；特性是开箱即用但“隐式装配”会带来排障复杂度。
    • 3分钟详答：出现频次：★★★★★（社招后端几乎必问）。
    • 开场：Boot 的核心价值是把“你每个项目都重复写的配置”变成“条件化的自动配置”。
    • 发生了什么：
      • 扫描你的业务组件（ComponentScan）。
      • 导入一批自动配置类（EnableAutoConfiguration），这些类里通过 @Bean 定义常用基础设施。
      • 是否生效由条件注解决定（classpath 是否有某类、是否配置了某属性、是否已有用户自定义 bean）。
    • 为什么这样设计：降低样板配置，提升一致性；同时允许用户通过“自定义 bean/属性”覆盖默认配置。
    • 实战：我排查“为什么多了一个 bean/为什么没装配”会先看自动配置报告（或条件评估），再看是否被用户配置覆盖。
    • 可能追问：
    • 追问：怎么让某个自动配置失效？
      15秒简答：常见做法是排除对应自动配置类（exclude）或提供你自己的同类型 bean 让条件 @ConditionalOnMissingBean 不成立。
    • 追问：为什么有时 bean 明明写了却注入不到？
      15秒简答：可能没被扫描到（包路径）、条件不满足、或被同名/同类型 bean 覆盖导致注入歧义。

    @ConditionalOnClass / @ConditionalOnMissingBean 你如何用一张图解释？¶

    • 15秒简答：条件装配本质是“按环境做 if-else 的 BeanDefinition 注册”；核心机制是启动期评估条件：类存在才装、容器里没同类 bean 才装；特性是默认配置可被用户覆盖，但条件链复杂时排障要看“为什么条件没通过”。
    • 3分钟详答：出现频次：★★★★☆（自动装配的核心追问点）。
    • 开场：我会把它类比成：Boot 在启动时跑一堆 if 判断，满足就注册默认 bean。
    • 两个最关键条件：
      • OnClass：classpath 有某依赖才启用（避免没引包也注册 bean 报错）。
      • OnMissingBean：你没自定义时才给你一个默认实现（可覆盖性）。
    • 实战：我写 starter 时会把“可覆盖点”全部用 MissingBean 包住，并让关键开关可配置（properties），保证用户能控。
    • 可能追问：
    • 追问：为什么说“自定义一个同类型 bean 就能覆盖默认”？
      15秒简答：因为默认 bean 多数受 @ConditionalOnMissingBean 保护，你注册了同类型 bean 后条件不成立，默认 bean 不再创建。
    • 追问：条件是何时评估的？
      15秒简答：在配置类解析/注册 bean 定义阶段评估（在 bean 实例化之前），决定哪些 BeanDefinition 进入容器。

    starter 你怎么设计？如何做到“引包即用、可配置、可覆盖、可禁用”？¶

    • 15秒简答：starter 本质是“自动配置的打包与约定”；核心机制是提供 auto-configuration 模块（条件化注册 bean）+ 可选依赖聚合模块，并通过 properties 暴露配置；特性是体验好但要严格控制默认值、覆盖点与开关，避免“引包就改行为”。
    • 3分钟详答：出现频次：★★★☆☆（5年+如果做过平台/中间件会问）。
    • 开场：一个好的 starter 要满足 4 件事：默认可用、参数可调、用户可覆盖、必要时可关闭。
    • 设计要点：
      • 默认实现用 @ConditionalOnMissingBean，给用户覆盖入口。
      • 用 @ConfigurationProperties 暴露配置，提供合理默认值。
      • 用 @ConditionalOnProperty 做总开关与子功能开关。
      • 依赖分层：starter（依赖聚合）与 autoconfigure（真正的配置类）分离，减少传递依赖污染。
    • 实战：我做过统一客户端（HTTP/RPC/Redis）starter 时，会强制线程池/连接池命名与监控埋点，默认安全但不自作主张，避免压垮业务。
    • 可能追问：
    • 追问：为什么要提供“禁用开关”？
      15秒简答：线上排障/灰度时必须能快速关闭某个自动配置，避免只能发版删除依赖。
    • 追问：如何避免 starter 把依赖传染给业务？
      15秒简答：用可选依赖/依赖分层，避免在 starter 里引入不必要的重依赖，必要时让业务显式引入实现依赖。

    @Configuration(proxyBeanMethods) 你怎么解释？lite 模式 vs full 模式在生产有什么影响？¶

    • 15秒简答：@Configuration 本质是“配置类的 bean 方法拦截器”；核心机制是 full 模式会用 CGLIB 代理配置类以保证 @Bean 方法多次调用仍返回同一单例，lite 模式不代理更快但方法间调用可能产生新实例；特性是 full 更安全、lite 启动更快但要求写法自律。
    • 3分钟详答：出现频次：★★★☆☆（新一点的面试点，能区分熟练度）。
    • 开场：很多人以为 @Bean 就一定是单例，其实是否“方法调用返回同一实例”取决于配置类是否被代理。
    • full 模式（proxyBeanMethods=true）：
      • 代理配置类，拦截 @Bean 方法，从容器取同名 bean，保证单例语义。
      • 代价：CGLIB 代理与方法拦截带来一点启动/运行期开销。
    • lite 模式（false）：
      • 不代理，@Bean 方法就是普通方法；如果你在一个 @Bean 方法里直接调用另一个 @Bean 方法，可能 new 出不同对象。
    • 实战：我在基础设施配置上更倾向 full 保守正确；在纯配置、无方法间调用依赖的场景才用 lite 提升启动速度。
    • 可能追问：
    • 追问：怎么避免 lite 模式下的多实例问题？
      15秒简答：不要在 @Bean 方法里直接调用另一个 @Bean 方法；改为方法参数注入（让容器注入依赖）或显式从容器获取。
    • 追问：为什么说“方法参数注入更推荐”？
      15秒简答：由容器负责解析依赖，不依赖配置类代理语义，行为更清晰且更容易测试。

    注入、Web 与工程化常见坑¶

    @Autowired 注入规则你怎么讲？@Primary、@Qualifier、@Resource 怎么取舍？¶

    • 15秒简答：注入本质是“按规则做依赖解析”；核心机制是 Autowired 默认按类型匹配，歧义时结合 @Primary/@Qualifier；@Resource 默认按名称优先再按类型；特性是规则清晰但多实现时要显式消歧，否则启动期就会失败。
    • 3分钟详答：出现频次：★★★★★（工程高频题）。
    • 开场：我会先说“注入失败基本就两类：找不到、找到了但不唯一”。
    • Autowired（按类型为主）：
      • 单实现：直接注入。
      • 多实现：需要 @Primary 或 @Qualifier 指定。
    • Resource（JSR-250）：
      • 通常按 name 优先（字段名/指定名），找不到再按类型。
    • 取舍：
      • 多实现场景我倾向 @Qualifier 明确依赖，减少隐式默认。
      • @Primary 更像“默认实现”的全局策略，适用于平台类组件（比如默认序列化器）。
    • 实战：我会避免“靠字段名注入”的隐式约定（易重构翻车），更喜欢显式 qualifier + 构造器注入。
    • 可能追问：
    • 追问：Optional/ObjectProvider 注入有什么用？
      15秒简答：用于可选依赖与延迟获取，避免启动期硬依赖，适合“有则用无则降级”的组件。
    • 追问：循环依赖和构造器注入冲突怎么办？
      15秒简答：优先重构拆环；必要时用接口抽象/事件解耦；实在不行用 setter 注入临时绕过，但要明确技术债。

    Spring MVC 里 Filter vs Interceptor vs AOP 怎么分工？别把它们混成一个¶

    • 15秒简答：它们本质是三层不同的拦截边界；核心机制是 Filter 在 Servlet 容器层拦请求，Interceptor 在 Spring MVC Handler 调用链拦控制器，AOP 在 bean 方法层拦业务调用；特性是边界越靠外越通用、越靠内越业务，选错层会让能力失焦或侵入过大。
    • 3分钟详答：出现频次：★★★★☆（面试常问“鉴权/日志/链路追踪放哪”）。
    • Filter：最外层，适合通用与协议级逻辑（跨框架）：编码、跨域、通用鉴权、请求体包装等。
    • Interceptor：面向 Spring MVC：登录态校验、限流、接口级埋点、根据 Handler 做策略。
    • AOP：面向业务方法：事务、幂等、业务审计、统一异常包装（但要小心切点过宽）。
    • 实战：链路追踪通常在 Filter/Interceptor（拿到请求入口）；事务一定在 service 层（AOP）；接口鉴权优先 Interceptor（能拿到 handler 与注解）。
    • 可能追问：
    • 追问：为什么不把所有东西都写成 AOP？
      15秒简答：AOP 粒度太细且容易误命中，Web 入口类问题更适合外层拦截；AOP 适合“方法级一致性能力”。
    • 追问：异常处理放哪更合适？
      15秒简答：Web 层用统一异常处理（ControllerAdvice 等）更自然；业务层异常用语义化异常，AOP 可做补充记录但别吞异常。

    Spring 事件机制你怎么讲？它和“直接调用”相比解决了什么痛点？¶

    • 15秒简答：事件机制本质是“容器内的发布-订阅解耦”；核心机制是 ApplicationEventPublisher 发布事件，Listener 按类型订阅处理（可同步/异步）；特性是解耦强、扩展好，但事件链路更隐式，需要监控与幂等设计。
    • 3分钟详答：出现频次：★★★☆☆（5年+常用来回答“怎么解耦”）。
    • 场景：下单后要发券、发消息、写审计。如果在下单方法里硬编码调用，会让主流程变胖且耦合多个下游。
    • 用事件做法：主流程只发布“订单已创建”事件；各个监听器独立处理各自职责，失败可独立重试/告警。
    • 为什么这样设计：在同一个应用内，事件是低成本解耦方式；比引入 MQ 更轻量（但也更容易隐式耦合）。
    • 实战：我会给事件加清晰命名与字段，监听器要幂等；如果事件链路跨系统或需要持久化/可追溯，通常升级为 MQ。
    • 可能追问：
    • 追问：同步事件会不会影响主流程延迟？
      15秒简答：会，同步监听器在同线程执行；延迟敏感时要用异步监听或把重操作下沉到 MQ。
    • 追问：异步事件有什么坑？
      15秒简答：线程池隔离、上下文传递、异常处理与重试都要自己兜住，否则容易变成“悄悄失败”。

    @Async / 线程池在 Spring 里怎么落地？为什么不建议直接用默认线程池？¶

    • 15秒简答：异步本质是“把任务从请求线程迁移到执行器”；核心机制是 @Async 通过代理把方法提交到 TaskExecutor；特性是提升吞吐但会带来线程池隔离、上下文丢失与拒绝策略问题，默认执行器不透明且不可控，生产应显式配置并监控。
    • 3分钟详答：出现频次：★★★☆☆（5年+经常从“异步化”聊到）。
    • 开场：异步不是免费午餐，核心是把延迟从请求线程挪走，但必须控制并发与失败策略。
    • 为什么不建议默认：
      • 没有业务隔离：不同任务抢同一池，容易互相拖死。
      • 无监控/无命名：排障难。
      • 拒绝策略不可控：高峰时可能直接丢任务或把上游拖慢。
    • 实战：我会为不同业务类型建不同线程池（IO/CPU/回调分离），线程命名、队列有界、拒绝策略可观测，并对异步任务加超时与失败告警。
    • 可能追问：
    • 追问：@Async 为什么也会有“自调用不生效”？
      15秒简答：同 AOP 原理，@Async 也是代理增强；同类内部调用绕过代理就不会提交到线程池。
    • 追问：异步任务怎么保证可靠执行？
      15秒简答：关键任务不要只靠线程池内存队列，需落库/落 MQ 做持久化与重试，并确保幂等。
