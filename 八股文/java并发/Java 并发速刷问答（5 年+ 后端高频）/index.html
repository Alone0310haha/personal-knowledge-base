    Java 并发速刷问答（5 年+ 后端高频）

    JMM 与并发基础¶

    线程安全你怎么拆解？JMM（Java 内存模型）到底解决了什么问题？¶

    • 15秒简答：线程安全本质是“多线程共享状态下的正确性约束”；核心机制是用 JMM 把读写抽象成“主内存/工作内存 + happens-before”来约束可见性与重排序；特性是它保证的是“可见性/有序性规则”，不直接保证业务原子性。
    • 3分钟详答：我会先给一个结论：线程安全通常拆成三件事——原子性、可见性、有序性；JMM 主要管后两件，并给出 happens-before 这种可推理规则。
    • 场景：两个线程同时读写同一个变量/对象字段，如果没有约束，就会出现“看不到别人写的新值”（可见性）或“指令重排导致半初始化对象被读到”（有序性）。
    • JMM 的抽象：每个线程有自己的“工作内存”（可理解为寄存器/缓存/重排序缓冲的综合抽象），共享变量在“主内存”。线程对共享变量的读写，需要在工作内存与主内存之间同步。
    • 它解决什么：给编译器/CPU 的重排序空间划边界，并定义哪些写对哪些读必须可见（happens-before）。
    • 为什么这样设计：要兼顾性能（允许重排序与缓存）与可推理性（给出可证明的可见性/有序性规则）。
    • 实战：我会用这个框架去解释“为什么加了锁就可见”“为什么 volatile 能修单例 DCL”，并在 code review 时更关注共享状态是否被正确发布、是否存在复合操作未加保护的问题。
    • 可能追问：
    • 追问：原子性/可见性/有序性分别怎么解决？
      15秒简答：原子性靠锁/原子类；可见性靠 volatile/锁的释放-获取；有序性靠 happens-before 约束（锁、volatile、final、安全发布等）。
    • 追问：JMM 是 JVM 的实现细节吗？
      15秒简答：不是，它是语言级规范；不同 JVM/硬件实现必须满足这些可见性与重排序规则。

    happens-before 规则你如何讲清楚？它和“可见性”是什么关系？¶

    • 15秒简答：happens-before 本质是“可见性与有序性的证明规则”；核心机制是如果 A happens-before B，则 A 的写对 B 必须可见且 A 的执行顺序先于 B；特性是它让并发正确性可以用规则推导而不是靠猜 CPU 缓存。
    • 3分钟详答：我会用“能推导出可见性”来讲：
    • 先说一句话：happens-before 不是时间先后，而是“可见性/顺序的约束关系”。
    • 常用规则（面试重点）：
      • 程序次序规则：同线程内，前面的操作 happens-before 后面的操作。
      • 监视器锁规则：对同一把锁的 unlock happens-before 后续的 lock。
      • volatile 规则：对 volatile 变量的写 happens-before 后续对同一变量的读。
      • 线程启动/终止：start() happens-before 线程内动作；线程内动作 happens-before join() 返回。
      • 传递性：A hb B 且 B hb C，则 A hb C。
    • 场景：用“写配置 → 启动线程 → 线程读取配置”解释为什么 start/join 能建立可见性链路。
    • 设计原因：允许 CPU/编译器做优化，但在关键同步点插入约束，保证程序员能推理。
    • 实战：排查并发 bug 时，我会画出 hb 链路：写入是否通过锁/volatile/线程启动建立了可见性；如果没有，就可能是“读到旧值/半初始化对象”的根因。
    • 可能追问：
    • 追问：happens-before 和内存屏障是什么关系？
      15秒简答：hb 是语言层规则；内存屏障是实现手段之一，用于禁止特定重排序并强制刷新/失效缓存。
    • 追问：只要满足 hb 就一定线程安全吗？
      15秒简答：不一定，hb 只保证可见性/顺序；复合更新还需要原子性（锁或 CAS/原子类）。

    volatile 的两大语义是什么？为什么它不能保证 i++ 的原子性？¶

    • 15秒简答：volatile 本质是“轻量级同步点”；核心机制是读写 volatile 会建立 happens-before 并插入内存屏障保障可见性与禁止部分重排序；特性是开销小但不提供复合操作原子性，所以 i++ 仍会丢更新。
    • 3分钟详答：我会按“语义 + 反例”讲清楚：
    • 开场：volatile 解决的是“看到最新值”和“不要乱重排”，不是“互斥”。
    • 两大语义：
      • 可见性：一个线程写 volatile，其他线程随后读同一 volatile 必须看到新值。
      • 有序性：volatile 写-读之间形成 hb，读写处会插入屏障，限制编译器/CPU 的重排序。
    • 为什么 i++ 不原子：i++ 是读-改-写三步，volatile 只能保证每一步的读写可见，但不能把三步合成一个不可分割操作；两个线程都读到 0，各自加 1 写回，最终还是 1。
    • 设计原因：volatile 走的是“内存可见性协议”，不做互斥，因此比锁轻，但能力也弱。
    • 实战：状态开关（停止标记）、配置热更新、发布单例引用我会用 volatile；计数/扣减这种复合更新我会用 AtomicLong/LongAdder 或锁。
    • 可能追问：
    • 追问：volatile 能替代 synchronized 吗？
      15秒简答：不能，volatile 不提供互斥与复合操作原子性；能替代的通常是“读多写少的状态标记/发布引用”类场景。
    • 追问：volatile 写为什么能让前面的写也对别的线程可见？
      15秒简答：因为 volatile 写会建立 hb，并通过屏障/缓存一致性让写入在语义上“先于”后续读可见（含传递性）。

    双重检查锁（DCL）单例为什么一定要 volatile？不加会怎样？¶

    • 15秒简答：DCL 的单例本质是“减少锁竞争的延迟初始化”；核心机制是 volatile 禁止对象创建时的指令重排，避免把“未初始化完成的引用”发布出去；特性是不加 volatile 可能出现读到半初始化对象的极低概率致命 bug。
    • 3分钟详答：我会用“new 的三步”解释：
    • 开场：DCL 的目的是让绝大多数请求走无锁路径，只在第一次初始化时加锁。
    • new Singleton() 可以抽象成三步：1) 分配内存；2) 调用构造初始化字段；3) 把引用赋值给变量。
    • 问题点：在没有约束时，2 和 3 可能重排，导致先把引用赋给 instance，另一个线程看到 instance != null 直接返回，但对象字段还没初始化完（半初始化）。
    • volatile 的作用：对 instance 的写建立 hb，禁止这类重排，并保证发布后的可见性。
    • 设计原因：DCL 本质是“正确发布”的问题；volatile 是发布引用的最低成本手段（相对每次都加锁）。
    • 实战：我更常用“静态内部类”或“枚举单例”绕开复杂性；如果必须 DCL，就强制 volatile + 单测/压测验证。
    • 可能追问：
    • 追问：为什么“静态内部类”天然线程安全？
      15秒简答：类初始化阶段由 JVM 保证串行执行，且类初始化的写对后续使用天然可见（安全发布）。
    • 追问：DCL 里外层 if 真的有意义吗？
      15秒简答：有，外层 if 避免每次都进入 synchronized，减少锁竞争；内层 if 防止并发初始化两次。

    false sharing（伪共享）是什么？怎么定位与缓解？¶

    • 15秒简答：伪共享本质是“缓存行粒度导致的无关写互相干扰”；核心机制是不同线程写不同变量但落在同一 cache line 上，引发 cache line 反复失效；特性是表现为 CPU 飙高但吞吐下降，定位难、缓解靠 padding/数据布局优化。
    • 3分钟详答：我会从 CPU 缓存一致性讲：
    • 开场：伪共享不是共享同一变量，而是共享同一缓存行。
    • 场景：两个线程分别更新 a、b 两个字段，看起来互不影响，但如果 a、b 在对象内存布局上相邻，可能同处一个 64B 缓存行；任一线程写都会让另一核心的该缓存行失效，导致频繁同步。
    • 为什么会慢：缓存行是最小一致性单位；无关字段被“绑在一起”就会被迫同步。
    • 怎么定位：看热点字段更新频率、perf/async-profiler 的 cache miss 指标（如果有）、以及对齐/分离字段后性能是否显著改善。
    • 怎么缓解：把高频写字段拆到不同对象/不同数组段；用 @Contended（需要 JVM 参数开启）或手动 padding；更常见的是用 LongAdder 这类分段计数减少单点写热点。
    • 实战：高并发计数、指标上报、环形队列这种场景最容易踩；我一般先用 LongAdder/分片统计，再在极限压测下考虑对齐优化。
    • 可能追问：
    • 追问：LongAdder 为什么能缓解伪共享/热点竞争？
      15秒简答：它把一个热点计数拆成多个 cell 分散到不同位置，让线程写不同 cell，减少对同一缓存行/同一内存位置的争抢。
    • 追问：伪共享和锁竞争怎么区分？
      15秒简答：锁竞争通常能在 dump/火焰图看到 monitor/park；伪共享更像“无锁但很慢”，CPU 高且 CAS/写热点集中。

    锁与同步器（synchronized / Lock / AQS）¶

    synchronized 到底锁的是什么？实例方法/静态方法/代码块分别锁哪里？¶

    • 15秒简答：synchronized 本质是“基于对象监视器的互斥 + 可见性边界”；核心机制是进入/退出 monitor（monitorenter/monitorexit），实例方法锁 this，静态方法锁 Class 对象，代码块锁括号里的对象；特性是语义简单、自动释放，但粒度与锁对象选择决定并发度。
    • 3分钟详答：我会按“三种用法 + 影响范围”讲：
    • 开场：synchronized 的锁是“对象级别”的，不是代码级别的。
    • 三种写法：
      • 修饰实例方法：等价于 synchronized(this)，同一个实例互斥，不同实例不互斥。
      • 修饰静态方法：锁的是 xxx.class 对应的 Class 对象，对所有实例全局互斥。
      • 修饰代码块：锁的是括号内引用指向的对象，锁对象选得好并发度就高。
    • 为什么要强调锁对象：锁范围过大等于把并发系统写成串行；锁对象过于“公共”（比如 String 常量）可能导致意外竞争。
    • 实战：我会把锁对象绑定到“真实共享资源”的生命周期上，比如针对一个订单/用户用分段锁（striped lock），避免全局锁把吞吐打穿。
    • 可能追问：
    • 追问：synchronized(new Object()) 有意义吗？
      15秒简答：基本没意义，每次 new 的锁对象不同，起不到互斥效果。
    • 追问：为什么不建议 synchronized(String)？
      15秒简答：String 常量池会复用对象，可能把本来无关的代码路径锁到一起，造成隐式全局锁。

    synchronized 的底层原理与“锁升级”怎么讲？（偏向/轻量/重量）¶

    • 15秒简答：synchronized 本质是 JVM 层的 monitor；核心机制是对象头 Mark Word 记录锁状态，竞争从偏向/轻量自旋到重量级 monitor；特性是低竞争快、高竞争会阻塞与上下文切换，且偏向锁在较新 JDK 中已逐步弱化/移除。
    • 3分钟详答：我会抓住“对象头 + 竞争形态”：
    • 开场：早期 synchronized 被认为重，但 JDK6 之后 JVM 做了大量优化，让它在低竞争下很快。
    • 对象头：锁相关信息主要在 Mark Word 里，记录偏向线程 ID、轻量锁指针等。
    • 典型演进（概念层面）：
      • 偏向锁：假设多数情况下同一线程反复进入同步块，避免 CAS 竞争；当有其他线程来抢，撤销偏向并升级。
      • 轻量级锁：通过 CAS 尝试把对象头指向栈上的 Lock Record，竞争不激烈时用自旋避免阻塞。
      • 重量级锁：竞争激烈或自旋不划算时，膨胀为 monitor，线程挂起/唤醒，发生内核态切换。
    • 设计原因：锁的目标是减少“无谓的阻塞与切换”，所以先自旋，打不过再阻塞。
    • 实战：我关注的是“是否存在长临界区/锁内 IO/锁内 RPC”；锁优化通常比纠结偏向/轻量细节更有收益。
    • 可能追问：
    • 追问：自旋一定更好吗？
      15秒简答：不一定，临界区长或竞争激烈时自旋会浪费 CPU，反而不如阻塞。
    • 追问：怎么减少 synchronized 竞争？
      15秒简答：缩小临界区、拆分锁（分段/细粒度）、避免锁内阻塞操作、用并发容器/原子类替代部分共享状态。

    ReentrantLock 和 synchronized 怎么选？说到工程味要点有哪些？¶

    • 15秒简答：它们本质都是互斥同步；核心机制是 synchronized 由 JVM 管理 monitor，ReentrantLock 基于 AQS 显式加解锁并支持更丰富控制；特性是 synchronized 简洁可靠，ReentrantLock 支持可中断/超时/公平/多条件队列，适合复杂并发控制。
    • 3分钟详答：我会从“能力差异 + 选型场景”讲：
    • 开场：如果只是保护临界区，优先 synchronized（可读性强、异常自动释放、JIT 优化成熟）。
    • ReentrantLock 的工程优势：
      • 可中断：lockInterruptibly()，适合可取消任务避免线程长期卡死。
      • 可超时：tryLock(timeout)，做降级/兜底策略，不让请求无限等待。
      • 公平锁：可选公平队列（吞吐通常低于非公平）。
      • Condition：一个锁可以创建多个条件队列，替代 wait/notify 的“单队列 + notifyAll 滥用”问题。
    • 为什么这些重要：线上最怕“线程堆积”和“不可控等待”，超时/可中断能把系统从雪崩边缘拉回来。
    • 实战：我在资源池、限流令牌、队列协调等场景会用 ReentrantLock+Condition；普通对象内的短临界区更倾向 synchronized。
    • 可能追问：
    • 追问：公平锁为什么吞吐更低？
      15秒简答：严格 FIFO 会增加排队与唤醒开销，且降低“刚释放锁的线程”快速再次获取的局部性优势。
    • 追问：忘记 unlock 会怎样？
      15秒简答：会造成永久占锁导致系统卡死；必须用 try-finally 保证释放。

    AQS 是什么？你怎么把它讲成“一个底座”？¶

    • 15秒简答：AQS 本质是“用 state + FIFO 队列封装获取/释放的模板框架”；核心机制是 CAS 修改 state，失败线程入队并 park，释放时 unpark；特性是复用性强，JUC 很多同步器（ReentrantLock、Semaphore、CountDownLatch）都基于它。
    • 3分钟详答：我会用“模板方法 + 队列”讲：
    • 开场：AQS 解决的是“写一个锁/同步器太复杂”，把排队、阻塞、唤醒这些通用逻辑收敛起来。
    • 三个核心元素：
      • state：一个 int 表示同步状态（占用次数、许可数、闩锁状态等），用 CAS 更新。
      • CLH 变体 FIFO 队列：把抢不到资源的线程封装成 Node 入队排队。
      • park/unpark：通过 LockSupport 挂起与唤醒线程，避免忙等。
    • 两种模式：独占（ReentrantLock）与共享（Semaphore/CountDownLatch）。
    • 设计原因：把“同步器的公共部分”做成框架，具体同步语义留给子类实现 acquire/release 相关方法。
    • 实战：面试我会把 AQS 的流程讲清楚即可，不强行背源码；工作里更多体现在我会正确选用这些同步器，而不是自己造轮子。
    • 可能追问：
    • 追问：CountDownLatch 和 CyclicBarrier 底层有什么不同？
      15秒简答：Latch 更像一次性门闩（共享模式 state 递减到 0 放行）；Barrier 是可复用的阶段栅栏（到齐后触发并可重置）。
    • 追问：为什么 AQS 用 CAS 而不是直接 synchronized？
      15秒简答：CAS 更适合实现可扩展的状态更新与低竞争快速路径，失败再走排队阻塞，减少无谓的内核切换。

    wait/notify、Condition、LockSupport 关系是什么？分别适合什么场景？¶

    • 15秒简答：它们本质都是线程阻塞/唤醒机制；核心机制是 wait/notify 绑定 monitor、Condition 绑定 Lock 并提供多条件队列、LockSupport 提供最底层的 park/unpark；特性是从高层到低层逐步“更灵活但更易用错”。
    • 3分钟详答：我会按“绑定对象 + 队列模型”讲：
    • wait/notify：
      • 绑定对象监视器，必须在 synchronized 内调用，否则抛异常。
      • 每个对象只有一个等待队列，复杂条件容易写成 notifyAll，导致惊群。
    • Condition：
      • 绑定 Lock（常见是 ReentrantLock），一个锁可创建多个 Condition，对应多个条件队列，表达力更强。
      • 常用写法是 while (!condition) await() 防止虚假唤醒。
    • LockSupport：
      • 更底层，AQS 就用它来 park/unpark。
      • “许可”语义：unpark 先发生也能生效（不会丢信号），比 wait/notify 更不容易丢通知，但也更需要自己管协议。
    • 实战：业务代码一般不用 LockSupport；需要复杂条件同步时用 Lock+Condition；简单的生产者消费者可直接用 BlockingQueue。
    • 可能追问：
    • 追问：为什么 await 要用 while 而不是 if？
      15秒简答：要防虚假唤醒与条件变化，醒来后必须重新检查条件是否满足。
    • 追问：notify 和 notifyAll 怎么选？
      15秒简答：能精确唤醒满足条件的线程就 notify，否则用 notifyAll 配合 while 防错，但要评估惊群开销。

    原子类与并发容器¶

    CAS 是什么？它为什么能“无锁并发”？有哪些典型坑？¶

    • 15秒简答：CAS 本质是“比较并交换的原子指令”；核心机制是用 CPU 原子指令把“读-比较-写”合成一个不可分割操作，失败就重试；特性是低竞争下快，但高竞争会自旋耗 CPU，且有 ABA 等语义坑。
    • 3分钟详答：我会用 AtomicInteger 举例：
    • 开场：CAS 的语义是“如果内存值仍等于期望值，就更新为新值，否则失败”。
    • 为什么无锁：它不阻塞线程，而是失败重试；因此减少了上下文切换。
    • 典型问题：
      • ABA：值从 A 变 B 又回 A，CAS 认为没变但语义已变；用版本号（AtomicStampedReference）或更高层协议解决。
      • 自旋开销：高竞争下大量线程反复失败，CPU 空转，吞吐可能比锁更差。
      • 只能保证一个变量的原子更新：多变量一致性要么用更高层结构，要么还是得锁。
    • 实战：库存扣减、指标计数、状态机更新我会优先考虑 CAS/原子类；但如果临界区涉及多字段一致性或外部 IO，就回到锁/事务。
    • 可能追问：
    • 追问：CAS 一定比锁快吗？
      15秒简答：不一定，低竞争短临界区 CAS 快；高竞争长尾下自旋可能更慢，锁的阻塞反而更稳。
    • 追问：ABA 在什么场景更危险？
      15秒简答：无锁栈/无锁队列这类依赖指针/引用语义的结构，值回到 A 但结构已变化，会出错。

    AtomicLong vs LongAdder 怎么选？为什么高并发计数推荐 LongAdder？¶

    • 15秒简答：它们本质都是线程安全计数器；核心机制是 AtomicLong 单点 CAS 更新同一个值，LongAdder 把计数拆成多个 cell 分散更新最后求和；特性是低并发 AtomicLong 更轻，高并发 LongAdder 吞吐更好但读值是“近似快照”且求和有成本。
    • 3分钟详答：我会用“写热点”解释：
    • 开场：计数是典型热点写，一个变量会被所有线程不断更新。
    • AtomicLong：每次 incrementAndGet 都 CAS 同一个内存位置，高并发下 CAS 失败率高，线程自旋重试导致抖动。
    • LongAdder：内部维护 base + cells，线程根据探针哈希落到不同 cell 上更新，减少对同一位置的争抢；读取时 sum 把 base 和所有 cell 相加。
    • 设计原因：用空间换时间，把热点拆散，减少缓存行争用与 CAS 冲突。
    • 实战：QPS/指标统计用 LongAdder；需要强一致的当前值（比如做序号/阈值判断）更倾向 AtomicLong 或锁。
    • 可能追问：
    • 追问：LongAdder 的 sum() 一定准确吗？
      15秒简答：它不是线性一致的强原子读，读取时可能与并发更新交错，因此更适合统计而非强一致控制。
    • 追问：什么时候不适合 LongAdder？
      15秒简答：需要“读-判断-写”强一致语义的场景（比如限流阈值硬判断），LongAdder 的读可能不够强。

    ConcurrentHashMap 的并发控制机制怎么讲？哪些操作仍然不是原子复合操作？¶

    • 15秒简答：ConcurrentHashMap 本质是“高并发下的分桶哈希表”；核心机制是 JDK8 主要用 CAS 插入 + 桶内 synchronized 保护链表/树结构，并配合扩容协作；特性是读高并发无锁，写在桶粒度上锁，但“先 get 再 put”这类复合逻辑仍需用 compute/merge 等原子 API。
    • 3分钟详答：我会抓住 JDK8 以后主流实现：
    • 开场：它追求的是“尽可能让读无锁，让写局部化”。
    • 写入流程（简化版）：
      • 定位桶位置，如果为空优先 CAS 放入节点。
      • 桶非空则对桶头节点 synchronized，在桶内做链表遍历/插入；链表过长会树化以降低退化。
      • 扩容时多个线程可以协作搬迁桶，避免单线程扩容长时间停顿。
    • 读操作通常无锁：依赖 volatile 读与结构约束，保证遍历能看到一致的结构。
    • 复合操作的坑：if (map.get(k)==null) map.put(k,v) 不是原子，会并发放入两次或覆盖；正确做法用 computeIfAbsent、putIfAbsent、merge。
    • 实战：缓存（本地热点、配置快照）常用 CHM；我会特别强调“复合操作用原子方法”以及避免在 compute 回调里做阻塞 IO（容易拖垮桶锁）。
    • 可能追问：
    • 追问：size() 一定是精准实时的吗？
      15秒简答：高并发下 size 计算会做近似/分段统计并最终收敛，强实时精度不一定有保证（看具体实现与并发程度）。
    • 追问：computeIfAbsent 回调里能再 put 同一个 map 吗？
      15秒简答：不建议，容易造成递归、长时间持有桶锁或死锁风险；回调应保持纯计算、短且无阻塞。

    ThreadLocal 原理是什么？为什么会内存泄漏？怎么规避？¶

    • 15秒简答：ThreadLocal 本质是“线程私有变量容器”；核心机制是每个 Thread 持有 ThreadLocalMap，key 是 ThreadLocal 的弱引用、value 是强引用；特性是线程池线程长生命周期下容易残留 value 导致泄漏，必须 remove 或使用正确的作用域管理。
    • 3分钟详答：我会用“线程池 + 上下文”场景讲：
    • 开场：ThreadLocal 不是解决共享数据的线程安全，而是让每个线程有一份自己的副本（比如 traceId、用户上下文、SimpleDateFormat 等）。
    • 原理：Thread 内部有 threadLocals（ThreadLocalMap），ThreadLocal.set/get 实际是在当前线程的 map 里读写。
    • 泄漏点：ThreadLocalMap 的 key 是弱引用，ThreadLocal 对象如果被 GC，key 变 null，但 value 仍被 map 强引用着；如果线程长期不结束（线程池），value 就长期占内存。
    • 规避：
      • 用 try-finally 在任务结束时 remove()。
      • 避免把大对象/连接/缓存塞进 ThreadLocal。
      • 对线程池场景，明确 ThreadLocal 的生命周期（一次请求/一次任务）并在边界清理。
    • 实战：我用 ThreadLocal 存 requestId/MDC 时，会统一在过滤器/拦截器/切面里 set 与 remove，避免业务代码散落。
    • 可能追问：
    • 追问：InheritableThreadLocal 能解决线程池的上下文传递吗？
      15秒简答：不完全能，它只在创建子线程时复制；线程池线程复用不会触发复制，仍会丢上下文或串号。
    • 追问：ThreadLocal 适合存数据库连接吗？
      15秒简答：不建议，连接生命周期与异常路径复杂，容易泄漏；更推荐连接池 + 显式传递/上下文管理。

    BlockingQueue 怎么选型？常见队列的差异点是什么？¶

    • 15秒简答：BlockingQueue 本质是“生产者-消费者的并发缓冲区”；核心机制是 put/take 在容量或元素条件不满足时阻塞，并用锁/条件队列协调；特性是用它能显著简化线程协作，但队列类型决定背压、吞吐与内存上限。
    • 3分钟详答：我会按“是否有界 + 是否直传 + 是否延迟”讲：
    • ArrayBlockingQueue：有界数组队列，内存上限明确，适合做背压；吞吐稳定。
    • LinkedBlockingQueue：可选有界，但默认近似无界（如果不传容量），易堆积导致内存风险；适合任务波峰较大但必须配容量。
    • SynchronousQueue：不存元素，生产者必须直接交给消费者；适合 cached 线程池这种“任务来就建/用线程”的策略，但可能创建大量线程。
    • DelayQueue：按延迟时间出队，适合定时任务、超时管理、重试调度。
    • 选型原则：优先有界（控内存）+ 明确背压策略；不要默认无界队列“把压力转嫁到内存”。
    • 实战：线程池任务队列我优先有界队列；需要严格限流/背压时用小容量队列 + 合理拒绝策略。
    • 可能追问：
    • 追问：为什么有界队列更安全？
      15秒简答：它把压力变成可控的“阻塞或拒绝”，而不是无限堆积到内存最终 OOM。
    • 追问：生产者消费者模型里为什么常用 BlockingQueue 而不是 wait/notify？
      15秒简答：BlockingQueue 封装了正确的阻塞/唤醒协议与并发控制，代码更少、更不易写错。

    线程池与异步编程¶

    ThreadPoolExecutor 的七大参数与工作流程怎么讲清楚？¶

    • 15秒简答：线程池本质是“用复用线程处理任务并提供背压点”；核心机制是按 corePoolSize→队列→maxPoolSize 的顺序接纳任务，最终触发拒绝策略；特性是参数决定吞吐/延迟/资源上限，配置不当会导致 OOM 或线程耗尽。
    • 3分钟详答：我会用“接单流程”讲：
    • 七大参数（高频）：
      • corePoolSize：核心线程数，常驻处理。
      • maximumPoolSize：最大线程数，上限兜底。
      • keepAliveTime：非核心线程空闲存活时间。
      • workQueue：任务队列（背压位置）。
      • threadFactory：线程命名/优先级/是否 daemon。
      • handler：拒绝策略。
      • unit：时间单位。
    • 工作流程（关键）：提交任务时
      • 若运行线程数 < core：创建核心线程执行。
      • 否则尝试入队。
      • 若队列满且线程数 < max：创建非核心线程执行。
      • 否则触发拒绝策略。
    • 设计原因：用队列与线程上限实现“弹性 + 保护”，避免无限建线程或无限堆任务。
    • 实战：我会把线程池当成资源池来管理：命名、监控、区分业务池、设置有界队列、明确拒绝策略与降级。
    • 可能追问：
    • 追问：corePoolSize 怎么估？
      15秒简答：CPU 密集型接近 CPU 核数（或核数+1），IO 密集型可更大；更关键是压测+监控迭代，而不是背公式。
    • 追问：为什么线程池要单独命名？
      15秒简答：线上排障靠线程名定位来源，没命名 thread dump/监控里很难分辨是哪类业务线程。

    线程池队列与拒绝策略怎么搭配？为什么不建议直接用 Executors？¶

    • 15秒简答：线程池配置本质是“背压策略选择题”；核心机制是队列容量决定压力是堆在队列还是扩线程，拒绝策略决定满载时如何失败；特性是 Executors 常用实现默认无界队列或极大线程上限，容易 OOM/线程风暴。
    • 3分钟详答：我会给一个工程化的选型路径：
    • 先定背压：系统满载时是“让调用方慢下来”（CallerRuns/阻塞）还是“快速失败”（Abort/自定义拒绝）还是“丢弃可降级任务”（Discard/DiscardOldest）。
    • 再定队列：
      • 有界队列：更可控，常用于业务线程池。
      • SynchronousQueue：更偏向扩线程，适合短任务、低延迟但必须控 max。
      • 默认无界队列风险：任务堆积把问题变成内存问题，最终 OOM。
    • 为什么不建议 Executors：例如 FixedThreadPool/SingleThreadExecutor 常用近似无界的 LinkedBlockingQueue；CachedThreadPool 可能创建大量线程；这些默认值在高峰下不安全。
    • 实战：我会用 ThreadPoolExecutor 显式声明：有界队列 + 自定义 ThreadFactory（命名）+ 自定义拒绝（打点告警、降级、返回明确错误）。
    • 可能追问：
    • 追问：CallerRunsPolicy 有什么副作用？
      15秒简答：会把压力传回提交线程，可能拖慢上游（甚至拖垮 IO 线程）；但它能形成天然背压，适合可接受变慢的链路。
    • 追问：AbortPolicy 线上能直接用吗？
      15秒简答：可以但要配合捕获与降级，否则 RejectedExecutionException 可能直冲用户；更常见是自定义拒绝做监控与熔断。

    execute vs submit 有什么差异？Future.get() 的坑怎么规避？¶

    • 15秒简答：它们本质都是提交任务；核心机制是 execute 只管执行且异常走线程未捕获路径，submit 会包装成 Future 并把异常封装在 get 里；特性是 submit 容易“吞异常”，get 还会阻塞导致线程堆积，必须配超时与降级。
    • 3分钟详答：我会分三点讲：
    • 返回值：execute 返回 void；submit 返回 Future（或 FutureTask）。
    • 异常行为：execute 任务抛异常通常会被线程的 UncaughtExceptionHandler 处理（或打印日志）；submit 的异常会被捕获并在 Future.get() 时以 ExecutionException 抛出，如果从不 get，就等于悄悄失败。
    • Future.get() 的坑：
      • 可能永久阻塞：任务卡死/死锁/外部 IO 超时没设，会导致 get 一直等。
      • 级联阻塞：在有限线程池里“任务 A 提交任务 B 然后 get B”，容易把线程都堵死（线程池饥饿死锁）。
    • 规避：
      • 只在需要结果时用 submit，并且 get(timeout) + 超时取消。
      • 对外部 IO 强制超时（HTTP/DB 客户端）。
      • 分离线程池：阻塞型任务与计算型任务不要混用同一个池。
    • 实战：我会给异步任务加统一的异常处理与指标（失败率/超时率/队列长度），避免“异步黑洞”。
    • 可能追问：
    • 追问：Future.cancel(true) 一定能停掉任务吗？
      15秒简答：不一定，它只是发中断信号；任务必须响应中断（检查中断位或阻塞点抛 InterruptedException）才能停。
    • 追问：为什么任务里不要直接吞 InterruptedException？
      15秒简答：吞掉会破坏取消协议；通常应恢复中断位或向上抛，让上层能正确终止流程。

    CompletableFuture 面试怎么讲？默认线程池是什么？有哪些踩坑点？¶

    • 15秒简答：CompletableFuture 本质是“可组合的异步 DAG”；核心机制是阶段（stage）链式编排，默认很多异步方法跑在 ForkJoinPool.commonPool 或调用方线程，显式传 executor 才可控；特性是组合能力强但线程池与阻塞混用容易把 commonPool 打穿。
    • 3分钟详答：我会用“组合替代回调地狱”讲：
    • 开场：它解决的是 Future “只能 get、不能编排”的问题，把异步流程变成可组合的阶段图。
    • 常见组合：
      • thenApply/thenCompose：map vs flatMap（是否返回新的 future）。
      • allOf/anyOf：并行聚合与竞争返回。
      • exceptionally/handle/whenComplete：异常兜底与观测。
    • 默认线程池：
      • supplyAsync/runAsync 不传 executor 时通常用 ForkJoinPool.commonPool。
      • 非 Async 后缀的方法可能在“完成前一阶段的线程”里执行（可能是业务线程）。
    • 踩坑：
      • 在 commonPool 里做阻塞 IO（HTTP/DB）会导致池被占满，影响全局。
      • 忘记统一异常处理，导致异常被封装但无人观测。
    • 实战：我会显式传入业务线程池（按业务隔离），并统一设置超时/降级与异常汇总处理。
    • 可能追问：
    • 追问：thenCompose 和 thenApply 怎么一句话区分？
      15秒简答：thenApply 把结果映射成值；thenCompose 把结果映射成另一个 CompletableFuture 并把它“拍平”。
    • 追问：如何给 CompletableFuture 加超时？
      15秒简答：用 orTimeout/completeOnTimeout（高版本 JDK）或自己用定时器/调度线程在超时后完成/取消，并配合下游兜底。

    ForkJoinPool 的“工作窃取”是怎么提升吞吐的？什么任务不适合？¶

    • 15秒简答：ForkJoinPool 本质是“面向可拆分任务的并行执行器”；核心机制是每个 worker 有本地双端队列，空闲线程从别人的队列尾部窃取任务减少负载不均；特性是适合 CPU 密集可分治任务，不适合长阻塞 IO 或任务粒度过粗导致窃取失效。
    • 3分钟详答：我会从“递归拆分”讲：
    • 开场：典型场景是分治算法（大任务拆小任务并行），比如大数组求和、递归遍历、批量计算。
    • 工作窃取的意义：把任务尽量留在本地队列（缓存友好），空闲时再去窃取，减少集中调度开销与负载倾斜。
    • 不适合的任务：
      • 长时间阻塞 IO：会占住 worker，导致可用并行度下降（除非使用 managedBlock 或把 IO 放到专门线程池）。
      • 任务不可拆/拆分成本高：拆了也不并行，反而增加调度成本。
    • 实战：我会把 ForkJoinPool 用在纯计算场景；业务 IO 异步更偏 CompletableFuture + 专用线程池隔离。
    • 可能追问：
    • 追问：parallelStream 默认用的是什么池？
      15秒简答：通常也是 ForkJoinPool.commonPool，因此在服务端慎用，避免把全局池与业务 IO 绑在一起。
    • 追问：为什么 commonPool 被打满会影响全局？
      15秒简答：它是 JVM 级共享池，多个组件/库可能共用，任何一方阻塞或占满都可能拖慢其他异步任务。

    排障与性能实战¶

    线上你如何排查：死锁、锁竞争、线程池耗尽？¶

    • 15秒简答：排障本质是“定位阻塞点与资源瓶颈”；核心机制是用 thread dump/jstack/监控指标确认线程状态（BLOCKED/WAITING）、锁持有关系与线程池队列/活跃度；特性是先分清“卡在锁、卡在 IO、还是卡在队列”，再对症下药。
    • 3分钟详答：我会给一套可执行流程：
    • 死锁：抓 thread dump（多次更好），看是否有 “Found one Java-level deadlock”，确认互相持有的锁与栈帧位置；若是业务锁顺序问题，按统一顺序加锁或合并锁域。
    • 锁竞争：看大量线程处于 BLOCKED，栈顶在同一 synchronized 或 LockSupport.park；确认临界区是否过大、是否有锁内 IO/RPC；必要时拆锁、缩短锁域、改并发容器/无锁结构。
    • 线程池耗尽：
      • 指标：activeCount 接近 max、queue size 持续增长、reject 次数上升。
      • 栈：线程多卡在外部 IO 或下游慢调用，导致任务处理不过来。
      • 处理：加超时、做隔离（不同业务不同池）、限流/降级、调整队列与拒绝策略。
    • 设计原因：并发问题很多是“等待放大”，必须用超时与隔离控制传播。
    • 实战：我会把线程池监控做成常态（队列长度、活跃线程、完成任务、拒绝次数、任务耗时分位数），让问题在雪崩前被发现。
    • 可能追问：
    • 追问：如何快速判断是 CPU 满还是线程阻塞？
      15秒简答：CPU 满通常 RUNNABLE 多且火焰图热点明显；阻塞问题则 WAITING/BLOCKED 多，栈顶在 park/wait/锁获取或 IO。
    • 追问：Thread dump 看起来一堆 WAITING 正常吗？
      15秒简答：可能正常（线程池空闲、队列等待）；关键看是否在业务高峰仍大量等待且队列增长/响应变慢。

    线程池如何“优雅关闭”？怎么保证任务一致性与可观测？¶

    • 15秒简答：优雅关闭本质是“停止接单、处理存量、可控退出”；核心机制是 shutdown() 停止接收新任务、awaitTermination 等待完成，超时再 shutdownNow() 中断并返回未执行任务；特性是要配合任务幂等、超时与中断响应，否则关不掉或关不干净。
    • 3分钟详答：我会按“协议 + 兜底”讲：
    • 开场：优雅关闭不是“立刻停”，而是有步骤地收敛。
    • 推荐流程：
      • shutdown()：不接收新任务，让队列里的任务继续跑。
      • awaitTermination(timeout)：等待一段时间。
      • 超时则 shutdownNow()：尝试中断正在执行的线程，并拿到未开始的任务列表做补偿或记录。
    • 任务层面的配合：
      • 任务必须响应中断：阻塞点可抛 InterruptedException，循环里检查中断位。
      • 外部调用必须有超时：否则任务可能永远卡住，线程池关不掉。
      • 幂等与补偿：shutdownNow 可能导致任务半途退出，必须可重试/可回滚。
    • 可观测：关闭期间打印/打点“剩余队列长度、活跃线程数、未执行任务数”，否则线上只知道“关不掉”。
    • 实战：定时任务/消费者服务升级发布时，这套流程能显著降低停机时间与数据不一致风险。
    • 可能追问：
    • 追问：shutdown() 和 shutdownNow() 的区别？
      15秒简答：shutdown 停止接收新任务但继续执行已提交；shutdownNow 尝试中断正在执行并返回未开始的任务列表。
    • 追问：为什么不建议在 finally 里吞 InterruptedException？
      15秒简答：会让关闭/取消失效，导致线程无法及时退出；应恢复中断位或向上抛并结束任务。

    “安全发布”怎么讲？哪些方式能保证别人拿到的对象一定初始化完成？¶

    • 15秒简答：安全发布本质是“把对象引用暴露给其他线程时建立 happens-before”；核心机制是通过 final 语义、volatile 写、锁释放、静态初始化等建立可见性与禁止重排；特性是没做安全发布就可能读到旧值或半初始化对象，问题隐蔽但致命。
    • 3分钟详答：我会把“发布点”作为核心：
    • 开场：并发 bug 很多不是写错算法，而是对象被发布得太早或没有可见性保证。
    • 常见安全发布方式：
      • 静态初始化：类初始化由 JVM 保证串行且可见。
      • volatile 写引用：发布引用时禁止重排并建立 hb。
      • 通过锁：在持锁写入并释放锁，随后另一个线程获取同一锁读取，unlock hb lock。
      • final 字段：构造完成后 final 字段有更强的可见性保证（前提是 this 没有在构造期间逸出）。
      • 线程启动：在 start 前写入，线程内读取可见。
    • 反例：构造函数里把 this 注册到全局容器/回调里（this 逸出），别的线程可能看到半初始化对象。
    • 实战：我会用“不可变对象 + 构造完成后一次性发布”的方式减少共享可变状态；对缓存对象，会用 volatile 引用或 CHM 的 compute 原子发布。
    • 可能追问：
    • 追问：final 一定线程安全吗？
      15秒简答：final 只对字段的发布有帮助，不等于对象整体线程安全；对象内部如果还有可变状态仍需同步。
    • 追问：this 逸出为什么危险？
      15秒简答：构造未完成就把引用暴露出去，其他线程可能看到默认值/未初始化字段，违反不变式。

    本地缓存/单例/懒加载在并发下最常见的坑是什么？怎么写得又快又稳？¶

    • 15秒简答：这类问题本质是“读多写少的共享状态发布与复合操作原子性”；核心机制是用正确的发布方式（volatile/类初始化）+ 原子复合 API（computeIfAbsent）避免竞态；特性是错误写法通常线上才爆（偶发空指针、重复初始化、缓存穿透放大）。
    • 3分钟详答：我会用三种常见坑讲：
    • 坑1：check-then-act 非原子：if (cache.get(k)==null) loadAndPut()，并发下会重复加载、击穿下游。
      • 方案：用 computeIfAbsent 或者用“互斥加载”（分段锁/单飞）保证同一个 key 只有一个线程加载。
    • 坑2：懒加载发布不安全：DCL 不加 volatile，或把可变对象直接暴露。
      • 方案：静态内部类/枚举；或 volatile 发布引用。
    • 坑3：锁内做慢操作：把 RPC/DB 放进 synchronized，导致锁竞争与雪崩。
      • 方案：锁只保护状态切换（比如放一个 Future/占位符），慢操作在锁外做，完成后再原子替换。
    • 实战：本地缓存我常用 CHM + computeIfAbsent（回调里避免 IO），更复杂的用“缓存 Future”实现单飞，显式超时与失败回收，避免把错误结果永久缓存。
    • 可能追问：
    • 追问：computeIfAbsent 回调里做 IO 为什么危险？
      15秒简答：回调可能在桶锁/同步路径内执行，IO 慢会拖住同桶的其他操作，放大延迟与锁竞争。
    • 追问：什么是“单飞”（singleflight）思想？
      15秒简答：同一个 key 的并发请求只让一个线程去加载，其余等待同一个结果，避免击穿与重复工作。

    异步与线程池下的“上下文丢失/串号”怎么处理？（ThreadLocal、MDC、链路追踪）¶

    • 15秒简答：上下文问题本质是“线程切换导致 ThreadLocal 绑定断裂或残留”；核心机制是在线程边界显式传递上下文并在任务结束清理，或使用框架能力做上下文捕获/恢复；特性是线程池复用会导致丢 traceId 或串号，必须把 set/remove 变成制度化边界逻辑。
    • 3分钟详答：我会用“请求链路”讲：
    • 开场：ThreadLocal 只对“同一个线程”生效，而异步/线程池会切线程，所以要么传参，要么做上下文搬运。
    • 常见问题：
      • 丢失：请求线程 set 的 traceId，在异步线程里拿不到。
      • 串号：线程池线程上一次任务留下的 ThreadLocal 没 remove，被下一个任务复用。
    • 处理策略：
      • 最稳：显式传参（把 traceId 放到方法参数/上下文对象）+ 统一清理。
      • 工程化：在提交任务时包装 Runnable/Callable，捕获当前线程上下文，在执行线程里 set，执行完 finally remove。
      • 与日志/链路：MDC（如果用）也属于 ThreadLocal，需要同样的捕获与清理。
    • 实战：我会把上下文传播做成基础设施（统一的 Executor 包装器），业务侧只使用，避免每个人自己 set/remove 导致漏清理。
    • 可能追问：
    • 追问：为什么线程池更容易串号？
      15秒简答：线程池线程生命周期长且复用，不清理 ThreadLocal 就会把上一次任务的上下文带到下一次。
    • 追问：上下文传递和性能怎么平衡？
      15秒简答：只传必要字段（traceId/tenant），避免大对象；并把传播逻辑集中在框架层，减少重复与出错。
