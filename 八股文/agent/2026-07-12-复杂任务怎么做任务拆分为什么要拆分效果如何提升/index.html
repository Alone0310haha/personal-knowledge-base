    AI Agent 任务拆分¶

    复杂任务怎么做任务拆分？为什么要拆分？效果如何提升？¶

    15秒简答

    任务拆分的本质是把一个复杂目标拆成多个边界清晰、可独立执行和验证的子步骤。它的核心机制是让 LLM 每一步只聚焦一个子目标，必要时用 Plan-and-Execute 先规划再执行，并基于步骤依赖做并行调度。它的特性是准确率更高、可验证可重试，但要控制粒度和规划成本。

    3分钟详答

    复杂任务拆分，本质上是把一个大目标拆成多个小步骤，让 LLM 每次只处理一件边界清晰的事，而不是把搜索、分析、写作、汇总等中间状态全塞进一次生成里。这样做能降低上下文混乱带来的错误，并让每一步的结果都能被单独检查和重试。

    原文用“竞品分析报告”说明问题：如果让 LLM 一次性完成，它可能在搜索阶段提前夹杂分析，在写表格时又引入新信息，甚至写到后面忘掉前面关键数据。根因是 context window 有限，任务越大，中间状态越多，模型越难持续追踪当前子目标。拆分之后，每一步的上下文更干净，模型注意力集中，质量自然更稳定。

    拆分方式主要有两类。静态拆分是人提前把流程写死，形成固定 Workflow，例如“搜索资料 -> 整理大纲 -> 逐段撰写 -> 润色校对”，优点是可预测、好排查，缺点是灵活性低。动态拆分是让 LLM 先根据目标生成计划，再按计划执行，也就是 Plan-and-Execute 的核心：先规划，再执行，最后汇总。它更适合开放或变化较大的复杂任务，但规划质量不稳定，计划错了会影响后续执行。

    拆完之后还要分析步骤之间的依赖关系。没有依赖的步骤可以并发执行，有依赖的步骤必须等待父步骤完成。原文用 DAG 表达这种关系：节点是步骤，边是依赖；并行优化降低的不是单步耗时，而是关键路径总耗时。若任务依赖稀疏、工具 I/O 占主要耗时，端到端延迟可能降低 40% 到 60%；如果所有步骤都强依赖上一步，并行空间就很小。

    粒度也不能一味变细。拆太粗会让每步仍然承担太多职责，容易出错且难定位；拆太细会增加 LLM 调用次数和 token 成本，还可能让每步失去全局感，最终衔接生硬。实践上可用“原子操作”判断：一个步骤只做一件独立的事，有明确输入输出，最好能写成清晰的函数签名。每个步骤还应配套验收标准，方便自动检查、失败重试和质量控制。

    原文的代码示例主要说明并行执行思路：用类似 asyncio.gather 的方式同时启动相互独立的步骤，等待全部完成后再进入依赖它们的后续步骤。它强调的不是具体代码，而是先识别依赖图，再按关键路径调度。

    追问：为什么任务拆分能提升准确率？

    15秒简答： 因为 LLM 的 context window 有限，复杂任务会堆积大量中间状态，导致模型难以追踪当前子目标；拆分后每步上下文更干净、目标更单一，错误率更低。

    追问：静态拆分和动态拆分有什么区别？

    15秒简答： 静态拆分是人提前写死 Workflow，稳定可控但灵活性低；动态拆分是让 LLM 先规划步骤再执行，适应性强但规划质量不稳定。

    追问：任务拆完后为什么还要做依赖分析？

    15秒简答： 因为有些步骤可以并行、有些必须串行；把步骤建成 DAG 后，并发执行无依赖节点，可以缩短关键路径，从而降低端到端延迟。

    追问：任务拆分是不是越细越好？

    15秒简答： 不是。太粗会让单步仍然复杂、难定位错误；太细会增加调用和 token 成本，还可能割裂上下文。合适粒度是“原子操作”：边界清晰、输入输出明确、可独立验证。

    追问：执行过程中发现计划不合适怎么办？

    15秒简答： 可以引入自适应拆分和 Replan：某步做不好就继续拆成更小子任务；某步结果让原计划失效时，把当前结果和剩余计划交给规划模块重新判断和调整。
