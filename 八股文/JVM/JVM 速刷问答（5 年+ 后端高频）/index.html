    JVM 速刷问答（5 年+ 后端高频）

    运行时数据区与对象分配¶

    JVM 运行时数据区你怎么讲？哪些线程私有、哪些共享？¶

    • 15秒简答：运行时数据区本质是“JVM 执行字节码所需的内存分区”；核心机制是按线程隔离分为线程私有（PC/栈/本地方法栈）与共享（堆/方法区(元空间)/直接内存等）；特性是私有区随线程生灭、共享区随 JVM 生灭，OOM 风险集中在堆/元空间/直接内存。
    • 3分钟详答：出现频次：★★★★★（JVM 第一题，后续所有问题的地图）。
    • 开场：我会先按“是否线程共享”划分，这样面试官能快速判断你有没有全局骨架。
    • 线程私有：
      • 程序计数器：记录当前线程执行到哪条字节码指令（恢复执行位置用）。
      • 虚拟机栈：每次方法调用压栈一个栈帧（局部变量表、操作数栈、返回地址等）。
      • 本地方法栈：服务于 JNI/Native 调用（HotSpot 常与虚拟机栈实现上合并）。
    • 线程共享：
      • 堆：对象实例/数组的主要分配区，也是 GC 主战场。
      • 方法区（HotSpot 的元空间实现）：类元数据、常量、JIT 代码缓存等。
      • 直接内存：NIO DirectBuffer/堆外内存映射，不属于规范数据区但常见 OOM 来源。
    • 为什么这样设计：线程私有区强调调用/执行的快速性与隔离性；共享区承载对象与类元信息，便于复用与 GC 管理。
    • 实战：线上定位 OOM/卡顿，我先判断是堆、元空间还是堆外，再决定是抓 heap dump、查类加载，还是查 native 内存与 direct buffer。
    • 可能追问：
    • 追问：为什么程序计数器几乎不会 OOM？
      15秒简答：它只保存当前执行位置（很小的地址/行号信息），随线程存在且占用极小空间。
    • 追问：堆和栈的根本差别是什么？
      15秒简答：堆存对象实例、线程共享、GC 管；栈存方法执行帧、线程私有、随调用出入栈自动释放。

    JDK 8 为什么把永久代换成元空间？这会改变你排查问题的方式吗？¶

    • 15秒简答：元空间本质是“用本地内存承载方法区的实现”；核心机制是类元数据从 JVM 堆内（PermGen）迁到 native memory（Metaspace），默认可扩展；特性是 PermGen OOM 少了，但 Metaspace OOM/容器内存压力更常见，排查要看类加载与 native 内存。
    • 3分钟详答：出现频次：★★★★☆（高频，尤其 Java8+ 生产环境）。
    • 开场：方法区是规范概念，永久代/元空间是 HotSpot 的实现选择。
    • 为什么替换：
      • PermGen 容量固定且容易爆（类多、动态生成类多时常见）。
      • 元空间用本地内存，扩展更灵活，降低“默认配置就炸”的概率。
    • 排查方式的变化：
      • 以前 PermGen OOM 常看 -XX:MaxPermSize；现在关注 -XX:MaxMetaspaceSize、类加载数量、动态代理/字节码增强（CGLIB、ASM、脚本引擎等）。
      • 以及：容器环境里 RSS/limits 约束更关键，元空间和 direct memory 都算进进程内存，容易被 OOMKilled。
    • 实战：我遇到过“热加载/频繁生成代理类”导致 Metaspace 上升，最终 Full GC 无法回收类元数据（类加载器泄漏），必须定位 ClassLoader 引用链而不是只加大上限。
    • 可能追问：
    • 追问：Metaspace OOM 的典型根因是什么？
      15秒简答：大量动态生成类或 ClassLoader 泄漏（旧类加载器无法回收），导致元空间持续增长。
    • 追问：MaxMetaspaceSize 不配会怎样？
      15秒简答：默认接近“只受本地内存限制”，可能把进程内存吃满导致系统/容器层面 OOM，而不是 JVM 先报错。

    Java 对象创建过程你怎么讲？（new 到可用经历了什么）¶

    • 15秒简答：对象创建本质是“在堆上分配 + 初始化 + 发布引用”；核心机制是类检查→分配内存（TLAB/指针碰撞/空闲链表）→零值填充→对象头设置→执行构造；特性是多数对象短命，分配走线程本地路径非常快，但发布不当会引发并发可见性问题。
    • 3分钟详答：出现频次：★★★★☆（常与 TLAB、逃逸分析、GC 一起问）。
    • 开场：面试我按“分配”和“初始化”两条线讲。
    • 分配前：先确保类已加载/初始化（必要时触发类加载）。
    • 分配：在 Eden 里给对象分配空间：
      • TLAB：线程本地分配缓冲，避免分配时加锁。
      • 如果没有 TLAB 或用尽：走全局分配（指针碰撞或空闲列表，取决于堆是否规整）。
    • 初始化：
      • 先对对象内存做零值填充，保证字段默认值可见。
      • 设置对象头（Mark Word、Klass 指针等）。
      • 执行 <init> 构造函数（含实例字段显式初始化、代码块等）。
    • 发布：把引用赋给变量/字段；并发下需要安全发布（volatile/锁/静态初始化等），否则可能看到半初始化对象。
    • 实战：排查“频繁 GC/对象创建过多”时，我会从热点分配路径入手（对象池/复用、减少临时对象、批量处理），而不是先调 GC。
    • 可能追问：
    • 追问：对象一定在堆上吗？
      15秒简答：大多数在堆上；但逃逸分析后可能被标量替换/栈上分配（本质是优化为不真实分配或更短生命周期）。
    • 追问：什么是 TLAB？
      15秒简答：线程本地的一小块 Eden 分配区，线程在里面 bump-the-pointer 分配，避免多线程争抢堆顶指针。

    TLAB、逃逸分析、标量替换这套优化你怎么“面试化”讲清楚？¶

    • 15秒简答：这套优化本质是“让对象分配更少、更本地、更可消除”；核心机制是 JIT 做逃逸分析判断对象是否逃出方法/线程，能不分配就标量替换，必须分配就尽量走 TLAB；特性是能显著减少 GC 压力，但只对热点代码与可优化形态生效。
    • 3分钟详答：出现频次：★★★☆☆（5年+经常加分题）。
    • 开场：我会用一句话串起来：TLAB 解决“分配时并发竞争”，逃逸分析解决“对象到底需不需要分配”。
    • 逃逸分析：
      • 方法逃逸：对象被返回或存到外部结构。
      • 线程逃逸：对象被其他线程访问（比如赋给共享字段）。
    • 能优化的结果：
      • 标量替换：把对象拆成若干标量字段放在栈帧/寄存器里（看起来像“没创建对象”）。
      • 栈上分配：某些实现会让对象更短命且不进堆（最终效果是减少堆分配/GC）。
      • 锁消除：如果对象不逃逸且锁无共享意义，JIT 可去掉同步。
    • 实战：性能优化时我更看重“减少临时对象”和“减少跨层装箱/字符串拼接”；JIT 优化是锦上添花，但别指望它救烂代码。
    • 可能追问：
    • 追问：为什么你说“栈上分配不是必然”？
      15秒简答：这是 JIT 的优化策略与实现细节，取决于 HotSpot 版本、逃逸分析结果与编译决策，并不保证发生。
    • 追问：锁消除的前提是什么？
      15秒简答：锁对象不逃逸且不会被并发访问，锁的互斥语义在当前线程内无意义，JIT 才可能去掉。

    JVM 里的“直接内存”是什么？为什么会出现 OOM 但堆没满？¶

    • 15秒简答：直接内存本质是“堆外的 native 内存映射/分配”；核心机制是 NIO 的 DirectByteBuffer 通过 Unsafe/本地分配拿到堆外内存，JVM 只持有引用；特性是堆不满也可能 OOM（Direct buffer memory 或被容器 OOMKilled），排查要看 native 内存与 direct buffer 配额。
    • 3分钟详答：出现频次：★★★★☆（线上问题很典型）。
    • 开场：很多人只盯堆，其实 Java 进程的 RSS 还包含：元空间、线程栈、JIT code cache、direct buffer、本地库等。
    • 直接内存典型来源：
      • NIO：网络框架（Netty）大量使用 direct buffer 减少一次拷贝。
      • mmap：文件映射也算进进程地址空间。
    • 为什么堆没满也会炸：
      • direct buffer 有单独上限（可配），或 native 内存耗尽。
      • 容器 memory limit 触发 OOMKilled，JVM 来不及抛 Java OOM。
    • 实战：我遇到过 Netty direct buffer 泄漏（未释放或缓存过大），表现是 RSS 持续涨、堆占用稳定；这类要看 NMT/进程内存、direct buffer 统计与框架配置。
    • 可能追问：
    • 追问：如何快速判断是不是堆外内存问题？
      15秒简答：堆使用率稳定但进程 RSS 持续增长；heap dump 看不出大对象；同时出现 Direct buffer OOM 或容器 OOMKilled。
    • 追问：DirectByteBuffer 什么时候释放？
      15秒简答：依赖对象回收触发 Cleaner（实现细节），不确定且可能滞后；框架通常用引用计数/池化显式管理更可控。

    StackOverflowError vs OutOfMemoryError（栈相关）怎么区分？线上怎么看懂？¶

    • 15秒简答：栈问题本质是“方法调用深度与线程栈内存的约束”；核心机制是递归/深调用导致栈帧爆掉抛 SOE；线程太多或栈扩展申请失败可能抛 OOM（unable to create new native thread）；特性是 SOE 更像代码逻辑错误，栈 OOM 更像资源/并发配置问题。
    • 3分钟详答：出现频次：★★★☆☆（不如 GC 高频，但问到就要答对）。
    • SOE：单线程栈深度超限（递归无终止、层级过深），通常很快复现。
    • 栈相关 OOM：
      • 创建线程失败：线程数太多或每个线程栈太大导致系统线程资源耗尽。
      • 栈可扩展但申请不到内存：极端情况下可能出现。
    • 实战：线上如果报 “unable to create new native thread”，我会看线程数、线程池是否泄漏、是否把每线程栈设太大，以及容器/系统的线程限制；如果是 SOE，多半是业务递归/序列化/解析导致的深栈。
    • 可能追问：
    • 追问：线程栈大小调小有什么副作用？
      15秒简答：能容纳的调用深度变小，更容易 SOE；但能创建更多线程（占用更少内存），要结合业务栈深评估。
    • 追问：怎么看到是“线程太多”导致的 OOM？
      15秒简答：报错通常直接提示无法创建新线程，且系统层面线程数/进程内存接近上限，线程 dump 会非常多。

    类加载与字节码执行¶

    类加载过程（加载→验证→准备→解析→初始化）你怎么讲？面试官在听什么？¶

    • 15秒简答：类加载本质是“把 .class 变成可执行的运行时类型”；核心机制是加载字节码、校验安全性、为静态字段分配并赋零值、解析符号引用、最后执行类初始化 <clinit>；特性是初始化只执行一次且有同步语义，常用于单例与框架启动。
    • 3分钟详答：出现频次：★★★★☆（框架/中间件方向常问）。
    • 开场：我会强调“类加载”和“类初始化”不同：前者把类型装进 JVM，后者执行静态初始化逻辑。
    • 五步要点：
      • 加载：找到 class 字节码并生成 Class 对象。
      • 验证：字节码合法性/安全性校验（防止恶意字节码破坏 JVM）。
      • 准备：为类变量（static）分配内存并设默认零值（常量可能在编译期内联是例外）。
      • 解析：把符号引用解析为直接引用（可延迟）。
      • 初始化：执行 <clinit>（静态变量显式赋值/静态代码块），并保证线程安全。
    • 实战：Spring 启动慢、动态代理多、热部署场景，类加载数量和类初始化逻辑会直接影响启动时间和元空间。
    • 可能追问：
    • 追问：new 一定会触发类初始化吗？
      15秒简答：通常会；另外访问静态字段/调用静态方法也会触发；但引用常量（编译期常量）可能不触发目标类初始化。
    • 追问：类初始化为什么线程安全？
      15秒简答：JVM 保证同一个类的 <clinit> 在多线程下只会被一个线程执行，其他线程会阻塞等待完成。

    双亲委派模型是什么？为什么要这么设计？哪些场景会“打破”它？¶

    • 15秒简答：双亲委派本质是“类加载的命名空间与安全隔离规则”；核心机制是先委派父加载器尝试加载，父加载不了子才加载；特性是避免重复加载与核心类被篡改，但在模块化/插件化/容器中可能需要打破以实现隔离与热加载。
    • 3分钟详答：出现频次：★★★★★（JVM 高概率必问）。
    • 开场：我会先讲“它解决什么”——安全与一致性：
      • 避免 java.lang.String 被业务 classpath 覆盖。
      • 避免同名类被加载多份导致类型不相等（ClassCastException）。
    • 打破场景（经典）：
      • SPI：JDBC 驱动、ServiceLoader，通常需要“父加载器定义接口，子加载器提供实现”，并由线程上下文类加载器加载实现。
      • 容器/插件：Tomcat、OSGi、IDE 插件，需要不同应用/插件隔离 classpath。
      • 字节码增强：运行期生成类并由自定义加载器定义。
    • 实战：排查 ClassNotFoundException/NoSuchMethodError/ClassCastException 时，我会先确认类由哪个 ClassLoader 加载（同名类多份是高频坑）。
    • 可能追问：
    • 追问：为什么同名类多份会导致 ClassCastException？
      15秒简答：JVM 判定类型不仅看类名，还看加载它的 ClassLoader；不同加载器加载的同名类视为不同类型。
    • 追问：什么是线程上下文类加载器（TCCL）？
      15秒简答：线程上绑定的 ClassLoader，用于让“父加载器加载的代码”在运行时能找到“子加载器提供的实现”（典型是 SPI）。

    new、反射、Class.forName、访问静态常量：哪些会触发类初始化？¶

    • 15秒简答：类初始化触发本质是“是否需要执行 <clinit> 以保证语义”；核心机制是主动使用（new/反射/调用静态方法/读写非编译期常量 static 字段）触发初始化；特性是编译期常量可能被内联导致不触发目标类初始化。
    • 3分钟详答：出现频次：★★★☆☆（常用来考细节与理解）。
    • 主动使用典型触发：
      • new、反射创建实例、Class.forName（默认会初始化）。
      • 调用静态方法、读写静态非 final 常量字段。
    • 不触发/可能不触发：
      • 访问编译期常量（static final 基本类型/字符串字面量）可能被编译器内联到调用方。
      • ClassLoader.loadClass 默认只加载不初始化（与 forName 区分点）。
    • 实战：排查“静态初始化副作用”（连接数据库、启动线程）导致的启动慢/死锁时，要关注触发路径是否在类初始化阶段发生。
    • 可能追问：
    • 追问：为什么建议静态初始化里不要做重 IO？
      15秒简答：类初始化有全局同步语义，重 IO 会放大启动延迟甚至导致类初始化死锁，影响整个应用可用性。
    • 追问：Class.forName(name, false, cl) 的意义？
      15秒简答：可以加载但不初始化，避免执行 <clinit> 的副作用，用于框架延迟初始化或探测类存在性。

    解释执行 vs JIT 编译：你怎么把“热点代码”讲清楚？¶

    • 15秒简答：执行模型本质是“先跑起来再变快”；核心机制是字节码先解释执行，JIT 监控热点方法/循环后编译成本地机器码并做内联等优化；特性是启动期可能慢、稳态更快，但 JIT 优化会让性能对代码形态敏感。
    • 3分钟详答：出现频次：★★★☆☆（中高级常问，尤其性能方向）。
    • 开场：JVM 追求的是“启动速度与长期吞吐”的平衡，所以采用解释+编译的混合模式。
    • 热点判定：基于方法调用次数/回边次数（循环）等计数器，达到阈值触发 C1/C2 编译（概念层面）。
    • 常见优化点：
      • 方法内联（最关键）：减少调用开销并暴露更多优化空间。
      • 逃逸分析相关：标量替换、锁消除。
      • 去虚拟化、范围检查消除等。
    • 实战：服务型应用我更关心“稳态吞吐与延迟”，会避免让热点路径产生大量临时对象与多态分派；启动时间敏感（Serverless）则会关注预热与类加载。
    • 可能追问：
    • 追问：为什么压测前要预热？
      15秒简答：让热点方法进入 JIT 编译后的稳态，否则冷启动阶段解释执行与未内联会让数据偏差很大。
    • 追问：JIT 会带来什么“不可预期”？
      15秒简答：同样逻辑因内联/分支预测/逃逸分析不同，性能可能差异很大；并且编译时机受运行负载影响，导致抖动。

    Safepoint（安全点）是什么？为什么 GC、Deopt、栈遍历都离不开它？¶

    • 15秒简答：安全点本质是“让线程停在可一致观察/修改的执行位置”；核心机制是 JVM 只在特定字节码边界让线程进入 safepoint，方便 GC 枚举根、走栈、撤销优化；特性是到达 safepoint 的延迟会影响 STW，典型问题是 safepoint 偏斜（线程长时间不进入安全点）。
    • 3分钟详答：出现频次：★★★☆☆（懂了很加分，尤其 GC 停顿异常场景）。
    • 开场：JVM 需要在某些时刻“暂停世界”，但不能在任意机器指令处停，否则上下文不完整（寄存器/栈映射难以恢复）。
    • 什么事需要 safepoint：
      • GC 枚举 GC Roots（尤其从栈、寄存器找引用）。
      • 去优化（Deoptimization）：从编译后的机器码回退到解释状态。
      • 获取一致的线程栈快照。
    • safepoint 偏斜场景：线程执行长时间 native 调用/长循环不触发 safepoint，会导致 STW 等待线程到点，表现为“停顿比预期长”。
    • 实战：线上遇到 STW 拉长，我会结合 GC 日志/诊断输出关注“到 safepoint 的时间”，区分是 GC 工作量大还是 safepoint 同步慢。
    • 可能追问：
    • 追问：为什么长循环会影响 safepoint？
      15秒简答：如果循环体没有 safepoint 检查或触发点少，线程可能很久不进入安全点，导致 JVM 等它停下来才能 STW。
    • 追问：safepoint 和 “stop-the-world” 的关系？
      15秒简答：STW 是全局停顿状态；safepoint 是让各线程能安全停下的落点机制，是实现 STW 的关键前提之一。

    GC 基础与收集器选型¶

    如何判断对象“可回收”？GC Roots 都有哪些典型来源？¶

    • 15秒简答：存活判定本质是“从根集合做可达性分析”；核心机制是以 GC Roots 为起点遍历引用图，能到达的对象存活，断链则可回收（再结合引用类型与 finalize/回收阶段）；特性是比引用计数更稳（避免循环引用），但可达性取决于根与引用链。
    • 3分钟详答：出现频次：★★★★★（GC 的第一题）。
    • 开场：主流 HotSpot 用可达性分析而不是引用计数，核心原因是循环引用会误判。
    • GC Roots 典型来源（面试常用）：
      • 线程栈帧里的局部变量引用（当前活跃方法的引用）。
      • 静态字段引用（类变量）。
      • 常量引用（运行时常量池）。
      • JNI 引用（native 代码持有的引用）。
    • 引用类型影响：软/弱/虚引用对“是否存活”的语义不同，会影响回收时机。
    • 实战：分析内存泄漏我会用 MAT 的“GC Roots 路径”，看是谁把对象链路保活了（常见是静态集合、线程本地、缓存、类加载器）。
    • 可能追问：
    • 追问：弱引用为什么更容易回收？
      15秒简答：弱引用在 GC 时只要没有强引用链就会被清理，不考虑内存是否充足。
    • 追问：finalize 能救活对象吗？
      15秒简答：理论上一次机会可以“自救”并重新建立引用，但不可靠且开销大，现代 Java 不建议依赖 finalize 语义。

    强/软/弱/虚引用分别适合什么场景？为什么说软引用缓存要谨慎？¶

    • 15秒简答：引用类型本质是“给 GC 提供不同强度的保活语义”；核心机制是强引用永不回收（除非断链），软引用内存紧张才回收，弱引用下次 GC 就回收，虚引用仅用于接收回收通知；特性是软引用缓存不稳定且难以预测回收时机，生产更常用显式容量/TTL 缓存。
    • 3分钟详答：出现频次：★★★★☆（经常从 GC Roots 延伸）。
    • 典型用途：
      • 强引用：正常业务对象图。
      • 软引用：可有可无的对象（历史上用于内存敏感缓存，但不可控）。
      • 弱引用：ThreadLocal key、弱键 map、避免监听器泄漏等。
      • 虚引用：配合 ReferenceQueue 做资源回收通知（堆外资源/清理动作），但一般框架层用得多。
    • 为什么软引用缓存谨慎：
      • 回收时机与“内存紧张”的判定依赖 JVM 策略，表现不可预测。
      • 在压力下可能把缓存全清，造成雪崩；更推荐 Caffeine/Guava 这种显式容量与淘汰策略。
    • 实战：我会优先用显式缓存（容量、TTL、监控命中率）而不是赌 GC 策略；弱引用更适合“避免不必要的强引用保活”。
    • 可能追问：
    • 追问：ThreadLocal 为什么用弱引用做 key？
      15秒简答：避免 ThreadLocal 本身不可达后仍被强引用 key 保活；但 value 仍可能泄漏，所以必须 remove。
    • 追问：虚引用为什么叫“形同虚设”？
      15秒简答：get() 拿不到对象，唯一用途是配合队列在对象被回收时得到通知信号。

    Minor GC / Major GC / Full GC 你怎么定义？线上你更关注哪个指标？¶

    • 15秒简答：GC 分类本质是“回收范围与停顿影响”；核心机制是 Minor 多发生在新生代，Major 多指老年代回收，Full 通常表示整堆（含元空间/老年代）或收集器退化；特性是 Full GC 停顿最长、对延迟最致命，线上优先盯 Full GC 次数与停顿分位。
    • 3分钟详答：出现频次：★★★★☆（面试常问概念，答法要带工程味）。
    • 开场：我会说明“不同 JVM/收集器对术语略有差异”，但工程上关注点一致：停顿与频率。
    • 经验判断：
      • Minor GC 频繁：通常意味着 Eden 太小或对象分配过快（短命对象太多），可能导致吞吐下降。
      • 老年代增长过快/Full GC 频繁：可能是晋升过快、大对象、内存泄漏或 GC 策略不适配。
    • 线上我关注：
      • Full GC 的频率与 P99 停顿。
      • 老年代占用趋势（是否可回收、是否持续上升）。
      • 分配速率与晋升速率（是否写放大到老年代）。
    • 实战：如果是“Full GC 回收不掉”，我优先按泄漏思路走（dump+GC roots）；如果是“Minor 很频繁”，优先按分配速率与对象生命周期优化。
    • 可能追问：
    • 追问：什么情况下会触发 Full GC？
      15秒简答：老年代/元空间分配失败、收集器退化（如 CMS 失败）、显式触发（System.gc，取决于配置）等。
    • 追问：Minor GC 很频繁一定要调 JVM 吗？
      15秒简答：不一定，先看分配热点与对象创建是否可优化；能减少临时对象往往比调堆更有效。

    常见 GC 算法（标记清除/复制/标记整理）分别解决什么问题？各自的代价是什么？¶

    • 15秒简答：算法本质是“如何回收并整理内存”；核心机制是复制用空间换时间适合新生代，标记清除实现简单但会碎片化，标记整理兼顾回收与连续性但移动成本高；特性是分代收集用对象生命周期差异来组合算法，追求吞吐与停顿平衡。
    • 3分钟详答：出现频次：★★★★☆（基础题，但要讲出“为什么分代”）。
    • 标记-清除：先标记存活，再清掉垃圾；缺点是碎片导致后续分配困难、可能触发整理或更慢的分配路径。
    • 复制：把存活对象复制到另一块区域；优点是分配快且无碎片，缺点是需要额外空间；适合“多数对象短命”的新生代。
    • 标记-整理：标记后把存活对象向一端移动并压缩；优点是解决碎片，缺点是移动对象成本高、停顿可能更长；常用于老年代。
    • 分代思想：新生代“死得快”用复制更划算；老年代“活得久”用标记整理/清除更合适。
    • 实战：调优时我更关心“对象分配与晋升”是否符合分代假设，假设被破坏（大量对象很快晋升）会导致老年代压力暴增。
    • 可能追问：
    • 追问：为什么老年代不直接用复制算法？
      15秒简答：老年代存活率高，复制意味着搬很多对象且需要巨大额外空间，代价太高。
    • 追问：碎片会带来什么现象？
      15秒简答：可用总空间够但无法分配大对象，触发 Full GC/整理或直接 OOM，表现为停顿与分配失败。

    CMS、G1、ZGC 你怎么做选型？回答里哪些点最容易被追问？¶

    • 15秒简答：选型本质是“在吞吐、停顿、内存占用之间取舍”；核心机制是 CMS 以并发标记清除降低停顿但可能碎片与退化，G1 以 Region+预测停顿做整体回收，ZGC 以更强并发降低停顿但对内存与版本要求更高；特性是默认优先 G1（Java 11+），极低延迟再评估 ZGC，CMS 逐步退出主流。
    • 3分钟详答：出现频次：★★★★☆（5年+常问“你们线上用啥”）。
    • 开场：我会先说“没有银弹”，先看业务目标：P99 延迟还是吞吐优先？堆多大？对象生命周期如何？
    • CMS：
      • 优点：并发回收，STW 相对短（适配某些中等堆延迟敏感场景）。
      • 风险：碎片、并发失败会退化 Full GC；对参数与场景敏感。
    • G1：
      • 思想：Region 化 + 预测停顿（按收益/成本选回收集合），既考虑吞吐也控停顿。
      • 风险：MaxGCPauseMillis 是目标不是硬保证；Mixed GC 阶段可能抖动。
    • ZGC：
      • 目标：更低 STW（大部分工作并发），适合大堆低延迟。
      • 代价：额外的内存映射/元数据开销，容器内存限制要留裕量；并且版本与生态适配要评估。
    • 实战：我通常“先把分配/泄漏解决，再谈换收集器”；换收集器属于成本更高的动作，需要压测与回滚预案。
    • 可能追问：
    • 追问：MaxGCPauseMillis 能保证停顿上限吗？
      15秒简答：不能，它是 G1 的目标值，G1 会尽量满足但不保证，尤其 Mixed/Full GC 时可能明显超过。
    • 追问：什么情况下 CMS 会退化成 Full GC？
      15秒简答：并发回收跟不上分配速度导致 concurrent mode failure 或碎片化导致分配失败，就会退化到 STW 的 Full GC。

    G1 的 Region、Remembered Set、Mixed GC 是什么？你怎么用一句话把它串起来？¶

    • 15秒简答：G1 本质是“以 Region 为单位的可预测整堆回收器”；核心机制是把堆切成 Region，通过 RSet 记录跨 Region 引用以便局部回收，先 Young GC 再进入 Mixed GC 按收益回收部分老年代；特性是能更好控制停顿，但元数据与维护 RSet 有额外开销。
    • 3分钟详答：出现频次：★★★☆☆（G1 细节题，适合拉开差距）。
    • Region：把堆划成许多小块，每块可扮演 Eden/Survivor/Old 的角色，便于按块回收。
    • RSet：记录“别的 Region 指向我”的引用集合，避免全堆扫描根。
    • 回收节奏：
      • Young GC：回收 Eden/Survivor，复制存活对象。
      • 并发标记：识别各 Region 的存活比例。
      • Mixed GC：在 Young 的同时按收益选择若干 Old Region 一起回收，逐步回收老年代。
    • 为什么这样设计：让回收工作量可以分摊到多次、可预测地控制停顿时间。
    • 实战：我更关注 Mixed GC 的频率与停顿分位，以及老年代是否能被持续回收；如果 Mixed 很频繁且效果差，往往是晋升太快或堆太紧。
    • 可能追问：
    • 追问：为什么要 RSet？
      15秒简答：为了局部回收时能找到跨 Region 的引用根，避免每次都全堆扫描导致停顿失控。
    • 追问：Mixed GC 的触发通常和什么有关？
      15秒简答：与老年代占用阈值、并发标记完成后可回收 Region 的收益评估有关，本质是“老年代压力上来了要开始回收 Old”。

    什么时候对象会进入老年代？你怎么用“晋升/大对象/担保”三句话讲清楚？¶

    • 15秒简答：晋升本质是“对象生命周期超过新生代承载能力”；核心机制是对象年龄达到阈值晋升、或大对象直接进老年代、或 Survivor 放不下触发分配担保把对象晋升；特性是晋升过快会把压力推到老年代，导致 Full GC/延迟抖动。
    • 3分钟详答：出现频次：★★★★☆（和 Minor/Full、调优强相关）。
    • 年龄晋升：对象在 Survivor 中每经历一次 Minor GC 年龄+1，达到 MaxTenuringThreshold 晋升老年代（还会受动态年龄判定影响）。
    • 大对象直接进老年代：比如很大的数组/缓存块，避免在新生代复制来复制去。
    • 分配担保（概念层）：Minor GC 后存活对象太多，Survivor 放不下，部分对象会直接晋升老年代。
    • 实战：如果我看到老年代增长快且 Mixed/Full 多，我会怀疑：
      • 分配速率过快（短命对象太多）。
      • 大对象（一次性拉很多数据、未分页、巨型 JSON）。
      • Survivor 太小导致早晋升（或阈值/动态策略导致）。
    • 可能追问：
    • 追问：大对象直接进老年代一定好吗？
      15秒简答：不一定，能避免复制开销，但会加速老年代膨胀；大对象多时应先优化业务分配（分页/流式处理/压缩结构）。
    • 追问：什么是“动态年龄判定”？
      15秒简答：如果 Survivor 中某年龄段对象总大小超过一定比例，JVM 会让该年龄及以上对象提前晋升，避免 Survivor 被挤爆。

    调优参数与观测工具¶

    你怎么解释 -Xms/-Xmx/-Xmn 这组参数？线上为什么常建议 Xms=Xmx？¶

    • 15秒简答：参数本质是“给堆空间设边界与分代比例”；核心机制是 Xms 初始堆、Xmx 最大堆、Xmn 新生代大小（或等价参数），设置 Xms=Xmx 可避免运行时扩缩容带来的停顿与抖动；特性是更稳定但会固定内存占用，需要结合容器 limits 与多实例密度。
    • 3分钟详答：出现频次：★★★★★（面试调优最常问的参数组）。
    • 开场：我会强调：调参不是为了“更大”，而是为了“更稳”和“更可控”。
    • Xms/Xmx：
      • 不相等时，堆可能动态扩容/收缩，扩容通常伴随停顿与更差的预测性。
      • 设为相等可以让 GC 行为更稳定，避免抖动，且更利于容量规划。
    • Xmn/新生代：
      • 新生代太小：Minor GC 频繁。
      • 新生代太大：老年代可能偏小，Mixed/Full 压力上升；并且 Young GC 单次停顿可能变长。
    • 实战：我会先用监控看分配速率、Minor GC 频率、晋升速率，再决定是否调新生代；并配合业务层减少临时对象与大对象。
    • 可能追问：
    • 追问：堆是不是越大越好？
      15秒简答：不是，堆大可能降低 GC 频率但提高单次回收成本与停顿尾部，还会挤压系统/容器内存；要结合延迟目标与实例密度。
    • 追问：如果堆>8G 你会怎么选收集器？
      15秒简答：通常优先 G1（现代默认），再根据 P99 延迟与内存预算评估是否需要更低停顿收集器（如 ZGC）。

    GC 日志你怎么“读”？你会重点关注哪 5 个指标？¶

    • 15秒简答：GC 日志本质是“回收行为的时间序列证据”；核心机制是记录每次 GC 的触发原因、回收前后堆占用、停顿时间与阶段耗时；特性是能区分“分配太快/晋升太快/并发跟不上/退化 Full”，但必须结合业务流量与内存曲线一起看。
    • 3分钟详答：出现频次：★★★★☆（面试常让你“说说怎么分析 GC 日志”）。
    • 我最关注的 5 类指标：
      • 停顿时间：P95/P99/最大值（延迟敏感服务最关键）。
      • GC 频率：Minor/Mixed/Full 的次数与间隔。
      • 回收效果：Old 是否能回收下来、回收后是否迅速反弹（泄漏/缓存）。
      • 晋升/分配速率：Young 回收后有多少对象晋升到 Old（promotion）。
      • 退化信号：并发标记跟不上、to-space exhausted、evacuation failure 等（不同收集器关键词不同）。
    • 怎么落地：
      • 把 GC 日志与 QPS、CPU、RT 对齐，看停顿是否导致 RT 尖刺。
      • 区分问题类型：频繁 Minor（分配太快）vs Old 持续上涨（泄漏/大对象/晋升过快）vs Full 退化（策略不匹配/空间太紧）。
    • 实战：我一般先定位“是不是业务分配问题”，能通过分页、复用、减少临时对象解决的，优先解决；调参只是兜底。
    • 可能追问：
    • 追问：Full GC 回收后 Old 还很高，意味着什么？
      15秒简答：强引用链保活（缓存/静态集合/类加载器泄漏）或存活对象确实很多；大概率需要 heap dump 找引用链，而不是继续加堆。
    • 追问：为什么 GC 停顿会导致 CPU 飙高？
      15秒简答：频繁 GC 会让回收线程与业务线程抢 CPU，且内存压力下分配/屏障成本上升；表现为吞吐下降但 CPU 反而更忙。

    你常用哪些 JVM 排障工具（jps/jstack/jmap/jstat/jcmd/VisualVM）？分别解决什么问题？¶

    • 15秒简答：工具体系本质是“把 JVM 状态可观测化”；核心机制是用 attach/JMX/SA 等接口读取进程的线程、堆、类、GC 统计与诊断命令；特性是线上优先低侵入（jcmd/jstat/jstack），需要证据再抓 dump（成本高）并在隔离环境分析。
    • 3分钟详答：出现频次：★★★★☆（5年+基本都会问“你怎么排查”）。
    • 我常用的组合拳：
      • jps：找 Java 进程 PID（LVMID）。
      • jstat：看 GC 概况（Young/Old 占用、GC 次数/时间），判断趋势与是否需要 dump。
      • jstack：抓线程栈，定位死锁/卡顿/线程池耗尽、以及 GC 线程是否占用异常。
      • jmap/jcmd：导出 heap dump、查看类直方图（histogram），快速判断大对象/类数量。
      • VisualVM/MAT：离线分析 dump（引用链、GC roots、对象分布）。
    • 为什么这样顺序：先用轻量统计判断方向，再用 dump 固化证据；避免一上来就大 dump 把线上打挂或生成超大文件。
    • 实战：CPU 100% 我会先 top→定位线程→jstack；内存增长我先 jstat 看 Old 曲线，再决定 dump；类加载问题我会看类数量与 ClassLoader 引用链。
    • 可能追问：
    • 追问：CPU 100% 时 jstack 看什么最关键？
      15秒简答：看热点线程的栈顶方法（是否业务死循环/锁竞争/GC 线程），以及是否大量线程处于 runnable/blocked/parking。
    • 追问：heap dump 太大怎么办？
      15秒简答：先用 histogram/采样定位大类与大对象，必要时只在非高峰抓、压缩传离线分析；同时先从业务指标缩小范围避免盲抓。

    线上 JVM 调优你如何给出“一套流程”？面试官想听到哪几步？¶

    • 15秒简答：调优本质是“基于证据的迭代实验”；核心机制是先明确目标（延迟/吞吐/成本）再采集数据（GC/RT/CPU/堆趋势）定位瓶颈，最后小步改动验证与回滚；特性是 JVM 调优是最后一颗子弹，先修业务分配/并发/IO，再谈参数与收集器。
    • 3分钟详答：出现频次：★★★★★（“你做过调优吗”必问）。
    • 我会按 6 步讲（面试最稳）： 1) 明确指标：P99 延迟、吞吐、内存上限、是否容器化。 2) 采集证据：GC 日志、jstat、监控（RT/QPS/CPU/RSS）、必要时 heap dump。 3) 定位类型：分配过快、晋升过快、并发回收跟不上、内存泄漏、堆外/元空间问题。 4) 先改业务：减少临时对象、分页/流式处理大对象、修缓存边界、修线程池/队列。 5) 再改 JVM：堆大小、代比例、换收集器、停顿目标与并发线程数（每次只改少量参数）。 6) 验证与回滚：压测/灰度，对比 GC 指标与业务指标，保留回滚开关与参数版本。
    • 实战：我会强调“调参要能解释为什么”，比如看到 Old 持续上涨且 Full 回收不掉，那就不是调参数能解决的，要先找泄漏根因。
    • 可能追问：
    • 追问：你如何避免“拍脑袋调参”？
      15秒简答：每次调参都要绑定一个假设与可观测指标（例如减少 FullGC 次数/降低 P99 停顿），并用对比实验验证，否则撤回。
    • 追问：什么情况下你会建议“换收集器”？
      15秒简答：当对象分配与泄漏问题已控制，且现有收集器在目标延迟/吞吐上确实达不到要求（有稳定证据）才换，并做充分压测。

    OOM 与线上故障排查¶

    线上 OOM 你怎么分类？为什么说“先分区再下手”？¶

    • 15秒简答：OOM 排查本质是“先确定是哪块内存耗尽”；核心机制是堆 OOM 看对象与引用链，元空间 OOM 看类加载与 ClassLoader，堆外 OOM 看 direct/native，线程 OOM 看线程数与栈；特性是不同 OOM 证据与工具完全不同，分区错误会导致白忙。
    • 3分钟详答：出现频次：★★★★★（5年+必问，且常要求讲案例）。
    • 我常见的 OOM 分类：
      • Java heap space：堆满。
      • Metaspace：类元数据/类加载器问题。
      • Direct buffer memory：堆外 direct buffer。
      • unable to create new native thread：线程资源耗尽/栈占用过大。
      • GC overhead limit exceeded：GC 花太多时间但回收太少，通常接近堆耗尽。
    • 为什么要先分类：因为“抓什么证据”不同：
      • 堆：heap dump + MAT + GC roots。
      • 元空间：类数量、ClassLoader 引用链、动态生成类来源。
      • 堆外：NMT/进程内存、direct buffer 统计、容器 limits。
      • 线程：线程 dump/线程池配置/系统限制。
    • 实战：我会配置 HeapDumpOnOutOfMemoryError，并把 dump 输出到可持久化盘；容器环境要同时留足 RSS 裕量，否则还没来得及 dump 就被 OOMKilled。
    • 可能追问：
    • 追问：GC overhead limit exceeded 说明什么？
      15秒简答：GC 时间占比过高但回收效果很差，通常是堆接近耗尽或泄漏导致“收不动”，需要看堆占用与 dump。
    • 追问：为什么有时 OOM 没有 heap dump？
      15秒简答：可能是容器/系统直接 OOMKill 进程或磁盘不足/权限问题导致 dump 写不出来，也可能是 OOM 发生在堆外区域。

    内存泄漏你怎么定位？“用 MAT 看 GC Roots”怎么讲得像真做过？¶

    • 15秒简答：泄漏定位本质是“找到谁在不该保活时仍保活对象”；核心机制是抓 heap dump，用 MAT/分析器看 Dominator Tree 与 GC Roots 路径，定位最大保留集与引用链；特性是“持续增长 + Full GC 回收不下去”是典型信号，修复往往是给缓存加边界、修 ThreadLocal、断开监听器/集合引用。
    • 3分钟详答：出现频次：★★★★★（面试高频深挖题）。
    • 开场：我先看症状是否符合泄漏：Old 区使用率随时间单调上升，Full GC 后回收幅度很小，最终 OOM。
    • 定位步骤（面试可复述）： 1) 抓 dump（最好在 OOM 现场或接近现场）。 2) 看直方图/Top class：哪些类实例数量或总大小异常。 3) 看 Dominator Tree：谁是“保留内存最大的支配者”（Retained Size 大）。 4) 对可疑对象做 GC Roots 路径：找到引用链（常见是静态 map、缓存、线程本地、全局单例）。 5) 回到代码：确认为何引用不断增长，补容量/TTL/移除、修 finally 清理、修监听器注销。
    • 实战例子（可讲）：无界 HashMap 当缓存、ThreadLocal 未 remove、集合缓存 key 无限增长、ClassLoader 泄漏（频繁生成代理类）等。
    • 可能追问：
    • 追问：如何区分“内存泄漏”与“正常的缓存增长”？
      15秒简答：看是否有上限与淘汰策略，是否能在稳定流量下达到平台期；泄漏通常不会平台且 Full GC 回收不下去。
    • 追问：ThreadLocal 泄漏的根因是什么？
      15秒简答：线程池线程长生命周期 + ThreadLocalMap 的 value 强引用残留，没 remove 就跨请求保活，导致堆持续增长或串号。

    CPU 飙高但 QPS 下降：你如何判断是 GC 线程在“吃 CPU”还是业务死循环？¶

    • 15秒简答：排障本质是“先定位热点线程再看栈”；核心机制是 top 定位高 CPU 线程→线程 id 转十六进制→jstack 找对应栈；特性是 GC 过载会看到 GC 线程活跃与 GC 日志频繁，业务死循环则栈顶稳定在某段业务代码。
    • 3分钟详答：出现频次：★★★★☆（非常贴近线上）。
    • 步骤：
      • 系统侧看 CPU：top/任务管理器定位进程与线程。
      • JVM 侧看栈：jstack 对应线程，判断是 GC、JIT、业务线程还是锁竞争。
    • 判断 GC 过载的信号：
      • GC 日志显示频繁 GC、停顿时间累积高。
      • jstat 显示 GC 次数快速增加、Old/Metaspace 压力大。
    • 判断业务死循环/热点：
      • 某个业务线程长期 RUNNABLE 且栈顶在同一方法。
      • 伴随线程池队列堆积、响应超时。
    • 实战：如果是 GC 过载，优先找分配热点/泄漏；如果是业务热点，优先修算法/限流/缓存；别一上来就调 GC 参数。
    • 可能追问：
    • 追问：为什么 GC 频繁会让 QPS 下降？
      15秒简答：停顿期间业务线程不工作，且 GC 线程与业务争抢 CPU；同时内存屏障与分配失败重试会增加开销。
    • 追问：如何快速判断是“锁竞争导致慢”而非 CPU 计算？
      15秒简答：jstack 里大量线程 BLOCKED/WAITING/parking，栈顶在锁/park/wait，而不是在纯计算热点。

    “频繁 Full GC”你如何定位根因？给一个你会现场讲的排查路径¶

    • 15秒简答：Full GC 频繁本质是“老年代/元空间压力不可控”；核心机制是看 Full GC 触发原因与回收效果，结合晋升速率、大对象、泄漏与收集器退化做归因；特性是回收不下去优先怀疑泄漏，回收得下去但很快又满优先怀疑分配/晋升与大对象。
    • 3分钟详答：出现频次：★★★★★（调优深挖题）。
    • 我会按 3 个问题走： 1) 为什么触发 Full GC：老年代分配失败？元空间？收集器并发失败？显式 GC？ 2) 回收效果如何：Full 后 Old 降多少？能不能回到安全水位？ 3) 反弹速度：多久又回到高水位？是否与流量峰值相关？
    • 归因模板：
      • Full 后 Old 几乎不降：泄漏/强引用保活，抓 heap dump 看 GC roots。
      • Full 后 Old 降很多但很快又满：分配/晋升过快或大对象，重点看分配热点、分页、批处理、缓存对象大小。
      • Full 伴随 “concurrent mode failure/to-space exhausted”：收集器策略不匹配或堆太紧，可能需要调整堆/并发线程/换收集器。
    • 实战：我通常能在 jstat+GC 日志里先把问题分成“泄漏型”还是“分配型”，再决定是否抓 dump。
    • 可能追问：
    • 追问：为什么大对象会导致 Full GC 频繁？
      15秒简答：大对象可能直接进老年代并快速占满 Old，且回收时成本高；更糟的是大对象存活时间长会让 Old 难以下降。
    • 追问：如何减少晋升速率？
      15秒简答：减少短命对象分配、增加新生代承载或优化对象生命周期（复用/池化、减少临时集合），并避免 Survivor 太小导致提前晋升。

    类加载器泄漏你怎么识别？为什么它会表现为 Metaspace 上升 + Full GC 无效？¶

    • 15秒简答：类加载器泄漏本质是“旧 ClassLoader 被强引用链保活，导致其加载的类元数据无法回收”；核心机制是每次热部署/动态生成类创建新加载器，但旧加载器仍被线程/静态缓存/回调持有；特性是 Metaspace 持续增长且 Full GC 也回收不下来，解决要断开引用链并避免频繁生成类。
    • 3分钟详答：出现频次：★★★☆☆（做过框架/容器/热部署的很常见）。
    • 症状：Metaspace 使用量随时间上升，触发 Full GC 但回收很少；类数量持续上升。
    • 常见根因：
      • 热部署/脚本引擎/动态代理频繁生成类。
      • Web 容器中应用重载，旧类加载器被线程池线程、ThreadLocal、日志/单例缓存引用。
    • 定位方式：
      • 看类数量与 ClassLoader 统计（工具或 jcmd 类相关命令）。
      • dump 中按 ClassLoader 分组，找 Retained Size 大的 ClassLoader，再看 GC Roots 引用链。
    • 实战：很多时候不是调大 metaspace 能解决的；只能修“谁还引用着旧 ClassLoader”，常见就是线程池/定时任务未停、ThreadLocal 未清、静态单例跨加载器引用。
    • 可能追问：
    • 追问：为什么 Full GC 回收不了类元数据？
      15秒简答：类卸载的前提是对应 ClassLoader 可回收；如果加载器仍可达，那么它加载的类元数据也必须存活。
    • 追问：什么场景最容易触发类加载器泄漏？
      15秒简答：热部署/插件化/容器重载 + 线程池复用（线程存活跨应用生命周期）是最经典组合。

    你如何在面试里讲一个“真实的 JVM 排障案例”？（给你一个可复述模板）¶

    • 15秒简答：案例表达本质是“症状→证据→定位→修复→验证”；核心机制是用监控与工具把问题从现象收敛到根因，再用小步变更验证；特性是面试官更看重你的诊断思路与证据链，而不是背参数名字。
    • 3分钟详答：出现频次：★★★★★（社招常要求讲案例）。
    • 我会按这个模板讲（通用且像真做过）：
      • 症状：比如 RT 抖动、CPU 高、Full GC 频繁、偶发 OOM。
      • 证据：监控图（Old 上升/GC 次数）、GC 日志（Full 触发原因与回收效果）、线程栈（GC 线程/业务热点）、必要时 heap dump（Dominator/GC roots）。
      • 定位：一句话点出根因（无界缓存、ThreadLocal 未清、一次性加载大对象、类加载器泄漏等）。
      • 修复：加上限/TTL、分页/流式、finally remove、停止旧线程、优化数据结构。
      • 验证：Full GC 消失或频率下降、Old 回到平台、P99 下降、CPU 恢复，并加上监控告警。
    • 实战：这种结构化表达能让面试官快速相信你具备“线上闭环能力”。
    • 可能追问：
    • 追问：为什么你说“先证据再调参”？
      15秒简答：因为调参只能改变症状表现，根因可能是泄漏或分配热点；没有证据会把问题拖更久且引入新风险。
    • 追问：你会加哪些长期监控？
      15秒简答：GC 次数/停顿分位、堆/Old/Metaspace 使用率、分配与晋升速率、RSS、以及 OOM/FullGC 告警与关联 QPS/RT。
