    微服务架构与拆分¶

    什么是微服务？它和 SOA/分布式 有什么区别？¶

    • 15秒简答：微服务本质是“按业务能力拆分的自治服务集合”；核心机制是独立部署+轻量通信+去中心化治理；特性是迭代快/可独立扩缩，但治理与一致性成本更高。
    • 3分钟详答：出现名次：#01（极高频）。我先给一句话结论：微服务是分布式的一种落地形态，强调“服务自治”和“独立演进”，而 SOA 更强调“服务复用与集成”。
    • 技术背景：分布式是把计算/存储分散到多节点；微服务是在分布式基础上，把系统按业务拆成多个可独立交付的服务单元。
    • 核心区别抓手：
      • 分布式：关注“部署形态”（多节点协作）。
      • SOA：关注“服务化思想”（复用、编排、ESB 等历史形态）。
      • 微服务：关注“服务粒度+自治团队+DevOps+服务治理”（注册发现、熔断限流、可观测性等）。
    • 为什么这样设计：系统变大后，单体发布慢、组织协作成本高；用“自治服务+契约”让团队并行。
    • 实战：我在电商/营销类系统里，把订单、库存、支付、营销拆成独立服务，配套网关、注册发现和链路追踪来做治理。
    • 可能追问：
    • 追问：微服务的“自治”具体指什么？
      15秒简答：自治指服务能独立开发/测试/部署/扩缩，最好数据也自洽（至少写路径不依赖别的库），通过契约对外提供能力而不是共享内部实现。
    • 追问：微服务一定要“每服务一库”吗？
      15秒简答：理想是每服务拥有数据；现实可分阶段，至少保证“写隔离”（写路径不跨库），读可以通过聚合/搜索/缓存/异步同步解决。

    为什么从单体拆微服务？收益和成本分别是什么？¶

    • 15秒简答：拆微服务本质是“用架构换组织与交付效率”；核心机制是让服务独立发布与弹性扩缩；特性是交付更快但引入分布式复杂性（链路、事务、运维、成本）。
    • 3分钟详答：出现名次：#02（极高频）。我先给一句话结论：拆微服务的第一目标通常不是性能，而是“团队并行+发布解耦”，性能是附带收益。
    • 场景：单体变大后，发布窗口长、回归成本高、一次上线影响全站；不同业务线互相等依赖、冲突频发。
    • 拆分带来的核心收益：
      • 研发协作：团队可并行开发，减少代码冲突。
      • 交付与风险：独立发布，故障影响面可控。
      • 资源利用：热点服务单独扩容，成本更可控。
    • 付出的关键成本：
      • 可靠性：网络不可靠带来超时、重试、雪崩。
      • 一致性：跨服务事务与数据一致性困难。
      • 可观测性：必须做日志/指标/链路追踪。
      • 运维复杂度：服务数量、配置、发布、权限、成本都上升。
    • 实战：拆分后我最先补齐“超时/重试/熔断/限流/追踪”，否则系统会从“单体难维护”变成“分布式难稳定”。
    • 可能追问：
    • 追问：什么情况下不建议上微服务？
      15秒简答：团队小、业务不复杂、需求迭代不快或缺少运维/平台能力时，微服务复杂性可能大于收益。
    • 追问：拆分失败最常见原因是什么？
      15秒简答：先拆服务不补治理（超时/熔断/观测/发布），以及边界不清导致强耦合（同步调用链太深、共享库/共享表泛滥）。

    服务边界怎么划分？如何避免“拆了但还是强耦合”？¶

    • 15秒简答：边界划分本质是“把变化点隔离成稳定契约”；核心机制是以领域/业务能力为中心定义服务与数据所有权；特性是边界清晰则高内聚低耦合，否则会变成分布式大泥球。
    • 3分钟详答：出现名次：#06（高频）。我会先给一句话结论：用“领域边界 + 数据所有权 + 组织结构”三条线一起定边界。
    • 技术背景：微服务把系统复杂度从“代码规模”转移到“服务协作”，所以边界错了会把协作成本放大。
    • 核心方法：
      • 领域/能力边界：围绕业务能力（下单、库存扣减、支付确认）而不是技术层（controller/service）。
      • 数据所有权：谁负责写这份数据，谁就拥有它；其他服务通过 API/事件拿到副本或查询视图。
      • 康威定律：团队边界往往决定系统边界，跨团队强依赖会变成跨组织扯皮。
    • 为什么这样设计：把“变化频繁的业务规则”关在一个服务里，降低联动改动。
    • 实战：我会限制同步调用深度（例如不超过 2-3 跳），更多用事件驱动做跨域协作；并且通过契约测试/版本治理减少联调成本。
    • 可能追问：
    • 追问：如何识别“边界错了”？
      15秒简答：表现是跨服务改动频繁、同一需求要改多个服务、同步调用链过长、数据回写互相依赖、接口语义模糊。
    • 追问：服务拆分粒度怎么把握？
      15秒简答：以“独立交付与独立扩缩”为主要尺度；粒度过细会治理成本爆炸，过粗会回到单体的发布瓶颈。

    “每服务一库”为什么重要？跨服务查询/报表怎么做？¶

    • 15秒简答：每服务一库本质是“用数据边界保证服务自治”；核心机制是写隔离+通过 API/事件同步构建读模型；特性是写一致性更清晰，但读侧需要冗余与最终一致。
    • 3分钟详答：出现名次：#10（中高频）。我先给一句话结论：不共享库的核心是避免“绕过服务直接改数据”，否则服务边界形同虚设。
    • 技术背景：一旦共享库，任何服务都能改别人的数据，业务规则会散落在多个服务里，后续很难治理与演进。
    • 跨服务查询常见解法（读写分离思路）：
      • API 聚合：由 BFF/聚合服务并发调用多个服务，拼装返回（注意超时与降级）。
      • 异步构建读模型：用 MQ/CDC 同步数据到“查询库/ES/宽表”，提供报表与复杂检索。
      • 物化视图/数仓：离线/准实时 ETL 做 BI。
    • 为什么这样设计：写路径保持清晰的所有权；读路径接受冗余换性能与解耦。
    • 实战：我更倾向把“跨域查询”下沉到读模型（ES/宽表），减少在线链路的同步依赖。
    • 可能追问：
    • 追问：共享库真的完全不行吗？
      15秒简答：过渡期可以，但要明确“共享只读”或“按 schema/表严格隔离写权限”，并规划迁移，否则最后会演化成强耦合单体。
    • 追问：数据冗余怎么保证一致性？
      15秒简答：用事件驱动/CDC+重试+幂等消费+对账补偿，接受最终一致；关键链路用事务消息或本地消息表保证不丢事件。

    服务通信与治理¶

    服务注册与发现是怎么工作的？CP/AP 怎么选？¶

    • 15秒简答：注册发现本质是“把实例列表从配置里挪到动态目录”；核心机制是服务启动注册、客户端订阅/拉取、健康检查摘除；特性是减少人工配置，但一致性/可用性要取舍。
    • 3分钟详答：出现名次：#03（极高频）。我先给一句话结论：注册中心提供“服务名 -> 实例列表”的动态映射，CP/AP 的取舍决定故障时是“宁可不可用”还是“宁可读到旧列表”。
    • 技术背景：微服务实例会弹性伸缩与频繁发布，静态配置地址不可维护，所以需要注册中心。
    • 核心流程：
      • 注册：实例上报服务名、IP/端口、元数据。
      • 发现：消费者订阅/定期拉取实例列表到本地缓存。
      • 健康检查：心跳或探活失败则摘除；恢复后再加入。
    • CP/AP 的直观理解：网络分区时，CP 倾向保证一致（可能拒绝服务）；AP 倾向保证可用（可能返回旧数据）。注册发现通常更关注“可用”，所以大量实现会更偏 AP 或提供可调策略。
    • 实战：我会在客户端做本地缓存+快速失败；对注册中心本身做高可用部署，避免它变成单点。
    • 可能追问：
    • 追问：注册中心挂了服务还能调用吗？
      15秒简答：看客户端是否有本地缓存；多数实现会用缓存继续调用一段时间，但新实例发现/下线摘除会变慢或失效。
    • 追问：心跳/探活有什么坑？
      15秒简答：心跳风暴与误判下线；要做随机抖动、合理超时阈值、以及故障摘除与恢复的冷却时间。

    负载均衡怎么做？客户端 LB vs 服务端 LB 有什么区别？¶

    • 15秒简答：负载均衡本质是“把请求分摊到多个实例”；核心机制是基于策略（轮询/权重/一致性哈希）选择目标；特性是提升吞吐与可用，但策略不当会引入热点与抖动。
    • 3分钟详答：出现名次：#07（高频）。我会先给一句话结论：客户端 LB 把选址逻辑放在调用方，服务端 LB 把选址集中在代理/网关，两者权衡是“全局统一 vs 客户端复杂度”。
    • 技术背景：同一服务会有多实例，直接随机请求会导致不均衡或无法摘除故障实例。
    • 两种形态：
      • 客户端 LB：调用方拿到实例列表后自行选择（常见于 RPC 框架/一些 Spring Cloud 形态）。
      • 服务端 LB：通过 Nginx/Envoy/SLB 统一转发，客户端只认一个入口。
    • 常见策略与场景：
      • 轮询/权重：通用；配合健康检查即可。
      • 一致性哈希：需要“同一 key 尽量落同一实例”（如本地缓存、会话亲和、分片路由）。
    • 实战：我会把“故障摘除”和“慢实例规避”当成负载均衡的一部分（例如结合实时指标做权重衰减）。
    • 可能追问：
    • 追问：一致性哈希解决什么问题？
      15秒简答：解决节点增减时大量 key 重映射导致缓存雪崩；它让只有少量 key 迁移，提升稳定性。
    • 追问：负载均衡怎么感知“慢实例”？
      15秒简答：结合熔断/统计 RT、错误率做实例级隔离，或用客户端侧的自适应负载（优先选健康且 RT 更低的实例）。

    RPC 调用必备的三件套：超时、重试、隔离，你怎么设计？¶

    • 15秒简答：RPC 调用本质是“把本地调用变成不可靠的网络调用”；核心机制是超时界定失败、重试提高成功率、隔离防止故障扩散；特性是能显著提稳但配置不当会放大流量与雪崩。
    • 3分钟详答：出现名次：#04（极高频）。我先给一句话结论：没有超时的 RPC 等于“无限占用资源”，没有隔离的重试等于“把故障放大”。
    • 技术背景：网络调用必然存在延迟抖动、丢包、服务端排队；“慢”会拖垮线程池与连接池。
    • 设计要点：
      • 超时：每一跳都要有 deadline（网关、BFF、服务间），并且下游超时要小于上游，避免无意义等待。
      • 重试：只对“幂等且短耗时”接口做有限次数重试；重试要加退避+jitter，避免同步洪峰。
      • 隔离：线程池隔离/舱壁隔离，把不同下游的资源池分开；必要时做熔断快速失败。
    • 为什么这样设计：把故障限制在局部，避免“一个下游慢 -> 上游线程堆满 -> 全链路不可用”。
    • 实战：我会先画出调用链，给每条链路定超时预算，并把重试从“默认开启”改成“按接口白名单启用”。
    • 可能追问：
    • 追问：重试会带来什么副作用？
      15秒简答：会放大流量、加剧下游拥塞、导致雪崩；还可能产生重复写，所以必须配合幂等与限流。
    • 追问：隔离除了线程池还有什么？
      15秒简答：还有连接池隔离、实例隔离（灰度/泳道）、数据隔离（分库分表），以及限流把系统压在可承受范围内。

    熔断、降级、限流分别是什么？线上一般怎么组合？¶

    • 15秒简答：它们本质都是“把系统从崩溃边缘拉回可控区间”；核心机制是限流控入口、熔断隔离故障下游、降级提供兜底结果；特性是牺牲部分功能/体验换整体可用。
    • 3分钟详答：出现名次：#05（极高频）。我先给一句话结论：限流治“进”，熔断治“出”（下游），降级治“结果”（给用户什么回应）。
    • 技术背景：微服务链路长，任何节点抖动都可能引发级联故障（雪崩）。
    • 三者定义与触发：
      • 限流：按 QPS/并发数/令牌桶/漏桶限制进入系统的请求。
      • 熔断：监控错误率/RT/并发等指标，超过阈值后短路请求，进入半开探测恢复。
      • 降级：在资源紧张或依赖不可用时，返回默认值/缓存/简化逻辑。
    • 组合套路（常见实践）：
      • 入口层（网关/Nginx）：全局限流+黑白名单。
      • 服务间调用：超时+隔离+熔断；熔断后触发降级返回兜底。
    • 实战：我会把降级结果设计成“对业务可接受”的最小可用（例如用缓存价/推荐兜底），并让前端能识别降级态。
    • 可能追问：
    • 追问：熔断为什么要半开？
      15秒简答：半开用于小流量试探恢复，避免一恢复就全量打过去导致二次击穿。
    • 追问：限流算法怎么选？
      15秒简答：要平滑就令牌桶/漏桶；要保护并发就并发限流；要精确到用户/接口就用滑动窗口+分布式计数（Redis/Lua）。

    网关在微服务里做什么？为什么说“网关不要太重”？¶

    • 15秒简答：网关本质是“统一入口与流量治理层”；核心机制是路由转发+鉴权+限流+协议适配；特性是收敛横切能力，但过重会变单点瓶颈和故障放大器。
    • 3分钟详答：出现名次：#11（中高频）。我先给一句话结论：网关负责“通用能力与策略”，业务逻辑尽量下沉到服务，否则网关会变成新的大单体。
    • 技术背景：没有网关时，客户端要直连多个服务，鉴权/限流/灰度重复建设且难统一。
    • 常见职责：
      • 路由与聚合：按路径/Host/版本路由；部分场景做轻量聚合（BFF 也可承担）。
      • 安全：JWT 校验、签名、黑白名单、WAF 规则等。
      • 治理：限流、熔断、灰度、A/B、header 注入、统一日志。
    • 为什么不要太重：网关链路最前，负载最大；业务逻辑放进去会导致发布风险与性能瓶颈集中。
    • 实战：我会把网关做成“策略执行器”，策略来自配置中心；重逻辑放到 BFF 或业务服务，并配套降级。
    • 可能追问：
    • 追问：网关和 BFF 有什么区别？
      15秒简答：网关偏通用治理与安全，面向全站；BFF 偏面向特定端的聚合与适配，更多带业务语义。
    • 追问：网关如何做鉴权与权限？
      15秒简答：网关做身份认证与粗粒度授权（token/角色）；细粒度数据权限通常在业务服务校验，避免网关承担过多业务规则。

    灰度/金丝雀发布怎么做？如何按用户/比例路由？¶

    • 15秒简答：灰度发布本质是“让新版本先吃小流量验证风险”；核心机制是按规则把流量分桶（比例/用户/地区/版本）并可随时回滚；特性是降低发布风险但需要可观测与自动化。
    • 3分钟详答：出现名次：#12（中高频）。我先给一句话结论：灰度的关键不是“怎么分流”，而是“怎么快速发现问题并回退”，所以必须配套监控与对比。
    • 技术背景：微服务频繁发布，如果全量上线，问题会瞬间放大。
    • 常见做法：
      • 比例灰度：1% -> 5% -> 20% -> 100%，监控核心指标。
      • 用户灰度：按 userId hash 分桶，保证同一用户体验一致。
      • 泳道/标签路由：header/metadata 标记，让同一链路请求落在同一套实例（避免混跑导致数据/逻辑不一致）。
    • 为什么这样设计：控制影响面，并让问题可复现（同一用户稳定命中同一版本）。
    • 实战：我会固定灰度窗口观察 SLO（错误率、P95、下游依赖失败率），并准备“一键回滚/降级”。
    • 可能追问：
    • 追问：为什么需要“泳道”而不仅是网关分流？
      15秒简答：因为链路中间可能还有多次服务间调用；泳道能把整条链路固定在同一版本/环境，避免 A 服务新、B 服务旧导致语义不一致。
    • 追问：灰度如何避免缓存污染？
      15秒简答：对关键缓存做版本隔离（key 加版本/泳道标签）或让灰度流量走独立缓存实例，避免新逻辑写入影响全量用户。

    配置中心如何做动态配置？怎么避免“配置把系统打挂”？¶

    • 15秒简答：配置中心本质是“把配置从代码发布中解耦”；核心机制是集中存储+版本管理+推送/拉取+灰度生效；特性是变更快但误配置风险高，需要审计与回滚。
    • 3分钟详答：出现名次：#15（中高频）。我会先给一句话结论：动态配置是把“上线风险”变成“配置风险”，所以要像对待代码一样对待配置。
    • 技术背景：微服务数量多，靠重启/重新发布更新配置会影响交付效率与稳定性。
    • 核心机制：
      • 配置集中管理：按应用/环境/命名空间隔离。
      • 动态推送：客户端监听变更，热更新本地配置。
      • 版本与审计：每次发布配置都有版本号、发布人、变更记录。
    • 防止配置事故的手段：
      • 配置校验：schema 校验、范围校验、关键项不允许空。
      • 灰度配置：先对少量实例生效观察。
      • 一键回滚：快速回到上一版本配置。
    • 实战：对“限流阈值、超时、开关”这类高风险项，我会加双重防护：发布审核+自动回滚触发条件（错误率飙升即回退）。
    • 可能追问：
    • 追问：配置热更新有什么坑？
      15秒简答：非线程安全的热更新、部分组件不支持动态刷新、以及变更风暴；要做变更节流、分批推送和幂等应用。
    • 追问：敏感配置（密钥）怎么管理？
      15秒简答：用专门的密钥管理（KMS/Vault）或加密存储+最小权限访问，严禁明文进仓库与日志。

    一致性与分布式事务¶

    CAP 定理怎么解释？你在项目里怎么做取舍？¶

    • 15秒简答：CAP 本质是“分区必然时只能在一致性和可用性二选一”；核心机制是网络分区下要么拒绝请求保一致，要么继续服务容忍不一致；特性是分布式系统默认必须选 P，然后按业务选 C 或 A。
    • 3分钟详答：出现名次：#08（高频）。我先给一句话结论：CAP 不是让你背定义，而是让你在“故障时刻”明确系统行为——是宁可失败，还是宁可读到旧值。
    • 技术背景：分布式一定会遇到网络分区/延迟抖动；你无法同时保证强一致与100%可用。
    • 典型取舍：
      • 金融扣款/库存强一致：倾向 CP（宁可失败/排队/降级，也不允许错账）。
      • 推荐/搜索/资讯：倾向 AP（可用更重要，允许短期不一致）。
    • 为什么这样设计：把一致性资源留给“必须正确”的链路，其他链路用最终一致换性能与可用。
    • 实战：我会把“强一致链路”做短做少（例如用本地事务+强约束），把跨服务协作通过事件驱动变成最终一致，并加对账补偿兜底。
    • 可能追问：
    • 追问：最终一致如何向业务解释？
      15秒简答：解释为“短时间可能看到旧状态，但最终会收敛到正确结果”，并给出可观测的状态机（处理中/成功/失败）避免用户困惑。
    • 追问：什么是线性一致/读写一致？
      15秒简答：线性一致像单机一样全局有序；读写一致通常指“读到自己刚写的结果”；工程上常用缓存/会话粘滞/版本号实现读一致体验。

    分布式一致性有哪些常见模型？强一致、最终一致、读写一致怎么落地？¶

    • 15秒简答：一致性模型本质是“多副本/多服务对同一事实的收敛约束”；核心机制是同步复制/共识/异步事件；特性是越强一致越慢越贵，最终一致更快但要处理补偿与对账。
    • 3分钟详答：出现名次：#14（中高频）。我会先给一句话结论：工程上最常见是“关键链路强一致 + 辅助链路最终一致 + 用户体验层读一致”。
    • 技术背景：微服务里常见“订单库、库存库、支付库”各自独立，多点写入需要定义一致性目标。
    • 落地方式示例：
      • 强一致：同库本地事务、强约束（唯一索引/外键约束有限）、或者把关键状态收敛在一个服务里写。
      • 最终一致：事件驱动（Outbox/CDC/MQ）传播状态，消费端幂等更新。
      • 读写一致（体验层）：刚下单后，读请求优先走主库/写库，或通过订单状态机给“处理中”态并轮询/推送。
    • 为什么这样设计：把一致性成本用在“不能错”的地方，其他用补偿保证可接受的正确性。
    • 实战：我会给每个关键业务对象定义状态机与对账口径，把“补偿入口”做成标准能力（重放事件/手工补偿/定时对账）。
    • 可能追问：
    • 追问：最终一致最怕什么？
      15秒简答：最怕事件丢失与重复、乱序、以及补偿无法收敛；因此必须有可靠投递、幂等、可重放和对账机制。
    • 追问：读一致如何和缓存配合？
      15秒简答：用写后删缓存+延迟双删/消息通知，或对关键读走主库；必要时用版本号/逻辑过期减少读到旧值的概率。

    分布式事务有哪些方案？你怎么选（2PC/TCC/Saga/本地消息表/事务消息）？¶

    • 15秒简答：分布式事务本质是“跨多个资源的原子提交”；核心机制要么用协调器强一致（2PC/AT），要么把原子性降级为补偿（TCC/Saga/消息）；特性是强一致更重更慢，补偿型更轻但业务复杂。
    • 3分钟详答：出现名次：#09（高频）。我先给一句话结论：大多数互联网业务会尽量避免强一致分布式事务，优先用“状态机+消息+补偿”实现最终一致。
    • 技术背景：跨服务+跨库时，单机 ACID 不再适用，网络故障会让“提交/回滚”变得不可靠。
    • 选型思路（按业务约束选）：
      • 强一致要求高且并发不极端：可考虑 2PC/AT，但要评估锁与性能。
      • 业务可拆分且可补偿：优先 TCC/Saga。
      • 以事件为中心的协作：本地消息表（Outbox）/事务消息，最常见也最通用。
    • 为什么这样设计：在分布式里追求“全局原子性”代价非常高，且容易成为可用性瓶颈。
    • 实战：订单创建我通常用“本地事务写订单+写Outbox”，再异步驱动库存/优惠/积分更新，失败走补偿或人工对账。
    • 可能追问：
    • 追问：本地消息表怎么保证“不丢不重”？
      15秒简答：同一数据库事务里写业务数据+Outbox；后台扫描未投递记录并重试；消费端幂等（按业务唯一键/消息 id 去重）。
    • 追问：2PC 最大问题是什么？
      15秒简答：阻塞与可用性差：参与者持锁等待协调者，协调者故障会导致长时间不确定状态，吞吐和可用性都受影响。

    TCC 的 Try/Confirm/Cancel 怎么保证幂等？“空回滚/悬挂”怎么处理？¶

    • 15秒简答：TCC 本质是“业务层两阶段提交”；核心机制是 Try 预留资源、Confirm 真正提交、Cancel 释放预留；特性是强一致可控但侵入业务、边界情况多。
    • 3分钟详答：出现名次：#18（中高频）。我先给一句话结论：TCC 最难不是三段流程，而是故障重试下的幂等、悬挂、空回滚三件事。
    • 技术背景：网络超时会导致“调用方以为失败但下游已成功”，所以 Confirm/Cancel 会重试。
    • 三个必做的工程化点：
      • 幂等：Confirm/Cancel 必须可重复执行（基于事务 id 记录执行状态）。
      • 空回滚：Cancel 先于 Try 到达时，要能识别并直接返回成功（记录空回滚标记）。
      • 悬挂：Try 超时后被重试，必须识别同一事务 id 的重复 Try，避免重复预留。
    • 为什么这样设计：TCC 把一致性下沉到业务资源层，必须显式处理分布式的不确定性。
    • 实战：我会为每个 TCC 资源表设计“冻结字段+事务状态表”，并用唯一约束保证同一事务 id 的状态只会推进一次。
    • 可能追问：
    • 追问：TCC 适合哪些场景？
      15秒简答：资源能预留/冻结且补偿明确的场景，比如余额冻结、库存冻结；不适合无法冻结或外部不可控资源。
    • 追问：TCC 为什么说“侵入业务”？
      15秒简答：因为每个资源都要写 Try/Confirm/Cancel 三套接口与状态表，业务代码需要围绕事务状态机设计。

    Saga 和 TCC 有什么区别？Saga 的补偿事务有哪些坑？¶

    • 15秒简答：Saga 本质是“长事务拆成一串本地事务+补偿动作”；核心机制是每步提交成功就推进，失败则按逆序补偿；特性是吞吐高但一致性是最终一致，补偿复杂且可能不可逆。
    • 3分钟详答：出现名次：#20（中高频）。我会先给一句话结论：TCC 更像“先冻结后确认”，Saga 更像“先做了再补偿”，所以 Saga 对补偿的可行性要求更高。
    • 技术背景：跨服务长流程（下单->锁库存->扣优惠->支付）如果用强一致会太重。
    • 核心区别：
      • TCC：先 Try 预留，后 Confirm；Cancel 释放预留，偏强一致。
      • Saga：每步本地事务直接提交，失败再做补偿，偏最终一致。
    • Saga 的典型坑：
      • 补偿不可逆：例如发送短信、第三方扣款很难撤销。
      • 并发冲突：补偿期间状态变化要靠版本号/状态机保护。
      • 乱序与重试：补偿动作也必须幂等。
    • 实战：我会优先选择“可逆/可补偿”的步骤纳入 Saga，把不可逆步骤放到末端或改为人工/对账处理。
    • 可能追问：
    • 追问：Saga 一般用编排还是协同？
      15秒简答：编排（集中式）更易控流程与补偿；协同（事件驱动）更松耦合但更难全局观测，常配合状态机与追踪。
    • 追问：补偿失败怎么办？
      15秒简答：设计可重试的补偿队列+人工介入通道；并把状态标记为“待补偿/异常”，用对账任务持续修复。

    Seata AT 模式大概原理是什么？线上常见坑有哪些？¶

    • 15秒简答：AT 本质是“基于代理数据源的自动补偿事务”；核心机制是记录 undo log、两阶段提交并用全局锁协调；特性是接入相对快但会引入锁竞争与性能开销。
    • 3分钟详答：出现名次：#22（中高频）。我先给一句话结论：AT 的优点是对业务侵入低，缺点是全局锁与回滚开销会在高并发场景放大。
    • 技术背景：希望像本地事务一样写代码，但跨多个服务/库也能回滚。
    • 原理（抓住关键点即可）：
      • 一阶段：各分支事务执行本地事务，同时写 undo log（前镜像/后镜像）。
      • 二阶段：全局提交时释放资源；全局回滚时用 undo log 反向恢复。
      • 全局锁：避免并发写导致回滚不安全。
    • 常见坑：
      • 热点行/热点表导致全局锁竞争，吞吐下降。
      • undo log 膨胀与回滚耗时，导致长事务拖慢系统。
      • 事务边界不清（把 RPC/IO 放进全局事务）导致超时与死锁风险。
    • 实战：我会严格控制全局事务范围，避免把长链路/外部调用放进去；并对热点写场景优先考虑 TCC/消息最终一致。
    • 可能追问：
    • 追问：AT 和 TCC 怎么选？
      15秒简答：追求快速接入且并发不极端可用 AT；高并发热点写或需更强可控性更适合 TCC/消息+补偿。
    • 追问：为什么不把远程调用放进全局事务？
      15秒简答：会拉长锁持有时间，导致锁竞争、超时与级联故障概率大增，得不偿失。

    分布式锁怎么实现？Redis vs ZooKeeper 怎么选？Redisson 看门狗是什么？¶

    • 15秒简答：分布式锁本质是“在多进程间实现互斥”；核心机制是用可靠存储记录锁的归属与过期，并确保解锁原子性；特性是 Redis 性能好但一致性弱些，ZK 更强一致但吞吐更低。
    • 3分钟详答：出现名次：#13（中高频）。我先给一句话结论：分布式锁的关键不是“加锁”，而是“不会误删别人的锁、能自动过期、故障下不死锁”。
    • 技术背景：多实例并发操作同一资源（如同一订单状态推进、同一优惠券库存扣减）需要互斥。
    • Redis 方案要点：
      • 加锁：SET key value NX PX ttl（value 放唯一标识）。
      • 解锁：校验 value 再删，用 Lua 保证原子性，避免误删。
      • 续期：业务执行时间不确定时需要续期机制。
    • ZooKeeper 方案要点：用临时顺序节点排队，监听前序节点删除获取锁，强一致但性能成本更高。
    • Redisson 看门狗：在未显式指定 ttl 时，自动周期续期，避免业务未完成锁过期导致并发进入。
    • 实战：需要强一致互斥（极度怕并发写错）我倾向 ZK；需要高吞吐且可接受少量风险/有补偿我用 Redis+严格解锁协议。
    • 可能追问：
    • 追问：RedLock（多 Redis 实例）怎么看？
      15秒简答：它试图用多数派提高可用性，但对时钟与网络条件敏感且争议大；工程上更常用单集群 Redis 锁+业务幂等/补偿，或直接用 ZK/etcd。
    • 追问：分布式锁一定要用吗？
      15秒简答：不一定，优先用数据库唯一约束/乐观锁/状态机幂等来“天然互斥”，锁只用于确实需要互斥且冲突高的场景。

    分布式 ID 怎么生成？雪花算法有哪些坑？时钟回拨怎么办？¶

    • 15秒简答：分布式 ID 本质是“在多节点下生成全局唯一且有序的主键”；核心机制是按时间戳+机器号+序列号组合（雪花）或号段分配；特性是雪花有序性能好但怕时钟问题，号段更稳但需要号段服务。
    • 3分钟详答：出现名次：#16（中高频）。我先给一句话结论：ID 方案要同时回答“唯一性、趋势递增、吞吐、可用性、时钟异常”五个问题。
    • 技术背景：分库分表/多服务写入时，数据库自增 id 会冲突或产生热点，需要独立 ID 方案。
    • 常见方案：
      • UUID：简单但无序、索引膨胀、写放大。
      • 雪花（timestamp+workerId+sequence）：趋势递增、吞吐高。
      • 号段（segment）：从数据库/号段服务批量领取区间，本地递增，性能好且时钟无关。
    • 雪花的坑与应对：
      • 时钟回拨：回拨会导致时间戳变小，可能重复或阻塞。
      • workerId 冲突：机器号分配不当会重复。
      • 解决思路：检测回拨后短暂阻塞到追平、切换备用 workerId、引入逻辑时钟/号段方案兜底。
    • 实战：业务强依赖趋势递增与高写入时我优先号段；雪花用于“可控机房时钟”的场景，并加回拨监控与熔断降级。
    • 可能追问：
    • 追问：为什么 UUID 不适合做主键？
      15秒简答：无序导致 B+Tree 索引频繁分裂，写放大严重，存储空间大，性能差。
    • 追问：号段服务会不会成单点？
      15秒简答：号段是低频操作（批量领取），可用主从+缓存+预取多段；就算短暂不可用，本地还有余量可继续发号一段时间。

    接口幂等怎么设计？如何防止重复提交/重复扣款/重复下单？¶

    • 15秒简答：幂等本质是“同一业务请求重复执行，结果只生效一次”；核心机制是用业务唯一键+状态机/去重表把重复请求吸收掉；特性是能抵抗重试与重复投递，但需要设计唯一语义与保存窗口。
    • 3分钟详答：出现名次：#17（中高频）。我先给一句话结论：分布式里“重试是常态”，所以写接口默认要可幂等，否则迟早出资金/库存事故。
    • 技术背景：网络超时会触发客户端/网关/SDK 重试；MQ 也可能重复投递。
    • 常见落地方式：
      • 业务唯一键：例如订单号/支付单号，配合数据库唯一索引，天然去重。
      • 去重表：记录 requestId -> 处理结果/状态，重复请求直接返回同结果。
      • 状态机：只允许状态按规定方向推进（NEW->PAID），重复推进无效。
      • 防重 token：前端先拿 token，提交时校验并一次性消费（更偏“防重复提交”）。
    • 为什么这样设计：把“重复”变成一次写入冲突/状态判断，而不是依赖分布式锁。
    • 实战：支付/扣减类接口，我会强制要求上游传业务幂等键，并在库里建唯一约束；同时对外返回可查询的业务单据状态，避免用户狂点重试。
    • 可能追问：
    • 追问：幂等和防重提交有什么区别？
      15秒简答：防重提交是入口层避免用户短时间重复点；幂等是系统层保证即使重复执行也只生效一次，范围更大。
    • 追问：去重表会不会无限增长？
      15秒简答：会，所以要设计 TTL/归档策略；只保留业务允许的重试窗口（比如 24 小时），或按业务单据生命周期清理。

    消息与异步架构¶

    为什么要引入消息队列？它解决什么问题，又带来什么新问题？¶

    • 15秒简答：MQ 本质是“把同步调用变成异步事件流”；核心机制是生产者写入 broker、消费者异步处理；特性是解耦/削峰/异步，但会引入最终一致、重复/乱序、积压与可观测难题。
    • 3分钟详答：出现名次：#19（中高频）。我先给一句话结论：MQ 的价值是把“强耦合链路”拆开，但你必须为此支付“消息可靠性与一致性”的工程成本。
    • 技术背景：同步链路越长越脆，任何慢依赖都会拖垮上游；高峰时同步调用会让系统被瞬时流量打爆。
    • 典型收益：
      • 解耦：下游挂了不影响上游写入（可控范围内）。
      • 削峰填谷：把峰值流量变成队列长度。
      • 异步化：用户请求快速返回，后台慢慢处理。
    • 新问题（面试高频）：
      • 可靠性：消息丢失/重复/乱序。
      • 一致性：业务与消息如何原子提交（Outbox/事务消息）。
      • 运维：积压、重试、死信、消费能力扩缩。
    • 实战：我会先把“消息语义”说清楚（至少一次/至多一次/恰好一次的目标与代价），并给每条消息定义幂等键与重试策略。
    • 可能追问：
    • 追问：不用 MQ 也能解耦吗？
      15秒简答：可以用异步任务/线程池/回调，但跨服务与跨进程场景下，MQ 的持久化与消费组能力更通用，容错也更好。
    • 追问：MQ 会让系统更“最终一致”，业务怎么接受？
      15秒简答：用状态机与可见状态（处理中/成功/失败）告知用户，并用补偿/对账保证最终正确，关键链路必要时用事务消息降低不一致窗口。

    如何保证消息不丢？生产端、Broker、消费端各怎么做？¶

    • 15秒简答：消息可靠性本质是“端到端确认”；核心机制是生产端确认写入、Broker 持久化与副本、消费端处理后再 ACK；特性是可靠性越强延迟越高，需要按业务分级。
    • 3分钟详答：出现名次：#21（中高频）。我会先给一句话结论：把链路拆成三段看：生产->存储->消费，任何一段没确认都可能丢。
    • 技术背景：网络抖动、Broker 宕机、消费者进程崩溃都会导致消息丢失或“看似丢失”。
    • 生产端：
      • 同步/异步发送确认（ack），失败重试。
      • 关键业务要记录发送日志/落库（Outbox）以便重放。
    • Broker 端：
      • 持久化（刷盘策略）与副本复制（至少写入多数/同步副本再 ack）。
      • 高可用集群，避免单点。
    • 消费端：
      • 处理成功后再 ACK/提交 offset；失败走重试与死信。
      • 消费逻辑必须幂等，避免重复投递造成重复生效。
    • 实战：我会给消息定义“可靠等级”：通知类可丢、业务状态变更不可丢；不可丢的必须配 Outbox/事务消息。
    • 可能追问：
    • 追问：如果消费者处理很慢，ACK 很晚会怎样？
      15秒简答：会导致消息堆积与重平衡风险；要做并发消费、扩容分区、以及把慢操作异步化或拆分。
    • 追问：Broker 刷盘策略怎么影响可靠性？
      15秒简答：异步刷盘吞吐高但断电可能丢；同步刷盘更稳但延迟高；关键链路通常更偏稳，非关键更偏性能。

    如何处理重复消费？“至少一次”语义下你怎么保证业务只生效一次？¶

    • 15秒简答：重复消费本质是“消息系统保证送达但不保证只送一次”；核心机制是消费端幂等（去重键/唯一索引/状态机）吸收重复；特性是实现简单可靠，但需要设计去重窗口与存储。
    • 3分钟详答：出现名次：#23（中高频）。我先给一句话结论：不要指望 MQ 给你恰好一次，工程上更可靠的是“至少一次 + 业务幂等”。
    • 技术背景：消费者超时、重平衡、网络抖动都会让同一条消息再次投递。
    • 常见幂等实现：
      • 去重表：messageId/业务单号 做唯一键，插入成功才执行，冲突则跳过。
      • 唯一索引：比如“订单状态变更表”用（orderId, state）唯一约束。
      • 状态机：只允许从特定前置状态推进，重复消息不会改变结果。
    • 提交 offset/ACK 的原则：业务落库成功后再提交，避免“提交了但业务没做”导致丢处理。
    • 实战：我会把“幂等键”写进消息体并贯穿全链路；同时提供重放工具，出现异常可按幂等键安全重放。
    • 可能追问：
    • 追问：去重表插入后业务执行失败怎么办？
      15秒简答：去重表要记录状态（处理中/成功/失败）并允许重试；失败时把状态改回可重试或删除记录，让下次重试能继续。
    • 追问：幂等是不是等于“不会重复扣款”？
      15秒简答：前提是幂等键语义正确且覆盖所有写路径；尤其是外部系统调用要用同一幂等键，并保证落库与外呼的顺序可恢复。

    顺序消息怎么保证？什么是“局部有序”，为什么全局有序很贵？¶

    • 15秒简答：顺序消息本质是“同一 key 的事件按产生顺序被处理”；核心机制是按 key 路由到同一分区/队列并单线程消费；特性是局部有序可用，全局有序吞吐差且可用性更难做。
    • 3分钟详答：出现名次：#24（中高频）。我先给一句话结论：面试里说“全局有序”通常是陷阱，工程上 99% 要的是“同一订单/同一用户的局部有序”。
    • 技术背景：订单状态变更（创建->支付->发货）要求按顺序处理，否则会出现状态回退或脏写。
    • 局部有序做法：
      • 生产端：用 orderId/userId 做 sharding key。
      • Broker：同 key 路由到同一分区/队列。
      • 消费端：该分区内按序消费（通常单线程或同 key 串行化）。
    • 全局有序为什么贵：所有消息进同一队列/分区，吞吐受单点限制，任何消费抖动都会放大。
    • 实战：我会优先设计为“按业务实体有序”，并把乱序容忍写入状态机（例如忽略过期状态事件）。
    • 可能追问：
    • 追问：局部有序下如何扩容？
      15秒简答：扩容通过增加分区提高并行度，但要接受分区变更时 key 可能迁移；要做消费者再均衡，并保证状态机能容忍短暂乱序/重复。
    • 追问：如果同一 key 的消费变慢怎么办？
      15秒简答：这是热点 key 问题；可拆分 key（更细粒度）、做分段队列、或把慢操作拆出去异步化，避免卡住整个 key 的顺序链路。

    延时消息、重试、死信队列怎么设计？如何避免“无限重试打爆下游”？¶

    • 15秒简答：重试体系本质是“失败可恢复的自动补偿”；核心机制是指数退避重试、超过阈值转死信、人工/任务补偿；特性是提高成功率但配置不当会造成重试风暴。
    • 3分钟详答：出现名次：#26（中高频）。我会先给一句话结论：重试不是越多越好，重试要“有边界、有退避、有隔离、有观测”。
    • 技术背景：失败通常分两类：可恢复（网络抖动、下游短暂过载）与不可恢复（参数错误、业务校验失败）。
    • 设计要点：
      • 退避：指数退避+jitter，避免集中重试。
      • 分类：可恢复才重试，不可恢复直接进死信或标记失败。
      • 上限：最大重试次数/最长重试窗口，避免无限重试。
      • 死信：进入 DLQ 后可人工处理或离线补偿。
    • 实战：我会在消息里带错误码与失败原因，做“可重试/不可重试”判定；并对 DLQ 做告警与处理 SOP。
    • 可能追问：
    • 追问：延时消息典型用在哪？
      15秒简答：订单超时关单、支付超时取消、缓存预热/延迟双删、异步补偿调度等。
    • 追问：怎么避免重试期间顺序被破坏？
      15秒简答：对需要顺序的 key，把重试也放在同一顺序通道（同分区串行重试），或用状态机忽略过期事件。

    消息积压怎么定位和治理？你会看哪些指标？¶

    • 15秒简答：积压本质是“生产速率长期大于消费速率”；核心机制是提升消费能力、降低生产、或拆分慢逻辑；特性是积压会放大延迟并可能触发重平衡/雪崩。
    • 3分钟详答：出现名次：#27（中高频）。我会先给一句话结论：先判断是“突发峰值”还是“持续失衡”，再选扩容/限流/降级/修复消费瓶颈。
    • 技术背景：积压会导致业务状态延迟，严重时会打满磁盘/内存，影响 Broker 可用性。
    • 排查路径（从外到内）：
      • 看积压曲线：队列长度/延迟是否持续上升。
      • 看消费端：并发度、CPU/IO、线程池、下游依赖 RT、是否频繁失败重试。
      • 看消息体：是否有“超大消息”或某类消息异常增多。
    • 治理手段：
      • 提升消费：增加消费者实例/分区、批量消费、优化慢 SQL、拆分耗时逻辑。
      • 降低生产：入口限流、业务降级、合并消息。
      • 临时手段：跳过非关键消息/旁路处理（要有审计与补偿）。
    • 实战：我会把“积压延迟”作为 SLO 指标（例如 P95 消费延迟），并对 DLQ/重试量设告警阈值。
    • 可能追问：
    • 追问：扩容消费者一定能解决吗？
      15秒简答：不一定，如果瓶颈在下游（数据库/外部接口）或消息需要局部有序（同 key 串行），盲目扩容只会加剧竞争。
    • 追问：如何快速止血？
      15秒简答：入口限流/降级减少生产，暂停非关键主题消费或旁路处理，同时优先修复消费端瓶颈（慢 SQL/下游超时/线程池配置）。

    可观测性与稳定性¶

    分布式链路追踪怎么做？traceId/spanId 是什么？日志怎么串起来？¶

    • 15秒简答：链路追踪本质是“给一次请求在多服务间打通同一条时间线”；核心机制是传播 traceId、为每段调用生成 span 并上报；特性是定位慢点/错误很强，但有采样与性能开销。
    • 3分钟详答：出现名次：#25（中高频）。我先给一句话结论：追踪要解决两个问题：跨服务定位（谁慢/谁错）和关联日志（一次请求的所有日志能聚合）。
    • 技术背景：微服务里一次用户请求可能经过网关、多个服务、缓存、数据库、MQ；没有 traceId 很难定位问题点。
    • 核心流程：
      • 入口生成 traceId（或沿用上游），放到 header（如 traceparent/b3 等）。
      • 服务间调用自动透传，并在每次 RPC/DB 操作上生成 span，记录开始/结束、tags（status、db.statement 摘要等）。
      • 上报到追踪系统（Zipkin/Jaeger/OTel Collector 等）。
    • 为什么这样设计：用统一的上下文把“分散的日志与调用”串成一条链路，减少 MTTR。
    • 实战：我会把 traceId 写入日志 MDC，并在网关/服务端都打印；线上排障先从 trace 找慢 span，再去对应服务的日志与指标确认根因。
    • 可能追问：
    • 追问：采样怎么做才合理？
      15秒简答：正常流量低采样保成本，异常流量（错误/慢请求）提高采样；也可按用户/接口分层采样。
    • 追问：异步链路（MQ）怎么串？
      15秒简答：把 trace 上下文写进消息头/消息体，消费端继续沿用并创建新的 span，把 producer/consumer 两段关联起来。

    微服务监控你会建哪些核心指标？SLO/告警怎么设计？¶

    • 15秒简答：监控本质是“用指标刻画系统健康并提前预警”；核心机制是围绕 QPS/延迟/错误率/饱和度建立 SLO 并配告警；特性是能提前止损，但指标过多会噪声大。
    • 3分钟详答：出现名次：#28（中高频）。我会先给一句话结论：优先把“用户可感知”的 SLO 建起来，再补充内部指标用于定位。
    • 技术背景：微服务数量多，靠人工看日志不可持续，必须用指标体系做“发现问题”和“定位问题”。
    • 我会优先关注四类黄金指标：
      • 流量：QPS、并发、队列长度。
      • 延迟：P95/P99、下游 RT。
      • 错误：5xx、超时、熔断次数、重试次数。
      • 饱和度：CPU、内存、线程池队列、连接池占用、GC、磁盘。
    • 告警设计原则：
      • 告警要可行动：触发后有明确处置（扩容/降级/回滚）。
      • 分级：P0（影响用户）与 P1/P2（风险上升）。
      • 结合窗口：避免瞬时抖动误报（滑动窗口、持续 N 分钟）。
    • 实战：我会把告警和发布/灰度联动：灰度期间错误率超阈值自动回滚或自动降级开关。
    • 可能追问：
    • 追问：为什么要用 P95/P99 而不是平均值？
      15秒简答：平均值会掩盖长尾，用户体验通常被长尾决定；P95/P99 更能反映高压下的真实延迟。
    • 追问：线程池/连接池你看哪些指标？
      15秒简答：队列长度、拒绝次数、活跃线程、任务等待时间；连接池看活跃/等待、超时、慢查询与下游 RT。

    同城双活/异地多活怎么理解？RPO/RTO 是什么？怎么做容灾演练？¶

    • 15秒简答：多活容灾本质是“用冗余换故障下的连续服务”；核心机制是多机房部署+流量切换+数据复制+故障隔离；特性是可用性更高但成本与数据一致性难度更大。
    • 3分钟详答：出现名次：#29（中高频）。我会先给一句话结论：容灾不是“部署两份”就完了，关键在于数据如何同步、故障如何自动/半自动切换、以及演练能不能跑通。
    • 技术背景：机房级故障（网络、机柜、机房断电）会让单活整体不可用，需要跨机房冗余。
    • 核心概念：
      • RPO：能接受丢多少数据（恢复点目标）。
      • RTO：能接受停多久（恢复时间目标）。
    • 常见架构：
      • 同城双活：延迟低，可做较强一致或准强一致，切换更快。
      • 异地多活：延迟高，通常最终一致，依赖异步复制与冲突处理。
    • 演练要点：
      • 演练切流（DNS/SLB/网关），验证回切。
      • 演练数据恢复：验证复制延迟、对账与补偿能力。
      • 演练降级：关键依赖不可用时系统能否保核心功能。
    • 实战：我会把容灾目标和业务分级绑定：支付/下单链路优先保可用与正确，非关键功能降级；并定期做故障注入验证。
    • 可能追问：
    • 追问：异地多活如何处理数据冲突？
      15秒简答：尽量避免双写（按业务分区/单写点），必须双写时用全局唯一 id、版本号/向量时钟或业务规则解决冲突，并配对账。
    • 追问：怎么验证容灾真的有效？
      15秒简答：靠演练与指标：切换是否在 RTO 内完成、数据是否在 RPO 内可接受、核心 SLO 是否满足，以及演练是否覆盖最坏路径（依赖不可用、链路降级）。
