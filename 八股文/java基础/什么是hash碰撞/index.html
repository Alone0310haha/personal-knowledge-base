    什么是 Hash 碰撞（Hash Collision）¶

    15 秒简答（贴标签）¶

    • 定结构（它是什么）：哈希碰撞就是“不同的 key 经过 hash 计算后，落到了同一个哈希值/同一个桶下标”的现象。
    • 讲机制（它怎么动）：因为 hash 把无限多的输入映射到有限的桶（多对一），所以必然可能冲突；冲突后需要靠冲突解决策略继续定位。
    • 说特性（它怎么样）：不可避免、可控；处理不好会导致性能退化。

    3 分钟简答¶

    开场白（润色版）：哈希碰撞指的是不同 key 在经过哈希函数计算后，得到相同的哈希结果或者相同的桶位置。它不是“bug”，而是散列表把大量 key 压缩映射到有限空间时的必然现象，关键在于如何降低发生概率以及如何在发生后高效处理。

    带着场景讲核心流程（以 HashMap 为例）：

    • 我们往 HashMap put(key, value) 时，会先根据 key 算 hash，再算桶下标（本质是对数组长度取模）。
    • 如果两个不同 key 计算出来的桶下标相同，它们就会进入同一个 bucket，这就是一次哈希碰撞。
    • 进入同一个 bucket 后，HashMap 会继续用 equals 判断是不是同一个 key：
    • 如果是同一个 key：覆盖 value；
    • 如果不是：就需要“把多个元素都放在这个桶里”，并保证后续 get 还能正确找到它。
    • JDK 8 的 HashMap 主要用“拉链法”（链表）解决碰撞；当同桶元素过多时会树化为红黑树，避免查询从平均 O(1) 退化到最坏 O(n)，树化后大致是 O(log n)。
    • 为了让碰撞更少，HashMap 还会在元素增多时扩容（resize），把桶变多，整体碰撞概率降低。

    为什么是这样设计的：

    • 碰撞不可避免：桶数量有限，而 key 的可能性几乎无限，理论上一定存在多对一映射（抽屉原理）。
    • 工程上追求“平均快”：散列表通过“先定位桶，再在桶内小范围查找”，让大多数情况下接近 O(1)。
    • 通过组合手段让最坏情况也可控：更均匀的 hash（降低热点桶）+ 扩容（降低装载密度）+ 链表/红黑树（保证冲突后的可查找性）。

    实战中我用过的地方：

    • 业务侧大量用 HashMap 做缓存、去重、聚合统计（例如 userId -> UserDTO 索引、统计维度计数）。这类场景我会特别关注 key 的 hashCode/equals 实现是否正确且稳定。
    • 线上排查过因 key 设计不当导致的碰撞偏高：比如把很多请求按相似前缀分组、或者 key 的哈希分布不均，导致某些桶特别“热”，此时会考虑优化 key 组成、必要时调整容量/负载策略或改用并发容器。

    比喻理解（帮助记忆）¶

    • 把 HashMap 想成“楼道里的储物柜”：柜子数量有限（bucket 数组），每个物品（key）先根据规则算一个柜子号（hash/index）。
    • 哈希碰撞就是“两个不同物品被分到了同一个柜子号”。
    • 解决碰撞就是“同一个柜子里怎么放、怎么找”：
    • 拉链法：柜子里挂一串挂钩（链表），一个个往后挂；
    • 桶里东西太多：给柜子加一本目录（树化），查起来更快；
    • 扩容：多装几排柜子，让大家分散存放。

    面试官可能追问（每题 15 秒简答）¶

    1）HashMap 遇到哈希碰撞怎么处理？¶

    JDK 8 的 HashMap 主要用拉链法：同一个桶里用链表/红黑树存多个节点；put/get 时先按 hash 定位桶，再用 equals 精确匹配。链太长会树化，元素太多会扩容降低碰撞。

    2）常见的哈希碰撞解决方案有哪些？¶

    典型三类：拉链法（链地址法）、开放寻址法（线性探测/二次探测等）、再哈希法（双重哈希）。工程里最常见的是拉链法（比如 HashMap），冲突多时再配合树化/扩容兜底。

    • 延伸阅读：常见的哈希碰撞解决方案有哪些

    3）哈希碰撞会带来什么影响？¶

    碰撞会让同桶查找从 O(1) 退化为桶内查找：链表最坏到 O(n)，碰撞集中时会明显拖慢 put/get；同时也可能被利用做 Hash Flood 攻击，所以需要控制分布、扩容策略，并在实现上做最坏情况兜底（如树化）。

    4）什么是 Hash Flood（哈希洪水）攻击？怎么防？¶

    攻击者构造大量 hashCode 相同的 key，让它们都落在同一个桶，逼 HashMap 查找退化到 O(n)。防护思路：JDK 8 树化（链表转红黑树）降低最坏复杂度；业务侧限制入参规模/频率、对不可信输入做校验或限流，必要时使用更稳健的数据结构。

    5）为什么 HashMap 需要扩容？它和碰撞的关系是什么？¶

    扩容是为了降低装载密度：元素多了但桶不变，碰撞会增加；超过阈值（capacity * loadFactor）就 resize，让桶变多，元素更分散，从而减少碰撞并稳定性能。
