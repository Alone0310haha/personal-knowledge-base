    Java 语言基础¶

    你怎么解释 ==、equals()、hashCode() 的关系？为什么重写 equals 必须重写 hashCode？¶

    • 15秒简答：它们本质是“对象相等语义三件套”：== 比引用，equals 比业务相等，hashCode 给哈希容器分桶；核心机制是 HashMap/HashSet 先用 hashCode 定桶再用 equals 精确匹配；特性是不满足一致性会导致“能找不到、能重复放”。
    • 3分钟详答：我会把它拆成“引用相等 / 逻辑相等 / 哈希定位”三层。
    • ==：对引用类型比较的是“是不是同一个对象地址”；对基本类型比较的是值。
    • equals()：语义可被重写，用来表达“业务上是否相等”（比如两个 User 的 userId 相同就相等）。
    • hashCode()：返回一个 int，用于哈希表快速定位桶位置。哈希容器（HashMap/HashSet）典型流程是：先算 key 的 hash 定位到桶，再在桶里用 equals 找到目标元素。
    • 为什么重写 equals 必须重写 hashCode：哈希容器依赖“不变式”——如果 a.equals(b) 为 true，则 a.hashCode()==b.hashCode() 必须为 true；否则相等对象会落到不同桶，导致 contains/get/remove 可能找不到。反过来 hashCode 相等不要求 equals 相等（允许碰撞）。
    • 设计原因：哈希表用 hashCode 换性能（平均 O(1)），再用 equals 保正确性；这就要求 equals/hashCode 在“作为 key 期间”稳定一致。
    • 实战里我踩过的坑：把可变对象（字段会变）当 key，用来做去重/索引，字段变了导致 hashCode 变化，后续删除/查询失效，最终只能重建 map。
    • 可能追问：
    • 追问：HashMap 查找时是先 equals 还是先 hashCode？
      15秒简答：先算 hash 定桶，再在桶内比较（通常先比 hash，再比 ==，最后 equals）。
    • 追问：equals 写不好会怎样？
      15秒简答：违反自反/对称/传递/一致性会让容器行为异常（比如 set 里出现“看似重复”的元素）。

    String 为什么不可变？StringBuilder/StringBuffer 怎么选？intern() 你在面试里怎么讲清楚？¶

    • 15秒简答：String 本质是不可变字符序列；核心机制是不可变带来线程安全、可缓存 hash、可做常量池复用；特性是读共享快但频繁拼接会产生大量临时对象，拼接用 StringBuilder，需要同步才用 StringBuffer。
    • 3分钟详答：String 不可变主要解决三件事：安全、性能、复用。
    • 安全：作为 URL、类加载、文件路径等关键参数时，不可变避免被篡改；作为 Map key 时 hash 可缓存且不会“变桶位”。
    • 性能：String 的 hashCode 可以缓存；多线程共享同一字符串无需锁。
    • 复用：编译期常量与运行时常量池（string pool）可以让相同字面量共享实例。
    • 拼接选择：循环拼接用 StringBuilder（无锁、快）；历史兼容或确实需要多线程共享同一个 builder 才用 StringBuffer（有锁、慢）。
    • intern()：把字符串“规范化到常量池里”，返回池中引用；面试我会强调它不是“去重神器”，滥用会把大量动态字符串常驻在池里，增加内存压力与 GC 压力（尤其是高基数字符串）。
    • 实战：日志拼接、SQL/JSON 构造都用 StringBuilder 或模板；需要跨线程共享就避免共享可变 builder，而是各线程各自构建再合并。
    • 可能追问：
    • 追问："a"+"b" 与 new String("ab") 有什么差别？
      15秒简答：前者编译期可优化并进常量池；后者一定在堆上新建对象（字面量 "ab" 仍可能在池里）。
    • 追问：为什么 String 适合当 HashMap 的 key？
      15秒简答：不可变 + hash 可缓存，作为 key 期间不会变化，容器语义稳定。

    final、static 各解决什么问题？static final 常量有什么坑？¶

    • 15秒简答：final 约束“不可变/不可扩展”，static 表示“类级别共享”；核心机制是编译期/类加载期绑定；特性是语义清晰、可优化，但 static final 的编译期常量会被内联，版本升级可能踩坑。
    • 3分钟详答：我会从“作用域与绑定时机”讲。
    • final：修饰类表示不可继承；修饰方法表示不可重写；修饰变量表示引用不可变（对象内容未必不可变）。它更多是“约束行为”，减少意外修改。
    • static：属于类而不是对象，典型用于工具方法、全局单例、共享缓存、常量。它依赖类加载阶段初始化顺序。
    • static final：表示常量。坑在于：编译期常量（如基本类型和 String 字面量）可能被编译器内联到调用方 class 文件里；如果你升级被依赖 jar 的常量值但不重编调用方，调用方仍可能用旧值。
    • 实战：配置常量我更倾向“可控的配置中心/环境变量”，而不是把可变配置写死成 static final；真正的协议常量、枚举值才适合。
    • 可能追问：
    • 追问：final 修饰对象引用，为什么对象还能改？
      15秒简答：final 保证引用不指向别的对象，不保证对象内部字段不可变。
    • 追问：为什么很多工具类方法用 static？
      15秒简答：无状态、无需实例化，减少对象创建与调用成本。

    自动装箱/拆箱会带来什么问题？Integer 缓存你怎么举例说明？¶

    • 15秒简答：装箱/拆箱本质是编译器插入 valueOf()/xxxValue()；核心机制是 Integer.valueOf 有缓存区间（常见 -128~127）；特性是写起来方便但有隐藏 NPE、性能与比较陷阱。
    • 3分钟详答：装箱拆箱发生在基本类型与包装类型互转时，比如把 int 放进 List<Integer>、或者 Integer + int 运算。
    • 性能：高频场景会创建大量包装对象（或触发缓存命中），增加 GC 压力；热路径尽量用基本类型或专门的 primitive collection（如果工程允许）。
    • NPE：Integer x = null; int y = x; 会在拆箱时 NPE。
    • 比较陷阱：Integer a=127; Integer b=127; a==b 可能为 true（缓存）；Integer a=128; Integer b=128; a==b 多半为 false（新对象）。因此包装类型比较语义要用 equals 或拆箱后比较。
    • 实战：我在做统计计数、聚合时会尽量用 long/int，避免把数值塞进 Map<String, Integer> 后频繁自增导致装箱抖动（可以用 LongAdder/AtomicLong 或者局部变量累加后再写回）。
    • 可能追问：
    • 追问：为什么 Integer.valueOf 不直接 new？
      15秒简答：小整数高频，缓存能减少对象创建与内存占用。
    • 追问：BigDecimal 为什么不推荐 new BigDecimal(double)？
      15秒简答：double 有二进制精度误差；用 BigDecimal.valueOf 或字符串构造更可控。

    深拷贝 vs 浅拷贝怎么区分？你在 Java 里常用哪些手段实现深拷贝？¶

    • 15秒简答：浅拷贝只复制引用，深拷贝复制整棵对象图；核心机制是“是否共享内部可变对象”；特性是深拷贝更安全但成本更高，通常只在边界层做。
    • 3分钟详答：面试我会先给判断标准：拷贝后改一个对象的内部字段，另一个是否受影响。
    • 浅拷贝：复制对象本身的字段值；如果字段是引用类型，两个对象会共享同一内部对象（风险：一改全改）。
    • 深拷贝：递归复制内部对象，保证互不影响（成本：CPU/内存都更高）。
    • 常见实现方式：
      • 手写拷贝构造器/工厂方法（最可控、最推荐，尤其是关键领域对象）。
      • clone()（不推荐滥用：语义复杂、容易漏拷贝、与继承交互不友好）。
      • 序列化/反序列化（简单但慢、依赖可序列化、对象图大时很重）。
    • 实战：我更倾向在“跨层边界”（比如 DTO ↔ DO、缓存对象 ↔ 业务对象）做明确转换，而不是在核心逻辑里深拷贝到处飞。
    • 可能追问：
    • 追问：为什么很多人不喜欢 clone()？
      15秒简答：默认是浅拷贝且易漏字段，异常/可见性/继承规则复杂，维护成本高。

    面向对象、异常与泛型¶

    重载（overload）和重写（override）有什么本质差异？这题怎么答出“动态分派/静态分派”？¶

    • 15秒简答：重载是编译期按参数签名选择（静态分派），重写是运行期按实际类型选择（动态分派）；核心机制是方法签名绑定 vs 虚方法表；特性是重写决定多态行为，重载更像语法糖。
    • 3分钟详答：我会把它和“多态发生在哪个阶段”绑定起来讲。
    • 重载：同名不同参（参数类型/个数/顺序不同），返回值不同不算重载；调用哪个方法在编译期就确定了（看声明类型+实参类型）。
    • 重写：子类提供与父类同签名的方法实现（返回值可协变），调用在运行期根据对象的实际类型决定，这是多态的核心。
    • 设计原因：Java 要在保持面向对象多态的同时，让方法调用仍能高效（虚方法表/内联优化）。
    • 实战：我会用接口+重写实现策略模式（如不同支付渠道），而不是靠 if-else；重载更多用于提升可用性（比如构造器重载、工具方法多签名）。
    • 可能追问：
    • 追问：private/static 方法能重写吗？
      15秒简答：不能参与虚方法重写；static 是隐藏（hiding），private 不可见。
    • 追问：为什么返回值不同不能重载？
      15秒简答：调用点仅靠参数无法区分目标方法，编译期会歧义。

    接口 vs 抽象类怎么选？default 方法引入后，面试官到底在听什么？¶

    • 15秒简答：接口表达“能力/契约”，抽象类表达“共性骨架”；核心机制是多实现 vs 单继承；特性是接口更解耦、抽象类更利于复用代码，default 是兼容演进手段不是让接口变成抽象类。
    • 3分钟详答：我会按三条判断：
    • 能力是否需要“多实现”：一个类可以实现多个接口，但只能继承一个抽象类；如果业务需要组合能力（如 Serializable、Comparable），接口更合适。
    • 是否需要共享状态/受保护字段：抽象类可以持有状态与构造器，适合模板方法模式，约束子类流程；接口通常不持有实例状态（除非通过外部组合）。
    • 演进兼容：default 方法解决“接口加新方法会破坏所有实现类”的兼容问题；面试里我会强调它是版本演进的妥协，同时也带来多继承冲突（两个接口同签名 default 时需要显式覆盖）。
    • 实战：我常用接口定义 SPI/领域能力，用抽象类封装通用流程（比如统一的校验、日志、埋点），让子类只实现差异点。
    • 可能追问：
    • 追问：两个接口都有同名 default 方法怎么办？
      15秒简答：实现类必须显式重写并选择调用哪个（或提供自己的实现）。

    Java 异常体系怎么分？checked vs unchecked 你怎么说出工程味？¶

    • 15秒简答：异常本质是“错误传播协议”；核心机制是受检异常强迫调用方处理，运行时异常用于编程错误；特性是受检异常提升显式性但会污染签名，现代工程更偏运行时异常 + 明确边界处理。
    • 3分钟详答：我会先讲分类，再讲“怎么用”。
    • Throwable 下分 Error（JVM/系统级，通常不处理）与 Exception。
    • Exception 分为受检异常（checked，必须 catch/throws）与运行时异常（RuntimeException，可不显式声明）。
    • 工程实践：业务代码里我倾向用“语义化运行时异常 + 统一异常处理层”（比如在接口层转成错误码/HTTP 状态），避免一路 throws 把签名污染；而对外部资源（IO、网络、DB）我会在边界层捕获并包装成业务可理解的异常。
    • finally/资源释放：try-with-resources 对可关闭资源更稳，避免忘记 close；finally 要避免吞异常（比如在 finally 再抛新异常覆盖原异常）。
    • 实战：在调用第三方接口时，我会把超时、限流、格式错误映射成不同业务异常，配合重试/降级，保证上层能做策略选择。
    • 可能追问：
    • 追问：try-with-resources 的原理是什么？
      15秒简答：编译器生成自动 close 的 finally 逻辑，并正确处理 suppressed 异常。
    • 追问：为什么不建议 catch Exception 然后啥也不做？
      15秒简答：会吞掉错误，导致问题隐蔽且状态不一致；至少要转换/记录/重试/返回明确失败。

    泛型为什么是“擦除实现”？? extends / ? super（PECS）怎么 15 秒讲清楚？¶

    • 15秒简答：Java 泛型本质是编译期类型检查；核心机制是类型擦除保证向后兼容；特性是运行时拿不到真实泛型参数，PECS 口诀是 Producer Extends / Consumer Super。
    • 3分钟详答：我会按“为什么这么设计 + 会带来什么限制”展开。
    • 擦除原因：Java 早期没有泛型，为了二进制兼容，选择让泛型只存在于编译期，运行时擦成原始类型（raw type），并在需要的地方插入强转与桥接方法。
    • 限制：运行时不能 new T()，不能直接 T.class，也不能用 instanceof List<String>；反射拿泛型要依赖 ParameterizedType 等元信息（且只在声明处可见）。
    • PECS：如果你只从集合读元素（生产者），用 ? extends T；如果你只往集合写元素（消费者），用 ? super T。这样类型系统才能保证安全。
    • 实战：我在封装通用工具（转换器、批处理器）时会用泛型表达输入输出类型，减少强转；但不会为了“炫技”把泛型层级堆到难读。
    • 可能追问：
    • 追问：什么是 raw type？为什么不建议用？
      15秒简答：不带泛型参数的类型会绕过编译期检查，运行期更容易 ClassCastException。

    反射与动态代理常见于哪些场景？JDK 动态代理和 CGLIB 的差异点是什么？¶

    • 15秒简答：反射/代理本质是“运行时元编程”；核心机制是通过 InvocationHandler（接口）或字节码生成（子类）织入增强；特性是灵活但有成本，常用于框架（AOP、RPC、ORM）。
    • 3分钟详答：我会从“为什么存在 + 什么时候用”讲。
    • 反射：运行时获取类、方法、字段信息并操作，常见于对象映射、序列化、框架注入、SPI 发现等。
    • 动态代理：在调用目标方法前后插入增强逻辑（日志、鉴权、事务、埋点、重试）。
    • JDK 动态代理：基于接口，生成实现同接口的代理类，通过 InvocationHandler 分发调用；前提是目标必须有接口。
    • CGLIB：通过生成目标类的子类来代理，不要求接口，但要求类/方法不能是 final；实现更重，生成字节码成本更高。
    • 实战：在项目里我更关注“代理链路是否清晰、异常如何传播、性能是否可控”，比如 RPC 调用的超时重试要避免放大流量。
    • 可能追问：
    • 追问：为什么 final 会影响 CGLIB？
      15秒简答：CGLIB 通过继承覆盖方法实现增强，final 禁止继承/重写。

    集合框架（List / Set / Queue）¶

    你怎么一句话讲清 List、Set、Queue/Deque 的差别？如何在面试里把“选型”讲出来？¶

    • 15秒简答：它们本质是不同“容器约束”：List 有序可重复，Set 唯一性，Queue/Deque 强调出入队规则；核心机制是底层结构差异（数组/链表/哈希/树/堆）；特性是选型看读写比例、是否需要去重/排序/TopN/两端操作。
    • 3分钟详答：我会用“需求 → 约束 → 结构”推导。
    • 需要按插入顺序访问、允许重复：List（ArrayList 多、LinkedList 少）。
    • 需要去重：Set（不关心顺序用 HashSet；要保留插入顺序用 LinkedHashSet；要排序用 TreeSet）。
    • 需要排队/栈：Queue/Deque（一般队列用 ArrayDeque；优先级队列用 PriorityQueue；线程安全队列另说）。
    • 实战：我常见是用 List 承载批量数据，用 Set 做去重，用 Deque 做 BFS/滑动窗口/缓存淘汰的双端结构。
    • 可能追问：
    • 追问：为什么很多场景不用 Stack？
      15秒简答：Stack 继承自 Vector、设计老且同步开销大，推荐用 ArrayDeque 作为栈。

    ArrayList vs LinkedList 的核心差异是什么？别只说“数组/链表”，说到什么程度算到位？¶

    • 15秒简答：ArrayList 本质是动态数组，随机访问快；LinkedList 本质是双向链表，插入删除节点快但随机访问慢；核心机制是内存连续性 vs 指针跳转；特性是大多数业务更适合 ArrayList。
    • 3分钟详答：我会强调“复杂度 + 常见误解”。
    • 访问：ArrayList.get(i) O(1)；LinkedList.get(i) O(n)（需要走指针）。
    • 插入删除：在“已定位到节点位置”的前提下，LinkedList 改指针 O(1)；但业务常常需要先按 index 找到位置，这一步就是 O(n)，因此整体未必比 ArrayList 快。
    • 内存与局部性：ArrayList 连续内存带来 CPU cache 友好；LinkedList 节点对象多、指针多，内存碎片与 GC 压力更大。
    • 实战：高频遍历、随机访问、批量追加场景几乎都选 ArrayList；只有在明确需要两端频繁插入删除且不依赖随机访问时，才考虑 LinkedList（或者直接用 Deque 的实现）。
    • 可能追问：
    • 追问：为什么 LinkedList 也实现了 Deque？
      15秒简答：它支持首尾 O(1) 插入删除，天然适合作为双端队列。
    • 追问：面试官问“链表插入删除快”你怎么补一句避免踩坑？
      15秒简答：快的前提是已拿到节点引用；如果按下标定位，定位成本会把优势吃掉。

    ArrayList 扩容机制你怎么讲？为什么建议初始化容量？¶

    • 15秒简答：ArrayList 本质是可扩容 Object[]；核心机制是容量不够时创建更大数组并拷贝；特性是摊还 O(1) 追加，但扩容和中间插入会 O(n)，提前预估容量能降拷贝成本。
    • 3分钟详答：我会从“触发点、策略、代价”讲。
    • 触发：add 发现 size+1 > elementData.length 就扩容。
    • 策略：通常按比例增长（不同 JDK 版本细节不同），核心思想是减少扩容次数。
    • 代价：扩容要分配新数组并 System.arraycopy 拷贝，瞬时 CPU 抖动；内存上旧数组等待 GC，可能造成峰值。
    • 初始化容量：批量装载（比如从 DB 查 10w 条）如果不设容量，会发生多次扩容拷贝；用 new ArrayList<>(expectedSize) 或先 ensureCapacity 可以显著降低成本。
    • 实战：分页/批处理我会预估容量；同时避免在大 list 中间频繁 remove(0) 这种 O(n) 行为（用 deque 或双指针替代）。
    • 可能追问：
    • 追问：remove(int index) 为什么慢？
      15秒简答：需要把 index 后面的元素整体左移，产生 O(n) 拷贝。

    HashSet 为什么能保证“不重复”？它和 HashMap 的关系是什么？¶

    • 15秒简答：HashSet 本质是“只存 key 的 HashMap”；核心机制是元素作为 map 的 key，利用 hashCode+equals 判重；特性是平均 O(1) 去重，但判重语义完全取决于元素的 equals/hashCode。
    • 3分钟详答：HashSet 内部一般用 HashMap 存储元素：把 set 的元素当作 map 的 key，value 放一个固定哨兵对象。加入元素时，等价于 map.put(e, PRESENT)；如果 key 已存在，put 会覆盖并返回旧值，从而 set 判断“已存在”。
    • 判重条件：同桶（hash 相同）且 equals 为 true 才认为重复。
    • 实战：我做去重时会特别注意 key 对象是否可变，以及是否正确实现 equals/hashCode；如果要排序去重，就换 TreeSet 或者先去重再排序。
    • 可能追问：
    • 追问：为什么 HashSet 的遍历顺序不稳定？
      15秒简答：桶分布与扩容会改变内部结构，遍历顺序不承诺稳定；需要稳定顺序用 LinkedHashSet。

    TreeSet/TreeMap 的排序规则是什么？Comparable 和 Comparator 你怎么把坑讲清楚？¶

    • 15秒简答：它们本质是有序集合/映射；核心机制是红黑树按比较器决定顺序与“相等”；特性是有序但 O(logN)，比较器写错会导致“丢元素/覆盖”。
    • 3分钟详答：TreeSet/TreeMap 依赖比较结果维护红黑树。
    • 排序来源：要么元素实现 Comparable（自然顺序），要么创建容器时传入 Comparator（外部顺序）。
    • 关键坑：在 TreeSet/TreeMap 里，“是否相等”由比较结果是否为 0 决定，而不一定看 equals。如果 comparator 只比较 userId，不比较其他字段，那么不同对象只要 userId 相同就会被当作同一个元素覆盖/去重。
    • 实战：我会保证 comparator 满足自反、传递、对称，并尽量与 equals 保持一致（除非业务明确要按某字段去重）。
    • 可能追问：
    • 追问：TreeMap 支持 null key 吗？
      15秒简答：取决于实现与比较器；自然顺序比较 null 会 NPE，通常不建议用 null key。

    ArrayDeque、LinkedList、PriorityQueue 分别适合什么场景？¶

    • 15秒简答：ArrayDeque 本质是循环数组双端队列，首尾操作快；LinkedList 是链表双端队列但内存重；PriorityQueue 是堆实现的优先队列；核心机制分别是环形数组/指针/堆有序性；特性是大多数队列/栈用 ArrayDeque，TopK/调度用 PriorityQueue。
    • 3分钟详答：我会从“操作模式”讲选型。
    • 双端队列/栈：ArrayDeque 通常最优，CPU cache 友好，常用作栈（push/pop）或队列（offer/poll）。
    • LinkedList：只有在需要频繁在两端插入删除且对内存不敏感时才考虑，但多数情况下 ArrayDeque 更好。
    • 优先级：PriorityQueue 适合做任务调度、TopK、合并多个有序流；它不保证遍历有序，只保证每次 poll 拿到最小/最大（取决于比较器）。
    • 实战：延迟任务、限流窗口、TopN 榜单我会结合 PriorityQueue（或配合 TreeMap）做简单实现。
    • 可能追问：
    • 追问：PriorityQueue 为什么不能 O(1) 找到任意元素？
      15秒简答：堆只保证父子局部有序，不保证全局有序，查找任意元素仍是 O(n)。

    Map 与 Hash（高频核心）¶

    说说 JDK 8 HashMap 的 put/get 核心流程（别背源码，讲关键分支）¶

    • 15秒简答：HashMap 本质是“数组桶 + 链表/红黑树”；核心机制是扰动 hash 后用 (n-1)&hash 定位桶，桶内用 equals 匹配，冲突走链/树，超过阈值触发扩容；特性是平均 O(1)，冲突严重会退化但 JDK8 树化把最坏压到 O(logN)。
    • 3分钟详答：我会按一次 put 拆成 4 个关键点：
    • 定位：对 key 的 hashCode 做扰动（让高位参与），再通过 (n-1)&hash 找到桶下标（n 为 2 的幂）。
    • 桶为空：直接放。
    • 桶不为空：先判断头节点是否命中（hash 相同且 key 相同），命中就覆盖 value；否则遍历链表或红黑树插入新节点。
    • 结构调整：插入后检查链表长度是否达到树化阈值（满足条件才树化）；同时检查 size 是否超过 threshold=capacity*loadFactor，超过就 resize 扩容并迁移元素。
    • 实战：我会强调扩容是“结构性大成本”，线上避免把 HashMap 当作无成本容器；容量预估、减少 key 过度离散/过度集中都很关键。
    • 可能追问：
    • 追问：为什么 table 长度是 2 的幂？
      15秒简答：便于用位运算取模 (n-1)&hash，更快且分布更均匀。

    HashMap 为什么容量必须是 2 的幂？扩容时元素怎么“重分布”？¶

    • 15秒简答：2 的幂让取桶变成按位与；核心机制是扩容翻倍后，元素要么留在原桶，要么移动到 oldIndex + oldCap；特性是迁移可用位运算完成，效率高但仍是 O(n)。
    • 3分钟详答：当容量是 2 的幂时，index = (n-1)&hash 等价于对 n 取模。扩容时 n 变成 2n，桶下标的变化只由 hash 的某一位（旧容量那一位）决定：
    • 如果 (hash & oldCap) == 0，位置不变；
    • 否则新位置是 oldIndex + oldCap。 这就是为什么 JDK8 扩容时可以把链表拆成高低位两条链，减少重新计算成本。
    • 实战：我会把“扩容代价”讲出来：一方面是迁移 O(n)，另一方面在并发场景会带来锁竞争/停顿风险，所以容量预估与避免无界增长很重要。
    • 可能追问：
    • 追问：扩容一定会 rehash 吗？
      15秒简答：会迁移节点到新表，但 JDK8 迁移更多用位判断拆分，避免对每个 key 重新完整计算。

    负载因子为什么默认 0.75？树化阈值为什么是 8？为什么 table 小于 64 先扩容不树化？¶

    • 15秒简答：这些本质是“空间、时间与对象开销的折中参数”；核心机制是 threshold=capacity*loadFactor 控制扩容点，链长过长树化降低最坏复杂度；特性是 0.75 兼顾冲突率与内存，8/64 是基于统计分布与 TreeNode 额外开销的工程权衡。
    • 3分钟详答：我会用三个“为什么”回答：
    • 为什么 0.75：装得更满（接近 1）会增加冲突与链长；太稀（很小）会浪费内存。0.75 是典型折中，让冲突概率可控且空间还算节省。
    • 为什么链表长度阈值 8：链表节点转 TreeNode 会增加对象与指针开销；只有当冲突确实明显时才值得用红黑树换查找性能。JDK 源码注释提到基于碰撞分布的估算，达到 8 的概率在 hash 分布良好时非常低。
    • 为什么 table < 64 先扩容：如果表很小导致冲突，扩容往往更能从根上降低冲突；过早树化会在小表期就引入高对象开销，得不偿失。
    • 实战：我会强调“参数不是圣经”：高并发/大 key 场景下，真正决定性能的是 key 分布、容量预估和是否频繁 resize。
    • 可能追问：
    • 追问：红黑树退化回链表的条件是什么？
      15秒简答：桶内元素变少（常见阈值 6）会反树化回链表，降低 TreeNode 开销。

    Hash 冲突怎么处理？面试官问“开放定址/拉链法”时你怎么把 HashMap 讲对？¶

    • 15秒简答：冲突本质是“不同 key 映射到同桶”；核心机制常见是拉链法（链表/树）或开放定址；特性是 Java HashMap 用拉链法，JDK8 在桶内链长过长时树化。
    • 3分钟详答：哈希表要做的是把 key 分散到桶里，但桶数有限必然有冲突。常见方案：
    • 拉链法：每个桶挂一条链（JDK8 可能树化），插入/查找都先定位桶，再在桶内比对。
    • 开放定址：冲突后按探测序列找下一个空槽（线性探测/二次探测/双重散列），更依赖装载因子，删除也更复杂。
    • Java HashMap 为什么选拉链：实现简单、删除自然、对负载因子更稳；再用树化降低最坏情况。
    • 实战：我会补充“冲突真正的根因通常是 key 设计/分布不佳”，比如用递增 id 的低位做 hash，容易集中；应尽量让 hash 分布均匀。
    • 可能追问：
    • 追问：HashMap 的扰动函数想解决什么？
      15秒简答：让 hash 高位信息参与桶定位，降低低位分布差导致的碰撞。

    为什么说 HashMap 线程不安全？能举一个“会出错”的具体点吗？¶

    • 15秒简答：它本质是无锁可变哈希表；核心机制是并发 put/resize 会导致数据覆盖、丢失，历史上甚至可能形成链表环；特性是读写并发不可用，解决要么外部同步，要么换并发容器。
    • 3分钟详答：线程不安全的根因是：HashMap 的 put、扩容、链/树转换都不是原子操作，也没有内存可见性与互斥保证。
    • 并发写：两个线程同时 put 到同桶，可能发生覆盖或链表结构被破坏。
    • 并发扩容：扩容过程中迁移节点与更新 table 引用，如果多个线程同时触发 resize，可能导致迁移重复、丢元素；在早期 JDK 的实现里还可能出现链表成环，导致 get 死循环（面试常考点）。
    • 实战：并发统计、缓存索引这种场景我会直接上 ConcurrentHashMap，或者在更高层用锁/分段锁保护临界区，并评估写比例与热点。
    • 可能追问：
    • 追问：只读并发安全吗？
      15秒简答：纯只读且构建后不再修改通常可用，但一旦有并发修改就不安全。

    ConcurrentHashMap（JDK 8）怎么保证线程安全？它和 JDK 7 分段锁的核心差异是什么？¶

    • 15秒简答：它本质是“桶级并发哈希表”；核心机制是读无锁（volatile/CAS）、写用 CAS + synchronized 锁桶头节点，并用 sizeCtl 协调初始化/扩容；特性是并发度高于 Hashtable，热点桶仍会竞争。
    • 3分钟详答：我会抓住三个面试官最在意的点：锁粒度、读路径、扩容协作。
    • 锁粒度：JDK7 典型是分段锁（Segment），锁住一段桶；JDK8 去掉 Segment，改为“锁桶头节点”，冲突只在热点桶上竞争，粒度更细。
    • 读路径：get 通常不加锁，依赖 volatile 可见性与节点结构的发布安全；这也是它吞吐高的关键。
    • 写路径：空桶用 CAS 放入；非空桶对桶头 synchronized，在桶内插入或树化。计数使用类似分段计数减少竞争。
    • 扩容：通过 sizeCtl 控制扩容状态，线程可以“协助扩容”搬迁桶，避免单线程扩容成为瓶颈。
    • 实战：我会说明选型：高并发读写用 ConcurrentHashMap；如果需要复合操作原子性（比如先判断再更新），仍需要额外同步或用 compute 系列方法。
    • 可能追问：
    • 追问：为什么 ConcurrentHashMap 不允许 null key/value？
      15秒简答：并发场景下 null 会和“找不到”语义冲突，难以区分且会增加分支与一致性复杂度。

    TreeMap、HashMap、LinkedHashMap 分别什么时候用？面试官为什么爱问这题？¶

    • 15秒简答：它们本质是三种“Map 约束”：HashMap 追求 O(1)；LinkedHashMap 在 O(1) 基础上保序；TreeMap 用红黑树按 key 有序；核心机制分别是哈希桶/哈希桶+双向链/红黑树；特性是顺序需求决定选型。
    • 3分钟详答：这题考的是你能否把“需求→结构→复杂度”串起来。
    • HashMap：无序、允许 null（实现层面），平均 O(1)，最常用。
    • LinkedHashMap：维护双向链表记录插入顺序或访问顺序；适合需要稳定遍历顺序、或做 LRU。
    • TreeMap：按 key 排序，firstKey/lastKey/subMap 等范围操作很强，复杂度 O(logN)。
    • 实战：我在做“配置合并/覆盖”时会用 LinkedHashMap 保持输出稳定；做时间范围、分段统计（比如按时间戳范围取数据）会用 TreeMap 的 range view；普通索引/映射默认 HashMap。
    • 可能追问：
    • 追问：LinkedHashMap 的“访问顺序”怎么用来做 LRU？
      15秒简答：开启 accessOrder 后，每次访问会把节点移到链表尾；再重写 removeEldestEntry 就能按容量淘汰最旧节点。

    LinkedHashMap 做 LRU 的实现思路是什么？为什么说它比“自己写双向链表+Map”更稳？¶

    • 15秒简答：它本质是“哈希表 + 双向链表”；核心机制是访问/插入会维护链表顺序，淘汰只需判断 eldest；特性是实现简单且语义清晰，但要注意线程不安全与容量控制。
    • 3分钟详答：LRU 需要两件事：O(1) 查找（map）+ O(1) 移动最近访问节点（双向链表）。
    • LinkedHashMap 正好内置了这两部分：底层还是哈希表定桶，额外维护 before/after 指针形成双向链表。
    • 开启 accessOrder：访问（get/put）会把节点移动到尾部，尾部表示“最近使用”。
    • 淘汰：重写 removeEldestEntry，当 size 超过容量时返回 true，自动删除链表头（最久未使用）。
    • 实战：如果是单机本地缓存小容量、无高并发写，我会优先用 LinkedHashMap 方案；如果并发和统计更复杂，会考虑专门缓存库或在外部加锁。
    • 可能追问：
    • 追问：这个 LRU 实现有什么明显缺陷？
      15秒简答：线程不安全、容量淘汰不是精确时间策略、缺少命中率/权重/过期等能力，需要业务自己补。

    迭代器、并发与性能陷阱（集合视角）¶

    什么是 fail-fast？为什么遍历时会抛 ConcurrentModificationException？怎么“正确地”边遍历边删除？¶

    • 15秒简答：fail-fast 本质是“检测到结构性修改就快速失败”；核心机制是容器维护 modCount，迭代器保存 expectedModCount，不一致就抛 CME；特性是尽早暴露并发/错误用法，但不是并发安全保证。
    • 3分钟详答：这题我会从“它解决什么问题”讲：
    • Java 大部分非并发容器的迭代器是 fail-fast：为了在迭代过程中有人（同线程或别线程）修改结构时，尽快报错，避免返回不一致结果。
    • 实现机制：容器每次结构性修改（add/remove/resize）会增加 modCount；迭代器创建时记录 expectedModCount；每次 next/remove 检查两者是否一致，不一致就抛 ConcurrentModificationException。
    • 正确删除方式：
      • 同线程遍历删除：用迭代器的 remove()（而不是集合的 remove），让迭代器同步更新期望值。
      • 或者用 removeIf、或者先收集待删元素再统一删除。
    • 并发场景：fail-fast 不是并发控制；多线程下要用并发容器或外部锁。
    • 可能追问：
    • 追问：fail-fast 和 fail-safe 有什么差别？
      15秒简答：fail-fast 发现修改就抛异常；fail-safe 常见是迭代时基于快照/复制，不抛但可能读到旧数据。

    Collections.synchronizedMap/List、CopyOnWriteArrayList、ConcurrentHashMap 怎么选？别只背结论¶

    • 15秒简答：它们本质是三种并发策略：外层粗锁、写时复制、桶级并发；核心机制分别是 synchronized 包装、复制数组、CAS+锁桶；特性是粗锁简单但竞争大，COW 读极快写很贵，CHM 读写均衡。
    • 3分钟详答：我会按“读写比例 + 操作类型”选。
    • Collections.synchronizedXxx：给每个方法加同一把锁；适合并发不高、改造成本低的场景。注意复合操作（遍历、先判后改）仍要手动在外层加锁包住整个过程。
    • CopyOnWriteArrayList：写时复制新数组，读不加锁；适合读远多于写、列表不大、写允许延迟的场景（比如监听器列表）。写频繁会导致复制与 GC 抖动。
    • ConcurrentHashMap：高并发 map 的常用解，读大多无锁，写锁粒度细；适合读写都比较高的场景。
    • 实战：配置监听、黑白名单这种“读多写少”我用 COW；并发计数/缓存索引用 CHM；低并发共享集合就同步包装或更上层锁。
    • 可能追问：
    • 追问：synchronizedMap 遍历时要注意什么？
      15秒简答：遍历需要手动对 map 加锁包住迭代过程，否则仍可能 CME 或读到不一致。

    为什么说“不要用可变对象当 Map 的 key”？除了 hashCode 变化，还有哪些隐藏坑？¶

    • 15秒简答：Map key 本质是“稳定的等价类代表”；核心机制是 key 的 equals/hashCode 必须稳定，否则会出现查不到、删不掉；特性是 bug 隐蔽且难复现，尤其在缓存/去重场景。
    • 3分钟详答：可变 key 的坑通常有三类：
    • hashCode 变化：对象被修改后落桶位置理论上变了，但容器不会搬它，导致 get/remove 失败。
    • equals 语义变化：即使 hash 不变，equals 变化也会导致集合里出现“逻辑重复”或无法命中。
    • 作为复合 key：很多人用 List、Map、Date 等可变对象做 key，或者用 Lombok/IDE 自动生成 equals/hashCode 把可变字段都包含进去，一修改就炸。
    • 实战：缓存 key 我更倾向用不可变类型（String/long/UUID），或自定义不可变 value object（只读字段、构造后不可变）。
    • 可能追问：
    • 追问：如果业务上 key 必须由多个字段组成怎么办？
      15秒简答：用不可变的复合 key（如 record/不可变类），或拼接成规范化字符串 key（注意分隔与转义）。

    遍历 Map 时为什么通常建议用 entrySet()？这题能答出“性能与可读性”层次吗？¶

    • 15秒简答：entrySet 本质是直接遍历键值对视图；核心机制是避免 keySet 再 get 的二次查找；特性是更快更清晰，尤其在 HashMap 下能减少哈希定位次数。
    • 3分钟详答：遍历 map 常见写法有：
    • for (Map.Entry<K,V> e : map.entrySet())：一次迭代同时拿到 key/value，通常最直观、也最省一次查找。
    • for (K k : map.keySet()) { V v = map.get(k); }：会多一次 get 查找（虽然在 HashMap 里平均 O(1)，但仍是额外计算；在某些 map 实现上可能更贵）。
    • values()：只关心 value 时更简洁。
    • 实战：我会优先 entrySet；如果是 ConcurrentHashMap 的遍历，还会说明它是弱一致迭代，读到的是遍历过程中的某个时间切片，不保证强一致。
    • 可能追问：
    • 追问：ConcurrentHashMap 遍历为什么不抛 CME？
      15秒简答：它迭代器是弱一致（不 fail-fast），允许并发修改，保证不抛异常但可能看见旧/新混合结果。
