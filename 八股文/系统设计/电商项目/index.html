    电商项目核心订单系统设计与实现：

    核心问题： 1. 如何避免重复下单？

    方式一：进前端页面的时候专门调用一个申请订单号的请求，先去请求一次订单号，后续下单的时候直接使用这个订单号。 但是后端在处理之前实际还是需要判断该订单号是否已经下过单，最简单的方式是直接在数据库中查询。但是当数据量很大的时候（量级多少？）数据库查询开销就比较大，此时可以使用布隆过滤器来实现。 具体实现布隆过滤器的方式有： 是否还有其他方式？

    1. 订单的ABA问题 它和CAS的ABA问题有点类似。

    2. 本质：同一条订单（或订单相关记录）在短时间内发生多次更新，请求在并发/异步/重试/乱序到达的情况下，可能出现“值从 A 变 B 又变回 A”，导致系统误以为“没变”，或者更严重的是“旧请求把新状态覆盖掉”。

    3. 典型触发条件（在订单领域很常见）：
    4. 前端/调用方重复提交（连点、网络慢、超时重试）
    5. 网络丢包导致调用方没收到“成功响应”而重试
    6. 多个更新请求在后端处理顺序与用户操作顺序不一致（乱序）
    7. 造成的结果：订单关键字段（例如收货地址、物流单号、备注、状态流转中的某些可回滚字段等）可能被“后到的旧版本更新”覆盖，最终落库值与用户最后一次操作不一致。

    视频给出的解决方案是用“版本号（version）”做多版本控制来避免并发更新产生的 ABA 覆盖：在订单（或订单相关）表里增加一个 version 字段，前端在读取数据时把当前 version 一并保存并随更新请求提交给后端；后端更新时不再直接按 id 覆盖写入，而是执行带版本条件的更新（例如 UPDATE ... SET 字段=新值, version=version+1 WHERE id=? AND version=? ），只有当数据库中的 version 与前端传来的 version 一致时才允许更新并将 version 自增；同时前端在发起一次更新后也将自己持有的 version 递增，这样如果出现“第一个请求尚未处理完、第二个请求又发出”或请求乱序/重试导致旧请求后到的情况，旧请求携带的 version 会与库中已变化的 version 不匹配，更新语句影响行数为 0，从而阻止旧版本把新版本覆盖，保证最终落库的是按版本顺序推进的一致结果。
