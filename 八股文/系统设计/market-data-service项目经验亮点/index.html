    Market Data Service 项目经验亮点包装¶

    适用身份：准备 Java 后端面试的业务系统/中后台/金融科技方向候选人
    项目定位：金融市场数据服务，负责标的物、行情、债券、汇率、利率曲线、基金 PCF、公司行动等主数据与行情数据的统一管理，对外提供 SDK/Feign/Dubbo 能力。

    一、项目一句话介绍¶

    Market Data Service 是一个面向金融交易和估值场景的市场数据中台，统一管理标的物主数据、收盘/日内/基金净值/股本市值行情、债券发行与债券行情、汇率、利率曲线、ETF PCF、公司行动等数据。项目采用 Java 8 + Spring Boot 2.5 + Gradle 多模块架构，拆分为 server、client、sdk 三层，对内通过 Dubbo/Feign 提供服务能力，并结合 Redis/Redisson 本地缓存、RabbitMQ 通知、Elasticsearch 检索、多数据库 profile、Excel 导入导出和批处理任务支撑高频查询与批量数据维护。

    二、技术栈与模块分层¶

    • 后端框架：Java 8、Spring Boot 2.5.14、Spring Data JPA、MapStruct、Lombok。
    • 服务接口：SDK/Client/Server 三模块分层，SDK 定义 DTO、枚举、Feign 接口，Client 封装缓存和工具能力，Server 承载业务实现。
    • 中间件：Redis/Redisson 本地缓存、RabbitMQ 数据变更通知、Elasticsearch 7.10.2 标的物搜索、Dubbo 服务引用。
    • 数据库适配：MySQL、Oracle、PostgreSQL、达梦等 profile 配置。
    • 工程能力：Gradle 多模块构建、Jacoco 测试覆盖率、Excel 模板导入导出、OpenSpec/PRD/测试用例文档沉淀。

    三、最值得包装的项目亮点¶

    亮点 1：高频行情查询缓存体系¶

    业务背景
    交易、估值、风控等系统会频繁按“标的物 + 日期 + 字段”查询行情。如果每次都走数据库或远程服务，会产生大量重复 IO，尤其是历史行情和空行情查询会造成缓存穿透。

    技术方案

    • 通用行情缓存 QuoteCache 使用 Map<Tuple, TimeSeriesDoubleArray> 组织数据，key 是“标的物 + 行情字段”，value 是按日期索引的时间序列数组。
    • 缓存未命中时按需从 InstrumentQuoteService 加载；批量场景支持 preload，一次拉取多个标的行情，减少 RPC 次数。
    • 使用 EMPTY 哨兵值缓存“查过但无数据”的结果，避免历史无行情数据被反复穿透。
    • 通过 RabbitMQ 订阅行情变更消息，对新增、更新、删除执行局部失效；清理/刷新消息则清空本地缓存。
    • 设置缓存大小阈值，超过阈值后定时缩容一半，防止长期运行造成内存膨胀。

    面试说法
    “我参与过行情查询缓存优化。原来行情查询会频繁访问 DB/RPC，且历史无数据会导致重复穿透。我把行情按标的和字段拆成时间序列数组缓存，用日期下标直接定位，同时引入 EMPTY 哨兵缓存空结果，批量场景先预加载再读缓存。缓存一致性通过 MQ 数据变更通知做局部失效，容量通过定时 shrink 控制。这个设计兼顾了查询性能、空值穿透和多实例一致性。”

    可被追问

    • 为什么用数组而不是 Map？数组按日期定位更紧凑，适合连续时间序列，内存和访问开销更低。
    • 空值缓存有什么风险？如果后续补数据，需要通过 MQ 变更通知或刷新机制失效，否则会读到旧的 EMPTY。
    • 多实例一致性怎么做？写入服务发送 MQ，所有实例消费后局部更新或失效缓存。

    亮点 2：债券行情专用 Caffeine 缓存与防穿透¶

    业务背景
    债券行情查询与普通股票/基金行情字段不完全一致，而且可能存在“请求日没有行情，需要取请求日之前最近一条估值行情”的历史回退逻辑。高频重复查询会对远程服务造成压力。

    技术方案

    • BondQuoteCache 使用 Caffeine 本地缓存，key 设计为 bondCode|quoteDate。
    • 缓存条目区分三种状态：未加载、有数据、已查询但无数据。
    • 无数据时放入 NO_DATA 哨兵，避免历史上完全无行情的债券被反复查询。
    • Caffeine 设置最大容量 50000，超出后按 LRU 淘汰。
    • 提供按债券、按债券+日期粒度的主动失效方法，方便写入后刷新。

    面试说法
    “债券行情我没有直接复用普通行情缓存，因为它有债券字段和历史回退语义。我单独做了一个扁平 Caffeine 缓存，用 bondCode 和 quoteDate 组成 key，并缓存无数据哨兵，解决重复查询不存在行情导致的穿透问题。容量交给 Caffeine 控制，写入侧提供精确失效接口。”

    亮点 3：标的物主数据缓存与缓存预热¶

    业务背景
    标的物是行情、债券、基金、公司行动等模块的共同依赖。很多导入、校验、查询场景都需要通过 instrumentId 与 UUID 双向映射，如果每次查询数据库会影响整体吞吐。

    技术方案

    • InstrumentCache 使用 Redisson RLocalCachedMap，同时维护三类映射：代码到 DTO、UUID 到代码、代码到 UUID。
    • 对不存在的标的物使用空值标记，避免批量校验时反复穿透。
    • 支持 warmup 和 refresh，可启动后或批处理前批量预热。
    • 保存标的物时主动删除相关缓存，包括期权/收益互换白名单缓存，避免脏读。
    • 对国际化场景返回前复制 DTO，避免修改缓存共享对象导致污染。

    面试说法
    “标的物是一个高复用主数据，所以我们做了 Redisson 本地缓存，既有代码到对象，也有 UUID 双向索引。为了避免不存在数据被重复查库，还做了空值缓存。写入时主动失效，批处理前可预热。这里我比较关注缓存对象污染问题，例如国际化字段转换会先 clone 再改，避免把展示态写回缓存。”

    亮点 4：行情批量保存的幂等与变更检测¶

    业务背景
    行情导入和同步经常重复推送同一标的、同一估值日的数据。如果不做幂等，可能造成重复记录、无意义更新和大量无效 MQ 通知。

    技术方案

    • InstrumentQuoteServiceImpl.batchSave 按行情实例类型分发：日内、收盘、其他、详细行情分别处理。
    • 收盘行情以“标的物 + 估值日”作为唯一语义，存在则更新，不存在则新增。
    • 更新时使用 setIfChanged 对字段逐个比较，只有真实变化才更新 quoteTimestamp 并加入待保存列表。
    • MQ 通知只在数据真实新增或变更时发送，避免下游缓存被无效刷爆。
    • 保存前统一补齐默认估值日、报价时间、时区，并按系统变量控制价格精度，默认保留 8 位小数。

    面试说法
    “行情保存不是简单 insert，而是按照行情类型做幂等 upsert。收盘行情会先查同标的同估值日的数据，再逐字段比较，只有真的变化才落库和发 MQ。这样能减少无效 DB update，也能避免下游缓存和订阅系统收到大量噪音消息。”

    可被追问

    • 为什么 MQ 只在变化时发？因为行情消费方通常会刷新缓存或触发计算，无变化通知会放大系统压力。
    • 日内行情为什么覆盖？业务语义是每个标的只保留最新一条日内行情。
    • 价格精度怎么控制？通过系统变量读取 scale，默认 8 位，统一四舍五入。

    亮点 5：汇率查询支持反向汇率和回溯查询¶

    业务背景
    外汇行情中不一定同时维护 USD/CNY 和 CNY/USD 两个方向；估值日当天也可能缺行情，需要按自然日或交易日向前回溯最近可用汇率。

    技术方案

    • FxRateServiceImpl.getFxRateQuoteWithReverse 优先查询正向货币对。
    • 正向缺失时查询反向货币对，并用 1 / 反向汇率 计算。
    • 反向计算时处理 BID/ASK 对调：正向 BID 来自反向 ASK，正向 ASK 来自反向 BID。
    • 支持 maxLookbackDays 限制最大回溯天数。
    • 支持自然日和交易日两种回溯方式，交易日通过交易日历工具对齐。
    • 批量查询允许每个 criteria 使用不同回溯参数，提升接口灵活性。

    面试说法
    “汇率接口我会重点讲反向查询和回溯。比如业务要查 USD/CNY，但库里只有 CNY/USD，我们会自动用倒数补齐，并且 BID/ASK 要反过来，不能简单字段照搬。如果当天没数据，还能按自然日或交易日回溯，且有最大回溯天数限制，避免无限查历史。”

    亮点 6：标的物保存链路的并发校验与复杂编排¶

    业务背景
    标的物字段多、资产类型多、校验规则复杂，批量导入时如果串行校验会很慢；同时还要处理分类、债券发行信息、ES 索引、自定义字段、白名单等联动逻辑。

    技术方案

    • InstrumentServiceImpl 中批量校验使用 CompletableFuture + instrumentValidationTaskExecutor 并发执行。
    • 线程池核心线程数按 CPU 核数设置，最大线程数为 2 倍 CPU，拒绝策略使用 CallerRunsPolicy。
    • 支持 failFast 模式：串行遇到第一个错误就停止；非 failFast 模式并发校验并保留原始顺序。
    • 保存链路中先做交易所/时区填充、国内交易所代码格式化，再进入校验和落库。
    • 更新标的物时合并 externalIds，避免外部系统代码被覆盖丢失。
    • 保存成功后联动分类移动、自定义字段保存、ES 索引更新、自动白名单维护。

    面试说法
    “标的物保存链路我主要负责/熟悉的是批量导入性能和一致性。因为标的物校验规则很多，非 failFast 场景会用 CompletableFuture 并发校验，失败的记录过滤掉，成功的保持原顺序继续落库。落库后还会更新 ES、自定义字段、分类、白名单，并且清理缓存。这里难点是不能只看单表保存，要把主数据变更对搜索、缓存和下游白名单的影响一起处理。”

    亮点 7：Elasticsearch 标的物搜索与索引初始化¶

    业务背景
    金融标的物数量大，用户会按代码、名称、外部代码、大小写变体进行搜索。单纯数据库模糊查询性能和体验都不理想。

    技术方案

    • 引入 Elasticsearch 作为标的物检索引擎。
    • 启动时扫描带 @Document 的实体类，检查 ES 健康状态，可用时自动创建索引和 mapping。
    • 搜索时对 instrumentId、instrumentName、externalIds 等字段做精确、大小写、前缀、通配、match 查询组合，并设置不同 boost。
    • 标的物保存成功后同步更新 ES 文档，保证搜索结果及时可见。

    面试说法
    “标的物搜索不是简单 like，而是接入 ES。我们会在启动时做 ES 健康检查和 mapping 初始化，查询时对代码、名称、externalIds 做多策略匹配，并通过 boost 提升精确匹配排序。写入标的物后同步更新 ES，保证主数据和搜索索引一致。”

    亮点 8：利率曲线行情接入的前置保障与幂等落库¶

    业务背景
    利率曲线行情接入时，某些曲线 Index 标的物可能还不存在。如果直接写行情，会因为主数据缺失失败；如果盲目创建，又可能产生脏标的或缓存不一致。

    技术方案

    • RateInstrumentPreflightServiceImpl 在普通行情落库前做前置保障。
    • 先校验映射出来的 SaveInstrumentRequest 是否符合利率曲线 Index 规则。
    • 按目标完整码判断已存在且合格的标的物，未就绪的统一批量 upsert。
    • 对比请求集合与 upsert 返回集合，识别成功和失败子集。
    • 仅对 upsert 成功的标的物刷新缓存；缓存刷新失败则不继续写普通行情。
    • RateCashQuotePersistServiceImpl 写入行情前只保留 CLOSE 实例，过滤无主数据标的，按估值日分组，并在同一估值日内按标的物去重后调用 batchSave。

    面试说法
    “利率行情同步我会讲前置保障。因为行情依赖标的物主数据，所以写行情前会先保证 Index 标的存在且属性合格。未就绪的标的批量 upsert，成功后只刷新成功子集缓存，再进入行情落库。落库时还会过滤非 CLOSE、过滤无标的物、按估值日分组和去重，保证这一条同步链路尽量幂等。”

    亮点 9：Excel 导入导出和复杂业务校验¶

    业务背景
    金融中后台大量数据由运营人员通过 Excel 维护，例如债券发行信息、债券行情、公司行动、基金 PCF、停复牌等。Excel 导入天然容易出现模板错误、重复数据、字段互斥、日期关系不合法等问题。

    技术方案

    • 项目内维护多类 Excel 模板，覆盖行情、债券、分红、基金、白名单等业务。
    • 债券发行信息支持主表 + 信用评级 + 利率变动 + 特殊条款多 Sheet 导入，子表通过 bondCode 关联主表。
    • 公司行动校验除权除息日、派息/转送/拆股/缩股至少一项、部分字段互斥、标的物必须存在、组合唯一等规则。
    • 利率行情导入会避免“空文件也提示成功”的误导，导入前做有效数据判断。
    • 行情导入对同一批次的“标的物 + 日期”去重，再进入批量保存。

    面试说法
    “这个项目里 Excel 导入不是普通 POI 读写，关键是业务校验。比如债券发行信息是多 Sheet，子表必须能在主表找到 bondCode；公司行动里派息、转送、拆股、缩股有至少一项和互斥规则；行情导入要先去重再批量保存。我会把导入流程拆成模板识别、字段转换、业务校验、幂等落库、错误提示这几层。”

    四、可以包装成简历项目描述¶

    项目名称：金融市场数据服务 Market Data Service

    项目描述：
    负责建设金融市场数据中台，统一管理标的物主数据、行情、债券、汇率、利率曲线、基金 PCF、公司行动等数据，对交易、估值、风控等系统提供 SDK/Feign/Dubbo 查询与批量维护能力。系统基于 Spring Boot + Gradle 多模块架构，结合 Redis/Redisson、Caffeine、RabbitMQ、Elasticsearch、JPA、MapStruct 和 Excel 导入导出能力，实现高频行情查询、主数据缓存、复杂业务校验、批量同步和数据变更通知。

    可写职责：

    • 设计并优化行情缓存体系，基于时间序列数组缓存历史行情，支持批量预加载、空值哨兵防穿透、MQ 变更通知失效和容量收缩。
    • 负责行情批量保存链路，按行情类型拆分处理，基于标的物和估值日实现幂等 upsert，并通过逐字段变更检测减少无效 DB 更新和 MQ 通知。
    • 参与标的物主数据缓存建设，使用 Redisson LocalCachedMap 维护代码、UUID 双向索引，支持缓存预热、空值缓存和写入失效。
    • 优化汇率查询接口，支持正反向货币对自动换算、BID/ASK 对调、自然日/交易日回溯和批量查询。
    • 参与标的物批量导入保存链路，使用 CompletableFuture 和自定义线程池提升批量校验性能，并联动 ES 索引、自定义字段、分类和白名单维护。
    • 参与利率曲线行情接入，设计标的物前置保障流程，在行情落库前完成 Index 标的 upsert、缓存刷新、失败子集隔离和按估值日幂等保存。

    五、STAR 话术模板¶

    话术 1：行情缓存优化¶

    S：行情查询被多个下游系统高频调用，尤其是历史行情和不存在行情，容易造成大量重复远程调用。
    T：需要降低数据库/RPC 压力，同时保证多实例缓存一致性。
    A：我将行情按“标的物 + 字段”组织成时间序列数组，按日期快速定位；对无数据结果使用 EMPTY 哨兵防穿透；批量查询先 preload；写入侧发送 RabbitMQ 变更通知，消费端做局部失效；同时用定时 shrink 控制缓存容量。
    R：减少了重复查询和无效远程调用，缓存能覆盖单条、批量、空结果和数据变更场景，整体查询链路更稳定。

    话术 2：行情幂等保存¶

    S：行情导入和同步存在重复推送，同一标的同一估值日可能被多次写入。
    T：需要避免重复记录、无效更新和下游重复通知。
    A：我按行情类型拆分保存策略。收盘/其他/详细行情按“标的物 + 估值日”做 upsert；更新前逐字段比较，只有值变化才更新报价时间并发送 MQ；日内行情按业务语义只保留最新一条。
    R：保存链路具备幂等性，减少了无意义 DB 写入和 MQ 通知，也降低了缓存刷新压力。

    话术 3：汇率反向与回溯查询¶

    S：估值系统查询汇率时，库里可能只有反向货币对，或者估值日当天没有行情。
    T：需要提供稳定的汇率查询能力，减少调用方处理复杂度。
    A：我在服务端封装正反向查询逻辑，正向缺失时自动查反向并取倒数，同时处理 BID/ASK 对调；当天无数据时支持自然日和交易日两种回溯，并用 maxLookbackDays 限制查询范围。
    R：调用方只需要传目标货币对和日期，服务端能返回更完整、业务语义正确的汇率结果。

    话术 4：标的物批量校验性能优化¶

    S：标的物批量导入字段多、规则多，串行校验耗时明显。
    T：需要提升批量导入性能，同时保留错误收集能力和顺序稳定性。
    A：我为非 failFast 模式引入 CompletableFuture 并发校验，并配置独立线程池；每条记录独立收集校验结果，最后只保留通过的数据，并按原始顺序继续保存。failFast 模式仍保留串行快速失败。
    R：批量导入在规则复杂时吞吐更好，同时错误处理和落库顺序可控。

    六、面试官可能深挖的问题¶

    1. 缓存一致性怎么保证？¶

    回答要点：

    • 写入侧在真实数据变更后发送 MQ。
    • 查询侧本地缓存消费 MQ，按标的物、字段、日期做局部失效或更新。
    • 对全量刷新/清空提供 refresh 和 clear 事件。
    • 主数据保存时主动删除 InstrumentCache，避免读到旧标的。
    • 空值缓存必须配合写入失效，否则会产生“补数据后仍读空”的问题。

    2. 为什么行情保存要做变更检测？¶

    回答要点：

    • 行情同步可能重复推送同一份数据。
    • 如果每次都 update，会造成数据库写放大。
    • MQ 通知会触发下游缓存刷新和计算，无变化通知会放大系统压力。
    • 逐字段比较后只对真实变化落库和通知，是一种轻量幂等优化。

    3. 反向汇率为什么 BID 和 ASK 要互换？¶

    回答要点：

    • 买入价/卖出价是有方向的。
    • 如果 USD/CNY 缺失，用 CNY/USD 反推时，正向买入价应使用反向卖出价的倒数，正向卖出价应使用反向买入价的倒数。
    • 中间价和收盘价可以直接取倒数，但 BID/ASK 不能简单字段同名转换。

    4. 批量导入如何处理部分失败？¶

    回答要点：

    • 校验阶段区分 failFast 和非 failFast。
    • 非 failFast 下每条数据独立校验，失败记录收集错误，通过记录继续保存。
    • 前置保障场景会对比请求集合和保存成功集合，得到失败子集。
    • 对保存成功的数据刷新缓存，失败数据不进入后续行情落库。

    5. 为什么要拆 server/client/sdk 三个模块？¶

    回答要点：

    • SDK 放 DTO、枚举、Feign 接口，供外部服务编译依赖。
    • Client 放缓存、转换器、本地工具类，对业务方提供更顺手的客户端能力。
    • Server 放真正业务实现、Controller、Repo、Mapper、Job。
    • 这样可以降低外部依赖体积，避免业务实现泄漏给调用方。

    八、项目亮点优先级¶

    最建议重点讲：

    1. 行情缓存体系：最有系统设计味道，能讲缓存结构、穿透、一致性、容量控制。
    2. 行情幂等保存：能体现工程细节、业务一致性和 MQ 降噪。
    3. 汇率反向与回溯：业务难点清晰，容易讲出金融场景理解。
    4. 标的物缓存和批量校验：能体现性能优化、并发编程和主数据治理。
    5. 利率曲线前置保障：适合高级一点的“链路编排 + 部分失败隔离 + 缓存刷新”话题。

    不建议过度包装：

    • 单纯 CRUD 模块，例如停复牌基础管理，除非面试官问业务覆盖面。
    • 单纯 Excel 读写，应该把重点放在多 Sheet 关联、业务校验、幂等落库、错误提示。
    • 单纯“用了 ES/Redis/RabbitMQ”，要讲清楚为什么用、怎么用、解决了什么问题。
