# 视频文案综合分析报告

[Google TL系统设计 - 支付系统](https://www.bilibili.com/video/BV1i43GzkEYW)

## 讨论视角（这期在解决什么）

这期是典型“系统设计面试题：设计支付系统”的拆解。主讲人把“支付系统”明确收敛为：不做真正资金清算/跨行划拨（那属于 PSP/银行/卡组织体系与金融牌照范围），而是站在商家/平台侧去设计一套可审计、正确、可靠的“支付流水记账 + 内部清算（ledger）+ 对账”的系统。

## 核心论点与分论点

### 核心论点

支付系统面试题的主线不是“怎么扣到钱”，而是“怎么把一笔支付从发起到落账的全过程记录正确、可追溯、可恢复”，围绕合规不可篡改、幂等防重复扣款、异步回调与补偿、账本（双重记账）与对账来完成端到端闭环。

### 分论点 1：先把业务边界讲清楚——PSP 负责扣款，你负责记录与结算

- 信用卡支付涉及持卡人/商户、发卡行/收单行、卡组织（Visa/Mastercard）等多方；商家不可能逐个对接
- 现实中商家通常接入 PSP（Payment Service Provider/Platform），PSP 提供统一 API 并收取通道费
- 本题“支付系统”更像内部的支付记录系统：创建支付请求 → 调 PSP 扣款 → 记录结果 → 通知用户/商户 → 日终清算与对账

### 分论点 2：支付集成有多种形态，定制自由度越高，成本与风险越高

- Server-to-Server 自收卡号：自主权大但需金融牌照/PCI 审计，合规与投入高（更适合大企业）
- PSP SDK（前端收集卡信息、返回 token）：降低 PCI 负担，但存在供应链/前端注入风险，需要 CSP/SRI 等前端安全策略
- Iframe/托管字段（更安全的隔离）：商家只拿 token；但 UI 受 iframe 限制，自由度下降
- Hosted Redirect（302 跳转到 PSP 页面支付）：最省心、PCI 负担最低，但 UI/品牌受限且多一步跳转带来转化损失

### 分论点 3：数据建模要服从合规——业务/交易/资金三态分离，账本不可变

- 单表记录“订单+支付状态”直观但不利于合规审计（金融数据不可随意覆盖修改）
- 典型拆分：
  - 业务状态：订单/商品/业务字段（Order）
  - 交易状态：支付尝试/方式/PSP/授权结果（Payment）
  - 资金状态：分录/手续费/商户结算（Ledger）
- Ledger 强调不可变（append-only），写入后不能改，只能用新增分录纠错

### 分论点 4：一致性核心是“避免重复扣款”——幂等键 + 状态机驱动

- 风险场景：用户网络卡顿、重复点击造成多次请求（double spending）
- 方案：Checkout 页面由服务端生成 idempotency key（幂等键），后续链路携带该键，服务端去重（dedup）
- Payment 表常见字段：charge_id、order_id、payment_method、psp、status（状态机）、amount/currency、created_at/updated_at、idempotency_key
- 状态机示意：created → started → processing → (success | failed)

### 分论点 5：外部 PSP 异步且不可靠——回调 + 主动查询 + 重试四要素

- PSP 通常先返回“已受理/处理中”，最终结果通过 callback URL 回调通知
- callback 可能丢失/失败，需要补一个 polling（超时后主动查询）
- 调外部依赖要带重试策略“四要素”：
  - timeout：超过阈值再重试
  - backoff：推荐 exponential backoff（1s/2s/4s/8s…）
  - jitter：加入抖动（如 ±15%）避免同频重试造成流量尖峰
  - DLQ（dead letter queue）：超过最大次数入死信队列，触发回滚/熔断/人工处理
- 幂等键最好穿透到 PSP（让外部也做去重），形成双保险

### 分论点 6：账本与对账是“正确性底座”——双重记账 + 日终结算文件对账

- 扣款成功后写 Ledger：行业标准是 double-entry bookkeeping（每笔交易影响两个账户，金额相等方向相反）
- Ledger 适合使用 append-only 存储/专用账本数据库
  - 视频举例：Amazon QLDB、阿里 LedgerDB（提供 SQL 接口，内置审计/加密等）
- 日终对账：银行/渠道提供 settlement file，每天跑一次对账任务；异常走自动补偿或人工审核，保证“账永远对”

### 分论点 7：状态冲突要能自洽——单调状态机/版本号/最终以对账为准

- 典型冲突：轮询超时把状态写成 failed，随后 PSP 回调成功导致“跳变”
- 解决思路：
  - 单调状态机：进入 failed 后不允许再覆盖为 success
  - 或引入版本号/条件更新（compare-and-swap）
  - 最差也要靠日终对账纠正，资金守恒不能错
- 实务上常见“实时状态表 + 不可变账务表”组合：前者可更新，后者保证资金正确

### 分论点 8：更高级的架构选项——事件溯源（event sourcing）提升可追溯与扩展

- 引用思路：Turning the database inside out（把数据库更新变成事件流）
- 用 event store（Kafka/Pulsar + append-only DB）作为 source of truth
- 查询用 projection/materialized view；纠错用新增事件而不是覆盖旧数据
- 优点：金融正确性 + 互联网可扩展性
- 代价：系统复杂度高，团队需同时熟悉流系统与金融账务

### 分论点 9（加分项）：支付风控/反欺诈（risk engine）分层演进

- 规则引擎：黑白名单/规则集，简单、易审计，但对复杂欺诈有限
- 统计/传统 ML：依赖数据与特征工程，对新型欺诈响应慢
- 无监督/图模型/GNN：能捕捉复杂关系，但开发/维护/推理开销更高，误报需人工审核

## 关键数据、案例与结论（逐条可复述）

- PSP 通道费：约 0.5% ～ 1%（视频给出的数量级）
- 合规关键词：PCI 审计；支付/金融数据需要可审计、不可随意修改
- 重复扣款防护：幂等键（idempotency key）+ 去重（dedup）+ 状态机
- 外部依赖可靠性：callback + polling；重试四要素 timeout/backoff/jitter/DLQ
- 账本模型：double-entry bookkeeping；Ledger append-only；写入后不可变
- 技术选型举例：Amazon QLDB、阿里 LedgerDB
- 正确性兜底：日终 settlement file 对账；异常走自动补偿或人工审核
- 进阶架构：event sourcing（Kafka/Pulsar 等）+ projection，纠错用新事件覆盖逻辑而非改旧记录
